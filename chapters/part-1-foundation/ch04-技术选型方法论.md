# Chapter 4 技术选型方法论

## 这套技术栈为什么成立

Rust + Tokio + SQLite + workspace 拆分这套组合，不是品味结果，而是几个具体约束共同推出来的。每一项选择都对应代码里能看到的一条决策痕迹。

要回答的核心问题：

1. 多轮工具调用如何稳定并发——尤其是 wave-based 并行
2. 本地部署如何把外部依赖压到最低
3. 状态、记忆、调度、subagent、Web 控制面如何共用一套数据模型
4. 功能扩张时如何控制代码库——主 binary 与子 crate 怎么划线
5. feature flag 如何让用户按场景裁剪二进制

## Workspace 拆分：内聚与边界并存

主 binary `microclaw` 加 7 个子 crate 共 8 个 workspace 成员：

| Crate | 职责 |
|-------|------|
| `microclaw-core` | 基础类型、LLM 类型、文本工具、错误类型 |
| `microclaw-storage` | SQLite 封装、schema 迁移、所有持久化 |
| `microclaw-tools` | Tool trait、sandbox、ToolConcurrencyClass、ToolRisk、path guard |
| `microclaw-channels` | ChannelAdapter trait、ChannelRegistry、消息投递 |
| `microclaw-observability` | OTLP metrics / traces / logs |
| `microclaw-clawhub` | 技能发现、安装、lockfile |
| `microclaw-app` | 日志配置、内置技能、转写支持 |
| `microclaw`（顶层 binary） | agent_engine、tool_executor、run_control、hooks、web、acp、a2a、gateway、setup、doctor |

划线原则：**对运行链路高度耦合的部分保持在顶层内聚，对可稳定抽象的能力域拆成独立 crate，只在收益明确时增加边界**。agent_engine 与 tool_executor 紧耦合 hooks、run_control、subagent 调度，把它们留在顶层 binary 里反而比强行拆分更易演进；而 storage、channels、tools 这三类有清晰对外契约，拆出去不会反复倒灌依赖。

## 为什么是 Rust + Tokio

MicroClaw 在单进程内同时运行多条并发线索：

```
┌── 15 渠道异步收发消息 ───────────────────────┐
├── wave-based 并行工具执行（JoinSet）         │
├── per-chat turn serialization（async mutex） │
├── run_control（AtomicBool + Notify）         │  单进程
├── Scheduler + MemoryReflector（Tokio task）  │  同时运行
├── Hooks 调用外部进程                         │
├── Subagent 并发（Semaphore）                 │
├── Web API（Axum）+ SSE + WebSocket           │
└── SQLite spawn_blocking                      │
```

Rust 在这种形态下提供三件别的语言难以复制的事：

1. **状态边界显式化**。`AppState` 字段中的 `Arc` 标记共享、`RwLock` 标记可变共享、`Option` 标记可缺省——很难"随手从全局拿东西"
2. **所有权与 Send/Sync 约束降低共享状态错误**。`Arc<AtomicBool>` 做取消信号、`Arc<Semaphore>` 限流 subagent、`LazyLock<Mutex<...>>` 维护 active runs，这些模式在编译期就被检查
3. **Tokio 让单进程多后台循环成为常态**。渠道 runtime + scheduler + web + 工具并行同时跑

```rust
use anyhow::Result;
use async_trait::async_trait;
use tokio::task::JoinHandle;

#[async_trait]
trait RuntimeLane: Send + Sync + 'static {
    async fn run(self: std::sync::Arc<Self>) -> Result<()>;
}

async fn boot_runtime(
    channels: std::sync::Arc<dyn RuntimeLane>,
    web: std::sync::Arc<dyn RuntimeLane>,
    scheduler: std::sync::Arc<dyn RuntimeLane>,
) -> Result<()> {
    let handles: Vec<JoinHandle<Result<()>>> = vec![
        tokio::spawn(channels.run()),
        tokio::spawn(web.run()),
        tokio::spawn(scheduler.run()),
    ];
    for h in handles {
        h.await??;
    }
    Ok(())
}
```

**为什么不是 Python 或 Node.js**：不是它们做不了 Agent，而是当目标变成"15 渠道 + 约 50 工具 + wave-based 并行 + turn serialization + run control + hooks + subagent + ACP/A2A"时，长期运行进程的状态边界、异步与阻塞 IO 的混用、并发分类的编译期保证，都是 Rust 更擅长的领域。

## 为什么是 SQLite

`crates/microclaw-storage/src/db.rs` 持有当前 schema v25+，超过 20 张表覆盖 session resume、memory lifecycle、scheduled task DLQ、subagent tracking、API key 管理、审计、metrics timeline、FTS5 全文索引（`session_search`，v21 引入）。这是事实源，不是 demo 存储。

| 优势 | 说明 |
|------|------|
| 部署门槛低 | bundled rusqlite，Homebrew / Docker / Nix / installer 四种安装路径都不依赖外部 DB |
| 本地一致性 | 20+ 表在同一 DB 文件，跨模块操作不需要分布式协调 |
| 可审计 | 单文件天然适合离线检查与备份 |
| 不阻塞 runtime | 所有 DB 操作包装在 `tokio::task::spawn_blocking` 中 |
| 迁移内嵌 | 升级时由 `db.rs` 的迁移函数顺序推进，无需独立工具 |

边界：横向扩展不是默认路径；高并发写入更早撞瓶颈。MicroClaw 的策略是先把单机 runtime 做厚做稳，再通过 Gateway 把它演化成长驻服务，而不是过早走多实例共享。

## Provider 抽象：只保留两类协议

`src/llm.rs` 收敛到两类协议：Anthropic 原生 + OpenAI 兼容。这个看似激进的取舍带来三件事：

1. agent_engine 完全模型无关——只关心消息、工具 schema、stop_reason、usage 统计
2. 新供应商只需贴近其中一类协议，不需要改核心循环
3. per-channel model override（Telegram 用 A 模型，飞书用 B 模型）通过 `ProviderRouter` 单点解决

代价是部分供应商特有能力不会第一时间暴露。但 runtime 最怕协议分叉——如果不同 provider 各自走一套 tool calling、streaming、usage 统计，统一循环、统一观测、统一 hooks 都会被持续削弱。

```rust
use std::collections::HashMap;
use anyhow::Result;
use async_trait::async_trait;
use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, Serialize, Deserialize)]
struct LlmRequest {
    model: String,
    messages: Vec<serde_json::Value>,
    tools: Vec<serde_json::Value>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
struct LlmResponse {
    text: String,
    tool_calls: Vec<serde_json::Value>,
    stop_reason: String,
    usage: serde_json::Value,
}

#[async_trait]
trait LlmProvider: Send + Sync {
    async fn complete(&self, req: LlmRequest) -> Result<LlmResponse>;
}

struct ProviderRouter {
    active: Box<dyn LlmProvider>,
    model_overrides: HashMap<String, String>, // channel -> model
}

impl ProviderRouter {
    fn resolve_model(&self, channel: &str, default_model: &str) -> String {
        self.model_overrides
            .get(channel)
            .cloned()
            .unwrap_or_else(|| default_model.to_string())
    }

    async fn run(
        &self,
        channel: &str,
        default_model: &str,
        messages: Vec<serde_json::Value>,
        tools: Vec<serde_json::Value>,
    ) -> Result<LlmResponse> {
        let model = self.resolve_model(channel, default_model);
        self.active
            .complete(LlmRequest { model, messages, tools })
            .await
    }
}
```

## 沙箱路由与工具风险

`microclaw-tools` 把每个工具的执行约束声明化：`ToolConcurrencyClass`（ReadOnly / SideEffect / Exclusive）决定是否能与同 wave 的其他工具并行；`ToolRisk` 枚举为 Low / Medium / High，高风险工具默认要求用户确认（由 `high_risk_tool_user_confirmation_required` 控制），中等风险用于会修改长期状态但仍可控的操作（`write_memory`、`pause_scheduled_task` 等）；path guard 决定哪些路径可以读写；sandbox 路由器把 bash 执行包进受控环境。这些约束的共同点是——**它们是 trait 上的属性，而不是循环里的临时判断**。tool_executor 因此可以纯粹按声明做调度，不用关心具体工具语义。

## 内嵌 schema 迁移

数据迁移没有依赖 sqlx 之类的外部框架，而是在 `db.rs` 内通过 `match` 分支顺序推进版本号。这一选择在工程上的好处：迁移与代码同版本发布，不会出现"数据库版本和二进制对不上"的窗口；二进制启动时可以决定是只读模式打开旧库还是直接拒绝启动。代价是迁移逻辑必须由开发者手写、走 review，而不是声明式生成。

## Feature flag 策略

```toml
[features]
default = []
sqlite-vec = ["microclaw-storage/sqlite-vec"]    # 向量检索
journald = ["microclaw-app/journald"]            # Linux systemd 日志
channel-matrix = ["dep:matrix-sdk"]              # Matrix（重依赖）
full = ["channel-matrix"]                        # 一键全开
```

默认最小编译。Matrix SDK 携带 e2e 加密的完整链路，不做 optional 则所有用户都要承担它的编译成本。系统在配置启用但二进制未编译某 feature 时会给出明确警告，避免静默降级。

## 依赖选择

| 库 | 工程理由 |
|----|---------|
| tokio + axum | 异步运行时与 Web 框架，与 JoinSet / spawn_blocking 无缝集成 |
| rusqlite（bundled） | 编译时不依赖系统 SQLite，简化交叉编译 |
| teloxide / serenity | Telegram、Discord 平台 SDK，吸收平台特有细节 |
| agent-client-protocol | ACP 协议库（仅 stdio），对接 IDE 与自动化生态 |
| rmcp | MCP 客户端，支持 child process 与 HTTP transport，动态注入工具 |
| opentelemetry-proto + prost | 轻量但可控的 OTLP 集成 |
| clap | CLI 参数解析（start / acp / setup / doctor / gateway / ...） |
| ratatui + crossterm | TUI 框架，承载 setup wizard |

## 关键权衡

| 决策 | 收益 | 代价 |
|------|------|------|
| Rust + Tokio | 并发清晰、状态安全、长期稳定 | 上手门槛高 |
| SQLite 单机状态库 | 部署简单、链路统一、易审计 | 不面向多实例共享 |
| 双协议 Provider 抽象 | 核心循环不被供应商差异污染 | 特有能力不会第一时间暴露 |
| 主 binary + 7 子 crate | 边界清晰但运行链路内聚 | 编译时间与依赖关系需管理 |
| 内嵌 schema 迁移 | 迁移与代码同版本，无外部工具 | 迁移逻辑必须手写并 review |
| Feature flag 裁剪 | 单一 crate 覆盖多场景 | 条件编译扩大测试矩阵 |

## 容易走错的地方

**失败模式 1：把技术选型理解成"追求最快或最新"**。Agent runtime 的核心约束是稳定执行，不是 benchmark。

**失败模式 2：过早上远程数据库与分布式组件**。主链路还没稳定就拆库拆服务，只会把调试难度放大。

**失败模式 3：Provider 品牌差异直接写进核心循环**。一旦 `agent_engine` 里充满"某模型例外"，统一行为就难以维持。

**失败模式 4：不做 feature flag**。Matrix SDK 等重依赖一旦默认开启，每个用户都要承担它的编译与体积成本。

**失败模式 5：用通用 ORM 替代手写迁移**。表结构与版本号绑定的强约束会被打破，启动时无法准确判断兼容性。

## 小结

选型顺序：Rust + Tokio（执行一致性）→ SQLite（本地状态压缩）→ 双协议 LLM 抽象 → 主 binary + 7 子 crate（结构化拆分）→ feature flag（按需裁剪）。一句话概括即"对内聚的保持内聚，对可拆的做稳定拆分"。

# Chapter 4 技术选型方法论

## 为什么这个系统会长成今天这样？

Rust + Tokio + SQLite + 8-crate workspace 这组选择背后的工程逻辑：

1. 多轮工具调用时能否稳定并发——特别是 wave-based parallel execution
2. 本地部署时能否把依赖压到足够低
3. 状态、记忆、调度、subagent、Web 控制面能否用同一套数据模型
4. 功能增加时代码库是否可控——8 个 crate 如何演进
5. feature flag 能否让用户按需裁剪

## 8-crate workspace：先控制复杂度，再追求广度

| Crate | 职责 |
|-------|------|
| `microclaw-core` | 基础类型、LLM 类型、文本工具、错误类型 |
| `microclaw-storage` | SQLite 封装、schema 迁移、所有持久化 |
| `microclaw-tools` | Tool trait、sandbox、concurrency class、path guard |
| `microclaw-channels` | ChannelAdapter trait、ChannelRegistry、消息投递 |
| `microclaw-observability` | OTLP metrics/traces/logs |
| `microclaw-clawhub` | 技能发现、安装、lockfile |
| `microclaw-app` | 日志配置、内置技能、转写 |
| 顶层 binary | agent_engine、tool_executor、run_control、hooks、web、acp、a2a、gateway、setup、doctor |

原则：对运行链路高度耦合的部分保持在顶层内聚，对可稳定抽象的能力域拆成独立 crate，只在收益明确时增加边界。

## 为什么选择 Rust + Tokio

MicroClaw 里同时存在的并发任务：

```
┌── 16 渠道异步收发消息 ──────────────────────┐
├── wave-based 并行工具执行 (JoinSet)          │
├── per-chat turn serialization (async mutex)  │
├── run_control (AtomicBool + Notify)          │  单进程
├── Scheduler + Reflector (Tokio task)         │  同时运行
├── hooks 调用外部进程                          │
├── subagent 并发 (Semaphore)                  │
├── Web API (Axum) + WebSocket                 │
└── SQLite spawn_blocking                      │
```

Rust 的价值：

1. **状态边界显式化**：`AppState` 的 16 个字段中，`Arc` 表示共享、`RwLock` 表示可变共享、`Option` 表示可选——很难"随手从全局拿东西"
2. **所有权降低共享状态错误**：`Arc<AtomicBool>` 做取消信号、`Arc<Semaphore>` 限流 subagent、`LazyLock<Mutex<...>>` 维护 active runs
3. **Tokio 让单进程多后台循环自然**：渠道 runtime + scheduler + web + tool parallel 同时跑

```rust
#[async_trait::async_trait]
trait RuntimeLane {
    async fn run(&self) -> anyhow::Result<()>;
}

async fn boot_runtime(
    channels: impl RuntimeLane,
    web: impl RuntimeLane,
    scheduler: impl RuntimeLane,
) -> anyhow::Result<()> {
    tokio::try_join!(channels.run(), web.run(), scheduler.run())?;
    Ok(())
}
```

```python
import asyncio
from typing import Protocol


class RuntimeLane(Protocol):
    async def run(self) -> None: ...


async def boot_runtime(
    channels: RuntimeLane, web: RuntimeLane, scheduler: RuntimeLane
) -> None:
    await asyncio.gather(channels.run(), web.run(), scheduler.run())
```

**为什么不是 Python / Node.js**：不是不能做 Agent，而是当目标变成"16 渠道 + 44 工具 + wave-based 并行 + turn serialization + run control + hooks + subagent + ACP/A2A"时，长期运行进程的状态边界、异步/同步混用、concurrency class 编译期保证都是 Rust 更擅长的。

## 为什么选择 SQLite

schema v19，20+ 表覆盖 session resume、memory lifecycle、scheduled task DLQ、subagent tracking、API key management、audit trail、metrics timeline。这不是 demo 存储。

| 优势 | 说明 |
|------|------|
| 部署门槛低 | bundled SQLite，Homebrew/Docker/Nix/installer 四种安装都不依赖外部 DB |
| 本地一致性 | 20+ 表在同一 DB，跨模块操作不需分布式协调 |
| 可审计 | SQLite 文件天然适合离线检查和备份 |
| spawn_blocking | 所有 DB 操作包装在 `tokio::task::spawn_blocking` 中 |

边界：横向扩展不是默认路径；高并发写入更快碰瓶颈。先围绕单机 runtime 把一致性做厚，通过 Gateway 让单机部署变长驻服务。

## Provider 抽象：只保留两类协议

`src/llm.rs` 收敛为 Anthropic 原生协议 + OpenAI-compatible 协议。

收益：(1) agent_engine 保持模型无关——只关心消息、工具 schema、stop_reason；(2) 新供应商只需适配既有协议；(3) per-channel model override 不需要改核心循环。

代价：某些 provider 特有能力不会第一时间暴露。但运行时最怕协议分叉——一旦不同 provider 各走一套 tool calling/streaming/usage 统计，统一循环、统一观测、统一 hooks 都会被削弱。

## 示例代码：Provider 抽象

```rust
#[async_trait::async_trait]
trait LlmProvider: Send + Sync {
    async fn send(&self, prompt: &str, model: &str) -> anyhow::Result<String>;
}

struct ProviderRouter {
    active: Box<dyn LlmProvider>,
    model_overrides: std::collections::HashMap<String, String>,
}

impl ProviderRouter {
    fn resolve_model(&self, channel: &str, default_model: &str) -> String {
        self.model_overrides
            .get(channel)
            .cloned()
            .unwrap_or_else(|| default_model.to_string())
    }

    async fn complete(
        &self, channel: &str, prompt: &str, default_model: &str,
    ) -> anyhow::Result<String> {
        let model = self.resolve_model(channel, default_model);
        self.active.send(prompt, &model).await
    }
}
```

```python
from dataclasses import dataclass, field
from typing import Protocol


class LlmProvider(Protocol):
    async def send(self, prompt: str, model: str) -> str: ...


@dataclass
class ProviderRouter:
    active: LlmProvider
    model_overrides: dict[str, str] = field(default_factory=dict)

    def resolve_model(self, channel: str, default_model: str) -> str:
        return self.model_overrides.get(channel, default_model)

    async def complete(
        self, channel: str, prompt: str, default_model: str
    ) -> str:
        model = self.resolve_model(channel, default_model)
        return await self.active.send(prompt, model)
```

## Feature Flag 策略

```toml
[features]
default = []
sqlite-vec = ["microclaw-storage/sqlite-vec"]    # 向量检索
journald = ["microclaw-app/journald"]             # Linux systemd 日志
channel-matrix = ["dep:matrix-sdk"]               # Matrix（重依赖）
full = ["channel-matrix"]                         # 一键全开
```

默认最小编译。Matrix SDK 带 e2e 加密，不做 optional 则所有用户都承担编译成本。系统会在配置启用但二进制未编译某 feature 时给出明确警告。

## 依赖选择

| 库 | 工程理由 |
|----|---------|
| tokio + axum | 异步运行时 + Web 框架，与 JoinSet/spawn_blocking 无缝集成 |
| rusqlite (bundled) | 编译时不需系统 SQLite，降低交叉编译复杂度 |
| teloxide / serenity | Telegram / Discord bot 框架，处理平台特有细节 |
| agent-client-protocol | ACP 协议库（v0.10.3），对接更广泛生态 |
| rmcp | MCP 客户端，支持 child process 和 HTTP transport |
| opentelemetry-proto + prost | 轻量但可控的 OTLP 集成 |
| clap | CLI 参数解析（start/acp/setup/doctor/gateway/...） |
| ratatui + crossterm | TUI 框架，用于 11,000 行 setup wizard |

## 关键权衡

| 决策 | 收益 | 代价 |
|------|------|------|
| Rust + Tokio | 并发清晰、状态安全、长期稳定 | 上手门槛高 |
| SQLite 单机状态库 | 部署简单、链路统一、易审计 | 不面向多实例共享 |
| 双协议 Provider 抽象 | 核心循环不被供应商差异污染 | 特有能力不会第一时间暴露 |
| 8-crate workspace | 边界清晰但运行链路内聚 | 编译时间和依赖关系需管理 |
| Feature flag 裁剪 | 一个 crate 覆盖所有场景 | 条件编译增加测试矩阵 |

## 容易走错的地方

### 失败模式 1：把技术选型理解成"追求最快或最新"
Agent Runtime 的核心约束是稳定执行，不是 benchmark。

### 失败模式 2：过早上远程数据库和分布式组件
主链路还没稳定就拆库拆服务，只会把调试难度放大。

### 失败模式 3：Provider 品牌差异直接写进核心循环
一旦 agent_engine 里充满"某模型例外"，统一行为就难以维持。

### 失败模式 4：不做 feature flag
Matrix SDK 的 e2e 依赖链很重，不做 optional 则每个用户都要承担成本。

## 小结

选型顺序：Rust + Tokio（执行一致性）→ SQLite（本地状态压缩）→ 双协议 LLM 抽象 → 8-crate workspace（结构化拆分）→ feature flag（按需裁剪）。"对内聚的保持内聚，对可拆的做稳定拆分"。

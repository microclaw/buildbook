# Chapter 4 技术选型方法论

## 这一章要回答什么问题

前三章解释了 MicroClaw 要解决什么问题，以及它怎样组织渠道、状态和数据流。接下来必须回答一个更现实的问题：为什么这个系统会长成今天这样，而不是另一种完全不同的工程形态？

这不是“喜欢 Rust 还是喜欢 Python”一类的泛讨论。对 Agent Runtime 来说，技术选型会直接改变四件事：

1. 多轮工具调用时能否稳定并发执行。
2. 本地部署时能否把依赖压到足够低。
3. 状态、记忆、调度、Web 控制面能否用同一套数据模型协作。
4. 当功能持续增加时，代码库会不会迅速失控。

这一章读完后，你应该能理解 MicroClaw 当前几项核心选择背后的工程逻辑：`Rust + Tokio`、`SQLite`、双协议 LLM 抽象，以及从单体源码向 workspace 演进的边界控制。

## 选型的第一原则：先控制复杂度，再追求广度

MicroClaw 的源码给出了一个很强的信号。根 `Cargo.toml` 已经不是“一个二进制 + 一堆模块”的最小形态，而是一个包含多个 crate 的 workspace：

- `crates/microclaw-core`
- `crates/microclaw-storage`
- `crates/microclaw-tools`
- `crates/microclaw-channels`
- `crates/microclaw-app`
- `crates/microclaw-observability`
- `crates/microclaw-clawhub`

同时，顶层 `src/` 仍然保留 `main.rs`、`runtime.rs`、`agent_engine.rs`、`llm.rs`、`scheduler.rs`、`web.rs` 这些高层装配入口。

这说明它的选型方法不是“先做一个超分布式平台”，也不是“把所有东西都塞回一个二进制文件”。它采取的是一种更务实的中间路线：

- 对运行链路高度耦合的部分保持内聚。
- 对可以稳定抽象的能力域拆成 crate。
- 只在收益明确时增加边界。

这条原则非常重要。很多 Agent 项目的失败，不是因为选错了语言，而是因为在功能还不稳定时，就过早引入服务化、消息总线、多数据库和复杂控制平面。MicroClaw 当前的做法更保守，但对一本强调“从零构建”的书来说，这恰恰是值得学习的地方。

## 为什么选择 Rust + Tokio

### Rust 解决的不是性能崇拜，而是执行可靠性

在 MicroClaw 里，模型调用只是执行链的一部分。真正复杂的是下面这些任务同时存在：

- 异步收消息和发消息
- 并发跑 Web API
- 周期性轮询 Scheduler
- 触发反射器（Reflector）
- 调用工具并处理超时
- 在本地 SQLite 上做同步持久化

如果运行时不能把“异步 IO、共享状态、错误传播、资源生命周期”稳定管理起来，Agent Runtime 很快就会在边角处失控。Rust 的价值就在这里。

第一，类型系统迫使状态边界显式化。`runtime.rs` 里的 `AppState` 把配置、数据库、技能、Hooks、LLM Provider、Memory Backend、ToolRegistry、OTLP exporter 等依赖全部集中声明。这个结构让你很难“随手从全局变量里拿一个东西”，也让装配路径更容易审计。

第二，所有权和借用规则降低了长期运行进程里的共享状态错误。MicroClaw 广泛使用 `Arc`、`RwLock`、`Mutex`，例如：

- `Arc<Database>` 共享 SQLite 包装器。
- `Arc<ChannelRegistry>` 共享渠道路由。
- `Arc<HookManager>` 共享策略钩子。
- `Arc<RwLock<HashMap<...>>>` 保存 provider/model override。

这些并发原语当然不是 Rust 独有，但 Rust 会强迫你把共享与可变性写清楚。对 Agent Runtime 这种“表面像聊天，底层其实是长期进程”的系统，这种约束是帮助，而不是负担。

第三，Tokio 让“一个二进制里同时跑多条后台循环”变得自然。`runtime.rs` 负责启动渠道 runtime，`scheduler.rs` 中的 `spawn_scheduler` 和 `spawn_reflector` 用 Tokio task 维持后台任务，而 `web.rs` 则用 Axum 在同一进程里挂出本地控制面。这个组合使得单进程方案依然能保持较高能力密度。

### 小例子：为什么一个 runtime 需要同时跑多条异步 lane？

对 Agent Runtime 来说，真正棘手的不是单次模型调用，而是前台消息、后台调度和本地控制面需要同时活着。只要这些 lane 之间的协作写得混乱，系统很快就会因为共享状态和生命周期问题失控。

Rust 版本用 `trait + struct + async` 表达两条后台 lane 的统一接口，再由一个引导函数把它们一起启动。这样 Tokio 的价值就不只是“快”，而是让长期运行的多角色进程有清晰边界。

```rust
#[async_trait::async_trait]
trait RuntimeLane {
    async fn run(&self) -> anyhow::Result<()>;
}

struct WebLane;
struct SchedulerLane;

async fn boot_runtime(web: WebLane, scheduler: SchedulerLane) -> anyhow::Result<()> {
    tokio::try_join!(web.run(), scheduler.run())?;
    Ok(())
}
```

Python 版本用 `Protocol` 和 `asyncio.gather` 保持相同结构。这样例子传达的重点不是“哪种语言更好”，而是一个成熟 runtime 必须把并发 lane 当成正式模型来管理。

```python
import asyncio
from typing import Protocol


class RuntimeLane(Protocol):
    async def run(self) -> None: ...


async def boot_runtime(web: RuntimeLane, scheduler: RuntimeLane) -> None:
    await asyncio.gather(web.run(), scheduler.run())
```

### 为什么不是 Python 或 Node.js

这里要避免神话化结论。MicroClaw 不选择 Python 或 Node.js，并不是说它们不能做 Agent，而是它们的默认工程代价不同。

如果用 Python 或 Node.js，最初几周的开发速度通常更快，生态里也有大量现成 SDK。但一旦目标变成“多渠道 + 多轮工具 + 持久记忆 + 调度 + Web 控制面 + 安全边界”，你就会更快碰到下面几个问题：

- 长期运行进程的状态边界容易散。
- 异步与同步混用时更容易出现隐藏阻塞。
- 高并发 IO 与本地资源管理靠团队纪律维持，而不是靠类型系统兜底。
- 后续想把核心能力抽成可复用模块时，边界更容易被历史兼容拖住。

MicroClaw 的选择，本质上是在“初期编码速度”和“中后期系统稳态”之间偏向后者。对于一个希望长期运行、可托管、可审计的 runtime，这个取舍是合理的。

## 为什么选择 SQLite

### SQLite 是运行时状态数据库，而不是临时 demo 存储

很多读者一看到 SQLite，就会下意识觉得这是一个“以后一定要换掉”的过渡方案。但从当前仓库看，MicroClaw 并不是把 SQLite 当成简陋替代品，而是把它作为单机优先架构的核心组成。

`crates/microclaw-storage/src/db.rs` 已经维护了相当完整的数据结构，包括但不限于：

- `messages`
- `sessions`
- `scheduled_tasks`
- `task_run_logs`
- `memories`
- `metrics_history`
- `audit_logs`
- `subagent_runs`
- `subagent_events`

这说明 SQLite 在这里承担的是统一事实源的角色。它不仅存聊天记录，还支撑：

- Session Resume
- 记忆生命周期
- 调度任务和 DLQ
- Web 侧历史查询
- 指标时间线
- 审计与 API key 相关状态

对于单机优先、多渠道运行时来说，这种“一库打通多条主链路”的收益很高。

### SQLite 为什么适合这一阶段

第一，部署门槛低。`rusqlite` 通过 bundled SQLite 让外部依赖非常少，用户不需要先搭 MySQL/Postgres/Redis 才能启动系统。书里的目标读者之一，就是想从零落地可运行 Agent 的工程师。SQLite 能显著降低第一步的复杂度。

第二，本地一致性更强。消息、会话、记忆、任务都在同一数据库里，很多跨模块操作不需要再做分布式协调。对当前规模而言，这比“每层一个独立存储”更容易保持正确性。

第三，可审计。SQLite 文件天然适合离线检查和备份。对于“为什么这轮对话会恢复成这样”“哪条任务为什么失败”“某条记忆何时被 supersede”这类问题，本地数据库比散落在多个组件里的状态更容易回溯。

### SQLite 的边界也必须承认

它当然不是没有代价。

- 横向扩展不是默认路径。
- 高并发写入场景会更快碰到瓶颈。
- 多实例共享一个状态库的部署模式需要额外设计。

MicroClaw 的做法不是否认这些边界，而是先围绕单机 runtime 把一致性做厚。只有当“一个实例已经无法承担核心负载”时，才讨论把部分能力拆出去。这种次序比“先上远程数据库和服务拆分”更适合当前产品阶段。

## 为什么 Provider 抽象只保留两类协议

### 统一到“Anthropic 原生”和“OpenAI-compatible”

`src/llm.rs` 的设计很克制。对外暴露的是统一的 `LlmProvider` trait，而在实现上主要收敛为两条通道：

- Anthropic 原生协议
- OpenAI-compatible 协议

`docs/generated/provider-matrix.md` 则把 OpenAI、OpenRouter、Ollama、Google、Azure、Bedrock、DeepSeek、xAI 等多家 provider 映射到“OpenAI-compatible”这条大类里。

这个决策非常关键。它意味着 MicroClaw 不追求“每接一个 provider 就引入一套新语义”，而是优先把差异压缩到协议兼容层。`docs/llm-provider-conventions.md` 进一步要求：

- 不要按具体模型名写分支。
- 用能力标志而不是品牌字符串控制行为。
- 把 provider/model 预设放在 config/setup 层，而不是核心循环里。

这会直接带来两个收益。

第一，`agent_engine.rs` 可以保持模型无关。核心循环只关心：

- 输入消息是什么
- 工具 schema 是什么
- 返回的 `stop_reason` 是什么
- 是否需要继续 tool loop

第二，provider 演进不需要重写 runtime。新的供应商只要尽量适配既有协议，MicroClaw 就能把新增工作控制在配置与翻译层，而不是在每个业务模块里开分支。

### 为什么不做“每个 provider 一套最优实现”

因为运行时最怕的是协议分叉。一旦不同 provider 在 tool calling、streaming、usage 统计、错误处理上各走一套，后面所有章节讲的“统一循环”和“统一观测”都会被削弱。

MicroClaw 选择的是一个更慢但更稳的方向：优先维持最小公共模型，再逐步修正兼容差异。这并不华丽，但非常符合 runtime 的本质。

## 为什么从单体源码演进到 workspace

书稿最初的目录里把 Chapter 5 的骨架写成 `core/storage/tools/channels/app`，而当前上游仓库恰好已经落到了这个方向。这不是巧合，而是工程复杂度自然增长的结果。

当系统只有 Telegram、一个 Provider、少量工具时，把所有东西放在 `src/` 里是合理的。但当能力扩展到：

- 多渠道适配
- 独立工具 runtime
- 独立存储层
- 独立可观测性适配
- 技能分发与 ClawHub 集成

继续维持“一个 crate 管一切”就会让边界越来越模糊。

workspace 的收益在于：

- `microclaw-storage` 可以独立管理 schema 和数据 API。
- `microclaw-tools` 可以沉淀执行策略、sandbox、path guard 等通用逻辑。
- `microclaw-channels` 可以把通道抽象与具体平台实现拆开。
- `microclaw-observability` 可以把 Metrics/Traces/Logs 统一封装。

但 MicroClaw 仍保留顶层 `src/agent_engine.rs`、`src/runtime.rs`、`src/web.rs` 等文件，说明它没有走向“万物微服务”，而是仍然用单二进制维持主链路一致性。这种“crate 级模块化，而非进程级服务化”的演进方式，是当前阶段非常合理的折中。

## 示例代码：为什么 Provider 抽象应该对齐协议，而不是对齐厂商名字？

下面的例子故意不接任何具体 SDK，只保留“协议边界”和“调用点”。这样读者能看到真正该稳定的不是某个品牌，而是核心循环调用模型时依赖的最小契约。

Rust 版本用 `trait` 表示统一协议，再用一个 `struct` 路由器持有当前 provider。这样未来即使更换厂商，主循环看到的仍然只是一个可等待的异步接口，而不是一串品牌分支。

```rust
#[async_trait::async_trait]
trait LlmProvider {
    async fn send(&self, prompt: &str) -> anyhow::Result<String>;
}

struct AnthropicProvider;
struct OpenAiCompatibleProvider;

struct ProviderRouter<P: LlmProvider> {
    active: P,
}
```

先把协议边界和路由器立住，才能看清这个抽象究竟稳定了什么。方法论上最重要的不是先看到厂商，而是先看到系统到底依赖哪一层能力。

```rust

impl<P: LlmProvider> ProviderRouter<P> {
    async fn complete(&self, prompt: &str) -> anyhow::Result<String> {
        self.active.send(prompt).await
    }
}

#[async_trait::async_trait]
impl LlmProvider for AnthropicProvider {
    async fn send(&self, prompt: &str) -> anyhow::Result<String> {
        Ok(format!("anthropic handled: {prompt}"))
    }
}

#[async_trait::async_trait]
impl LlmProvider for OpenAiCompatibleProvider {
    async fn send(&self, prompt: &str) -> anyhow::Result<String> {
        Ok(format!("openai-compatible handled: {prompt}"))
    }
}
```


Python 版本维持同一结构，但用 `Protocol` 和 `@dataclass` 保证接口与装配层都可读。动态语言里最容易走偏的地方，就是把“支持多个 provider”写成一连串 `if/else`，这个例子正好反过来示范。

```python
from dataclasses import dataclass
from typing import Protocol


class LlmProvider(Protocol):
    async def send(self, prompt: str) -> str: ...


class AnthropicProvider:
    async def send(self, prompt: str) -> str:
        return f"anthropic handled: {prompt}"


class OpenAICompatibleProvider:
    async def send(self, prompt: str) -> str:
        return f"openai-compatible handled: {prompt}"


@dataclass
class ProviderRouter:
    active: LlmProvider
```

Python 这里也先展示统一接口，再落到具体 provider。这样版面上先出现的是判断标准，而不是品牌分支，论证顺序和阅读顺序会更一致。

```python

    async def complete(self, prompt: str) -> str:
        return await self.active.send(prompt)
```

## 关键权衡

### 决策一：选择 Rust + Tokio 作为统一 runtime

优点是并发模型清晰、状态边界显式、长期运行更稳定。代价是上手门槛高于脚本语言，需求验证初期的编码速度会稍慢。

### 决策二：优先 SQLite 单机状态库

优点是部署简单、链路统一、易于审计。代价是默认不面向多实例共享状态，需要后续演进时再拆分。

### 决策三：Provider 抽象只收敛到两类协议

优点是核心循环不被供应商差异污染。代价是某些 provider 的特有能力不会第一时间在核心层暴露。

### 决策四：用 workspace 控制复杂度，而不是直接服务化

优点是边界更清晰但运行链路仍然内聚。代价是 crate 数增加后，依赖关系和文档治理必须跟上。

## 容易走错的地方

### 失败模式 1：把技术选型理解成“追求最快或最新”

Agent Runtime 的核心约束是稳定执行，而不是 benchmark 漂亮。如果只用“性能最好”或“生态最热”来做判断，很容易忽视状态一致性和治理成本。

### 失败模式 2：过早上远程数据库和分布式组件

如果消息恢复、工具循环、记忆注入这些主链路还没稳定，先拆库拆服务只会把调试难度放大。

### 失败模式 3：Provider 接入时把品牌差异直接写进核心循环

一旦 `agent_engine.rs` 里开始充满“某模型例外”“某平台特判”，后面很难再维持统一行为。

## 小结

MicroClaw 的技术栈并不神秘，但它的选型顺序很有代表性：先选择能保持执行一致性的语言和运行时，再选择能把状态压缩在本地的存储，再选择足够克制的协议抽象，最后才通过 workspace 把不断增长的能力做结构化拆分。

这套方法论决定了后面几章的展开方式。

# Chapter 4 技术选型方法论

## 为什么这个系统会长成今天这样，而不是另一种形态？

前三章解释了 MicroClaw 要解决什么问题，以及它怎样组织渠道、状态和数据流。接下来必须回答一个更现实的问题：Rust + Tokio + SQLite + 8-crate workspace 这组选择背后的工程逻辑是什么？

这不是"喜欢 Rust 还是喜欢 Python"一类的泛讨论。对 Agent Runtime 来说，技术选型会直接改变五件事：

1. 多轮工具调用时能否稳定并发执行——特别是 wave-based parallel execution。
2. 本地部署时能否把依赖压到足够低。
3. 状态、记忆、调度、subagent、Web 控制面能否用同一套数据模型协作。
4. 当功能持续增加时，代码库会不会迅速失控——8 个 crate 的 workspace 怎么演进。
5. feature flag 策略能否让用户按需裁剪二进制。

这一章读完后，你应该能理解 MicroClaw v0.1.38 几项核心选择背后的工程逻辑：`Rust + Tokio`、`SQLite`、双协议 LLM 抽象、8-crate workspace 和 feature flag 体系。

## 选型的第一原则：先控制复杂度，再追求广度

MicroClaw v0.1.38 的根 `Cargo.toml` 是理解其选型方法论的最佳入口。workspace 包含 8 个成员：

- `crates/microclaw-core`：基础类型、LLM 类型定义、文本工具、错误类型
- `crates/microclaw-clawhub`：ClawHub 技能发现、安装、lockfile
- `crates/microclaw-storage`：SQLite 数据库封装、schema 迁移、所有持久化操作
- `crates/microclaw-tools`：工具运行时抽象、sandbox 策略、concurrency class、path guard、web fetch/search
- `crates/microclaw-channels`：渠道适配器抽象、ChannelRegistry、消息投递
- `crates/microclaw-app`：应用层封装（日志配置、内置技能、转写支持）
- `crates/microclaw-observability`：OTLP metrics/traces/logs 导出
- 顶层 `.`：主二进制，包含 `src/agent_engine.rs`、`src/runtime.rs`、`src/tool_executor.rs`、`src/hooks.rs`、`src/run_control.rs`、`src/chat_turn_queue.rs`、`src/acp.rs`、`src/a2a.rs`、`src/gateway.rs`、`src/setup.rs`、`src/doctor.rs`、`src/web.rs` 等高层装配

这说明它的选型方法不是"先做一个超分布式平台"，也不是"把所有东西都塞回一个文件"。它采取的是一种更务实的中间路线：

- 对运行链路高度耦合的部分（agent engine、tool executor、run control、hooks）保持在顶层 crate 内聚。
- 对可以稳定抽象的能力域（storage、tools runtime、channels、observability、clawhub）拆成独立 crate。
- 只在收益明确时增加边界。

从 v0.1.16 到 v0.1.38，crate 数量从约 5 个增长到 8 个。新增的 `microclaw-clawhub` 承载了技能发现和供应链追踪，`microclaw-app` 承载了日志和转写。这些拆分不是事先规划的，而是在功能增长过程中自然产生的。

这条原则至关重要。很多 Agent 项目的失败，不是因为选错了语言，而是因为在功能还不稳定时就过早引入服务化、消息总线、多数据库和复杂控制平面。

## 为什么选择 Rust + Tokio

### Rust 解决的不是性能崇拜，而是执行可靠性

在 MicroClaw 里，模型调用只是执行链的一部分。真正复杂的是下面这些任务同时存在：

- 异步收消息和发消息（16 个渠道 + ACP + A2A）
- wave-based 并行工具执行（tokio::JoinSet）
- per-chat turn serialization（ChatTurnQueue 的 async mutex）
- run control 的 cancelled 标记和 notify（AtomicBool + Notify）
- 周期性轮询 Scheduler 和 Reflector
- hooks 调用外部进程并等待超时
- subagent 并发执行（Semaphore 限流）
- Web API（Axum）和 WebSocket bridge
- 在本地 SQLite 上做 spawn_blocking 持久化

如果运行时不能把"异步 IO、共享状态、错误传播、资源生命周期"稳定管理起来，Agent Runtime 很快就会在边角处失控。Rust 的价值就在这里。

第一，类型系统迫使状态边界显式化。`runtime.rs` 里的 `AppState` 把所有运行时依赖集中声明：

```rust
pub struct AppState {
    pub config: Config,
    pub channel_registry: Arc<ChannelRegistry>,
    pub db: Arc<Database>,
    pub memory: MemoryManager,
    pub skills: SkillManager,
    pub hooks: Arc<HookManager>,
    pub llm: Box<dyn LlmProvider>,
    pub llm_provider_overrides: Arc<RwLock<HashMap<String, String>>>,
    pub llm_model_overrides: Arc<RwLock<HashMap<String, String>>>,
    pub embedding: Option<Arc<dyn EmbeddingProvider>>,
    pub memory_backend: Arc<MemoryBackend>,
    pub tools: ToolRegistry,
    pub chat_turn_queue: Arc<ChatTurnQueue>,
    pub metric_exporter: Option<Arc<OtlpMetricExporter>>,
    pub trace_exporter: Option<Arc<OtlpTraceExporter>>,
    pub log_exporter: Option<Arc<OtlpLogExporter>>,
}
```

这个结构让你很难"随手从全局变量里拿一个东西"。每一个依赖的生命周期和共享方式都是显式的：`Arc` 表示共享、`RwLock` 表示可变共享、`Option` 表示可选能力。

第二，所有权和借用规则降低了长期运行进程里的共享状态错误。v0.1.38 广泛使用并发原语：

- `Arc<Database>` 共享 SQLite 包装器
- `Arc<ChannelRegistry>` 共享渠道路由
- `Arc<HookManager>` 共享策略钩子
- `Arc<ChatTurnQueue>` 共享 turn 序列化器
- `Arc<AtomicBool>` + `Arc<Notify>` 在 run_control 中做取消信号
- `Arc<Semaphore>` 在 subagent 模块中限流并发 subagent 数
- `LazyLock<Mutex<HashMap<...>>>` 在 run_control 中维护全局 active runs

这些并发原语不是 Rust 独有，但 Rust 会强迫你把共享与可变性写清楚。对 Agent Runtime 这种"表面像聊天，底层其实是长期进程"的系统，这种约束是帮助。

第三，Tokio 让"一个二进制里同时跑多条后台循环"变得自然。`runtime.rs` 负责启动所有渠道 runtime，`scheduler.rs` 用 Tokio task 维持定时任务和 reflector，`tool_executor.rs` 用 `tokio::JoinSet` 做 wave 内并行，`web.rs` 用 Axum 在同一进程里挂出本地控制面。这个组合使得单进程方案依然能保持很高的能力密度。

### 一个 runtime 为什么需要同时跑多条异步 lane？

v0.1.38 比 v0.1.16 多了更多需要同时存在的后台活动：ACP stdio 循环、A2A HTTP 端点、subagent 并发执行、hooks 外部进程调用。

Rust 版本用 `trait + struct + async` 表达多条后台 lane 的统一接口。

```rust
#[async_trait::async_trait]
trait RuntimeLane {
    async fn run(&self) -> anyhow::Result<()>;
}

struct ChannelsLane;    // 16 渠道 adapters
struct WebLane;         // Axum Web UI + API
struct SchedulerLane;   // cron + reflector
struct AcpLane;         // Agent Client Protocol stdio
struct GatewayLane;     // service management

async fn boot_runtime(
    channels: ChannelsLane,
    web: WebLane,
    scheduler: SchedulerLane,
) -> anyhow::Result<()> {
    tokio::try_join!(channels.run(), web.run(), scheduler.run())?;
    Ok(())
}
```

Python 版本用 `Protocol` 和 `asyncio.gather` 保持相同结构。

```python
import asyncio
from typing import Protocol


class RuntimeLane(Protocol):
    async def run(self) -> None: ...


async def boot_runtime(
    channels: RuntimeLane,
    web: RuntimeLane,
    scheduler: RuntimeLane,
) -> None:
    await asyncio.gather(channels.run(), web.run(), scheduler.run())
```

### 为什么不是 Python 或 Node.js

MicroClaw 不选择 Python 或 Node.js，并不是说它们不能做 Agent，而是它们的默认工程代价不同。

如果用 Python 或 Node.js，最初几周的开发速度通常更快。但一旦目标变成"16 渠道 + 44 工具 + wave-based 并行执行 + per-chat turn serialization + run control + hooks + subagent + ACP/A2A + Web 控制面 + 安全边界"，就会更快碰到下面几个问题：

- 长期运行进程的状态边界容易散。Python 没有 `Arc<RwLock<...>>` 这种显式标注来提醒你"这里有并发共享"。
- 异步与同步混用时更容易出现隐藏阻塞。SQLite 操作在 Rust 中通过 `spawn_blocking` 显式隔离，在 Python 中更容易不小心阻塞 event loop。
- wave-based parallel execution 需要精确的 concurrency class 控制和 JoinSet 语义。Python 的 `asyncio.gather` 可以做类似的事，但缺少类型系统对 concurrency class 的编译期保证。
- 后续想把核心能力抽成可复用 crate 时，Rust 的模块系统和 workspace 提供了比 Python packages 更严格的边界控制。

MicroClaw 的选择，本质上是在"初期编码速度"和"中后期系统稳态"之间偏向后者。

## 为什么选择 SQLite

### SQLite 是运行时状态数据库，而不是临时 demo 存储

`crates/microclaw-storage/src/db.rs` 的 schema 已经演进到第 19 版。它维护的表包括：

- 基础会话：`chats`、`messages`、`sessions`
- 调度：`scheduled_tasks`、`task_run_logs`、`scheduled_task_dlq`
- 记忆：`memories`、`memory_reflector_runs`、`memory_injection_logs`、`memory_supersede_edges`、`memory_reflector_state`
- Subagent：`subagent_runs`、`subagent_events`、`subagent_announces`、`subagent_focus_bindings`
- 认证：`auth_passwords`、`auth_sessions`、`api_keys`、`api_key_scopes`
- 观测：`metrics_history`、`audit_logs`、`llm_usage_logs`
- 元数据：`db_meta`、`schema_migrations`

这不是一个"demo 级别的本地存储"。它是一个完整的运行时状态数据库，支撑着 session resume、memory lifecycle、scheduled task DLQ、subagent tracking、API key management、audit trail 和 metrics timeline。

### SQLite 为什么适合这一阶段

第一，部署门槛低。`rusqlite` 通过 bundled SQLite 把外部依赖压到极少。MicroClaw 的安装方式包括 Homebrew、Docker、Nix 和 one-line installer——这些都要求启动时不依赖外部数据库服务。如果换成 PostgreSQL，四种安装方式中至少两种会变得复杂得多。

第二，本地一致性更强。消息、会话、记忆、任务、subagent、审计都在同一数据库里，很多跨模块操作不需要做分布式协调。v0.1.38 的 schema 有 20+ 个表，如果分散在多个存储后端，一致性维护成本会急剧上升。

第三，可审计。SQLite 文件天然适合离线检查和备份。对于"为什么这轮对话会恢复成这样""哪条 subagent run 为什么失败""某条记忆何时被 supersede"这类问题，本地数据库比散落在多个组件里的状态更容易回溯。

第四，与 Rust 的 spawn_blocking 配合良好。`call_blocking` 函数把所有数据库操作包装在 `tokio::task::spawn_blocking` 中，避免阻塞 async runtime。这个模式在 v0.1.38 中已经成熟。

### SQLite 的边界也必须承认

- 横向扩展不是默认路径。
- 高并发写入场景会更快碰到瓶颈（虽然 SQLite 的 WAL 模式在多数 Agent 场景下足够）。
- 多实例共享一个状态库的部署模式需要额外设计。

MicroClaw 的做法不是否认这些边界，而是先围绕单机 runtime 把一致性做厚。Gateway service 管理（`src/gateway.rs`）让单机部署更容易变成长期运行的系统服务，而不是急于跳到多实例集群。

## 为什么 Provider 抽象只保留两类协议

### 统一到"Anthropic 原生"和"OpenAI-compatible"

`src/llm.rs` 的设计很克制。对外暴露的是统一的 `LlmProvider` trait，而在实现上主要收敛为两条通道：

- Anthropic 原生协议
- OpenAI-compatible 协议

v0.1.38 在 config 层通过 provider name 做路由。`config.rs` 的 `default_model_for_provider_name` 函数展示了映射关系：anthropic 走原生协议，openai/ollama/openai-codex 等走 OpenAI-compatible。

这个决策直接影响系统的可维护性。MicroClaw 不追求"每接一个 provider 就引入一套新语义"，而是优先把差异压缩到协议兼容层。

这会直接带来两个收益。

第一，`agent_engine.rs` 可以保持模型无关。核心循环只关心：输入消息是什么、工具 schema 是什么、返回的 `stop_reason` 是什么、是否需要继续 tool loop。它不需要知道底层是 Claude 还是 GPT。

第二，provider 演进不需要重写 runtime。新的供应商只要尽量适配既有协议，MicroClaw 就能把新增工作控制在配置与翻译层。v0.1.38 还支持 per-channel model override（`llm_model_overrides`），让不同渠道可以使用不同模型，而核心循环完全不需要改动。

### 为什么不做"每个 provider 一套最优实现"

因为运行时最怕的是协议分叉。一旦不同 provider 在 tool calling、streaming、usage 统计、错误处理上各走一套，后面所有章节讲的"统一循环""统一观测""统一 hooks"都会被削弱。

MicroClaw 选择的是一个更慢但更稳的方向：优先维持最小公共模型，再逐步修正兼容差异。

## 为什么用 8-crate workspace 而不是更多或更少

### workspace 演进的逻辑

从 v0.1.16 到 v0.1.38，workspace 的变化可以追踪出清晰的演进逻辑：

- `microclaw-core`：最底层的类型定义。LLM 请求/响应类型（`llm_types.rs`）、错误类型（`error.rs`）、文本处理（`text.rs`）。这些是所有其他 crate 都可能依赖的基础。
- `microclaw-storage`：schema 管理、所有数据库操作。这个 crate 需要独立于 agent engine，因为 tools、channels 和 scheduler 都需要直接访问数据库。
- `microclaw-tools`：工具运行时抽象。`Tool` trait、`ToolResult`、`ToolRisk`、`ToolConcurrencyClass`、sandbox 策略、path guard。这些定义需要被顶层的 tool_executor 和各个具体工具实现共同使用。
- `microclaw-channels`：`ChannelAdapter` trait、`ChannelRegistry`、消息投递逻辑。独立于具体渠道实现（具体渠道在顶层 `src/channels/` 里）。
- `microclaw-observability`：OTLP metrics、traces、logs 导出。独立于业务逻辑，但被 agent_engine、tool_executor 等多个模块使用。
- `microclaw-clawhub`：ClawHub 客户端、技能安装、lockfile 管理。这是 v0.1.38 新增的 crate，因为供应链追踪和技能发现的复杂度已经足以独立。
- `microclaw-app`：日志配置、内置技能、转写支持。这是应用层辅助功能的集合。
- 顶层 crate：agent engine、tool executor、run control、hooks、chat turn queue、all channels、web、acp、a2a、gateway、setup、doctor。这些高层装配逻辑依赖所有子 crate，放在顶层最合理。

### workspace 的收益

`microclaw-storage` 可以独立管理 schema 和数据 API，schema migration 不需要理解 agent engine 的内部逻辑。

`microclaw-tools` 沉淀了 concurrency class、sandbox、path guard 等通用逻辑。当 tool_executor 需要查询一个工具是 ReadOnly 还是 Exclusive 时，它调用的是 `microclaw-tools::runtime::tool_concurrency_class`，不需要引入 agent engine 的任何依赖。

`microclaw-channels` 把通道抽象拆出来，让 `ChannelRegistry` 可以在 tools（send_message 需要它）和 subagent（announce 需要它）中使用，而不用引入整个 runtime。

`microclaw-observability` 封装了 OpenTelemetry protobuf 序列化、SpanData 构建等细节。agent_engine 和 tool_executor 只需要 `use microclaw_observability::traces::*`，不需要直接依赖 `opentelemetry-proto` 和 `prost`。

### 为什么没有拆得更细

MicroClaw 仍保留顶层 `src/agent_engine.rs`、`src/tool_executor.rs`、`src/runtime.rs`、`src/hooks.rs`、`src/run_control.rs` 等文件，说明它没有走向"万物微 crate"。

原因很直接：这些模块之间有高度耦合的运行时交互。agent_engine 需要调用 tool_executor，tool_executor 需要检查 hooks，hooks 需要访问 config 和 db，run_control 需要被 agent_engine 和 channels 共同使用。如果把它们拆成独立 crate，循环依赖和 trait 抽象的复杂度会急剧上升，收益却不明显。

这种"crate 级模块化，而非进程级服务化"的演进方式，是当前阶段最合理的折中。

## Feature Flag 策略

v0.1.38 的 `Cargo.toml` 定义了四个 feature：

```toml
[features]
default = []
sqlite-vec = ["microclaw-storage/sqlite-vec"]
journald = ["microclaw-app/journald"]
channel-matrix = ["dep:matrix-sdk"]
full = ["channel-matrix"]
```

这个策略的逻辑很清晰：

- **默认最小**：`default = []`，不开启任何可选功能。这让基础编译和部署尽可能简单。
- **sqlite-vec**：向量检索支持，穿透到 storage crate。只有需要 embedding-based memory search 的用户才开启。
- **journald**：Linux systemd 日志集成，穿透到 app crate。只有 Linux 服务部署才需要。
- **channel-matrix**：Matrix 协议支持。`matrix-sdk` 是一个重量级依赖（带 e2e 加密和 SQLite），作为可选 feature 可以显著减小默认二进制体积。
- **full**：一键开启所有可选功能。

这种 feature flag 策略的工程价值在于：它让用户可以按部署场景裁剪二进制，而不是被迫接受所有功能的编译成本和依赖链。对于 Docker 部署，可能想开 `full`；对于嵌入式或 minimal 部署，默认就够了。

同时，feature flag 也影响了 runtime 的条件编译。在 `src/runtime.rs` 中：

```rust
#[cfg(not(feature = "channel-matrix"))]
fn warn_missing_feature(config: &Config, channel_key: &str, feature_name: &str) {
    if config.channel_enabled(channel_key) {
        warn!(
            "Channel '{}' is enabled in config, but this binary was built without the '{}' feature",
            channel_key, feature_name
        );
    }
}
```

这段代码说明系统会在配置要求但二进制未编译某个 feature 时给出明确警告，而不是静默忽略。

## 示例代码：为什么 Provider 抽象应该对齐协议，而不是对齐厂商名字？

下面的例子故意不接任何具体 SDK，只保留"协议边界"和"调用点"。

Rust 版本用 `trait` 表示统一协议，再用一个路由器持有当前 provider。v0.1.38 新增了 per-channel model override，所以路由器需要根据 channel 选择不同的 model。

```rust
#[async_trait::async_trait]
trait LlmProvider: Send + Sync {
    async fn send(&self, prompt: &str, model: &str) -> anyhow::Result<String>;
}

struct AnthropicProvider;
struct OpenAiCompatibleProvider;

struct ProviderRouter {
    active: Box<dyn LlmProvider>,
    model_overrides: std::collections::HashMap<String, String>,
}
```

```rust

impl ProviderRouter {
    fn resolve_model(&self, channel: &str, default_model: &str) -> String {
        self.model_overrides
            .get(channel)
            .cloned()
            .unwrap_or_else(|| default_model.to_string())
    }

    async fn complete(
        &self,
        channel: &str,
        prompt: &str,
        default_model: &str,
    ) -> anyhow::Result<String> {
        let model = self.resolve_model(channel, default_model);
        self.active.send(prompt, &model).await
    }
}

#[async_trait::async_trait]
impl LlmProvider for AnthropicProvider {
    async fn send(&self, prompt: &str, model: &str) -> anyhow::Result<String> {
        Ok(format!("anthropic/{model}: {prompt}"))
    }
}

#[async_trait::async_trait]
impl LlmProvider for OpenAiCompatibleProvider {
    async fn send(&self, prompt: &str, model: &str) -> anyhow::Result<String> {
        Ok(format!("openai-compat/{model}: {prompt}"))
    }
}
```

```python

    def resolve_model(self, channel: str, default_model: str) -> str:
        return self.model_overrides.get(channel, default_model)

    async def complete(
        self, channel: str, prompt: str, default_model: str
    ) -> str:
        model = self.resolve_model(channel, default_model)
        return await self.active.send(prompt, model)
```

## 依赖选择：关键库的工程理由

v0.1.38 的依赖列表比 v0.1.16 长得多，但每一个关键依赖都有明确的工程理由。

**tokio + axum**：异步运行时 + Web 框架。Axum 是 Tokio 生态的原生选择，它与 `tokio::JoinSet`、`tokio::sync::Mutex`、`tokio::task::spawn_blocking` 等原语无缝集成。

**rusqlite (bundled)**：SQLite 绑定。`bundled` feature 确保编译时不需要系统级 SQLite 安装，降低交叉编译和 Docker 构建的复杂度。

**teloxide**：Telegram bot 框架。选择它而不是直接调 HTTP API，是因为它处理了 long polling、webhook、消息格式化等大量 Telegram 特有细节。

**serenity**：Discord bot 框架。类似 teloxide 的角色。

**agent-client-protocol**：ACP 协议库。这是一个外部依赖（v0.10.3），说明 ACP 不是 MicroClaw 自己发明的协议，而是对接了一个更广泛的生态。

**rmcp**：MCP（Model Context Protocol）客户端。支持 child process 和 streamable HTTP 两种 transport。

**opentelemetry-proto + prost**：OTLP 指标和 trace 导出。选择直接使用 protobuf 定义而不是 OpenTelemetry SDK，说明团队选择了更轻量但更可控的集成方式。

**clap**：CLI 参数解析。MicroClaw 的 CLI 结构（start/acp/setup/doctor/gateway/skill/hooks/...）足够复杂，需要一个结构化的命令解析框架。

**ratatui + crossterm**：TUI 框架。用于 setup wizard 的交互式界面。11,000 行的 setup 代码说明这不是一个简单的命令行问答。

## 安装与分发策略

v0.1.38 提供了四种安装方式：

- **Homebrew**：`brew install microclaw`（macOS/Linux）
- **Docker**：`Dockerfile` + `docker-compose.yaml`
- **Nix**：`flake.nix`
- **One-line installer**：`install.sh`（Unix）、`install.ps1`（Windows）

这种多通道分发策略的选型理由是降低"第一步"的摩擦。不同用户群有不同的工具链偏好——macOS 开发者习惯 Homebrew，DevOps 习惯 Docker，NixOS 用户需要 flake，Windows 用户需要 PowerShell installer。

`build.rs` 和 `packaging/` 目录处理跨平台编译细节。`deny.toml` 做依赖许可证和安全审计。这些都是一个想要被真正部署的系统必须面对的工程现实。

## 关键权衡

### 决策一：选择 Rust + Tokio 作为统一 runtime

优点是并发模型清晰、状态边界显式、长期运行更稳定。代价是上手门槛高于脚本语言。v0.1.38 的复杂度（wave-based parallel execution、per-chat turn serialization、run control cancellation）证明了这个选择在中后期的回报。

### 决策二：优先 SQLite 单机状态库

优点是部署简单、链路统一、易于审计。代价是默认不面向多实例共享状态。v0.1.38 的 schema v19（20+ 表）证明 SQLite 在这个规模下仍然可行。

### 决策三：Provider 抽象只收敛到两类协议

优点是核心循环不被供应商差异污染。代价是某些 provider 的特有能力不会第一时间在核心层暴露。per-channel model override 是一个折中方案。

### 决策四：用 8-crate workspace 控制复杂度

优点是边界更清晰但运行链路仍然内聚。代价是 crate 数增加后，依赖关系和编译时间需要管理。

### 决策五：Feature flag 裁剪而不是多二进制

优点是一个 crate 覆盖所有场景，用户按需开启。代价是条件编译增加了测试矩阵。`full` feature 作为"全开"选项简化了 CI。

## 容易走错的地方

### 失败模式 1：把技术选型理解成"追求最快或最新"

Agent Runtime 的核心约束是稳定执行，而不是 benchmark 漂亮。如果只用"性能最好"或"生态最热"来做判断，很容易忽视状态一致性和治理成本。

### 失败模式 2：过早上远程数据库和分布式组件

如果消息恢复、工具循环、记忆注入、subagent tracking 这些主链路还没稳定，先拆库拆服务只会把调试难度放大。

### 失败模式 3：Provider 接入时把品牌差异直接写进核心循环

一旦 `agent_engine.rs` 里开始充满"某模型例外""某平台特判"，后面很难再维持统一行为。v0.1.38 的 `config.rs` 把 provider/model 预设放在配置层，而不是核心循环里。

### 失败模式 4：不做 feature flag，强迫用户编译所有功能

Matrix SDK 的 e2e 加密依赖链很重。如果不做 optional feature，每个用户——包括那些完全不需要 Matrix 的用户——都要承担这个编译和运行时成本。

## 小结

MicroClaw v0.1.38 的技术栈并不神秘，但它的选型顺序很有代表性：先选择能保持执行一致性的语言和运行时（Rust + Tokio），再选择能把状态压缩在本地的存储（SQLite），再选择足够克制的协议抽象（双协议 LLM），然后通过 8-crate workspace 把不断增长的能力做结构化拆分，最后用 feature flag 让用户按需裁剪。

从 v0.1.16 到 v0.1.38，crate 数量从约 5 个增长到 8 个，工具从少量增长到 44 个，渠道从约 6 个增长到 16 个，schema 从早期版本演进到 v19。但 workspace 的组织原则——"对内聚的保持内聚，对可拆的做稳定拆分"——始终没有变。

这套方法论决定了后面几章的展开方式。下一章，我们进入具体工程骨架，看看这些选型是如何在 `main.rs`、`runtime.rs`、`AppState` 和各个 crate 的职责边界中落地的。

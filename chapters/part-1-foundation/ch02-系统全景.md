# Chapter 2 系统全景

## 一次请求如何穿过系统

如果只把 MicroClaw 想象成 Ingress → LLM → Egress 的三段路径，会漏掉所有让它在生产环境活下来的机制：turn serialization、run control、hooks、wave-based 并行、subagent 调度、取消检查。读完本章应该能画出主路径，并清楚识别五个边界——渠道适配、turn serialization、工具执行（含并行调度）、数据持久化、协议桥接（ACP/A2A）。

## 从一条 Telegram 消息开始

用户在 Telegram 里发了一条消息。这条消息的完整链路：

```
Ingress ──► TurnQueue ──► RunControl ──► Session/Memory/SOUL
                                              │
                                              ▼
                                     Hooks(BeforeLLMCall)
                                              │
                                              ▼
                              ┌──────── Agent Loop ◄──────┐
                              │               │           │
                              │               ▼           │
                              │          LLM 调用         │
                              │          │    │           │
                              │     end_turn  tool_use    │
                              │          │    │           │
                              │          ▼    ▼           │
                              │    Egress  Wave 并行执行 ──┘
                              │               │
                              │               ▼
                              │         Cancelled?
                              │               │
                              ▼               ▼
                         Persistence ──► Cleanup/OTLP
```

展开 16 步：(1) 渠道接收 → (2) 解析 chat_id 与 session key → (3) ChatTurnQueue 检查活跃 run → (4) run_control 注册 → (5) session 恢复 → (6) 记忆与 SOUL 注入 → (7) BeforeLLMCall hooks → (8) 调用 LLM → (9) 工具请求则 wave-based 并行执行 → (10) 工具结果回灌，检查 cancelled → (11) 循环至 end_turn 或触达 max iterations → (12) 结果发回渠道 → (13) 持久化 → (14) run_control 注销 → (15) turn lock 释放 → (16) OTLP 指标更新。

### 入口事件规范化

所有渠道事件在入口处被翻译成统一形状，内核看到的事件结构与渠道无关：

```rust
use anyhow::Result;
use async_trait::async_trait;
use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, Serialize, Deserialize)]
struct IngressEvent {
    channel: String,           // "telegram" | "discord" | "feishu" | ...
    external_chat_id: String,  // 渠道方原生 chat 标识
    sender_id: String,
    text: String,
    chat_type: String,         // "private" | "group" | "channel"
    source_message_id: Option<String>,
}

#[async_trait]
trait ChannelIngress: Send + Sync {
    async fn normalize(&self, raw: &[u8]) -> Result<IngressEvent>;
}
```

15 个渠道（Telegram、Discord、Slack、Matrix、Email、Weixin、WhatsApp、Signal、IRC、iMessage、钉钉、飞书、QQ、Nostr、Web）都通过这个 trait 把外部协议归一为同一结构，再交给后续层处理。

## 四层结构

```
┌─────────────────────────────────────────────────────┐
│  第一层：接入层                                       │
│  15 渠道 + Web(SSE/WebSocket) + ACP(stdio) + A2A(HTTP)│
├─────────────────────────────────────────────────────┤
│  第二层：并发控制层                                   │
│  ChatTurnQueue（per-chat turn lock）                 │
│  run_control（register / cancel / notify）           │
├─────────────────────────────────────────────────────┤
│  第三层：核心执行层                                   │
│  agent_engine（统一 Agent Loop）                     │
│  tool_executor（wave-based 并行）                    │
│  hooks（BeforeLLM/BeforeTool/AfterTool）             │
│  llm.rs + tools/*.rs + subagents                    │
├─────────────────────────────────────────────────────┤
│  第四层：基础设施层                                   │
│  SQLite（schema v25+）| scheduler | memory          │
│  Web UI | OTLP | gateway | setup | doctor           │
└─────────────────────────────────────────────────────┘
```

**第一层（接入层）**把外部消息规范化为内部请求。Web 控制面通过 `src/web/stream.rs`（SSE）与 `src/web/ws.rs`（WebSocket）实时推送 AgentEvent。ACP 仅走 stdio，A2A 走 HTTP，二者与渠道并列在同一层。

**第二层（并发控制层）**。ChatTurnQueue 保证同一 chat 不并发 run，run_control 保证可中断。没有这层，多渠道高并发下会出现状态竞争与死循环。

**第三层（核心执行层）**。agent_engine 统一循环 + tool_executor wave 调度。tool_executor 的关键逻辑：

1. 为每个工具查询 `ToolConcurrencyClass`（ReadOnly / SideEffect / Exclusive）
2. 全部 ReadOnly → 合并成一个 wave 并行执行
3. 混合类型 → ReadOnly 先并行（wave 1），SideEffect 逐个执行，Exclusive 独占
4. wave 内由 `tokio::JoinSet` 并行，wave 间严格串行

**第四层（基础设施层）**。SQLite（schema v25+，20+ 表）、scheduler、Web 控制面、OTLP、gateway 服务管理、setup wizard、doctor 诊断。`db.rs` 在 v21 引入 FTS5 `session_search` 表，让 `session_search` 工具可以做全文检索。

## 关键模块职责

| 模块 | 职责 | 关键细节 |
|------|------|---------|
| `agent_engine.rs` | 统一 Agent Loop | 发出 8 种 AgentEvent（Iteration / ToolStart / ToolResult / TextDelta / ToolWaveStart / ToolWaveComplete / Cancelled / FinalResponse） |
| `tool_executor.rs` | 并行工具调度 | `partition_into_waves` + `PendingToolCall` + `ToolBatchContext` |
| `hooks.rs` | 执行拦截 | 调外部脚本，返回 allow / block / modify |
| `run_control.rs` | 运行控制 | `HashMap<(channel, chat_id), Vec<ActiveRun>>` + `AtomicBool` + `Notify` |
| `chat_turn_queue.rs` | Turn 序列化 | async mutex + pending message coalesce |
| `web/stream.rs` + `web/ws.rs` | 实时推送 | SSE 与 WebSocket 双通道 |
| `crates/microclaw-storage/src/db.rs` | 状态事实源 | schema v25+，覆盖完整运行时状态 |

## 五个系统边界

| 边界 | 分离什么 | 为什么 |
|------|---------|-------|
| 渠道适配 vs 核心逻辑 | 15 个 adapter 通过 ChannelAdapter trait 汇入统一循环 | 防止 15 套行为分支 |
| turn serialization vs 工具执行 | chat 级并发控制 vs turn 级工具调度 | 各自规则简单，组合可预测 |
| 工具执行 vs 模型决策 | 模型决定调什么，工具负责执行，hooks 插入拦截 | 权限、安全、审计 |
| 状态存储 vs 界面展示 | SQLite 是真源，Web/ACP/CLI 仅是界面 | 多入口共享同一事实 |
| 内部执行 vs 外部协议 | ACP/A2A 只做格式翻译，不改内核行为 | 内核不感知请求来源 |

## 主路径骨干

下面是把上述各层骨架抽出来后的最小可运行结构，足以体现 turn lock、cancelled 检查、持久化三件事的层级关系。真实代码在 `src/runtime.rs` 与 `src/agent_engine.rs` 中分散展开。

```rust
use std::sync::Arc;
use std::sync::atomic::{AtomicBool, Ordering};
use anyhow::Result;
use async_trait::async_trait;
use serde_json::Value;
use tokio::sync::OwnedMutexGuard;

#[async_trait]
trait MessageStore: Send + Sync {
    async fn load_recent(&self, chat_id: i64) -> Result<Vec<Value>>;
    async fn store_reply(&self, chat_id: i64, reply: &str) -> Result<()>;
}

#[async_trait]
trait MemoryStore: Send + Sync {
    async fn load_context(&self, chat_id: i64) -> Result<Vec<Value>>;
}

#[async_trait]
trait Agent: Send + Sync {
    async fn process(
        &self,
        chat_id: i64,
        prompt: Value,
        cancelled: Arc<AtomicBool>,
    ) -> Result<String>;
}

struct TurnGuard {
    _lock: OwnedMutexGuard<()>,
}

struct AppState<D, M, A> {
    db: Arc<D>,
    memory: Arc<M>,
    agent: Arc<A>,
}

impl<D, M, A> AppState<D, M, A>
where
    D: MessageStore + 'static,
    M: MemoryStore + 'static,
    A: Agent + 'static,
{
    async fn handle_message(
        &self,
        chat_id: i64,
        text: &str,
        _turn_guard: TurnGuard,
        cancelled: Arc<AtomicBool>,
    ) -> Result<String> {
        let history = self.db.load_recent(chat_id).await?;
        let memories = self.memory.load_context(chat_id).await?;
        let prompt = serde_json::json!({
            "history": history,
            "memory": memories,
            "user_text": text,
        });
        if cancelled.load(Ordering::Relaxed) {
            return Ok("Cancelled.".into());
        }
        let reply = self.agent.process(chat_id, prompt, cancelled).await?;
        self.db.store_reply(chat_id, &reply).await?;
        Ok(reply)
        // TurnGuard 在这里 drop，turn lock 释放，下一条排队消息可以进入
    }
}
```

`_turn_guard` 必须在函数签名上显式存在——它编码的是"调用者必须先拿到 turn lock 才能进入这条主路径"，是把第二层并发控制提升到类型签名的一种做法。`cancelled` 在每个 await 之间被检查；这是把第二层 run_control 的取消能力贯穿到主路径的方式。

## 关键权衡

| 决策 | 收益 | 代价 |
|------|------|------|
| 四层而不是三层 | turn 序列化与可取消性是所有渠道的默认行为 | 主链路多一层间接 |
| wave-based 工具并行偏保守 | 不需分析依赖关系也能安全并行 | 放弃部分理论可并行的写操作 |
| ACP/A2A 与渠道共享同一层 | 内核不必感知请求来源 | 协议适配不能调核心层之外的能力 |
| 基础设施层 Day 1 投入 | 运维能力从一开始就存在 | 前期投入更多"非功能"工作 |

## 容易走错的地方

**失败模式 1：把 Web UI 当成系统主体**。Web 只是控制面。以 UI 为中心会耦合到前端交互细节，难以支持 ACP、A2A 等其他入口。

**失败模式 2：在渠道层塞业务逻辑**。15 个渠道的复用就会失效。看起来是"灵活"，实际上是技术债。

**失败模式 3：忽略并发控制层**。没有 ChatTurnQueue，连发 3 条消息会产生 3 个并发 run 互相竞争会话状态；没有 run_control，无法取消卡住的 run。

**失败模式 4：忽略基础设施层**。把注意力全押在 prompt 与模型上，忽视 setup/doctor/OTLP，最后多花十倍时间在排查环境配置上。

## 小结

MicroClaw 的系统全景是一组围绕统一执行内核组织的能力层，由 turn serialization 与 run control 提供并发安全。决定它形态的三个设计选择：并发控制层（ChatTurnQueue + run_control）、wave-based 并行工具执行、协议桥接层（ACP + A2A）。

# Chapter 2 系统全景

## 一次请求从进入系统到离开，中间到底发生了什么？

知道"MicroClaw 是一个多渠道 Agent Runtime"还不够。真正要理解它，必须回答一个更具体的问题：一次请求从进入系统到离开系统，中间到底发生了什么？

MicroClaw 的请求路径远不只是 Ingress - LLM - Egress。它需要经过 turn serialization、run control 注册、hooks 拦截、wave-based 并行工具执行、subagent 生命周期管理、取消检查等多个阶段。

本章的目标，是建立 MicroClaw 的整体视图。读完后，你应该能画出这样一条主路径：

`Ingress → TurnQueue → RunControl → Session/Memory/SOUL → Hooks → Agent Loop → ParallelTools/Subagent → Persistence → Egress → Observability`

你还应该能识别五个关键边界：

1. 渠道适配边界
2. turn serialization 边界
3. 工具执行边界（含并行调度）
4. 数据持久化边界
5. 协议桥接边界（ACP/A2A）

这五个边界，决定了系统为什么能扩展、为什么能恢复、为什么不容易失控，以及为什么能被外部程序编排。

## 从一次请求开始看系统

假设用户在 Telegram 里发了一条消息："帮我检查当前仓库里最近的 TODO 并总结风险。"

这条消息进入系统后，不会直接调用模型。它会经过一条更长的链路：

1. 渠道适配器接收消息。
2. 系统解析 chat_id、thread、account、session key。
3. ChatTurnQueue 检查当前 chat 是否有活跃 run。如果有，消息被排队等待；如果没有，获取 turn lock。
4. run_control 注册一个新 run，分配 run_id、cancelled 标记和 notify 通知器。
5. 历史消息与 session 被恢复。
6. 文件记忆（AGENTS.md）、SOUL.md 人格文件、结构化记忆被注入。
7. Hooks 在 BeforeLLMCall 点位被触发。如果某个 hook 返回 block，请求不会到达模型。
8. Agent Loop 组装系统提示词和消息上下文，调用 LLM。
9. 模型返回文本，或请求调用工具。
10. 如果是工具请求，BeforeToolCall hooks 被触发。系统通过 tool_executor 将多个 tool_use 块按 concurrency class 分成 wave 并行/串行执行。AfterToolCall hooks 在每个工具完成后触发。
11. 工具结果追加回会话上下文，agent loop 检查 cancelled 标记，如果未取消则继续调用模型。
12. 循环继续，直到模型返回 `end_turn` 或达到 max_tool_iterations。
13. 最终结果经渠道适配器发回用户。
14. 消息、usage、memory 变化、subagent 状态被持久化。
15. run_control 注销当前 run。ChatTurnQueue 释放 turn lock，处理排队消息。
16. OTLP 指标、traces、审计日志被更新。

这 16 步可以压缩成以下主线：

```
Ingress ──► TurnQueue ──► RunControl ──► Session/Memory/SOUL
                                              │
                                              ▼
                                     Hooks(BeforeLLMCall)
                                              │
                                              ▼
                              ┌──────── Agent Loop ◄──────┐
                              │               │           │
                              │               ▼           │
                              │        LLM 调用           │
                              │          │    │           │
                              │     end_turn  tool_use    │
                              │          │    │           │
                              │          ▼    ▼           │
                              │    Egress  Wave 并行执行 ──┘
                              │               │
                              │               ▼
                              │         Cancelled?
                              │               │
                              ▼               ▼
                         Persistence ──► Cleanup/OTLP
```

MicroClaw 的主要复杂度不在"调用模型"这一步，而在于如何组织围绕模型的整个执行过程——特别是并发控制、取消语义和工具调度。

### 入口事件为什么要先被规范化？

系统全景之所以能画成一条稳定主路径，前提是所有渠道事件在入口处就被翻译成统一形状。MicroClaw 有 16 个渠道适配器，如果每个渠道都把自己的平台细节一直带进内核，代码会迅速失控。

Rust 版本把入口对象压成一个 `IngressEvent`，再用 `ChannelIngress` trait 负责规范化。这样内核看到的是统一事件，而不是 Telegram 或 Slack 的私有 payload。

```rust
struct IngressEvent {
    channel: String,
    external_chat_id: String,
    sender_id: String,
    text: String,
    chat_type: String,
}

#[async_trait::async_trait]
trait ChannelIngress {
    async fn normalize(&self, raw_body: &str) -> anyhow::Result<IngressEvent>;
}
```

Python 版本保留同一边界，但用 `Protocol + @dataclass` 让规范化步骤更一眼可见。

```python
from dataclasses import dataclass
from typing import Protocol


@dataclass
class IngressEvent:
    channel: str
    external_chat_id: str
    sender_id: str
    text: str
    chat_type: str


class ChannelIngress(Protocol):
    async def normalize(self, raw_body: str) -> IngressEvent: ...
```

## 四层结构：接入层、并发控制层、核心执行层、基础设施层

为了控制复杂度，MicroClaw 可以被理解为四层结构。并发控制层的存在是架构中最重要的设计决策之一。

### 第一层：接入层

这一层负责把不同渠道的输入统一成系统内部能理解的上下文。

典型文件分布在 `src/channels/` 目录下——telegram、discord、slack、feishu、email、irc、nostr、weixin、whatsapp、signal、imessage、dingtalk、qq——以及需要 `channel-matrix` feature flag 的 matrix 适配器。此外，`src/web.rs`（Web UI + API）、`src/acp.rs`（Agent Client Protocol，headless stdio 入口）和 `src/a2a.rs`（Agent-to-Agent 协议入口）也属于这一层。

这些模块做的事情并不是"实现 Agent 智能"，而是：接收外部消息、识别路由元数据、规范化 sender/chat/thread/account、调用共享的核心处理流程、把输出结果按渠道规则发回去。

ACP 和 A2A 也属于接入层。ACP 把 stdio 上的 JSON-RPC 消息转成 runtime 内部请求；A2A 把来自其他 MicroClaw 实例的 HTTP 请求转成内部消息。它们和 Telegram adapter 在架构上是同一层。

### 第二层：并发控制层

这一层解决的核心问题是：当多条消息同时涌入同一个 chat 时，怎么保证 Agent 行为的一致性？

- `src/chat_turn_queue.rs`：ChatTurnQueue 为每个 (channel, chat_id) 维护一个 turn lock。同一时刻只有一个 agent run 可以执行。后续消息被 coalesce 后在 run 结束时统一处理。
- `src/run_control.rs`：管理 active run 的注册、取消和通知。每个 run 持有 `Arc<AtomicBool>` cancelled 标记，支持干净取消。

这两个机制配合工作：ChatTurnQueue 保证序列化，run_control 保证可中断。没有这一层，多渠道 Agent 在高并发场景下会出现状态竞争和不可预期行为。

### 第三层：核心执行层

这是 MicroClaw 的主体。它主要由以下部分组成：

- `src/agent_engine.rs`：统一 Agent Loop，负责上下文组装、LLM 调用、tool loop 编排、cancelled 检查、subagent 结果收集。
- `src/tool_executor.rs`：wave-based 并行工具执行引擎。将多个 tool_use 按 concurrency class 分成 wave，同一 wave 内的 ReadOnly 工具并行执行。
- `src/llm.rs`：屏蔽 Anthropic 和 OpenAI-compatible 两类 provider 差异。
- `src/tools/*.rs`：44 个内置工具的实现。
- `src/hooks.rs`：BeforeLLMCall/BeforeToolCall/AfterToolCall 三个拦截点，每个 hook 可以 allow/block/modify。
- `src/tools/subagents.rs`：session-native subagent 体系。spawn → queue → run → complete/fail/timeout/cancel 生命周期，restricted tool registry，announce callbacks。

其中 `tool_executor.rs` 的 wave 调度逻辑值得特别说明。当 LLM 返回 N 个 tool_use 块时：

1. 系统为每个工具查询其 `ToolConcurrencyClass`（ReadOnly / SideEffect / Exclusive）。
2. 如果全部是 ReadOnly，合并成一个 wave 并行执行。
3. 如果有混合类型，ReadOnly 先并行执行（wave 1），SideEffect 逐个执行（wave 2+），Exclusive 独占一个 wave。
4. wave 之间严格串行，wave 内部通过 `tokio::JoinSet` 并行。

这个设计的关键权衡是：宁可保守也不冒险。SideEffect 工具（如 `write_file`、`send_message`）不会被并行执行，因为它们的副作用之间可能有依赖关系。只有 `read_file`、`glob`、`grep` 这类确认无副作用的工具才会被放入并行 wave。

### 第四层：基础设施层

这一层提供运行所需的持久化、配置与运维能力：

- `crates/microclaw-storage/src/db.rs`：SQLite 数据库，schema v19，覆盖 chats、messages、sessions、memories、scheduled_tasks、subagent_runs、subagent_events、subagent_announces、audit_logs、metrics_history 等。
- `src/scheduler.rs`：计划任务和后台 reflector。
- `src/web.rs` + `web/`：React+Vite 嵌入式 Web UI，支持 session tree/fork/delete、abort、streaming。
- `src/memory_service.rs` + `src/memory_backend.rs`：记忆注入与反射器。
- `src/skills.rs` + `src/mcp.rs`：技能管理与 MCP 集成。
- `crates/microclaw-observability/`：OTLP metrics、traces、logs 三合一。
- `crates/microclaw-clawhub/`：ClawHub 技能发现、lockfile 供应链追踪。
- `src/gateway.rs`：跨平台 service 管理（install/start/stop/status/logs）。
- `src/setup.rs`：11,000+ 行交互式配置向导。
- `src/doctor.rs`：跨平台环境诊断。
- `SOUL.md`：人格注入系统。

## 关键模块职责

四层结构提供了一个静态的分层视角。但要真正理解系统的行为，还需要进入关键模块的内部，看它们各自承担什么责任、如何协作。下面按执行路径上的重要性逐一展开。

### `src/agent_engine.rs`：统一 Agent Loop

这是整个系统最重要的文件之一。它负责把"一个请求"扩展成"一个多轮执行过程"。

它的职责包括：

- 恢复会话
- 构建系统提示词（包括 SOUL.md 注入）
- 注入文件记忆和结构化记忆
- 触发上下文 compaction
- 在每次 LLM 调用前触发 BeforeLLMCall hooks
- 调用 LLM
- 处理 tool_use：委托给 tool_executor 做 wave-based 并行执行
- 检查 cancelled 标记，支持中途取消
- 处理 subagent 结果
- 发送 AgentEvent 流（Iteration、ToolStart、ToolResult、ToolWaveStart、ToolWaveComplete、TextDelta、Cancelled、FinalResponse）
- 持久化最终状态

AgentEvent 流意味着系统不再只是"最后返回一段文本"，而是在执行过程中持续发出结构化事件。这让 Web UI 的 streaming 显示、ACP 的流式响应和渠道层的中间状态更新都成为可能。

### `src/tool_executor.rs`：并行工具调度

这个模块解决的是"一次 LLM 返回多个 tool_use 时怎么高效执行"的问题。

核心数据结构是 `PendingToolCall`（待执行的工具调用）和 `ToolBatchContext`（跨工具执行的累积状态，包括失败工具集合、连续 send_message 计数、用户审批等待状态等）。

`partition_into_waves` 函数是调度逻辑的核心：它接收一组 PendingToolCall，查询每个工具的 concurrency class，然后产出一组 wave（每个 wave 是一组可以并行的工具索引）。配置层还支持 `tool_concurrency_overrides`，允许用户覆盖特定工具的默认并发类。

### `src/hooks.rs`：执行拦截器

Hooks 系统支持三个事件点：BeforeLLMCall、BeforeToolCall、AfterToolCall。每个 hook 是一个外部可执行程序（通常是 shell 脚本），通过 frontmatter 声明触发事件、超时和优先级。

系统在相应时机调用 hook 程序，传入当前上下文（工具名、输入参数等），hook 返回 JSON 响应，可以是 allow（放行）、block（阻止）或 modify（修改输入）。这个机制让用户可以在不修改 MicroClaw 源码的情况下，插入安全审查、日志记录或输入改写逻辑。

### `src/run_control.rs`：运行控制

run_control 管理一个全局的 `ACTIVE_RUNS: HashMap<(channel, chat_id), Vec<ActiveRun>>` 和一个 `ABORTED_SOURCE_MESSAGE_IDS` 集合。

每个 run 注册时获得一个 run_id、一个 `Arc<AtomicBool>` cancelled 标记和一个 `Arc<Notify>` 通知器。取消操作通过 `abort_runs` 函数触发，它会设置 cancelled 标记并通知所有等待者。Agent loop 在每次迭代时检查这个标记，如果发现取消就发出 `AgentEvent::Cancelled` 并终止循环。

### `src/chat_turn_queue.rs`：Turn 序列化

ChatTurnQueue 确保同一 (channel, chat_id) 同一时刻只有一个 agent run 在执行。它的 `ChatSlot` 包含一个 async mutex（turn lock）和一个 pending messages 队列。

当 run 完成后释放 turn lock 时，如果有排队消息，系统会把它们 coalesce 成一条合并消息，然后开始下一个 run。这避免了"用户连发 3 条消息导致 3 个并发 run 互相竞争会话状态"的问题。

### `crates/microclaw-storage/src/db.rs`：状态事实来源

MicroClaw 选择 SQLite 作为本地事实来源。schema 已经到第 19 版，包含的核心表：

- `chats`、`messages`、`sessions`：基础会话状态
- `scheduled_tasks`、`task_run_logs`、`scheduled_task_dlq`：调度任务与死信队列
- `memories`、`memory_reflector_runs`、`memory_injection_logs`、`memory_supersede_edges`：完整记忆生命周期
- `subagent_runs`、`subagent_events`、`subagent_announces`、`subagent_focus_bindings`：subagent 全生命周期
- `audit_logs`：审计日志
- `metrics_history`：指标时间线
- `auth_passwords`、`auth_sessions`、`api_keys`、`api_key_scopes`：认证与授权
- `llm_usage_logs`：LLM 使用统计

这意味着系统不是"无状态 API"，而是一个有完整持久状态的本地运行时。

## 系统边界：哪些东西必须分开

### 边界一：渠道适配与核心逻辑分开

渠道层不应该持有业务主逻辑。MicroClaw 有 16 个渠道适配器，如果每个都自定义 Agent 行为，系统就不可维护了。MicroClaw 通过 `ChannelAdapter` 抽象（`crates/microclaw-channels/src/channel_adapter.rs`）和 `ChannelRegistry`，把大部分智能行为收敛在核心层。

### 边界二：turn serialization 与工具执行分开

ChatTurnQueue 保证"同一 chat 不会有并发 run"，tool_executor 保证"同一 run 内的工具按正确顺序并行/串行执行"。这两层解决不同问题：前者是 chat 级并发控制，后者是 turn 级工具调度。把它们分开，让每层的规则更简单。

### 边界三：工具执行与模型决策分开

模型负责决定"是否调用工具、调用哪个工具"，但工具负责执行实际副作用。Hooks 在两者之间插入了拦截点。这三方边界必须清楚，否则权限、安全和审计就无法建立。

### 边界四：状态存储与界面展示分开

Web UI 很重要，但它不应该成为状态真源。状态真源在 SQLite 和 runtime 数据目录里。Web UI（React+Vite）只是读取与操作这些状态的界面。ACP 也是如此。这个分离让 CLI、Web、渠道消息、ACP 客户端都能共享同一套底层事实。

### 边界五：内部执行与外部协议分开

ACP 和 A2A 是外部协议，它们让 MicroClaw 可以被程序化驱动和跨实例通信。但它们不改变内核行为——它们只是把请求翻译成内部格式，再把结果翻译回协议格式。这样做的好处是，内核不需要知道"这个请求来自 IDE 还是来自另一个 Agent"。

## 一次请求的主路径

为了把系统看得更清楚，我们把一次请求拆成十二个阶段。

### 阶段 1：Ingress

消息从某个渠道进入系统，渠道适配器提取 chat id、sender、account、thread/topic、外部 chat 标识、chat type。

### 阶段 2：Turn Serialization

ChatTurnQueue 检查当前 chat 是否有活跃 run。如果有，消息进入 pending 队列。如果没有，获取 turn lock，进入下一阶段。

### 阶段 3：Run Registration

run_control 注册新 run，返回 (run_id, cancelled, notify)。从此刻起，这个 run 可以被外部取消。

### 阶段 4：Session 解析

系统根据 chat 和 session key 识别当前会话。如果有可恢复 session，恢复它；如果没有，根据历史消息重建。session settings（label、thinking_level、verbose_level、reasoning_level）也在这一步装载。

### 阶段 5：历史与记忆装载

系统读取最近消息、AGENTS.md（global/bot/chat 三级）、SOUL.md（全局 + per-chat override）、结构化记忆，为后续构建上下文做准备。

### 阶段 6：Hooks - BeforeLLMCall

如果配置了 BeforeLLMCall hooks，系统在调用 LLM 前触发它们。如果任何 hook 返回 block，请求不会到达模型。

### 阶段 7：LLM 调用

模型接收上下文后，要么直接给出最终答复（`end_turn`），要么请求工具（`tool_use`）。流式响应通过 AgentEvent::TextDelta 实时传递给调用方。

### 阶段 8：Parallel Tool Execution

如果模型返回 tool_use，tool_executor 将所有 pending tool calls 按 concurrency class 分成 wave。每个工具执行前触发 BeforeToolCall hook，执行后触发 AfterToolCall hook。wave 内的 ReadOnly 工具并行执行，wave 之间串行。AgentEvent::ToolWaveStart 和 ToolWaveComplete 事件被发出。

### 阶段 9：Cancellation Check

每次 tool wave 完成后、每次 LLM 调用前，系统检查 cancelled 标记。如果发现取消，发出 AgentEvent::Cancelled，携带已累积的文本，然后终止循环。

### 阶段 10：Persistence

消息、usage、memory 变化、subagent run 状态、task 记录等被写入 SQLite。这一步是系统可恢复性的关键。

### 阶段 11：Egress

最终文本或中间消息经渠道层发送出去。这里会处理各平台的格式差异、长度限制和多段发送。

### 阶段 12：Cleanup & Observability

run_control 注销当前 run。ChatTurnQueue 释放 turn lock，如果有 pending messages 则 coalesce 后开始下一个 run。OTLP 指标、traces、审计日志被更新。

## 示例代码：为什么一次消息必须沿着同一条主路径穿过系统？

这组示例不是完整实现，而是把主路径骨干压成可读代码。关键设计是引入了 turn guard 和 cancellation token。

Rust 版本把四类依赖抽成独立 trait，再由 `AppState` 持有它们。TurnGuard 是 RAII 对象，离开作用域时自动释放 turn lock。

```rust
use std::sync::Arc;
use std::sync::atomic::{AtomicBool, Ordering};

#[async_trait::async_trait]
trait MessageStore {
    async fn load_recent_messages(&self, chat_id: i64) -> anyhow::Result<Vec<String>>;
    async fn store_reply(&self, chat_id: i64, reply: &str) -> anyhow::Result<()>;
}

#[async_trait::async_trait]
trait MemoryStore {
    async fn load_context(&self, chat_id: i64) -> anyhow::Result<Vec<String>>;
}

#[async_trait::async_trait]
trait Agent {
    async fn process(
        &self,
        chat_id: i64,
        prompt: &str,
        cancelled: Arc<AtomicBool>,
    ) -> anyhow::Result<String>;
}

struct TurnGuard {
    _lock: tokio::sync::OwnedMutexGuard<()>,
}

struct AppState<D, M, A> {
    db: D,
    memory: M,
    agent: A,
}
```

这段先把主路径依赖固定下来，包括 cancellation token。系统全景最怕的是边界松动，所以这里先定义"谁提供历史、谁提供记忆、谁负责推理、谁负责取消"。

```rust
impl<D: MessageStore, M: MemoryStore, A: Agent> AppState<D, M, A> {
    async fn handle_message(
        &self,
        chat_id: i64,
        text: &str,
        _turn_guard: TurnGuard,
        cancelled: Arc<AtomicBool>,
    ) -> anyhow::Result<String> {
        let history = self.db.load_recent_messages(chat_id).await?;
        let memories = self.memory.load_context(chat_id).await?;
        let prompt = build_prompt(&history, &memories, text);
        if cancelled.load(Ordering::Relaxed) {
            return Ok("Cancelled.".to_string());
        }
        let reply = self.agent.process(chat_id, &prompt, cancelled).await?;
        self.db.store_reply(chat_id, &reply).await?;
        Ok(reply)
        // TurnGuard drops here, releasing the turn lock
```

## 为什么这个结构适合扩展

MicroClaw 的规模正好验证了这个结构的扩展能力。能力扩展大体只有几种路径：

- 新渠道：加 adapter（核心层几乎不需要改动）
- 新工具：加 tool，声明 concurrency class
- 新 provider：加 llm backend
- 新控制面：加 API/route，复用现有 runtime
- 新拦截逻辑：加 hook 脚本，不改源码
- 新协议入口：ACP 和 A2A 就是这样加入的——它们是新的 adapter，不是新的内核

这让系统的扩展面保持清晰。并行工具执行是一个特别好的例子：它没有修改 agent_engine 的主循环结构，而是在 tool execution 阶段插入了一个新的调度层。老工具不需要改代码，只需要确认自己的 concurrency class。

## 关键权衡

### 决策一：四层而不是三层

MicroClaw 在接入层和核心执行层之间插入了专门的并发控制层（ChatTurnQueue + run_control）。优点是 turn serialization 和可取消性成为所有渠道的默认行为，而不是每个渠道自己实现。代价是主链路多了一层间接，状态机更复杂。但考虑到 16 个渠道在高并发下的竞态问题，这个代价是值得的。

### 决策二：wave-based 工具并行宁可保守也不冒险

SideEffect 工具（如 `write_file`、`send_message`）绝不并行，只有确认无副作用的 ReadOnly 工具才进入并行 wave。优点是不需要分析工具之间的依赖关系就能安全并行。代价是放弃了一部分理论上可以并行的写操作。但在生产系统里，简单规则比复杂分析更可靠。

### 决策三：ACP 和 A2A 与渠道共享同一层

把 ACP（stdio 协议）和 A2A（跨实例 HTTP 协议）放在接入层而不是核心层，意味着它们和 Telegram adapter 在架构上是同一层。优点是内核不需要知道请求来自 IDE 还是另一个 Agent。代价是协议适配逻辑不能调用核心层以外的能力，否则会破坏分层。

### 决策四：基础设施层不是附属件

11,000 行的 setup wizard、跨平台 doctor、gateway service 管理和 OTLP 三合一导出都属于第四层。优点是运维能力从 Day 1 就存在。代价是前期开发必须投入更多"看起来不直接产出功能"的工作。

## 容易走错的地方

### 失败模式 1：把 Web UI 当成系统主体

Web 很重要，但它只是控制面。如果设计反过来，以 UI 为中心，系统会很快耦合到前端交互细节，难以支持 ACP、A2A 和渠道消息等其他入口。

### 失败模式 2：在渠道层塞入业务逻辑

一旦每个渠道都开始自定义自己的 Agent 行为，16 个渠道的复用就会失效。看起来是"灵活"，实际上是技术债。

### 失败模式 3：忽略并发控制层

没有 ChatTurnQueue，用户连发 3 条消息就会产生 3 个并发 run 互相竞争同一个 chat 的会话状态。没有 run_control，用户就无法取消一个卡住的 run。这些问题在 demo 场景里不明显，但在长期运行的生产系统里是致命的。

### 失败模式 4：忽略基础设施层

很多 Agent 项目把注意力全放在 prompt 和模型上，忽略调度、存储、观测、setup、doctor。短期看很快，长期看不可维护。MicroClaw 的结构提醒我们：基础设施不是附属件，而是 runtime 的一部分。一个有 11,000 行 setup wizard 的系统，显然不认为配置是次要问题。

## 小结

MicroClaw 的系统全景可以被概括为一句话：它不是"很多功能拼在一起"，而是"围绕统一执行内核组织起来的一组能力层，中间由 turn serialization 和 run control 保障并发安全"。

架构中最重要的三个设计决策是：并发控制层的引入（ChatTurnQueue + run_control）、wave-based 并行工具执行（tool_executor）、以及协议桥接层（ACP + A2A）。

下一章，我们继续向下钻一层，不再只谈模块，而是谈领域模型：Chat、Session、Message、Task、Memory、SubagentRun 这六个核心实体是如何协作的。

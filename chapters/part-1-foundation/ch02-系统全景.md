# Chapter 2 系统全景

## 一次请求从进入到离开，中间发生了什么？

MicroClaw 的请求路径远不只 Ingress → LLM → Egress。它需要经过 turn serialization、run control、hooks、wave-based 并行工具执行、subagent 管理、取消检查等阶段。

读完本章，你应该能画出主路径并识别五个关键边界：渠道适配、turn serialization、工具执行（含并行调度）、数据持久化、协议桥接（ACP/A2A）。

## 从一次请求开始看系统

用户在 Telegram 里发了一条消息。这条消息经过的完整链路：

```
Ingress ──► TurnQueue ──► RunControl ──► Session/Memory/SOUL
                                              │
                                              ▼
                                     Hooks(BeforeLLMCall)
                                              │
                                              ▼
                              ┌──────── Agent Loop ◄──────┐
                              │               │           │
                              │               ▼           │
                              │        LLM 调用           │
                              │          │    │           │
                              │     end_turn  tool_use    │
                              │          │    │           │
                              │          ▼    ▼           │
                              │    Egress  Wave 并行执行 ──┘
                              │               │
                              │               ▼
                              │         Cancelled?
                              │               │
                              ▼               ▼
                         Persistence ──► Cleanup/OTLP
```

16 步展开：(1) 渠道接收 → (2) 解析 chat_id/session key → (3) ChatTurnQueue 检查活跃 run → (4) run_control 注册 → (5) session 恢复 → (6) 记忆/SOUL 注入 → (7) BeforeLLMCall hooks → (8) 调用 LLM → (9) 工具请求则 wave-based 并行执行 → (10) 工具结果追加，检查 cancelled → (11) 循环至 end_turn 或 max iterations → (12) 结果发回 → (13) 持久化 → (14) run_control 注销 → (15) turn lock 释放 → (16) OTLP 更新。

## 四层结构

```
┌─────────────────────────────────────────────────────┐
│  第一层：接入层                                       │
│  16 渠道 + Web + ACP(stdio) + A2A(HTTP)              │
├─────────────────────────────────────────────────────┤
│  第二层：并发控制层                                    │
│  ChatTurnQueue (per-chat turn lock)                  │
│  run_control (register/cancel/notify)                │
├─────────────────────────────────────────────────────┤
│  第三层：核心执行层                                    │
│  agent_engine (统一循环)                              │
│  tool_executor (wave-based 并行)                     │
│  hooks (BeforeLLM/BeforeTool/AfterTool)              │
│  llm.rs + tools/*.rs + subagents                     │
├─────────────────────────────────────────────────────┤
│  第四层：基础设施层                                    │
│  SQLite(schema v19) | scheduler | memory             │
│  Web UI | OTLP | gateway | setup | doctor            │
└─────────────────────────────────────────────────────┘
```

**第一层（接入层）**：把外部消息规范化为内部请求。ACP/A2A 也属于此层。

**第二层（并发控制层）**：ChatTurnQueue 保证同一 chat 不并发 run，run_control 保证可中断。没有这层，多渠道高并发下会出现状态竞争。

**第三层（核心执行层）**：agent_engine 统一循环 + tool_executor wave 调度。tool_executor 的关键逻辑：

1. 为每个工具查询 `ToolConcurrencyClass`（ReadOnly / SideEffect / Exclusive）
2. 全部 ReadOnly → 合并成一个 wave 并行
3. 混合类型 → ReadOnly 先并行（wave 1），SideEffect 逐个执行，Exclusive 独占
4. wave 内 `tokio::JoinSet` 并行，wave 间严格串行

**第四层（基础设施层）**：SQLite（20+ 表）、scheduler、Web 控制面、OTLP、gateway 服务管理、11,000 行 setup wizard、doctor 诊断。

## 关键模块职责

| 模块 | 职责 | 关键细节 |
|------|------|---------|
| `agent_engine.rs` | 统一 Agent Loop | 发出 8 种 AgentEvent（Iteration/ToolStart/ToolResult/TextDelta/ToolWaveStart/ToolWaveComplete/Cancelled/FinalResponse） |
| `tool_executor.rs` | 并行工具调度 | `partition_into_waves` + `PendingToolCall` + `ToolBatchContext` |
| `hooks.rs` | 执行拦截 | 外部脚本，返回 allow/block/modify |
| `run_control.rs` | 运行控制 | `HashMap<(channel, chat_id), Vec<ActiveRun>>` + AtomicBool + Notify |
| `chat_turn_queue.rs` | Turn 序列化 | async mutex + pending message coalesce |
| `db.rs` | 状态事实源 | schema v19, 20+ 表覆盖完整运行时状态 |

## 系统边界

| 边界 | 分离什么 | 为什么 |
|------|---------|-------|
| 渠道适配 vs 核心逻辑 | 16 个 adapter 通过 ChannelAdapter trait 汇入统一循环 | 防止 16 套分裂系统 |
| turn serialization vs 工具执行 | chat 级并发控制 vs turn 级工具调度 | 各自规则更简单 |
| 工具执行 vs 模型决策 | 模型决定调什么，工具负责执行，hooks 插入拦截 | 权限、安全、审计 |
| 状态存储 vs 界面展示 | SQLite 是真源，Web/ACP/CLI 只是界面 | 多入口共享同一事实 |
| 内部执行 vs 外部协议 | ACP/A2A 只做格式翻译，不改内核行为 | 内核不感知请求来源 |

## 示例代码：主路径骨干

```rust
use std::sync::Arc;
use std::sync::atomic::{AtomicBool, Ordering};

#[async_trait::async_trait]
trait MessageStore {
    async fn load_recent_messages(&self, chat_id: i64) -> anyhow::Result<Vec<String>>;
    async fn store_reply(&self, chat_id: i64, reply: &str) -> anyhow::Result<()>;
}

#[async_trait::async_trait]
trait MemoryStore {
    async fn load_context(&self, chat_id: i64) -> anyhow::Result<Vec<String>>;
}

#[async_trait::async_trait]
trait Agent {
    async fn process(
        &self,
        chat_id: i64,
        prompt: &str,
        cancelled: Arc<AtomicBool>,
    ) -> anyhow::Result<String>;
}

struct TurnGuard {
    _lock: tokio::sync::OwnedMutexGuard<()>,
}

struct AppState<D, M, A> {
    db: D,
    memory: M,
    agent: A,
}

impl<D: MessageStore, M: MemoryStore, A: Agent> AppState<D, M, A> {
    async fn handle_message(
        &self,
        chat_id: i64,
        text: &str,
        _turn_guard: TurnGuard,
        cancelled: Arc<AtomicBool>,
    ) -> anyhow::Result<String> {
        let history = self.db.load_recent_messages(chat_id).await?;
        let memories = self.memory.load_context(chat_id).await?;
        let prompt = build_prompt(&history, &memories, text);
        if cancelled.load(Ordering::Relaxed) {
            return Ok("Cancelled.".to_string());
        }
        let reply = self.agent.process(chat_id, &prompt, cancelled).await?;
        self.db.store_reply(chat_id, &reply).await?;
        Ok(reply)
        // TurnGuard drops here, releasing the turn lock
    }
}
```

```python
from dataclasses import dataclass
from typing import Protocol
import threading


class MessageStore(Protocol):
    async def load_recent_messages(self, chat_id: int) -> list[str]: ...
    async def store_reply(self, chat_id: int, reply: str) -> None: ...


class MemoryStore(Protocol):
    async def load_context(self, chat_id: int) -> list[str]: ...


class Agent(Protocol):
    async def process(
        self, chat_id: int, prompt: str, cancelled: threading.Event
    ) -> str: ...


@dataclass
class AppState:
    db: MessageStore
    memory: MemoryStore
    agent: Agent

    async def handle_message(
        self,
        chat_id: int,
        text: str,
        cancelled: threading.Event,
    ) -> str:
        history = await self.db.load_recent_messages(chat_id)
        memories = await self.memory.load_context(chat_id)
        prompt = build_prompt(history, memories, text)
        if cancelled.is_set():
            return "Cancelled."
        reply = await self.agent.process(chat_id, prompt, cancelled)
        await self.db.store_reply(chat_id, reply)
        return reply
```

## 关键权衡

| 决策 | 收益 | 代价 |
|------|------|------|
| 四层而不是三层 | turn serialization 和可取消性成为所有渠道的默认行为 | 主链路多一层间接 |
| wave-based 工具并行宁可保守 | 不需要分析工具依赖关系就能安全并行 | 放弃部分理论并行写操作 |
| ACP/A2A 与渠道共享同一层 | 内核不需要知道请求来源 | 协议适配不能调核心层外能力 |
| 基础设施层 Day 1 投入 | 运维能力从一开始就存在 | 前期投入更多"非功能"工作 |

## 容易走错的地方

### 失败模式 1：把 Web UI 当成系统主体

Web 只是控制面。以 UI 为中心，会耦合到前端交互细节，难以支持 ACP、A2A 等其他入口。

### 失败模式 2：在渠道层塞入业务逻辑

16 个渠道的复用就会失效。看起来是"灵活"，实际上是技术债。

### 失败模式 3：忽略并发控制层

没有 ChatTurnQueue，连发 3 条消息就会产生 3 个并发 run 互相竞争会话状态。没有 run_control，无法取消卡住的 run。

### 失败模式 4：忽略基础设施层

很多项目把注意力全放在 prompt 和模型上。11,000 行 setup wizard 说明配置不是次要问题。

## 小结

MicroClaw 的系统全景：围绕统一执行内核组织的能力层，由 turn serialization 和 run control 保障并发安全。最重要的三个设计决策：并发控制层（ChatTurnQueue + run_control）、wave-based 并行工具执行、协议桥接层（ACP + A2A）。

# Chapter 2 系统全景

## 这一章要回答什么问题

知道“MicroClaw 是一个多渠道 Agent Runtime”还不够。真正要理解它，必须回答一个更具体的问题：一次请求从进入系统到离开系统，中间到底发生了什么？

本章的目标，是建立 MicroClaw 的整体视图。读完后，你应该能画出这样一条主路径：

`Ingress -> Session/Memory -> Agent Loop -> Tools -> Persistence -> Egress`

你还应该能识别三个关键边界：

1. 渠道适配边界
2. 工具执行边界
3. 数据持久化边界

这三个边界，决定了系统为什么能扩展、为什么能恢复、以及为什么不容易失控。

## 从一次请求开始看系统

假设用户在 Telegram 里发了一条消息：“帮我检查当前仓库里最近的 TODO 并总结风险。”

这条消息进入系统后，不会直接调用模型。它会经过一条更长的链路：

1. 渠道适配器接收消息。
2. 系统解析 chat_id、thread、account、session key。
3. 历史消息与 session 被恢复。
4. 文件记忆和结构化记忆被注入。
5. Agent Loop 组装系统提示词和消息上下文。
6. 模型返回文本，或请求调用工具。
7. 如果是工具请求，系统执行工具并把结果追加回会话。
8. 循环继续，直到模型返回 `end_turn`。
9. 最终结果经渠道适配器发回用户。
10. 整个过程中的状态、usage、memory、副作用被持久化或记录。

从这个路径可以看出，MicroClaw 的主要复杂度不在“调用模型”这一步，而在于如何组织围绕模型的整个执行过程。

### 小例子：为什么入口事件要先被规范化？

系统全景之所以能画成一条稳定主路径，前提是所有渠道事件在入口处就被翻译成统一形状。否则每个渠道都会把自己的平台细节一直带进内核。

Rust 版本把入口对象压成一个 `IngressEvent`，再用 `ChannelIngress` trait 负责规范化。这样内核看到的是统一事件，而不是 Telegram 或 Slack 的私有 payload。

```rust
struct IngressEvent {
    channel: String,
    external_chat_id: String,
    sender_id: String,
    text: String,
}

#[async_trait::async_trait]
trait ChannelIngress {
    async fn normalize(&self, raw_body: &str) -> anyhow::Result<IngressEvent>;
}
```

Python 版本保留同一边界，但用 `Protocol + @dataclass` 让规范化步骤更一眼可见。这样读者更容易理解接入层到底在“做什么抽象工作”。

```python
from dataclasses import dataclass
from typing import Protocol


@dataclass
class IngressEvent:
    channel: str
    external_chat_id: str
    sender_id: str
    text: str


class ChannelIngress(Protocol):
    async def normalize(self, raw_body: str) -> IngressEvent: ...
```

## 三层结构：接入层、核心执行层、基础设施层

为了控制复杂度，MicroClaw 可以被理解为三层结构。

### 第一层：接入层

这一层负责把不同渠道的输入统一成系统内部能理解的上下文。

典型文件包括：

- `src/channels/telegram.rs`
- `src/channels/discord.rs`
- `src/channels/slack.rs`
- `src/channels/feishu.rs`
- `src/web.rs`

它们做的事情并不是“实现 Agent 智能”，而是：

- 接收外部消息
- 识别路由元数据
- 规范化 sender/chat/thread/account
- 调用共享的核心处理流程
- 把输出结果按渠道规则发回去

这意味着渠道层主要解决的是 ingress/egress 问题，而不是推理问题。

### 第二层：核心执行层

这是 MicroClaw 的主体。它主要由三个部分组成：

- `src/agent_engine.rs`
- `src/llm.rs`
- `src/tools/mod.rs` 及各个具体工具实现

其中：

- `agent_engine.rs` 管理循环、上下文压缩、memory 注入、工具调用编排
- `llm.rs` 屏蔽不同 provider 差异
- `tools` 提供系统可以执行的外部能力

这一层决定了系统的“智能执行”能力。无论消息来自 Telegram 还是 Web，只要进入这一层，行为就应该尽量一致。

### 第三层：基础设施层

这一层提供运行所需的持久化与控制面能力：

- SQLite 数据库
- Scheduler
- Web 控制台与 API
- OTLP 指标
- Hooks
- Skills / MCP

代表性文件包括：

- `crates/microclaw-storage/src/db.rs`
- `src/scheduler.rs`
- `src/web.rs`
- `src/hooks.rs`
- `src/skills.rs`
- `src/mcp.rs`

这一层不直接决定每一轮推理“说什么”，但它决定了系统是否可观察、可配置、可恢复。

## 关键模块职责

### `src/agent_engine.rs`：统一 Agent Loop

这是整个系统最重要的文件之一。它负责把“一个请求”扩展成“一个多轮执行过程”。

在高层上，它的职责包括：

- 恢复会话
- 构建系统提示词
- 注入记忆
- 触发上下文压缩
- 调用 LLM
- 处理 tool_use
- 持久化最终状态

如果把 MicroClaw 比作操作系统，那么 `agent_engine.rs` 很像调度核心。

### `src/llm.rs`：模型抽象层

MicroClaw 支持 Anthropic 和 OpenAI-compatible provider。`src/llm.rs` 的价值在于，它把 provider 差异尽量压平，避免上层逻辑被具体厂商 API 绑住。

这层通常处理：

- 请求格式适配
- stream 与非 stream 模式
- tool schema 传递
- usage 信息回收

如果没有这一层，整个系统会被 provider-specific 细节污染。

### `src/tools/*.rs`：工具系统

工具系统让 Agent 从“会说”变成“会做”。但工具并不是随意执行命令，而是通过统一的 `Tool` trait 和 registry 调度。

这种设计有三个价值：

- 工具定义结构化，可生成文档
- 工具执行可插入权限控制
- 工具返回值可统一回灌到模型循环

对运行时来说，工具系统就是“能力总线”。

### `crates/microclaw-storage/src/db.rs`：状态事实来源

MicroClaw 选择 SQLite 作为本地事实来源。对系统全景来说，这个决定非常关键，因为它让这些能力成为可能：

- 聊天历史查询
- session 恢复
- memory 生命周期管理
- 调度任务存储
- API key / auth session 管理
- 观测与 usage 汇总

这意味着系统不是“无状态 API”，而是一个有持久状态的本地运行时。

## 系统边界：哪些东西必须分开

### 边界一：渠道适配与核心逻辑分开

渠道层不应该持有业务主逻辑。否则 Telegram、Slack、Web 会逐渐分裂成几套系统。

MicroClaw 通过 `ChannelAdapter` 抽象和共享 Agent Loop，把大部分智能行为收敛在核心层。这是它支持多渠道而不失控的前提。

### 边界二：工具执行与模型决策分开

模型负责决定“是否调用工具、调用哪个工具”，但工具负责执行实际副作用。这两个边界必须清楚，否则权限、安全和审计就无法建立。

这也是为什么工具系统要有自己的定义、执行、超时和错误处理通道。

### 边界三：状态存储与界面展示分开

Web UI 很重要，但它不应该成为状态真源。状态真源在数据库和 runtime 数据目录里。UI 只是读取与操作这些状态的界面。

这个分离让 CLI、Web、渠道消息都能共享同一套底层事实。

## 一次请求的主路径

为了把系统看得更清楚，我们把一次请求拆成九个阶段。

### 阶段 1：Ingress

消息从某个渠道进入系统，渠道适配器提取：

- chat id
- sender
- account
- thread / topic
- 外部 chat 标识

### 阶段 2：Session 解析

系统根据 chat 和 session key 识别当前会话。对于 Web 和 hooks，这一步尤其重要，因为 session key 可以被显式指定。

### 阶段 3：历史与记忆装载

系统读取最近消息、可恢复 session、`AGENTS.md` 和结构化 memory，为后续构建上下文做准备。

### 阶段 4：System Prompt 构建

系统并不是把所有历史直接塞给模型，而是构建一个更精细的 prompt，包括：

- 系统规则
- 记忆内容
- 技能说明
- 工具定义

### 阶段 5：LLM 调用

模型接收上下文后，要么直接给出最终答复，要么请求工具。

### 阶段 6：Tool Loop

如果模型返回 `tool_use`，系统执行工具，把结果作为新消息追加回上下文，再继续调用模型。

### 阶段 7：Persistence

消息、usage、memory 变化、任务记录等被写入 SQLite 或 runtime 文件。

### 阶段 8：Egress

最终文本或中间消息经渠道层发送出去。这里会处理各平台的格式差异、长度限制和多段发送。

### 阶段 9：Observability

指标、审计、失败信息、memory 反射器记录被更新，为后续运营与排错提供依据。

## 为什么这个结构适合扩展

如果一个系统的扩展方式是“每新增能力就加一条特殊分支”，它迟早会失控。MicroClaw 的结构之所以健康，是因为新增能力大体只有几种路径：

- 新渠道：加 adapter
- 新工具：加 tool
- 新 provider：加 llm backend
- 新控制面：加 API/route，但尽量复用现有 runtime

这让系统的扩展面保持清晰。最近补上的 Mission Control WebSocket bridge 就是一个好例子：它没有重写执行内核，而是复用既有 async run + stream hub，再翻译成兼容帧。

## 示例代码：为什么一次消息必须沿着同一条主路径穿过系统？

这组示例不是完整实现，而是把“装载历史、装载记忆、调用 Agent、落库回复”压成一条看得见的骨干路径。只要这条路径始终统一，系统扩展时就不会把状态事实散落到渠道、前端或脚本里。

Rust 版本把三类依赖抽成独立 trait，再由 `AppState` 持有它们。这样主路径上的每一步都可以替换实现，但消息处理顺序本身保持稳定，不会因为接入新渠道而变形。

```rust
#[async_trait::async_trait]
trait MessageStore {
    async fn load_recent_messages(&self, chat_id: i64) -> anyhow::Result<Vec<String>>;
    async fn store_reply(&self, chat_id: i64, reply: &str) -> anyhow::Result<()>;
}

#[async_trait::async_trait]
trait MemoryStore {
    async fn load_context(&self, chat_id: i64) -> anyhow::Result<Vec<String>>;
}

#[async_trait::async_trait]
trait Agent {
    async fn process(&self, chat_id: i64, prompt: &str) -> anyhow::Result<String>;
}

struct AppState<D, M, A> {
    db: D,
    memory: M,
    agent: A,
}
```

第一段先把主路径依赖固定下来。系统全景最怕的是边界松动，所以这里先定义“谁提供历史、谁提供记忆、谁负责推理”，再进入消息处理本身。

```rust
impl<D: MessageStore, M: MemoryStore, A: Agent> AppState<D, M, A> {
    async fn handle_message(&self, chat_id: i64, text: &str) -> anyhow::Result<String> {
        let history = self.db.load_recent_messages(chat_id).await?;
        let memories = self.memory.load_context(chat_id).await?;
        let prompt = build_prompt(&history, &memories, text);
        let reply = self.agent.process(chat_id, &prompt).await?;
        self.db.store_reply(chat_id, &reply).await?;
        Ok(reply)
    }
}
```

Python 版本保留同样的边界，但用 `Protocol` 把依赖契约写显式，用 `@dataclass` 保持装配层是一个可读对象而不是松散命名空间。这样读者会更容易看出“系统全景”不是目录图，而是一条稳定的协作链。

```python
from dataclasses import dataclass
from typing import Protocol


class MessageStore(Protocol):
    async def load_recent_messages(self, chat_id: int) -> list[str]: ...
    async def store_reply(self, chat_id: int, reply: str) -> None: ...


class MemoryStore(Protocol):
    async def load_context(self, chat_id: int) -> list[str]: ...


class Agent(Protocol):
    async def process(self, chat_id: int, prompt: str) -> str: ...


@dataclass
class AppState:
    db: MessageStore
    memory: MemoryStore
    agent: Agent
```

Python 也按同样顺序拆成两步。先看装配对象，后看主路径执行，读者会更容易把“协作关系”和“执行顺序”分开理解，同时避免整块代码硬跨页。

```python

    async def handle_message(self, chat_id: int, text: str) -> str:
        history = await self.db.load_recent_messages(chat_id)
        memories = await self.memory.load_context(chat_id)
        prompt = build_prompt(history, memories, text)
        reply = await self.agent.process(chat_id, prompt)
        await self.db.store_reply(chat_id, reply)
        return reply
```

## 容易走错的地方

### 失败模式 1：把 Web UI 当成系统主体

Web 很重要，但它只是控制面。如果设计反过来，以 UI 为中心，系统会很快耦合到前端交互细节，难以支持其他入口。

### 失败模式 2：在渠道层塞入业务逻辑

一旦每个渠道都开始自定义自己的 Agent 行为，多渠道复用就会失效。看起来是“灵活”，实际上是技术债。

### 失败模式 3：忽略基础设施层

很多 Agent 项目把 attention 全放在 prompt 和模型上，忽略调度、存储、观测。短期看很快，长期看不可维护。MicroClaw 的结构提醒我们：基础设施不是附属件，而是 runtime 的一部分。

## 小结

MicroClaw 的系统全景可以被概括为一句话：它不是“很多功能拼在一起”，而是“围绕统一执行内核组织起来的一组能力层”。

这张全景图非常重要，因为后续每一章都会落回这里：工具系统、记忆系统、多渠道架构、调度、Web 控制面，本质上都只是这张图里某一层或某一条边界的展开。

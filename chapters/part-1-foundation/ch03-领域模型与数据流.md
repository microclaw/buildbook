# Chapter 3 领域模型与数据流

## 系统里的核心对象是什么，它们之间是什么关系？

系统全景解决的是"模块怎么分层"，但还没有回答另一个更底层的问题：运行时里的六个关键实体是什么，它们之间如何协作？

对于一个 Agent Runtime 来说，这不是纸上建模。领域模型直接决定：

- 会话能不能恢复
- 消息能不能追踪
- 任务能不能调度
- 记忆能不能演化
- subagent 能不能被管理和观测

MicroClaw v0.1.38 有六个关键实体：

- Chat
- Session
- Message
- Task
- Memory
- SubagentRun

相比 v0.1.16 的五个实体，SubagentRun 是新增的第六个。它不是一个边角概念，而是 session-native subagent 体系的核心——它有自己的生命周期、状态机、token 预算、深度限制和 announce 机制。

这一章读完后，你应该能够：

1. 说清这六个实体的职责与边界。
2. 理解一次请求是如何在这些实体之间流动的，包括 subagent 的嵌套执行。
3. 理解系统为什么能够做到中断、恢复、重试和观测。

## 为什么领域模型比你想的更重要

很多项目的失败，不是因为模型不够好，而是因为状态模型太模糊。最常见的错误有两种。

第一种错误是把所有内容都塞进"消息历史"。这样做短期最简单，但很快就会遇到问题：哪些消息是用户输入，哪些是工具结果？哪些是临时过程，哪些是最终事实？一个 subagent 的执行轨迹应该放在主会话的消息流里，还是放在独立的 run 记录里？一个任务失败后，如何定位原因？记忆是从历史里提炼出来的，还是和历史混在一起的？

第二种错误是完全无状态。这样做会让每次请求看似独立，实际上系统无法积累经验，也无法恢复中断状态。

MicroClaw 的做法是把不同职责拆成不同实体，让每类状态有明确归属。

## 实体一：Chat

Chat 是最外层的对话容器。它通常对应一个真实聊天空间：

- Telegram 私聊或群组
- Discord 频道或线程
- Slack 频道或线程
- Feishu/Lark 群组
- Email 线程
- IRC 频道
- Web 中的一个 session key 对应聊天
- ACP 中的一个 session

Chat 的核心作用，是承载"这个上下文属于谁、来自哪里"。它不是推理状态本身，而是运行时的外部归属。

在数据库层面（`chats` 表），Chat 至少承担这些职责：

- 绑定渠道与外部 chat 标识
- 提供人类可识别的标题或 session key
- 记录 chat_type（private、group、acp 等）
- 作为 Message、Session、Task、Memory、SubagentRun 的归属锚点

如果没有 Chat 这个容器，系统就只能围绕"消息文本"工作，而无法处理跨渠道和长期运行问题。v0.1.38 的 16 个渠道加上 ACP 和 A2A，每个都有自己的外部标识格式，Chat 是把它们统一起来的锚。

## 实体二：Session

Session 是 Agent Loop 的恢复对象。很多读者第一次看这个概念，会误以为它和 Chat 是同一个东西。它们并不是。

Chat 解决的是"对话属于哪个外部空间"，Session 解决的是"当前推理状态是什么"。

v0.1.38 的 `sessions` 表和 `SessionSettings` 结构包含：

- 当前消息上下文
- 压缩后的历史摘要
- session label（人类可读的会话名称）
- thinking_level（模型思考深度设置）
- verbose_level（输出详细度）
- reasoning_level（推理模式）

为什么要把 Session 单独拿出来？因为聊天历史和推理状态并不总是一一对应。

聊天里可能有很多无关闲聊，但 Session 里只保留当前任务相关内容。历史消息可能很长，但 Session 可以经过 compaction。一个 Chat 可以在重置后继续存在，而 Session 可以被清空并重建。v0.1.38 的 Web UI 支持 session tree、fork 和 delete——这些操作之所以成立，正是因为 Session 和 Chat 是两个独立实体。

### Session 为什么不该直接等于 Chat？

Rust 版本让 `Chat` 只负责外部归属，让 `SessionState` 负责当前推理状态。v0.1.38 增加了 `SessionSettings` 来携带 per-session 配置。

```rust
#[derive(Debug, Clone)]
struct Chat {
    id: i64,
    channel: String,
    chat_type: String,
}

#[derive(Debug, Clone)]
struct SessionSettings {
    label: Option<String>,
    thinking_level: Option<String>,
    verbose_level: Option<String>,
    reasoning_level: Option<String>,
}

#[derive(Debug, Clone)]
struct SessionState {
    chat_id: i64,
    summary: String,
    settings: SessionSettings,
    pending_tool_call: Option<String>,
}
```

Python 版本用 `@dataclass` 保持同样的边界。

```python
from dataclasses import dataclass


@dataclass
class Chat:
    id: int
    channel: str
    chat_type: str


@dataclass
class SessionSettings:
    label: str | None = None
    thinking_level: str | None = None
    verbose_level: str | None = None
    reasoning_level: str | None = None


@dataclass
class SessionState:
    chat_id: int
    summary: str
    settings: SessionSettings
    pending_tool_call: str | None = None
```

## 实体三：Message

Chat 和 Session 分别解决了"对话属于谁"和"当前推理状态是什么"的问题。但系统里还有第三类东西需要被记录：用户说了什么、系统回了什么、工具执行了什么。这就是 Message 的职责。Message 是最直观的实体，但也是最容易被误用的实体。

在 MicroClaw 的 `messages` 表里，每条 `StoredMessage` 至少包含：

- `id`（UUID）
- `chat_id`（归属 chat）
- `sender_name`
- `content`
- `is_from_bot`
- `timestamp`

Message 在系统中承担四种角色：

1. 用户输入的持久化痕迹
2. 助手输出的持久化痕迹
3. 工具执行结果的可审计记录
4. 系统事件的可见表示

它的价值在于提供审计性和可回放性。用户说了什么，系统回复了什么，某个时刻发生了什么，这些都必须落在 Message 这一层。

但是 Message 不等于全部状态。它不应该独占以下职责：记忆事实库（那是 Memory 的事）、调度任务定义（那是 Task 的事）、subagent 执行记录（那是 SubagentRun 的事）、可恢复 session 状态（那是 Session 的事）。否则系统会变成"只有日志，没有模型"。

## 实体四：Task

前三个实体——Chat、Session、Message——覆盖了"交互式对话"的完整生命周期。但真实场景中，用户很快会提出一个新要求：能不能明天早上帮我跑一次巡检？能不能每天定时汇总一次仓库风险？这就把系统推向了时间维度。Task 就是为此而存在的——它表示被调度、被追踪、可重试的工作单元。v0.1.38 的 `scheduled_tasks` 表结构包含：

- `id`
- `chat_id`（归属 chat）
- `prompt`（要执行的指令）
- `schedule_type`（"cron" 或 "once"）
- `schedule_value`（cron 表达式或 ISO 时间戳）
- `timezone`（IANA 时区）
- `next_run`、`last_run`
- `status`（active / paused / completed / cancelled）

配套的 `task_run_logs` 表记录每次执行的开始时间、结束时间、耗时、是否成功和结果摘要。`scheduled_task_dlq`（死信队列）表记录失败的执行，支持事后 replay。

Task 的存在让系统不再被限制在"收到消息才运行"。一旦任务被纳入领域模型，MicroClaw 就可以处理这些场景：定时提醒、定时抓取、后台巡检、延时执行。

v0.1.38 提供了完整的调度工具族：`schedule_task`、`list_scheduled_tasks`、`pause_scheduled_task`、`resume_scheduled_task`、`cancel_scheduled_task`、`get_scheduled_task_history`、`list_scheduled_task_dlq`、`replay_scheduled_task_dlq`。这意味着 Agent 自己就能管理自己的定时任务，而不是依赖外部 cron 或人工操作。

## 实体五：Memory

Memory 是本书中最重要、也最容易被泛化的概念之一。MicroClaw 选择把它拆成两层：

### 文件记忆

使用 `AGENTS.md` 作为可读、可编辑、可解释的记忆层，分为：

- global（全局偏好和约束）
- bot/account（per-bot 行为规则）
- chat（per-chat 项目背景和上下文）

v0.1.38 新增了 `SOUL.md` 人格注入系统。SOUL.md 不是记忆，而是人格定义——它描述 Agent "是什么样的人"，而不是 Agent "记住了什么"。全局 SOUL.md 在根目录，per-chat override 可以让同一实例在不同聊天中表现不同性格。

### 结构化记忆

存放在 `memories` 表中，带有：

- `category`
- `confidence`（浮点数，表示可靠度）
- `source`（来源标注）
- `embedding_model`（可选，用于向量检索）
- `is_archived` / `archived_at`（归档状态）
- `last_seen_at`（最后活跃时间）
- `created_at` / `updated_at`

配套的观测设施非常完整。`memory_reflector_runs` 表记录每次反射器提取的结果（extracted_count、inserted_count、updated_count、skipped_count、dedup_method）。`memory_injection_logs` 表记录每次记忆注入的过程（candidate_count、selected_count、omitted_count、tokens_est）。`memory_supersede_edges` 表记录记忆之间的替代关系。

`MemoryObservabilitySummary` 结构体把这些观测数据聚合起来：总记忆数、活跃数、归档数、低置信度数、平均置信度、24 小时内的 reflector 运行次数和注入事件次数。

双层设计的关键意义在于：可读性和可操作性并不总是同一个维度。文件记忆更容易被人维护，结构化记忆更容易被系统使用。v0.1.38 的 `structured_memory_search`、`structured_memory_delete`、`structured_memory_update` 三个工具让 Agent 可以直接操作结构化记忆，而 `read_memory` 和 `write_memory` 操作文件记忆。

## 实体六：SubagentRun

前五个实体已经覆盖了单 Agent 的完整工作模型。但当任务复杂到一个 Agent 循环无法高效完成时——比如同时做代码审查、写测试和更新文档——系统需要把工作委托出去。SubagentRun 正是为此而生。它是 v0.1.38 新增的第六个关键实体，让 MicroClaw 从"单 Agent 循环"进化到"session-native subagent 体系"。

`subagent_runs` 表的字段说明了它的完整生命周期：

- `run_id`（UUID，唯一标识）
- `parent_run_id`（可选，支持嵌套 subagent）
- `depth`（嵌套深度，防止无限递归）
- `chat_id`（归属 chat）
- `caller_channel`
- `task`（子任务描述）
- `context`（传递给 subagent 的上下文）
- `status`（queued / running / completed / failed / timeout / cancelled）
- `token_budget`（token 预算限制）
- `provider` / `model`
- `input_tokens` / `output_tokens` / `total_tokens`
- `error_text` / `result_text`
- `artifact_json`（结构化输出）
- `cancel_requested`
- `created_at` / `started_at` / `finished_at`

配套表包括：

- `subagent_events`：记录 run 生命周期中的关键事件（event_type + detail）
- `subagent_announces`：管理 subagent 完成后的回调通知（支持重试、状态追踪）
- `subagent_focus_bindings`：管理"聚焦"关系，允许主 Agent 持续关注某个 subagent

SubagentRun 为什么必须是独立实体，而不是"特殊的 Message"？

第一，它有自己的状态机。从 queued 到 running 到 completed/failed/timeout/cancelled，这个生命周期不是 Message 的"发送/接收"语义能表达的。

第二，它有自己的资源约束。token_budget 和 depth 是 SubagentRun 的属性，不是 Message 的属性。当系统需要判断"这个 subagent 是不是用完了预算"时，它查的是 SubagentRun，不是 Message。

第三，它有嵌套关系。parent_run_id 让系统可以追踪 subagent 之间的父子关系。一个 orchestrate 操作可能同时 spawn 多个 subagent，每个 subagent 又可能 spawn 自己的子 subagent（受 depth 限制）。这种树形结构只有作为独立实体才能清晰建模。

第四，它有独立的 announce 机制。当 subagent 完成时，系统通过 `subagent_announces` 表管理回调通知。如果通知失败，会自动重试。这套重试逻辑与 Message 的发送逻辑完全不同。

`SubagentObservabilitySummary` 结构体聚合了 subagent 体系的运行状态：活跃 run 数、排队数、运行中数、pending/retry/failed announce 数、24 小时完成/失败/超预算数、平均耗时。

## 六个实体之间的关系

如果把整个系统想成一张图，最重要的关系可以概括为：

- Chat 拥有很多 Message
- Chat 关联一个可恢复的 Session（可 fork、可 reset）
- Chat 关联很多 Task（scheduled tasks）
- Chat 关联文件记忆（AGENTS.md per-chat）与结构化 Memory
- Chat 关联很多 SubagentRun
- Session 以 Message 历史和 Memory 注入为输入，驱动一次或多次 Agent Loop
- SubagentRun 可以有 parent_run_id 形成树形嵌套
- SubagentRun 完成时通过 announce 机制通知关联的 Chat
- Memory 有 supersede_edges 记录替代关系
- Task 有 task_run_logs 记录执行历史和 dlq 记录失败历史

两个关键点：

第一，Session 不是 Message 的简单集合，而是对历史与记忆的一次可恢复投影。

第二，SubagentRun 不是 Message 或 Task 的变种，而是独立的执行实体。它和 Task 的区别在于：Task 是被调度的（cron 或 once），SubagentRun 是被 spawn 的（在 Agent Loop 内由工具调用触发）。Task 在主 Agent 的上下文中执行，SubagentRun 拥有自己的 restricted tool registry 和 token 预算。

## 一次请求的数据流

下面我们用一次涉及 subagent 的请求来走完数据流。

### 第一步：用户消息进入 Chat

用户消息首先被写入 Message 存储，并归属到某个 Chat。这一阶段解决的问题是：记录事实、建立归属、为后续恢复和审计提供基础。

### 第二步：系统恢复 Session

如果当前 Chat 已经有可恢复 Session，系统就恢复它（包括 SessionSettings）；如果没有，就根据历史消息重建。

### 第三步：系统装载 Memory 和 SOUL

此时系统读取 AGENTS.md（global/bot/chat）、SOUL.md（global + per-chat override）、结构化记忆（通过 `memory_backend` 检索）、技能说明。然后把这些内容与 Session 组合成模型可理解的上下文。

### 第四步：Agent Loop 运行

模型根据当前上下文决定直接回答、调用工具或 spawn subagent。

如果模型请求调用多个工具，tool_executor 按 concurrency class 分成 wave 执行。工具结果回灌给 Session，run_control 检查 cancelled 标记。

### 第五步：Subagent Spawn

如果模型调用了 `sessions_spawn` 工具，系统创建一个新的 SubagentRun 记录（状态为 queued），分配 run_id，记录 parent_run_id 和 depth。

subagent 使用 restricted tool registry——它不能访问所有 44 个内置工具，只能使用一个受限子集。这通过 `ToolRegistry::new_sub_agent` 实现。如果 `allow_session_tools` 为 true，subagent 还可以 spawn 自己的子 subagent（受 depth 限制）。

subagent 的执行在自己的 Agent Loop 中完成，有独立的 token 预算。执行完成后，状态更新为 completed/failed/timeout/cancelled，result_text 和 token 统计被写入 SubagentRun 记录。

### 第六步：Announce Callback

subagent 完成后，系统通过 announce 机制把结果通知回主 chat。announce 记录存在 `subagent_announces` 表中，支持重试。如果通知失败，系统会在后续尝试中自动重试。

### 第七步：结果持久化

当一轮请求结束时，系统会更新：

- Message（用户输入、助手输出、工具结果）
- Session（上下文状态、compaction 摘要）
- usage（LLM token 消耗）
- Memory（如果触发了 reflector 或显式 memory 操作）
- SubagentRun（如果有 subagent 活动）
- Task 状态（如果是 scheduled task 触发的执行）

## stop_reason、失败路径与恢复

一个健康的领域模型，不只描述成功路径，还必须描述失败路径。

在 MicroClaw v0.1.38 的 Agent Runtime 里，至少要区分以下几种结束状态：

- 正常结束（end_turn）
- 工具错误（某个工具返回 is_error = true）
- 用户中断（通过 /stop、Web UI abort 或 ACP cancel）
- 超时（tool timeout 或 compaction timeout）
- 预算超限（max_tool_iterations 或 subagent token_budget）
- hook 阻止（BeforeLLMCall 或 BeforeToolCall 返回 block）
- subagent 失败/超时/取消

这些状态为什么重要？因为它们决定了系统后续能做什么。

如果是用户中断，run_control 设置 cancelled 标记，agent loop 发出 `AgentEvent::Cancelled`，已有历史保留，当前 run 终止。

如果是 subagent 失败，主 Agent 收到错误信息，可以决定重试、报告或放弃。SubagentRun 的状态被更新为 failed，error_text 记录失败原因。

如果是 hook 阻止，系统可以把 hook 返回的拒绝理由告知用户，而不是静默失败。

这说明 Session、SubagentRun 和 Task 不只是"有没有"，还必须有生命周期语义。

## 幂等与重试：为什么要在模型外建立规则

Agent 系统天然容易发生重试：网络抖动、渠道重投、WebSocket 重连、webhook 重复调用。

如果领域模型里没有幂等意识，系统就会出现重复消息、重复工具调用、重复任务副作用。

MicroClaw 的设计启示是：幂等不应该交给模型记忆，而应该交给运行时规则来控制。

ChatTurnQueue 通过 turn lock 防止同一 chat 的并发 run。run_control 的 `ABORTED_SOURCE_MESSAGE_IDS` 追踪已取消的源消息，避免重复处理。SubagentRun 的 run_id 是 UUID，announce 机制有状态追踪。scheduled_task_dlq 的 replay 操作有 `replayed_at` 时间戳，防止重复 replay。

换句话说，领域模型不仅是数据结构，也是抗故障策略的基座。

## 为什么双层记忆值得单独强调

在六个实体里，Memory 最容易被误解成"只是 RAG 的一部分"。这其实低估了它的作用。

在 MicroClaw 里，Memory 更像"系统长期事实层"。它承接的是那些不适合每轮都由用户重述、又不能完全依赖模型短期上下文记住、并且需要被观测和修正的知识。

v0.1.38 的记忆可观测性远超 v0.1.16。`MemoryObservabilitySummary` 可以告诉你：系统里有多少条记忆、多少是活跃的、多少被归档了、平均置信度多少、最近 24 小时 reflector 提取了多少条新记忆、注入了多少次。这些数据让记忆系统不再是黑盒。

文件记忆（AGENTS.md + SOUL.md）负责可解释性和人格，结构化记忆负责可计算性。这个分工，反映了 MicroClaw 在领域建模上对可读性和可计算性两个维度的刻意分离。

## 关键权衡

### 决策一：六个独立实体而非一个大对象

把 Chat、Session、Message、Task、Memory、SubagentRun 拆成独立实体，每个都有自己的表和生命周期。优点是职责边界清晰——归属、恢复、审计、持续执行、长期事实、委托执行互不混淆。代价是实体之间的关系需要显式维护，查询时需要跨表 join。但对于一个需要恢复、取消、审计的长期运行系统，松散的大对象远比分不清边界更危险。

### 决策二：Session 和 Chat 分离

Chat 负责"这个对话属于谁"，Session 负责"当前推理状态是什么"。优点是支持 session fork、tree view、compaction 和 reset 等操作——这些在 Chat 和 Session 合一的模型里无法表达。代价是概念上多了一层抽象，新用户需要理解两者的区别。

### 决策三：SubagentRun 作为独立实体而非特殊 Message

SubagentRun 有自己的状态机（queued → running → completed/failed/timeout/cancelled）、资源约束（token_budget、depth）、嵌套关系（parent_run_id）和 announce 机制。优点是所有这些语义都可以被独立查询、观测和管理。代价是 schema 复杂度增加（需要 subagent_runs、subagent_events、subagent_announces、subagent_focus_bindings 四张表）。

### 决策四：双层记忆而非单层

文件记忆（AGENTS.md + SOUL.md）负责可读可编辑，结构化记忆负责可检索可归档。优点是两层各自适合不同的使用场景，人类运维和机器检索都有合适的接口。代价是两层系统需要持续解释职责边界，且修改时需要决定改哪一层。

### 决策五：失败路径纳入领域模型

在实体定义中显式包含取消、超时、预算超限、hook 拒绝、subagent 失败等状态语义。优点是恢复和重试逻辑有明确的状态基础。代价是每个实体的状态空间更大，但不定义这些状态只会让系统在运行时以不可预期的方式失败。

## 容易走错的地方

### 失败模式 1：把 Chat 和 Session 混为一谈

这样会导致你无法解释为什么"聊天还在，但上下文已经重置"，也无法解释 session fork、compaction、resume、tree view 等行为。

### 失败模式 2：把 Message 当作全部事实来源

一旦所有东西都靠消息日志表达，系统会很难做结构化恢复、memory 生命周期管理、subagent 状态追踪和任务控制。

### 失败模式 3：把 Memory 做成单层

只有文件记忆，检索和自动化会很弱；只有结构化记忆，可读性和手工修正会很差。双层虽然复杂，但在长期系统里更稳。

### 失败模式 4：把 SubagentRun 当成"特殊的 Message"

SubagentRun 有自己的状态机（queued → running → completed/failed/timeout/cancelled）、自己的资源约束（token_budget、depth）、自己的 announce 机制、自己的嵌套关系。把它塞进 Message 会让这些语义全部丢失。

### 失败模式 5：没有失败路径模型

如果系统只定义"成功时怎么走"，没有取消、超时、预算超限、hook 拒绝、subagent 失败等状态语义，运行时很快会失去一致性。

## 小结

MicroClaw v0.1.38 的领域模型之所以成立，不是因为实体很多，而是因为每个实体都承担了不同职责：

- Chat 负责归属
- Session 负责恢复（含 per-session settings）
- Message 负责审计
- Task 负责持续执行（含 DLQ 和 replay）
- Memory 负责长期事实（双层：文件 + 结构化，含完整观测）
- SubagentRun 负责委托执行（含嵌套、预算、announce、focus）

当这六个实体被分清楚之后，系统的数据流、恢复机制和失败处理逻辑就会变得可解释。后续章节无论讲工具、记忆还是调度，都会回到这里。

# Chapter 3 领域模型与数据流

## 这一章要回答什么问题

系统全景解决的是“模块怎么分层”，但还没有回答另一个更底层的问题：系统里的核心对象到底是什么，它们之间又是什么关系？

对于一个 Agent Runtime 来说，这不是纸上建模。领域模型直接决定：

- 会话能不能恢复
- 消息能不能追踪
- 任务能不能调度
- 记忆能不能演化

本章会定义 MicroClaw 的五个关键实体：

- Chat
- Session
- Message
- Task
- Memory

这一章读完后，你应该能够：

1. 说清这五个实体的职责与边界。
2. 理解一次请求是如何在这些实体之间流动的。
3. 理解系统为什么能够做到中断、恢复、重试和观测。

## 为什么领域模型比你想的更重要

很多项目的失败，不是因为模型不够强，而是因为状态模型太模糊。最常见的错误有两种。

第一种错误是把所有内容都塞进“消息历史”。这样做短期最简单，但很快就会遇到问题：

- 哪些消息是用户输入，哪些是工具结果？
- 哪些是临时过程，哪些是最终事实？
- 一个任务失败后，如何定位原因？
- 记忆是从历史里提炼出来的，还是和历史混在一起的？

第二种错误是完全无状态。这样做会让每次请求看似独立，实际上系统无法积累经验，也无法恢复中断状态。

MicroClaw 的做法是把不同职责拆成不同实体，让每类状态有明确归属。

## 实体一：Chat

Chat 是最外层的对话容器。它通常对应一个真实聊天空间：

- Telegram 私聊或群组
- Discord 频道或线程
- Slack 频道或线程
- Web 中的一个 session key 对应聊天

Chat 的核心作用，是承载“这个上下文属于谁、来自哪里”。它不是推理状态本身，而是运行时的外部归属。

在数据库层面，Chat 至少承担这些职责：

- 绑定渠道与外部 chat 标识
- 提供人类可识别的标题或 session key
- 作为 Message、Session、Task、Memory 的归属锚点

如果没有 Chat 这个容器，系统就只能围绕“消息文本”工作，而无法处理跨渠道和长期运行问题。

## 实体二：Session

Session 是 Agent Loop 的恢复对象。很多读者第一次看这个概念，会误以为它和 Chat 是同一个东西。它们并不是。

Chat 解决的是“对话属于哪个外部空间”，Session 解决的是“当前推理状态是什么”。

Session 一般包含这些东西：

- 当前消息上下文
- 最近的工具交互
- 压缩后的历史摘要
- 当前运行所需的中间状态

为什么要把 Session 单独拿出来？因为聊天历史和推理状态并不总是一一对应。

例如：

- 聊天里可能有很多无关闲聊，但 Session 里只保留当前任务相关内容。
- 历史消息可能很长，但 Session 可以经过 compaction。
- 一个 Chat 可以在重置后继续存在，而 Session 可以被清空并重建。

这就是为什么 `/clear`、`/reset`、session fork 这些功能在设计上成立。系统不是只删“消息”，而是在操作更清晰的对象。

### 小例子：为什么 Session 不该直接等于 Chat？

很多系统一开始会把“聊天容器”和“推理状态”揉成一个对象，短期省事，长期就会分不清哪些字段是平台归属，哪些字段是运行时状态。MicroClaw 把这两层分开，正是为了避免这种混淆。

Rust 版本让 `Chat` 只负责外部归属，让 `SessionState` 只负责当前推理状态。这样哪怕同一个 chat 被清空、压缩或 fork，外层归属也不会跟着混乱。

```rust
#[derive(Debug, Clone)]
struct Chat {
    id: i64,
    channel: String,
}

#[derive(Debug, Clone)]
struct SessionState {
    chat_id: i64,
    summary: String,
    pending_tool_call: Option<String>,
}
```

Python 版本用 `@dataclass` 保持同样的边界。这样读者会更直观地看到，Session 不是“另一个名字的 Chat”，而是另一种生命周期对象。

```python
from dataclasses import dataclass


@dataclass
class Chat:
    id: int
    channel: str


@dataclass
class SessionState:
    chat_id: int
    summary: str
    pending_tool_call: str | None
```

## 实体三：Message

Message 是最直观的实体，但也是最容易被误用的实体。

在 MicroClaw 里，Message 至少承担四种角色：

1. 用户输入
2. 助手输出
3. 工具执行结果的持久化痕迹
4. 系统事件的可见表示

它的价值在于提供审计性和可回放性。用户说了什么，系统回复了什么，某个时刻发生了什么，这些都必须落在 Message 这一层。

但是 Message 不等于全部状态。它不应该独占以下职责：

- 记忆事实库
- 调度任务定义
- 可恢复 session 状态

否则系统会变成“只有日志，没有模型”。

## 实体四：Task

Task 表示被调度、被追踪、可重试的工作单元。它包括但不限于：

- cron 任务
- 一次性调度任务
- 后台反射器相关工作
- 某些异步控制面动作

Task 的存在让系统不再被限制在“收到消息才运行”。一旦任务被纳入领域模型，MicroClaw 就可以处理这些场景：

- 定时提醒
- 定时抓取
- 后台巡检
- 延时执行

这使得 MicroClaw 从“交互式助手”进一步接近“持续运行系统”。

## 实体五：Memory

Memory 是本书中最重要、也最容易被泛化的概念之一。MicroClaw 选择把它拆成两层：

### 文件记忆

使用 `AGENTS.md` 作为可读、可编辑、可解释的记忆层，分为：

- global
- bot/account
- chat

这种记忆适合保存长期偏好、行为约束、项目说明等“人也愿意直接编辑”的内容。

### 结构化记忆

存放在 `memories` 表中，带有：

- category
- confidence
- source
- archived
- last_seen

这种记忆适合机器处理：检索、去重、归档、观测。

双层设计的关键意义在于：可读性和可操作性并不总是同一个维度。文件记忆更容易被人维护，结构化记忆更容易被系统使用。把两者混成一种模型，通常两头都做不好。

## 五个实体之间的关系

如果把整个系统想成一张图，最重要的关系可以概括为：

- Chat 拥有很多 Message
- Chat 关联一个可恢复的 Session
- Chat 关联很多 Task
- Chat 关联文件记忆与结构化 Memory
- Session 以 Message 历史和 Memory 注入为输入，驱动一次或多次 Agent Loop

其中最值得注意的是：Session 不是 Message 的简单集合，而是对历史与记忆的一次可恢复投影。

这个设计避免了两个极端：

- 极端一：全靠消息历史回放
- 极端二：把运行态藏在内存里，进程一挂就丢

MicroClaw 选择的是中间路线：状态可恢复，但恢复对象是经过抽象的 Session。

## 一次请求的数据流

下面我们用一次普通请求来走完数据流。

### 第一步：用户消息进入 Chat

用户消息首先被写入 Message 存储，并归属到某个 Chat。

这一阶段解决的问题是：

- 记录事实
- 建立归属
- 为后续恢复和审计提供基础

### 第二步：系统恢复 Session

如果当前 Chat 已经有可恢复 Session，系统就恢复它；如果没有，就根据历史消息重建。

这里体现了一个重要原则：Session 可以被推导，也可以被持久化恢复。

### 第三步：系统装载 Memory

此时系统读取：

- `AGENTS.md`
- 结构化记忆
- 技能说明

然后把这些内容与 Session 组合成模型可理解的上下文。

### 第四步：Agent Loop 运行

模型根据当前上下文决定：

- 直接回答
- 调用工具
- 多轮继续

如果发生工具调用，工具结果会回灌给 Session，而相关痕迹也可能转成 Message 或 usage 记录。

### 第五步：结果持久化

当一轮请求结束时，系统会更新：

- Message
- Session
- usage
- 可能的 Memory
- 可能的 Task 状态

这一步是系统可恢复性的关键。如果没有这一层持久化，前面的运行就只是短暂内存行为。

## stop_reason、失败路径与恢复

一个健康的领域模型，不只描述成功路径，还必须描述失败路径。

在 Agent Runtime 里，至少要区分以下几种结束状态：

- 正常结束
- 工具错误
- 用户中断
- 超时
- 预算超限
- 需要压缩或恢复

这些状态为什么重要？因为它们决定了系统后续能做什么。

例如：

- 如果是用户中断，系统应保留已有历史，但终止当前运行。
- 如果是预算超限，系统应记录状态并允许后续调整后继续。
- 如果是工具失败，系统可能要继续重试，或把失败摘要反馈给用户。

这说明 Session 和 Task 不只是“有没有”，还必须有生命周期语义。

## 幂等与重试：为什么要在模型外建立规则

Agent 系统天然容易发生重试：

- 网络抖动
- 渠道重投
- WebSocket 重连
- webhook 重复调用

如果领域模型里没有幂等意识，系统就会出现重复消息、重复工具调用、重复任务副作用。

MicroClaw 的设计启示是：幂等不应该交给模型记忆，而应该交给运行时规则来控制。Message、Session、Task 的持久化存在，正是实现这类规则的基础。

换句话说，领域模型不仅是数据结构，也是抗故障策略的基座。

## 为什么双层记忆值得单独强调

在五个实体里，Memory 最容易被误解成“只是 RAG 的一部分”。这其实低估了它的作用。

在 MicroClaw 里，Memory 更像“系统长期事实层”。它承接的是那些：

- 不适合每轮都由用户重述
- 又不能完全依赖模型短期上下文记住
- 并且需要被观测和修正

的知识。

文件记忆负责可解释性，结构化记忆负责可计算性。这个分工，是 MicroClaw 在领域建模上非常成熟的一步，因为它承认了一个现实：可读和可检索是两种不同需求。

## 示例代码：为什么 Chat、Session、Message、Task、Memory 不能混成一个大对象？

一旦把这几个对象的职责拆清楚，后面的持久化、恢复和重试规则都会立刻简单很多。反过来，如果它们被折叠成一个“万能会话对象”，失败路径和生命周期就会很快失控。

Rust 版本先把领域对象定义成稳定 `struct`，再用一个异步 projector 负责从事实重建会话。这样“对象是什么”和“对象怎样被恢复”是两套规则，不会被揉在同一层里。

```rust
#[derive(Debug, Clone)]
struct Chat {
    id: i64,
    channel: String,
}

#[derive(Debug, Clone)]
struct Message {
    role: String,
    content: String,
}

#[derive(Debug, Clone)]
struct Session {
    chat: Chat,
    transcript: Vec<Message>,
}
```

这里先只放最基本的三层对象：聊天、消息、会话。先看懂“会话由什么组成”，再看任务和记忆，领域模型就不会被误读成一张大而平的字段表。

```rust

#[derive(Debug, Clone)]
struct Task {
    id: i64,
    prompt: String,
    status: String,
}
```

这里先把“会话之外还要长期存在什么状态”单独拿出来。`Task` 的意义不在字段多少，而在于它让系统开始具备会话外执行能力。

```rust

#[derive(Debug, Clone)]
struct Memory {
    chat_id: Option<i64>,
    content: String,
    category: String,
}

#[async_trait::async_trait]
trait SessionProjector {
    async fn rebuild(&self, chat: Chat, history: Vec<Message>, memories: Vec<Memory>) -> Session;
}
```

Python 版本保留相同的对象边界，但用 `@dataclass` 显式表达状态载体，再用 `Protocol` 把恢复行为抽出来。这样读者能直接看出：领域模型不是 ORM 字段表，而是运行时语义的最小集合。

```python
from dataclasses import dataclass, field
from typing import Protocol


@dataclass
class Chat:
    id: int
    channel: str


@dataclass
class Message:
    role: str
    content: str


@dataclass
class Session:
    chat: Chat
    transcript: list[Message] = field(default_factory=list)
```

这一段也先把会话骨架单独放稳，再引入任务、记忆和 projector。这样读者会先建立最关键的对象关系，后面的恢复逻辑才有落脚点。

```python


@dataclass
class Task:
    id: int
    prompt: str
    status: str
```

Python 这里也先把任务对象单独立住，再补记忆和 projector。这样读者会先看到运行时为什么不只有聊天状态，然后再理解恢复逻辑依赖哪些额外实体。

```python


@dataclass
class Memory:
    chat_id: int | None
    content: str
    category: str


class SessionProjector(Protocol):
    async def rebuild(self, chat: Chat, history: list[Message], memories: list[Memory]) -> Session: ...
```

## 容易走错的地方

### 失败模式 1：把 Chat 和 Session 混为一谈

这样会导致你无法解释为什么“聊天还在，但上下文已经重置”，也无法解释 session fork、compaction、resume 等行为。

### 失败模式 2：把 Message 当作全部事实来源

一旦所有东西都靠消息日志表达，系统会很难做结构化恢复、memory 生命周期管理和任务控制。

### 失败模式 3：把 Memory 做成单层

只有文件记忆，检索和自动化会很弱；只有结构化记忆，可读性和手工修正会很差。双层虽然复杂，但在长期系统里更稳。

### 失败模式 4：没有失败路径模型

如果系统只定义“成功时怎么走”，没有中断、超时、预算超限、重试等状态语义，运行时很快会失去一致性。

## 小结

MicroClaw 的领域模型之所以成立，不是因为实体很多，而是因为每个实体都承担了不同职责：

- Chat 负责归属
- Session 负责恢复
- Message 负责审计
- Task 负责持续执行
- Memory 负责长期事实

当这五个实体被分清楚之后，系统的数据流、恢复机制和失败处理逻辑就会变得可解释。后续章节无论讲工具、记忆还是调度，都会回到这里。

# Chapter 3 领域模型与数据流

## 系统里的核心对象是什么？

系统全景解决了"模块怎么分层"，本章回答更底层的问题：运行时里的六个关键实体是什么，它们之间如何协作？

MicroClaw 有六个关键实体：Chat、Session、Message、Task、Memory、SubagentRun。

```
                        Chat
                     (外部归属)
                    ┌────┼────┐
                    │    │    │
                    ▼    ▼    ▼
              Message  Task  Memory
              (审计)  (调度) (长期事实)

         Chat ──── Session (可恢复推理状态)
                    │
                    ▼
              Agent Loop ──► SubagentRun
                             (委托执行)
                              │
                         parent_run_id
                         (树形嵌套)
```

## 实体一：Chat

Chat 是最外层的对话容器，对应真实聊天空间（Telegram 私聊/群组、Discord 频道、Slack 线程、Web session、ACP session 等）。

核心作用：承载"这个上下文属于谁、来自哪里"。它是 Message、Session、Task、Memory、SubagentRun 的归属锚点。16 个渠道加上 ACP 和 A2A，每个都有自己的外部标识格式，Chat 把它们统一。

## 实体二：Session

Session 是 Agent Loop 的恢复对象。Chat 解决"对话属于哪个外部空间"，Session 解决"当前推理状态是什么"。

包含：当前消息上下文、压缩历史摘要、session label、thinking_level、verbose_level、reasoning_level。支持 fork、tree view、compaction、reset——这些操作正是因为 Session 和 Chat 是独立实体才成立。

```rust
#[derive(Debug, Clone)]
struct Chat {
    id: i64,
    channel: String,
    chat_type: String,
}

#[derive(Debug, Clone)]
struct SessionState {
    chat_id: i64,
    summary: String,
    settings: SessionSettings,
    pending_tool_call: Option<String>,
}
```

```python
from dataclasses import dataclass


@dataclass
class Chat:
    id: int
    channel: str
    chat_type: str


@dataclass
class SessionState:
    chat_id: int
    summary: str
    settings: SessionSettings
    pending_tool_call: str | None = None
```

## 实体三：Message

每条 `StoredMessage` 包含 id(UUID)、chat_id、sender_name、content、is_from_bot、timestamp。承担四种角色：用户输入痕迹、助手输出痕迹、工具结果审计记录、系统事件表示。

Message 不等于全部状态：记忆事实库是 Memory 的事，调度定义是 Task 的事，subagent 执行记录是 SubagentRun 的事，可恢复状态是 Session 的事。

## 实体四：Task

Task 表示被调度、被追踪、可重试的工作单元。`scheduled_tasks` 表包含 id、chat_id、prompt、schedule_type（cron/once）、schedule_value、timezone、next_run、last_run、status（active/paused/completed/cancelled）。

配套：`task_run_logs`（执行历史）、`scheduled_task_dlq`（死信队列，支持 replay）。

MicroClaw 提供完整调度工具族：`schedule_task`、`list_scheduled_tasks`、`pause/resume/cancel_scheduled_task`、`get_scheduled_task_history`、`list/replay_scheduled_task_dlq`。Agent 自己就能管理定时任务。

## 实体五：Memory

双层设计：

| 层 | 载体 | 职责 | 操作工具 |
|----|------|------|---------|
| 文件记忆 | AGENTS.md（global/bot/chat）+ SOUL.md（人格） | 可读、可编辑、可解释 | `read_memory`、`write_memory` |
| 结构化记忆 | `memories` 表（category/confidence/source/embedding/archived） | 可检索、可归档、可自动提取 | `structured_memory_search/delete/update` |

配套观测：`memory_reflector_runs`（提取统计）、`memory_injection_logs`（注入统计）、`memory_supersede_edges`（替代关系）、`MemoryObservabilitySummary`（聚合仪表盘）。

## 实体六：SubagentRun

SubagentRun 让 MicroClaw 从"单 Agent 循环"进化到"session-native subagent 体系"。

`subagent_runs` 表字段：run_id(UUID)、parent_run_id（嵌套）、depth、chat_id、task、status（queued/running/completed/failed/timeout/cancelled）、token_budget、provider/model、input/output/total_tokens、error_text/result_text、artifact_json、cancel_requested。

SubagentRun 必须是独立实体的四个理由：

1. **状态机**：queued → running → completed/failed/timeout/cancelled，不是 Message 的"发送/接收"语义
2. **资源约束**：token_budget 和 depth 是 SubagentRun 的属性
3. **嵌套关系**：parent_run_id 追踪父子关系，orchestrate 可同时 spawn 多个
4. **独立 announce**：完成后通过 `subagent_announces` 表管理回调通知，支持重试

```rust
#[derive(Debug, Clone)]
enum SubagentStatus {
    Queued, Running, Completed, Failed, Timeout, Cancelled,
}

#[derive(Debug, Clone)]
struct SubagentRun {
    run_id: String,
    parent_run_id: Option<String>,
    depth: i64,
    chat_id: i64,
    task: String,
    status: SubagentStatus,
    token_budget: i64,
    tokens_used: i64,
}
```

```python
from dataclasses import dataclass
from enum import Enum


class SubagentStatus(Enum):
    QUEUED = "queued"
    RUNNING = "running"
    COMPLETED = "completed"
    FAILED = "failed"
    TIMEOUT = "timeout"
    CANCELLED = "cancelled"


@dataclass
class SubagentRun:
    run_id: str
    parent_run_id: str | None
    depth: int
    chat_id: int
    task: str
    status: SubagentStatus
    token_budget: int
    tokens_used: int
```

## 一次涉及 subagent 的数据流

```
用户消息 ──► Message 存储 ──► Session 恢复
                                │
                    Memory/SOUL 装载 ──► Agent Loop
                                          │
                         ┌────────────────┼────────────────┐
                    直接回答        调用工具(wave)      spawn subagent
                         │               │                 │
                         ▼               ▼                 ▼
                      Egress     工具结果回灌        SubagentRun(queued)
                                    │                      │
                                    ▼                 restricted tools
                              继续循环               独立 token 预算
                                                          │
                                                     完成/失败
                                                          │
                                                     announce → Chat
                                                          │
                                                     持久化全部状态
```

## stop_reason 与失败路径

| 结束状态 | 后续行为 |
|---------|---------|
| end_turn | 正常持久化并发回 |
| 工具错误 | 错误信息回灌模型，继续循环或结束 |
| 用户中断（/stop, Web abort, ACP cancel） | cancelled 标记 → AgentEvent::Cancelled → 终止 |
| 超时 | tool timeout 或 compaction timeout |
| 预算超限 | max_tool_iterations 或 subagent token_budget |
| hook 阻止 | 拒绝理由告知用户 |
| subagent 失败 | 主 Agent 收到错误，可重试/报告/放弃 |

## 幂等与重试

Agent 系统天然容易发生重试。MicroClaw 的幂等设计：ChatTurnQueue（turn lock 防并发 run）、run_control（`ABORTED_SOURCE_MESSAGE_IDS` 防重复处理）、SubagentRun（UUID run_id + announce 状态追踪）、scheduled_task_dlq（`replayed_at` 防重复 replay）。

## 关键权衡

| 决策 | 收益 | 代价 |
|------|------|------|
| 六个独立实体而非一个大对象 | 职责边界清晰 | 跨表 join |
| Session 和 Chat 分离 | 支持 fork/tree/compaction/reset | 多一层抽象 |
| SubagentRun 独立实体 | 独立状态机、资源约束、嵌套、announce | schema 复杂（4 张表） |
| 双层记忆 | 人类运维和机器检索各有接口 | 需持续解释两层边界 |
| 失败路径纳入领域模型 | 恢复和重试有明确状态基础 | 每个实体状态空间更大 |

## 容易走错的地方

### 失败模式 1：把 Chat 和 Session 混为一谈
无法解释"聊天还在，但上下文已经重置"，也无法解释 session fork、compaction、resume。

### 失败模式 2：把 Message 当作全部事实来源
无法做结构化恢复、memory 生命周期管理、subagent 状态追踪。

### 失败模式 3：把 Memory 做成单层
只有文件记忆 → 检索弱；只有结构化记忆 → 可读性差。

### 失败模式 4：把 SubagentRun 当成"特殊的 Message"
状态机、资源约束、announce 机制、嵌套关系全部丢失。

### 失败模式 5：没有失败路径模型
只定义成功路径，运行时很快失去一致性。

## 小结

六个实体各司其职：Chat（归属）、Session（恢复）、Message（审计）、Task（持续执行）、Memory（长期事实，双层）、SubagentRun（委托执行，含嵌套和 announce）。分清这六个实体后，数据流、恢复机制和失败处理才可解释。

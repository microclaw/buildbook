# Chapter 9 多渠道架构

## 这一章要回答什么问题

多渠道支持看起来像“多接几个 SDK”，但真正难的是：同一个 runtime 能否在不同渠道上保持大体一致的行为，同时保留各个平台必要的差异处理。MicroClaw 的多渠道设计有一个很明确的方向：渠道是入口，不是内核。

这一章读完后，你应该已经能回答：

1. Channel Adapter 的抽象边界是什么。
2. 为什么渠道差异必须被约束在 ingress/egress 层。
3. 多账号、多模型和权限差异如何在统一 runtime 中共存。
4. 新渠道接入时，哪些行为必须回归验证。

## `ChannelAdapter`：渠道抽象要足够薄

`crates/microclaw-channels/src/channel_adapter.rs` 定义的 `ChannelAdapter` trait 很克制。一个适配器主要回答这些问题：

- 渠道名是什么。
- 它负责哪些 `chat_type`。
- 这些 `chat_type` 属于私聊还是群聊。
- 是否支持真正外发消息。
- 是否允许跨 chat 操作。
- 如何发送文本与附件。

这个接口的价值在于，它把渠道抽象收敛到“消息进出”和“会话路由”上，而不是把 Agent 行为逻辑塞进去。

这正是多渠道系统稳定的关键。只要统一循环、工具、记忆、调度都不下沉到适配器里，不同平台之间的行为就更容易维持一致。

## `ChannelRegistry`：统一路由如何建立

适配器真正被系统使用时，会进入 `ChannelRegistry`。这个注册器维护了两类映射：

- `chat_type -> channel_name`
- `chat_type -> ConversationKind`

配合 `crates/microclaw-channels/src/channel.rs`，系统可以根据数据库中的 `chat_type` 和 `chat_channel` 字段恢复出一条消息究竟属于哪个渠道、对应私聊还是群聊、是否允许跨 chat 行为。

这一步的重要性，往往被低估。因为一旦状态库里没有统一路由语义，后面所有这些能力都会变得困难：

- `send_message` 把结果发回正确渠道
- Scheduler 在后台任务中找到正确回复出口
- Web 面板查询历史时知道消息来源
- 子代理和导出工具理解会话归属

多渠道架构之所以能成立，前提不是“支持很多适配器”，而是“所有会话都能被统一路由”。

### 小例子：为什么渠道事件要先变成统一 `InboundMessage`？

多渠道运行时真正需要的不是十几种平台事件，而是一种可以稳定进入内核的统一消息形状。只有先做这一步，后面的会话恢复、权限和消息落库才可能保持一致。

Rust 版本把平台事件压成 `InboundMessage`，其中路由字段和内容字段分开保存。这样适配器负责翻译，内核负责处理。

```rust
struct RouteKey {
    channel: String,
    external_chat_id: String,
}

struct InboundMessage {
    route: RouteKey,
    sender_id: String,
    text: String,
}
```

Python 版本用两个 `@dataclass` 表示同样的边界。这样读者很容易看出，多渠道支持首先是数据规范化问题，而不是平台 SDK 技巧。

```python
from dataclasses import dataclass


@dataclass
class RouteKey:
    channel: str
    external_chat_id: str


@dataclass
class InboundMessage:
    route: RouteKey
    sender_id: str
    text: str
```

## `runtime.rs` 如何装配多渠道

当前 `runtime.rs` 已经支持相当多的渠道 runtime context，包括：

- Telegram
- Discord
- Slack
- Feishu
- Matrix
- WhatsApp
- iMessage
- Email
- Nostr
- Signal
- DingTalk
- QQ
- IRC
- Web

装配时的模式也很一致：

1. 读取 config，判断该渠道是否启用。
2. 根据配置构造一个或多个 runtime context。
3. 向 `ChannelRegistry` 注册适配器。
4. 必要时记录该渠道的 model override。
5. 为每个 runtime context 启动独立任务。

这种设计有两个明显收益。

第一，它允许一个渠道配置多个账号或多个 runtime 实例，而不破坏统一抽象。

第二，它把“渠道启动失败不影响其他渠道”作为显式目标。`runtime.rs` 里的 `spawn_guarded` 会捕获 panic，使单个渠道任务失败后，其它任务仍可继续运行。

这正是长期运行系统必须有的弹性。

## 统一行为来自哪里

如果多渠道系统想保持一致，真正的秘诀不是在每个适配器里写更多业务逻辑，而是让一致行为来自共享内核。

在 MicroClaw 中，一次消息最终都会被收敛到：

- 同一个 `process_with_agent`
- 同一套工具系统
- 同一套记忆注入
- 同一套 session 恢复
- 同一套存储层

因此，多渠道一致性的来源并不是“所有适配器写得一样”，而是“绝大部分行为根本不在适配器里”。

这也是为什么新渠道接入时，真正需要写的主要是：

- 如何接收原始事件
- 如何映射为 `chat_id`、`chat_type`、发送者、文本/附件
- 如何把最终回复发回去

核心循环本身通常不需要重写。

## 渠道差异必须保留在哪些地方

渠道抽象再统一，也不可能消除所有差异。关键是差异要被关在边界层。

### 长度限制与消息拆分

README 和 `DEVELOP.md` 都给出了各渠道不同的消息长度上限，例如 Telegram 4096、Discord 2000、Slack 4000。消息拆分因此属于 egress 责任，而不应让统一循环直接感知具体平台上限。

### 群聊语义与 mention 策略

群聊中 bot 何时响应、读取多少上下文、如何识别 mention，是典型的平台相关行为。MicroClaw 当前一个很典型的实现细节是 Telegram 群组 mention catch-up：被提及后，会读取自上次 bot 回复以来的消息，而不是固定最近 N 条。

### 本地渠道和远程渠道的差异

Web 适配器在 `web.rs` 中被声明为 `is_local_only = true`、`allows_cross_chat = false`。这说明 Web 作为本地控制面，有意采用和远程聊天渠道不同的权限姿态。

### 附件能力与外发能力

`ChannelAdapter` 默认不支持附件发送，具体平台再按能力覆盖。这种默认值很合理，因为不是每个平台都支持同样的多媒体交互方式。

## 多账号、多模型与租户隔离

多渠道系统一旦进入生产环境，通常不会停留在“一个 bot token 对一个平台”。MicroClaw 当前至少已经体现了两个关键方向。

### 渠道级模型覆盖

`runtime.rs` 会从 runtime context 中收集 channel 对应的 model override，并写入 `llm_model_overrides`。这意味着：

- 同一个内核可以被不同渠道绑定不同默认模型。
- 渠道差异不必侵入核心循环。

### 控制 chat 与普通 chat 权限边界

在工具系统中，某些能力只允许 control chat 调用，例如跨 chat 操作或全局记忆写入。多渠道支持越多，这个边界越重要。因为渠道扩展会放大攻击面和误操作面，必须依靠统一权限模型兜底，而不是依赖某个平台本身的权限概念。

## 新渠道接入的最小改动面

如果你要为 MicroClaw 增加一个新渠道，最佳实践不是“复制一个现有实现然后乱改”，而是遵循一张很清楚的接入清单。

1. 定义该渠道的 runtime context。
2. 实现 `ChannelAdapter`，明确 `chat_type_routes` 和消息发送能力。
3. 把平台事件映射为统一会话字段。
4. 在 `runtime.rs` 中注册并启动。
5. 验证消息入库、会话恢复、工具调用、最终回复和异常恢复。

这份清单之所以有效，是因为 MicroClaw 已经把渠道之外的复杂性都收敛到了共享内核。你要做的，主要是做好映射，而不是重造 Agent。

## 回归测试要测什么

多渠道系统最容易出的问题，不是启动失败，而是“看起来能跑，但行为悄悄变了”。新增渠道或修改现有适配器时，至少应该验证以下几类回归：

- 私聊与群聊的响应条件是否正确。
- 发送长消息时拆分是否正常。
- 消息和 bot 回复是否正确入库。
- Scheduler/`send_message` 是否能找到正确路由。
- 权限边界是否因渠道差异被绕过。

如果这些测试缺失，多渠道支持就很容易从“统一 runtime”退化成“一堆互不完全一致的 bot”。

## 示例代码：为什么渠道适配器必须保持薄，而不能自己承载 Agent 逻辑？

这一层代码的目标不是承载 Agent 推理，而是把平台差异翻译成统一 runtime 看得懂的事件和路由。只要适配器开始私自决定业务逻辑，多渠道复用就会很快失效。

Rust 版本把发送能力抽成 trait，再让具体 adapter 只保留平台字段和异步发送动作。这样渠道层的职责被压缩在“翻译”和“投递”两件事上，而不是偷偷长成第二个执行内核。

```rust
#[derive(Debug, Clone)]
struct OutboundMessage {
    external_chat_id: String,
    text: String,
}

#[async_trait::async_trait]
trait ChannelAdapter {
    fn name(&self) -> &str;
    async fn send(&self, message: OutboundMessage) -> anyhow::Result<()>;
}

struct TelegramAdapter {
    bot_token: String,
}
```

先把消息载体和适配器契约定住，读者才能清楚看到渠道层真正稳定的是什么。这样后面的 Telegram 实现只是在履行接口，而不是自带一套新的业务流程。

```rust

#[async_trait::async_trait]
impl ChannelAdapter for TelegramAdapter {
    fn name(&self) -> &str {
        "telegram"
    }

    async fn send(&self, message: OutboundMessage) -> anyhow::Result<()> {
        println!("{} -> {}: {}", self.bot_token, message.external_chat_id, message.text);
        Ok(())
    }
}
```

Python 版本同样把边界收紧到 `Protocol + dataclass + async` 这三层。这样动态语言里的示例也能明确表达：渠道适配器负责把消息送出去，不负责决定消息为什么产生。

```python
from dataclasses import dataclass
from typing import Protocol


@dataclass
class OutboundMessage:
    external_chat_id: str
    text: str


class ChannelAdapter(Protocol):
    def name(self) -> str: ...
    async def send(self, message: OutboundMessage) -> None: ...


@dataclass
class TelegramAdapter:
    bot_token: str
```

Python 也按同样顺序拆开：先定义发送对象和渠道接口，再给出具体适配器实现。这样版面更紧凑，论点也更集中。

```python

    def name(self) -> str:
        return "telegram"

    async def send(self, message: OutboundMessage) -> None:
        print(f"{self.bot_token} -> {message.external_chat_id}: {message.text}")
```

## 关键权衡

### 决策一：渠道适配器保持薄层

优点是行为大多来自共享内核。代价是某些平台特性不能无损映射，必须接受抽象损耗。

### 决策二：统一路由信息进入数据库

优点是后台任务、导出、子代理等共享能力都能复用。代价是迁移旧数据和兼容历史 `chat_type` 会增加一点复杂度。

### 决策三：支持渠道级模型覆盖

优点是更灵活。代价是运维人员需要理解“同一 runtime 不同入口可能使用不同模型”的事实。

### 决策四：Web 采用本地控制面姿态

优点是安全边界更清晰。代价是 Web 渠道不能像其它外部渠道一样天然支持跨 chat 操作。

## 容易走错的地方

### 失败模式 1：在适配器里复制核心逻辑

这样做短期快，长期一定会造成各渠道行为分叉。

### 失败模式 2：没有统一路由模型

如果状态库里不能稳定映射 chat 到渠道，调度、发消息和导出都会持续出错。

### 失败模式 3：只测 Happy Path，不测群聊、长消息和权限边界

多渠道 bug 大多出现在这些边角语义上，而不是主链路本身。

## 小结

MicroClaw 的多渠道设计并不追求“把所有平台都做成一样”，而是追求“让大部分行为来自共享内核，只把真正不可避免的差异留在边界层”。这正是多渠道 runtime 能持续演进的前提。

下一章我们把视角转向后台执行能力。因为一旦系统支持多渠道和长期会话，用户很快就会希望它不仅响应消息，还能主动在未来某个时间点继续工作。

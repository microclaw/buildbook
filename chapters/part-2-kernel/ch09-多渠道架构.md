# Chapter 9 多渠道架构

## 这一章要回答什么问题

同一个 runtime 如何在 16 种渠道上保持一致行为，同时保留各平台必要的差异处理？

## 16 个渠道的实际面貌

| 分类 | 渠道 | 传输方式 | 特殊说明 |
| --- | --- | --- | --- |
| 完整实现 | Telegram | teloxide 0.17 | 多 bot token 支持 |
| 完整实现 | Discord | serenity 0.12 | embed 消息 |
| 完整实现 | Slack | Socket Mode | workspace 级别 |
| 完整实现 | Feishu/Lark | WebSocket + webhook | ACK reaction |
| 完整实现 | IRC | 标准 IRC 协议 | 380 字符限制 |
| 完整实现 | Web | 嵌入式 React+Vite | 本地控制面 |
| 完整实现 | Email | IMAP polling + sendmail | 异步投递模型 |
| Feature-gated | Matrix | matrix-sdk | E2E 加密，需 `channel-matrix` feature |
| Stub | iMessage, Signal, Weixin, WhatsApp, DingTalk, Nostr, QQ | 各异 | 契约已定义，底层实现成熟度不同 |

先让 adapter 抽象和路由体系稳定，各渠道按自己的节奏填充底层实现。

## `ChannelAdapter`：渠道抽象要足够薄

适配器只需回答：渠道名、负责哪些 `chat_type`、是否本地渠道、是否允许跨 chat、如何发送文本和附件。职责被压缩在"翻译"和"投递"两件事上。

## `ChannelRegistry`：统一路由

注册器维护三类核心映射：

- `adapters`: `channel_name -> Arc<dyn ChannelAdapter>`
- `type_to_channel`: `chat_type -> channel_name`（例如 `"slack_dm" -> "slack"`）
- `type_to_conversation`: `chat_type -> ConversationKind`（私聊或群聊）

统一路由使以下能力成立：`send_message` 回投正确渠道、Scheduler 找到回复出口、Web 面板查询消息来源、子代理 announce 回传。

## `runtime.rs` 装配流程

```
config.channel_enabled(key) → prepare_channel_runtimes(build, register, model_override)
  → spawn_channel_runtimes / spawn_guarded
```

- `prepare_channel_runtimes` 接受三个闭包（build、register、model_override），新增渠道只需提供闭包。
- `spawn_guarded` 用 `catch_unwind` 捕获 panic，单渠道失败不影响其他渠道。
- 同一渠道支持多实例（多 bot token / 多账号）。

## 各渠道的关键差异处理

差异要被关在边界层，不泄漏到内核。

### Feishu/Lark：WebSocket + ACK 反应

连接使用 `tokio-tungstenite`，收到消息后立即发送 emoji reaction 作为 ACK——解决用户等待几十秒误以为 bot 挂了的体验问题。Feishu 还有 `system_prompt_extension`，精确限定在渠道边界层。

### Matrix：feature flag + E2E 加密

`matrix-sdk` 引入大量加密依赖，通过 `channel-matrix` feature flag 隔离编译开销：

```rust
#[cfg(not(feature = "channel-matrix"))]
warn_missing_feature(&config, "matrix", "channel-matrix");
```

### Email：异步投递

IMAP polling 收取 + sendmail 发送。无"已读"回执、无"正在输入"。`ChannelAdapter` 的异步接口天然适配。

### Stub 渠道：先定义契约后填充实现

iMessage、Signal 等已有完整 adapter 和消息投递语义定义，底层协议完整度不同。先注册让路由体系提前验证，避免后期 big-bang。

### 消息拆分

各渠道长度上限不同（Telegram 4096、Discord 2000、IRC 380 等）。拆分用 `microclaw_core::text::split_text`，属 egress 责任。

## 渠道级模型覆盖

同一内核可被不同渠道绑定不同默认模型（Telegram 用 Claude、Discord 用 GPT-4、Feishu 用国产模型），模型选择在 `AgentRequestContext` 层完成，不侵入核心循环。

## 统一行为的来源

```
平台事件 → ChannelAdapter 翻译 → InboundMessage
  → ChannelRegistry 路由 → ChatTurnQueue 串行化
    → process_with_agent（统一循环）
      → ChannelAdapter 外发回复
```

一致性来自共享内核：同一套 `process_with_agent`、`ToolRegistry`、记忆注入、session 恢复、存储层、`ChatTurnQueue`。

## Per-Chat Turn 串行化

`ChatTurnQueue` 确保同一 `(channel, chat_id)` 任意时刻只有一个 agent run：

- 每 chat 一个 `turn_lock`（`Arc<Mutex<()>>`）
- 当前 run 期间新消息收集为 `PendingMessage`
- run 完成后 pending messages 合并为下一轮输入

消除用户连续发消息导致并行 run 踩踏 session 状态的竞态。

## 示例代码：薄适配器

```rust
#[derive(Debug, Clone)]
struct OutboundMessage {
    external_chat_id: String,
    text: String,
}

#[async_trait::async_trait]
trait ChannelAdapter {
    fn name(&self) -> &str;
    fn is_local_only(&self) -> bool { false }
    fn allows_cross_chat(&self) -> bool { true }
    async fn send(&self, message: OutboundMessage) -> anyhow::Result<()>;
}

struct FeishuAdapter {
    channel_name: String,
    app_id: String,
    app_secret: String,
    domain: String,
}
```

```rust

#[async_trait::async_trait]
impl ChannelAdapter for FeishuAdapter {
    fn name(&self) -> &str {
        &self.channel_name
    }

    async fn send(&self, message: OutboundMessage) -> anyhow::Result<()> {
        println!("feishu/{} -> {}: {}", self.app_id, message.external_chat_id, message.text);
        Ok(())
    }
}
```

## 新渠道接入清单

1. `src/channels/` 下新增模块，定义 `RuntimeContext` 和 `Adapter`。
2. 实现 `ChannelAdapter`，明确 `chat_type_routes` 和消息发送。
3. 提供 `build_*_runtime_contexts` 函数。
4. `src/channels/mod.rs` 中 re-export。
5. `runtime.rs` 中调用 `prepare_channel_runtimes` + `spawn_channel_runtimes`。
6. 验证消息入库、会话恢复、工具调用、回复和异常恢复。

## 容易走错的地方

| 失败模式 | 后果 |
|---------|------|
| 在适配器里复制核心逻辑 | 渠道行为分叉，测试难发现，生产难修复 |
| 没有统一路由模型 | 调度、send_message、导出持续出错 |
| 忽略 per-chat 串行化 | 并行 run 踩踏 session state |
| 把 feature flag 渠道当默认可用 | 运行时只得到警告日志而非连接 |

## 关键权衡

| 决策 | 优点 | 代价 |
|------|------|------|
| 适配器保持薄层 | 行为来自共享内核，不 diverge | 部分平台特性无损映射受限 |
| Matrix feature flag | 不需要者免受编译影响 | CI 需 feature 矩阵防 bit-rot |
| Stub 先注册后完善 | 路由体系提前验证 | 用户可能误以为全部生产就绪 |
| Per-chat turn 串行化 | 消除并行竞态 | 单 chat 吞吐受限于串行 |
| 渠道级模型覆盖 | 不同渠道用不同最优模型 | 运维需理解同 runtime 多模型 |

# Chapter 9 多渠道架构

## 同一个 runtime 如何在 16 种渠道上保持一致行为？

多渠道支持看起来像"多接几个 SDK"，但真正难的从来不是适配器数量，而是：同一个 runtime 能否在 16 种渠道上保持大体一致的行为，同时保留各平台必要的差异处理。MicroClaw v0.1.38 已经将渠道阵列扩展到 Telegram、Discord、Slack、Feishu/Lark、IRC、Web、Email、Matrix、iMessage、Signal、Weixin、WhatsApp、DingTalk、Nostr、QQ——并且还保留了更多 stub。这不只是工程量的膨胀，它逼出了一系列必须回答的架构问题。

这一章读完后，你应该已经能回答：

1. `ChannelAdapter` 的抽象边界为什么必须保持薄。
2. 16 个渠道如何在 `ChannelRegistry` 中统一路由，而不让内核膨胀。
3. 哪些渠道差异（Feishu ACK、Matrix E2E、Email 异步）必须被关在边界层。
4. 渠道级模型覆盖、feature flag 和 stub channel 的设计权衡。
5. 新渠道接入时的最小改动面和必须回归的测试清单。

## 16 个渠道的实际面貌

从 `src/channels/mod.rs` 可以看到当前的渠道清单：

**完整实现的渠道**：Telegram（teloxide 0.17）、Discord（serenity 0.12）、Slack（Socket Mode）、Feishu/Lark（WebSocket 默认 + webhook）、IRC、Web（嵌入式 React+Vite）、Email（IMAP polling + sendmail）、Matrix（可选 feature `channel-matrix`，支持 E2E 加密）。

**Stub 渠道**：iMessage、Signal、Weixin、WhatsApp、DingTalk、Nostr、QQ。这些渠道已有完整的 adapter 注册、runtime context 构建和 `start_*_bot` 入口，但底层协议实现的成熟度各有不同。

这种分层非常重要。它说明 MicroClaw 不是要等所有渠道都成熟了再统一注册，而是先让 adapter 抽象和路由体系稳定，然后各渠道按自己的节奏填充底层实现。这正是多渠道系统能持续演进而不崩溃的前提。

## `ChannelAdapter`：渠道抽象要足够薄

`crates/microclaw-channels/src/channel_adapter.rs` 定义的 `ChannelAdapter` trait 非常克制。一个适配器只需要回答以下问题：

- 渠道名是什么（`name()`）。
- 它负责哪些 `chat_type`，以及每个 type 是私聊还是群聊（`chat_type_routes()`）。
- 是否为本地渠道（`is_local_only()`，Web 和 ACP 为 true）。
- 是否允许跨 chat 操作（`allows_cross_chat()`，Web 为 false）。
- 如何发送文本和附件。

这个 trait 的价值在于，它把渠道抽象收敛到"消息进出"和"会话路由"上，而不是把 Agent 行为逻辑塞进去。只要统一循环、工具、记忆、调度都不下沉到适配器里，16 个渠道之间的行为就更容易维持一致。

### 渠道事件为什么要先变成统一 `InboundMessage`？

多渠道运行时真正需要的不是十几种平台事件，而是一种可以稳定进入内核的统一消息形状。只有先做这一步，后面的会话恢复、权限和消息落库才可能保持一致。

Rust 版本把平台事件压成 `InboundMessage`，其中路由字段和内容字段分开保存。这样适配器负责翻译，内核负责处理。

```rust
struct RouteKey {
    channel: String,
    external_chat_id: String,
}

struct InboundMessage {
    route: RouteKey,
    sender_id: String,
    text: String,
}
```

Python 版本用两个 `@dataclass` 表示同样的边界。这样读者很容易看出，多渠道支持首先是数据规范化问题，而不是平台 SDK 技巧。

```python
from dataclasses import dataclass


@dataclass
class RouteKey:
    channel: str
    external_chat_id: str


@dataclass
class InboundMessage:
    route: RouteKey
    sender_id: str
    text: str
```

## `ChannelRegistry`：统一路由如何建立

适配器真正被系统使用时，会进入 `ChannelRegistry`。这个注册器维护了三类核心映射：

- `adapters`: `channel_name -> Arc<dyn ChannelAdapter>`
- `type_to_channel`: `chat_type -> channel_name`（例如 `"slack_dm" -> "slack"`）
- `type_to_conversation`: `chat_type -> ConversationKind`（私聊或群聊）

配合 `crates/microclaw-channels/src/channel.rs` 中的 `resolve()` 和 `resolve_routing()`，系统可以根据数据库中的 `chat_type` 字段恢复出一条消息究竟属于哪个渠道、是私聊还是群聊、是否允许跨 chat 行为。

这一步的重要性往往被低估。一旦状态库里没有统一路由语义，后面所有这些能力都会变得困难：

- `send_message` 把结果发回正确渠道
- Scheduler 在后台任务中找到正确回复出口
- Web 面板查询历史时知道消息来源
- 子代理的 announce 回传找到正确投递渠道

多渠道架构之所以能成立，前提不是"支持很多适配器"，而是"所有会话都能被统一路由"。

## `runtime.rs` 如何装配 16 个渠道

`src/runtime.rs` 是渠道装配的枢纽。v0.1.38 的装配流程已经非常成熟：

1. 读取 config，通过 `config.channel_enabled(channel_key)` 判断该渠道是否启用。
2. 调用 `prepare_channel_runtimes` 泛型函数，统一完成 build、register 和 model override 三步。
3. 必要时记录该渠道的 model override 到 `llm_model_overrides`。
4. 使用 `spawn_channel_runtimes` 或 `spawn_guarded` 为每个 runtime context 启动独立任务。

`prepare_channel_runtimes` 这个泛型函数是关键。它接受三个闭包——build、register、model_override——统一完成从配置到注册的全流程。新增渠道时，核心装配逻辑完全不需要变动，只需提供三个闭包。

`spawn_guarded` 会用 `AssertUnwindSafe + catch_unwind` 捕获 panic，使单个渠道任务失败后，其它任务仍可继续运行。这正是长期运行系统必须有的弹性——你不能因为 IRC 服务器断开就把 Telegram 也拖下线。

### 多实例支持

同一渠道可以配置多个账号或多个 runtime 实例。例如 Telegram 允许配置多个 bot token，每个 token 生成独立的 `(Bot, TelegramRuntimeContext)` 对。Discord、Slack、Feishu 等也都支持多实例。这种设计允许一个 runtime 同时服务多个 bot 身份。

## 各渠道的关键差异处理

渠道抽象再统一，也不可能消除所有差异。关键是差异要被关在边界层，而不是泄漏到内核。

### Feishu/Lark：WebSocket 默认与 ACK 反应

Feishu 渠道默认使用 WebSocket 连接飞书开放平台，这是 v0.1.38 的一个重要变化。`src/channels/feishu.rs` 显示，连接使用 `tokio-tungstenite`，并且在收到消息后立即发送 ACK 反应（通常是一个 emoji reaction），让用户知道 bot 已经收到消息并开始处理。

这个 ACK 机制解决了一个非常实际的体验问题：飞书消息从发出到 Agent 完成响应可能需要几十秒，用户在等待期间如果看不到任何反馈，会误以为 bot 挂了。ACK reaction 是一个非常轻量但有效的解决方案。

Feishu 还有一个 `system_prompt_extension` 特殊处理。`src/channels/mod.rs` 中的全局 `system_prompt_extension` 函数会检查 caller channel 是否为 feishu，如果是则注入额外的系统提示。这说明某些渠道确实需要特殊的 prompt 调整，但这种调整被精确限定在渠道边界层。

### Matrix：可选 feature 与 E2E 加密

Matrix 渠道是唯一一个使用 feature flag 的渠道——`channel-matrix`。这不是偶然的。Matrix 依赖 `matrix-sdk`，后者会引入大量加密和同步相关的依赖，对编译时间和二进制体积影响很大。通过 feature flag，不需要 Matrix 的用户可以完全跳过这些开销。

在 `runtime.rs` 中可以看到，当 feature 未启用但配置中开启了 matrix 时，系统会通过 `warn_missing_feature` 发出明确警告：

```rust
#[cfg(not(feature = "channel-matrix"))]
warn_missing_feature(&config, "matrix", "channel-matrix");
```

`src/channels/matrix.rs` 使用 `matrix_sdk` 并完整支持 E2E 加密。这意味着 Matrix 渠道中的消息在传输和存储层面都可以端到端加密，这是其他渠道目前不具备的安全特性。

### Email：异步投递模型

Email 渠道的特殊性在于它天然是异步的。`EmailAdapter` 通过 `from_address` 和 `sendmail_path` 配置，使用系统 sendmail 命令发送邮件。IMAP polling 则负责收取消息。

这意味着 Email 渠道的响应延迟模型和即时通讯渠道完全不同——发出消息后没有"已读"回执，也没有"正在输入"状态。适配器层面不需要做特殊处理，因为 `ChannelAdapter` 的 `send_text` 接口本身就是异步的。但运维层面需要理解这个差异。

### Stub 渠道：占位而不是空壳

iMessage、Signal、Weixin、WhatsApp、DingTalk、Nostr、QQ 这些渠道已经有完整的 adapter 实现和 runtime context 构建逻辑。以 `NostrAdapter` 为例，它使用 `publish_command` 来外发消息；`SignalAdapter` 使用 `send_command`。这说明 stub 渠道不是空壳，而是已经把消息投递语义定义清楚，只是底层协议对接的完整度不同。

这种"先定义契约，后填充实现"的方式比"等全部做好再注册"更务实。因为它让渠道注册表和路由体系能提前验证正确性，避免后期集中集成时的 big-bang 风险。

### 长度限制与消息拆分

各渠道有不同的消息长度上限，例如 Telegram 4096、Discord 2000、Slack 4000、IRC 380。消息拆分使用 `microclaw_core::text::split_text`，属于 egress 责任，而不是统一循环的关注点。`IrcAdapter` 在构造时就传入了 380 这个字符限制：`IrcAdapter::new(380)`。

## 渠道级模型覆盖

`runtime.rs` 在 `prepare_channel_runtimes` 中会从每个 runtime context 中收集 channel 对应的 model override，并通过 `normalize_model_name` 验证后写入 `llm_model_overrides`。这意味着：

- 同一个内核可以被不同渠道绑定不同默认模型。
- Telegram 可以用 Claude，Discord 用 GPT-4，Feishu 用国产模型——全在同一个进程内。
- 渠道差异不必侵入核心循环，模型选择在 `AgentRequestContext` 层完成。

## 统一行为来自哪里

如果多渠道系统想保持一致，真正的秘诀不是在每个适配器里写更多业务逻辑，而是让一致行为来自共享内核。

在 MicroClaw 中，无论消息从哪个渠道进入，最终都会被收敛到：

- 同一个 `process_with_agent` / `process_with_agent_with_events_guarded`
- 同一套工具系统（`ToolRegistry`）
- 同一套记忆注入
- 同一套 session 恢复
- 同一套存储层
- 同一个 `ChatTurnQueue` 做 per-chat 串行化

因此，多渠道一致性的来源并不是"所有适配器写得一样"，而是"绝大部分行为根本不在适配器里"。

## Per-Chat Turn 串行化

v0.1.38 引入了 `ChatTurnQueue`，这是多渠道架构的一个关键基础设施。它确保同一个 `(channel, chat_id)` 在任意时刻只有一个 agent run 在执行。

`src/chat_turn_queue.rs` 实现了一个精巧的设计：

- 每个 chat 有一个 `ChatSlot`，包含一个 `turn_lock`（`Arc<Mutex<()>>`）。
- 当前 run 持有 lock 期间，新到达的消息被收集为 `PendingMessage`。
- run 完成后，pending messages 被合并（coalesced）为下一轮输入。

这解决了一个在多渠道系统中特别容易出现的竞态问题：用户在 bot 还没回复时连续发送多条消息，导致同一个 chat 出现并行 agent run，互相踩踏 session 状态。Turn 串行化把这个风险从根本上消除了。

## 示例代码：渠道适配器为什么必须保持薄？

这一层代码的目标不是承载 Agent 推理，而是把平台差异翻译成统一 runtime 看得懂的事件和路由。只要适配器开始私自决定业务逻辑，多渠道复用就会很快失效。

Rust 版本把发送能力抽成 trait，再让具体 adapter 只保留平台字段和异步发送动作。渠道层的职责被压缩在"翻译"和"投递"两件事上。

```rust
#[derive(Debug, Clone)]
struct OutboundMessage {
    external_chat_id: String,
    text: String,
}

#[async_trait::async_trait]
trait ChannelAdapter {
    fn name(&self) -> &str;
    fn is_local_only(&self) -> bool { false }
    fn allows_cross_chat(&self) -> bool { true }
    async fn send(&self, message: OutboundMessage) -> anyhow::Result<()>;
}

struct FeishuAdapter {
    channel_name: String,
    app_id: String,
    app_secret: String,
    domain: String,
}
```

注意 `is_local_only` 和 `allows_cross_chat` 这两个默认值的设计。大部分外部渠道都不是 local-only 且允许 cross-chat，只有 Web 和 ACP 需要覆盖默认值。这种默认策略让新渠道接入时不需要思考权限模型中不相关的维度。

```rust

#[async_trait::async_trait]
impl ChannelAdapter for FeishuAdapter {
    fn name(&self) -> &str {
        &self.channel_name
    }

    async fn send(&self, message: OutboundMessage) -> anyhow::Result<()> {
        println!("feishu/{} -> {}: {}", self.app_id, message.external_chat_id, message.text);
        Ok(())
    }
}
```

## 新渠道接入的最小改动面

如果你要为 MicroClaw 增加一个新渠道，遵循这张清单：

1. 在 `src/channels/` 下新增模块，定义 `RuntimeContext` 和 `Adapter`。
2. 实现 `ChannelAdapter`，明确 `chat_type_routes` 和消息发送能力。
3. 提供 `build_*_runtime_contexts` 函数从 config 构建 runtime context。
4. 在 `src/channels/mod.rs` 中 re-export adapter。
5. 在 `runtime.rs` 中调用 `prepare_channel_runtimes` 注册并 `spawn_channel_runtimes` 启动。
6. 验证消息入库、会话恢复、工具调用、最终回复和异常恢复。

这份清单之所以有效，是因为 `prepare_channel_runtimes` 的泛型设计把装配逻辑标准化了。你要做的主要是填充三个闭包，而不是到处改代码。

## 关键权衡

### 决策一：渠道适配器保持薄层

优点是行为大多来自共享内核，16 个渠道不会各自 diverge。代价是某些平台特性不能无损映射，必须接受抽象损耗——例如 Telegram 的 inline keyboard、Discord 的 embed 都无法通过纯文本接口完整表达。

### 决策二：Matrix 使用 feature flag

优点是不需要 Matrix 的用户不受编译和体积影响。代价是 CI 需要额外的 feature 矩阵来保证 Matrix 代码不会因为长期不编译而 bit-rot。

### 决策三：stub 渠道先注册后完善

优点是路由体系和适配器抽象可以提前验证。代价是用户可能会误以为所有列出的渠道都已生产就绪。

### 决策四：per-chat turn 串行化

优点是彻底消除并行 run 踩踏 session 状态的竞态。代价是同一 chat 的消息必须排队等待上一轮完成，吞吐量受限于单 chat 串行。

### 决策五：渠道级模型覆盖

优点是更灵活，不同渠道可以用不同的最优模型。代价是运维人员需要理解"同一 runtime 不同入口可能使用不同模型"的事实。

## 容易走错的地方

### 失败模式 1：在适配器里复制核心逻辑

一旦某个适配器开始自己处理 session 恢复或记忆注入，就会和其他渠道的行为分叉。这种分叉在测试中很难发现，在生产中很难修复。

### 失败模式 2：没有统一路由模型

如果状态库里不能稳定映射 chat 到渠道，调度、send_message 和导出都会持续出错。这是 16 个渠道系统中最致命的基础设施缺陷。

### 失败模式 3：忽略 per-chat 串行化

没有 turn queue 的多渠道系统，在用户连续发消息时会出现并行 agent run，导致 session state 互相覆盖、工具调用重复执行。

### 失败模式 4：把 feature flag 渠道当成默认可用

Matrix 需要 `channel-matrix` feature 编译。如果部署脚本没有包含这个 feature，运行时启用 Matrix 配置只会得到一条警告日志，而不是预期的连接。

## 小结

MicroClaw v0.1.38 的多渠道设计从 v0.1.16 的几个渠道扩展到了 16 个渠道，但核心架构原则没有变：渠道是入口，不是内核。`ChannelAdapter` 保持薄层，`ChannelRegistry` 统一路由，`ChatTurnQueue` 消除并行竞态，共享内核提供一致行为。新增的 Feishu ACK、Matrix E2E、Email 异步、stub channel 机制，都是在不破坏这个架构的前提下，逐步扩展平台覆盖面。

下一章我们把视角转向后台执行能力。因为一旦系统支持 16 个渠道和长期会话，用户很快就会希望它不仅响应消息，还能主动调度、编排子代理、在后台持续工作。

# Chapter 9 多渠道架构

## 这一章要回答什么问题

同一个 runtime 如何在 15 种渠道上保持一致行为，同时保留各平台必要的差异处理？

## 15 个渠道的实际面貌

仓库 `src/channels/` 下落地了完整的 15 个适配器，加上嵌入式 Web 控制面构成对外的 15 条入口。

| 渠道 | 底层 crate / 协议 | 备注 |
| --- | --- | --- |
| Telegram | `teloxide 0.17` | 多 bot token，`channel-telegram` feature 默认开启 |
| Discord | `serenity 0.12` | gateway + REST，embed 消息 |
| Slack | Socket Mode（自研 client） | 工作区级 bot |
| Matrix | `matrix-sdk 0.16`（optional） | E2E 加密，`channel-matrix` feature gate |
| Email | IMAP polling + sendmail | 异步投递 |
| Weixin | 自研 web 协议 | 私聊 / 群聊 |
| WhatsApp | WhatsApp Cloud API | 商业号 webhook |
| Signal | signal-cli bridge | 本地进程桥接 |
| IRC | 标准 IRC | 380 字符上限 |
| iMessage | macOS AppleScript bridge | 仅在 macOS 启用 |
| 钉钉（DingTalk） | webhook + outgoing | 企业群 |
| 飞书（Feishu） | `tokio-tungstenite` + open API | WebSocket + ACK reaction |
| QQ | NapCat / OneBot 协议 | 反向 WebSocket |
| Nostr | `nostr-sdk` | NIP-04 私信 |
| Web | Axum + 嵌入式 React | 本地控制面，`is_local_only=true` |

15 条入口共享同一份 `process_with_agent`、同一份 `ToolRegistry`、同一份会话存储；适配器只负责把外部协议翻译为 `InboundMessage`，再把内核产物投递回去。

## `ChannelAdapter`：统一抽象只问五件事

`crates/microclaw-channels/src/channel_adapter.rs` 把每个渠道压缩到一个 trait：

```rust
#[async_trait]
pub trait ChannelAdapter: Send + Sync {
    fn name(&self) -> &str;
    fn chat_type_routes(&self) -> Vec<(&str, ConversationKind)>;
    fn is_local_only(&self) -> bool { false }
    fn allows_cross_chat(&self) -> bool { true }
    async fn send_text(&self, external_chat_id: &str, text: &str) -> Result<(), String>;
    async fn send_attachment(
        &self,
        _external_chat_id: &str,
        _file_path: &Path,
        _caption: Option<&str>,
    ) -> Result<String, String> {
        Err(format!("attachments not supported for {}", self.name()))
    }
}
```

`ConversationKind` 只有 `Private` 和 `Group` 两个变体——决定记忆作用域、群内提及策略、群成员可见性，渠道差异不外溢到内核。

## `ChannelRegistry`：`db_chat_type → (adapter, ConversationKind)`

注册器维护三张哈希表：

```rust
pub struct ChannelRegistry {
    adapters: HashMap<String, Arc<dyn ChannelAdapter>>,
    type_to_channel: HashMap<String, String>,        // "slack_dm" -> "slack"
    type_to_conversation: HashMap<String, ConversationKind>,
}
```

注册一次（`reg.register(Arc::new(SlackAdapter::new(...)))`），路由层就可以同时给出"该用哪个适配器"和"这是私聊还是群"，外部调用方不需要再 `match` 渠道。Scheduler、子代理 announce、Web `/api/sessions` 都共用这张表。

Web 适配器是该模型的反例：`is_local_only = true`、`allows_cross_chat = false`，意味着 Web 消息只入库，不会主动外发；任何 cross-chat 工具调用都不能从 Web 入口发起，规避运维台变成攻击面。ACP 适配器（`AcpAdapter`，第 12 章）共享相同姿态。

## 平台事件先归一为 `InboundMessage`

```rust
struct RouteKey {
    channel: String,          // "telegram"、"discord"
    external_chat_id: String, // 平台原生 chat id
}

struct InboundMessage {
    route: RouteKey,
    sender_id: String,
    text: String,
}
```

每个适配器把 webhook payload、Socket Mode 帧、IMAP letter、WebSocket JSON 等等翻译为同一份结构。数据库里只看到 `(channel, external_chat_id, sender_id, text)`，不感知平台细节。

## 装配流程：闭包注入新渠道

`runtime.rs` 用 `prepare_channel_runtimes` 接受三个闭包（build、register、model_override），新增渠道只需补三段：

```
config.channel_enabled(key)
  → prepare_channel_runtimes(build, register, model_override)
    → spawn_channel_runtimes / spawn_guarded
```

- `spawn_guarded` 用 `catch_unwind` 包裹任务，单渠道 panic 不波及其他渠道。
- 同一渠道支持多实例（多 bot token、多账号）。
- 未启用 feature 时打印 `warn_missing_feature` 而非崩溃：

```rust
#[cfg(not(feature = "channel-matrix"))]
warn_missing_feature(&config, "matrix", "channel-matrix");
```

## 各渠道差异治理：约束在边界层

差异越界一次，内核就要为它退让一次；约束在边界层是不可让步的纪律。

### 飞书：WebSocket + ACK reaction

`src/channels/feishu.rs` 用 `tokio-tungstenite` 维持长连接，收到消息后立即发送 emoji reaction 作为"已收到"——避免几十秒推理过程中用户以为 bot 挂了。`system_prompt_extension` 也只生效在飞书入口，不会污染其他渠道。

### Matrix：feature gate 隔离编译开销

`matrix-sdk` 引入 sqlite + 加密栈，会让默认编译时间上涨一倍以上；通过 `channel-matrix` feature 让默认构建省掉这部分。CI 必须跑一次 `--features channel-matrix` 防止 bit-rot。

### Email：异步投递语义

IMAP polling 收件 + `sendmail` 出件，没有"正在输入"也没有已读回执。`ChannelAdapter` 的 `async fn send_text` 天然适配；适配器内部把多段长回复拆为一封邮件，靠邮件正文格式承载信息。

### IRC、Telegram、Discord：消息长度差异

各平台长度上限不同（Telegram 4096、Discord 2000、IRC 380）。所有适配器在 egress 调用 `microclaw_core::text::split_text` 按渠道阈值拆分，主循环输出不感知。

### Stub 渠道：先注册后填充

iMessage、Signal、Weixin 等渠道 adapter 已实现，但底层 bridge 成熟度不一。先注册让路由体系提前承认它们，避免后期 big-bang 接入。

### 附件与引用

附件接口在 `ChannelAdapter::send_attachment` 默认返回 `Err`，由各适配器选择性实现：Telegram、Discord、飞书原生支持 multipart 上传；IRC、Email 走链接降级。引用（reply-to）在数据库层用 `parent_message_id` 关联，Adapter 决定是否在外发时还原为原生引用——例如 Discord embed 引用、Slack thread_ts。

### Markdown 兼容性

Telegram 用 MarkdownV2 + 转义，Discord 用 GFM 子集，飞书用富文本块。统一在 egress 做"轻量 Markdown → 平台格式"的最小映射，超出能力直接退化为纯文本——一个 bug 在一个渠道，比所有渠道都半截好看代价小。

## Per-Chat Turn 串行化

`ChatTurnQueue` 保证同一 `(channel, chat_id)` 任意时刻只有一个 agent run：

- 每 chat 一个 `turn_lock`（`Arc<Mutex<()>>`）
- 当前 run 期间新消息收集为 `PendingMessage`
- run 完成后 pending messages 合并为下一轮输入

否则用户连续两条消息会触发并行 run 踩踏 session、记忆、todo 状态。

## 渠道级模型覆盖

同一内核可以让 Telegram 用 Claude、Discord 用 GPT-4、飞书用国产模型——选择在 `AgentRequestContext` 层完成，不侵入主循环。

## 接入新渠道的最小改动面

1. `Cargo.toml` 加 feature gate（`channel-foo = ["dep:foo-sdk"]`）。
2. `src/channels/foo.rs` 实现 `ChannelAdapter`、定义 `chat_type_routes`、提供 `build_*_runtime_contexts`。
3. `src/channels/mod.rs` re-export。
4. `runtime.rs` 在 `prepare_channel_runtimes` 流程里 `reg.register(Arc::new(FooAdapter::new(...)))`。
5. `config` 增加 `[channel.foo]` section + `channel_enabled` 检查。
6. 验证消息入库、会话恢复、工具调用、回复路径、异常恢复。

## 示例代码：薄适配器实现

```rust
struct FeishuAdapter {
    channel_name: String,
    app_id: String,
    app_secret: String,
}

#[async_trait::async_trait]
impl ChannelAdapter for FeishuAdapter {
    fn name(&self) -> &str { &self.channel_name }

    fn chat_type_routes(&self) -> Vec<(&str, ConversationKind)> {
        vec![
            ("feishu_p2p", ConversationKind::Private),
            ("feishu_group", ConversationKind::Group),
        ]
    }

    async fn send_text(&self, external_chat_id: &str, text: &str) -> Result<(), String> {
        feishu_api::send(&self.app_id, &self.app_secret, external_chat_id, text)
            .await
            .map_err(|e| e.to_string())
    }
}
```

## 统一行为的来源

```
平台事件 → ChannelAdapter 翻译 → InboundMessage
  → ChannelRegistry 路由 → ChatTurnQueue 串行化
    → process_with_agent（统一循环）
      → ChannelAdapter 外发回复
```

一致性来自共享内核：同一套 `process_with_agent`、`ToolRegistry`、记忆注入、session 恢复、存储层、`ChatTurnQueue`。

## 容易走错的地方

| 失败模式 | 后果 |
|---------|------|
| 在适配器里复制核心逻辑 | 渠道行为分叉，测试难发现，生产难修复 |
| 没有统一路由模型 | 调度、send_message、子代理 announce 持续出错 |
| 忽略 per-chat 串行化 | 并行 run 踩踏 session state |
| 把 feature gate 渠道当默认可用 | 运行时只得到 warn 日志而非连接 |
| 在内核 import 平台 SDK | 把渠道差异传染到主循环 |

## 关键权衡

| 决策 | 优点 | 代价 |
|------|------|------|
| 适配器保持薄层 | 行为来自共享内核，不 diverge | 部分平台特性无损映射受限 |
| Matrix feature flag | 不需要者免受编译影响 | CI 需 feature 矩阵防 bit-rot |
| Stub 先注册后完善 | 路由体系提前验证 | 用户可能误以为全部生产就绪 |
| Per-chat turn 串行化 | 消除并行竞态 | 单 chat 吞吐受限于串行 |
| 渠道级模型覆盖 | 不同渠道用不同最优模型 | 运维需理解同 runtime 多模型 |

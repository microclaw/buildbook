# Chapter 10 调度与后台任务

## 这一章要回答什么问题

一个真正可用的 Agent Runtime，不能只会“收到消息再回答”。用户很快就会提出更复杂的要求：

- 明天上午提醒我跟进这件事。
- 每天早上汇总一次仓库风险。
- 后台继续跑一个子任务，完成后再回来通知我。

这就把系统从“交互式问答”推进到“具备时间维度和后台执行能力的 runtime”。MicroClaw 在这一点上已经具备相当成熟的雏形：调度器、反射器、子代理和回传链路都已经进入主干。

这一章读完后，你应该理解：

1. 调度任务如何从持久化数据变成实际执行。
2. 为什么 Scheduler 必须和统一循环共享同一内核。
3. 失败补偿、DLQ 和时区处理为何是生产级必需品。
4. 后台 lane 不只有 cron，还有 Reflector 和 subagents 这样的异步执行通道。

## Scheduler 的主循环：先保证“会执行”，再谈花样

`src/scheduler.rs` 的 `spawn_scheduler` 体现了一条很典型的工程原则：后台任务首先要保证可靠触发。

当前实现做了几件很重要的事：

1. 启动时恢复上次进程异常退出时仍处于 running 状态的任务。
2. 启动后立刻执行一次 `run_due_tasks`，避免过期任务等到下一分钟。
3. 用 wall-clock 对齐分钟边界，让“每分钟”调度更加稳定。
4. 使用 `MissedTickBehavior::Skip`，避免系统落后后突发补跑。

这些细节说明 MicroClaw 并没有把 Scheduler 当成“简单 while loop + sleep 60s”。它把对齐、恢复和补偿都纳入了第一版语义，这是一个很成熟的取舍。

## 任务模型：一次性任务和周期任务如何统一

存储层的 `ScheduledTask` 结构已经明确区分：

- `schedule_type`: `cron` 或 `once`
- `schedule_value`: cron 表达式或 ISO 时间
- `timezone`
- `next_run`
- `last_run`
- `status`

这个模型的好处是简单且够用。无论是一次性提醒还是重复任务，最终都可以统一表示为“下一次何时应当执行”。这让 claim、执行、更新状态的流程保持一致。

很多系统会在这里过早引入复杂工作流引擎、队列系统和 DAG 概念。MicroClaw 当前没有这样做，而是优先把最核心的“可靠时间驱动执行”做好。这对单机优先 runtime 是合理的。

### 小例子：为什么任务模型首先要把 `next_run` 和时区写清楚？

调度系统最容易犯的错误，不是“不会执行”，而是“在错误的时间执行”。只要 `next_run` 和 `timezone` 没有成为任务的一等字段，调度器迟早会在跨时区或重启恢复时出现语义漂移。

Rust 版本把这两个时间语义直接放进 `ScheduledTask`。这样调度器拿到任务后，不需要再猜“这条任务当时是按哪个时间解释的”。

```rust
struct ScheduledTask {
    id: i64,
    prompt: String,
    schedule_type: String,
    timezone: String,
    next_run: i64,
}
```

Python 版本用 `@dataclass` 保持同样的任务事实。例子虽然短，但它点出了调度系统最该先稳定下来的字段是什么。

```python
from dataclasses import dataclass


@dataclass
class ScheduledTask:
    id: int
    prompt: str
    schedule_type: str
    timezone: str
    next_run: int
```

## 调度任务如何复用统一循环

`run_due_tasks` 中最关键的一步，是当任务到期后，它并不会调用某个“专门给 Scheduler 写的推理路径”，而是直接调用：

- `process_with_agent`

只不过这一次的输入不是用户即时消息，而是任务保存下来的 `prompt`，并且通过 `get_chat_routing` 找到目标 chat 对应的渠道和会话语义。

这意味着 Scheduler 不是第二套 Agent 系统，而只是统一循环的另一种触发器。这个设计极其重要，因为它保证了以下行为在定时任务和即时消息之间保持一致：

- session 恢复
- 记忆注入
- 工具调用
- 权限边界
- 最终消息回投

如果调度器另写一套执行链，它和交互式链路很快就会在行为上分叉。MicroClaw 通过复用统一循环，避免了这类系统性漂移。

## 任务回传：后台执行不能变成黑洞

后台任务最大的风险之一，是“任务确实运行了，但用户不知道结果去了哪里”。MicroClaw 在成功和失败两条路径上都显式做了回传。

### 成功路径

如果 `process_with_agent` 返回非空文本，Scheduler 会通过 `deliver_and_store_bot_message` 把结果投递回原 chat，并同时入库。

### 失败路径

如果运行报错，系统也会给目标 chat 发送一条失败说明，而不是默默吞掉。

这意味着后台任务在用户视角里仍然是会话的一部分，而不是一个完全不可见的系统子进程。对聊天型 runtime 来说，这一点非常重要。

## 失败补偿：DLQ 为什么不能省

光有日志不够。调度任务一旦失败，如果不留下可重放的补偿入口，运维就只能手工重建现场。MicroClaw 当前已经把失败任务写入 DLQ，并提供了：

- `list_scheduled_task_dlq`
- `replay_scheduled_task_dlq`

同时，任务运行日志也会记录：

- 开始时间
- 结束时间
- 持续时间
- 成功与否
- 结果摘要

这套设计的价值在于，它把“失败之后怎么办”前置到了主干，而不是等线上出事故后再补脚本。

对于生产系统来说，DLQ 的核心意义不是“以后也许能用”，而是：

- 可以批量查看失败面
- 可以定向重放某个 task
- 可以把 replay 行为本身纳入审计

这会极大降低后台任务系统的运维焦虑。

## 时区与时间语义：调度系统最容易被低估的难点

`scheduler.rs` 中专门有 `resolve_task_timezone`。任务结构也把时区作为一等字段保存，而不是隐含在服务器本地时间里。

这背后的原因很朴素：聊天系统的用户和部署机器未必在同一个时区。即使是单机部署，也不能默认“服务器时区就是用户意图”。

因此，生产级调度至少要做两件事：

1. 任务创建时记录明确时区。
2. 下一次执行时间的计算要优先使用任务时区，而不是全局默认。

这类设计虽然不起眼，却是从“能跑”迈向“可信”的分水岭。

## Reflector：第二条后台 lane

在 MicroClaw 中，后台执行不只属于 Scheduler。Reflector 同样是一条后台 lane。它按固定周期扫描对话、提取 durable memory、应用去重和 supersede 规则。

把 Reflector 放在 `scheduler.rs` 中并不奇怪，因为它和调度任务共享很多后台系统特征：

- 定时触发
- 读取持久化状态
- 调用模型或解析结果
- 写回数据库
- 需要独立观测

这说明 MicroClaw 的后台任务设计并不是“只有 cron”，而是在逐步形成一个统一的后台执行框架。未来要增加别的后台能力，例如健康修复、缓存预热、离线索引重建，也会自然地沿这条路径扩展。

## Subagents：第三条后台 lane

RFC `0005-subagents-runtime-v1.md` 给出了另一种后台执行思路：把子代理运行做成 session-native 异步任务，而不是嵌在单个工具调用里死等。

这条设计非常值得注意，因为它把“后台执行”从纯时间驱动扩展到了工作驱动：

- `sessions_spawn` 立即返回
- 子代理进入 `accepted -> queued -> running -> completed|failed|timed_out|cancelled`
- 父 chat 保持可交互
- 完成后通过 announce 回传

这与传统 Scheduler 不同，但共享同样的后台系统原则：

- 持久化生命周期
- 可取消
- 可观测
- 有结果回传

从架构视角看，Scheduler、Reflector、Subagents 已经共同构成了 MicroClaw 的后台执行平面。

## 后台任务与权限边界

后台任务的危险之处在于，它们运行时用户并不在线，误操作更难即时阻断。因此，后台系统必须严格复用已有的权限和路由语义，而不是偷偷绕过。

MicroClaw 的 current design 有两个重要好处：

- Scheduler 通过 chat 路由找到正确渠道，不需要伪造一个新上下文。
- 子代理使用受限的 `ToolRegistry::new_sub_agent`，避免子任务拥有和主会话同等的无界能力。

这说明后台能力不是“主系统开个后门”，而是“在更严格边界内复用主系统”。

## 示例代码：为什么 Scheduler 最好复用统一循环，而不是另起一套执行器？

实际的 Scheduler 还要处理时区、恢复、DLQ 和回传，但它的骨架依然很简单：拿到到期任务，交给统一执行能力，再回写结果。关键点不在循环本身，而在后台任务有没有继续复用主系统的状态和观测规则。

Rust 版本把任务存储和任务执行拆成两个 trait，再由 `Scheduler` 统一编排。这样调度系统不会因为“只是后台任务”就开始直接越过主循环，写出第二套执行语义。

```rust
struct Task {
    id: i64,
    prompt: String,
}

#[async_trait::async_trait]
trait TaskStore {
    async fn claim_due_tasks(&self) -> anyhow::Result<Vec<Task>>;
    async fn log_task_run(&self, task_id: i64, ok: bool) -> anyhow::Result<()>;
    async fn update_next_run(&self, task_id: i64) -> anyhow::Result<()>;
}

#[async_trait::async_trait]
trait TaskExecutor {
    async fn execute_prompt(&self, prompt: &str) -> anyhow::Result<()>;
}

struct Scheduler<S, E> {
    store: S,
    executor: E,
}
```

这一段先把任务、存储契约和调度器对象放在一起。先明确谁负责取任务、谁负责执行任务，后面的 `tick` 才能被看成统一循环的后台延伸，而不是另起炉灶。

```rust

impl<S: TaskStore, E: TaskExecutor> Scheduler<S, E> {
    async fn tick(&self) -> anyhow::Result<()> {
        for task in self.store.claim_due_tasks().await? {
            let ok = self.executor.execute_prompt(&task.prompt).await.is_ok();
            self.store.log_task_run(task.id, ok).await?;
            self.store.update_next_run(task.id).await?;
        }
        Ok(())
    }
}
```

Python 版本也把“取任务”和“执行任务”分离，再用 `@dataclass` 把调度器装配成一个明确对象。这样你很容易看清后台任务为什么应该复用已有 runtime，而不是在 cron 脚本里重新长出一套系统。

```python
from dataclasses import dataclass
from typing import Protocol


@dataclass
class Task:
    id: int
    prompt: str


class TaskStore(Protocol):
    async def claim_due_tasks(self) -> list[Task]: ...
    async def log_task_run(self, task_id: int, ok: bool) -> None: ...
    async def update_next_run(self, task_id: int) -> None: ...


class TaskExecutor(Protocol):
    async def execute_prompt(self, prompt: str) -> None: ...


@dataclass
class Scheduler:
    store: TaskStore
    executor: TaskExecutor
```

Python 这里也先停在装配边界，再给出轮询逻辑。这样读者先看到对象分工，再看到异常处理和回写流程，长代码块也不会再压到跨页。

```python

    async def tick(self) -> None:
        for task in await self.store.claim_due_tasks():
            ok = True
            try:
                await self.executor.execute_prompt(task.prompt)
            except Exception:
                ok = False
            await self.store.log_task_run(task.id, ok)
            await self.store.update_next_run(task.id)
```

## 关键权衡

### 决策一：Scheduler 直接复用统一循环

优点是即时消息和定时任务行为一致。代价是后台执行的复杂度会继承主循环的全部特性。

### 决策二：失败任务进入 DLQ，而不是只打日志

优点是可补偿、可重放。代价是需要额外维护状态机和运维工具。

### 决策三：时区作为任务字段显式保存

优点是时间语义更可信。代价是创建和调试任务时要多考虑时区输入的合法性。

### 决策四：后台 lane 多样化，但都走统一持久化与观测思路

优点是后续扩展更自然。代价是后台子系统会越来越像一个独立平面，需要持续治理其复杂度。

## 容易走错的地方

### 失败模式 1：给 Scheduler 单独写一套执行逻辑

这样做会让定时任务和即时对话行为快速分叉。

### 失败模式 2：失败只记日志，不做 DLQ

一旦任务多起来，运维基本无法高效补偿。

### 失败模式 3：忽略时区问题

系统可能“按时执行”，但执行的是错误时区下的时间。

## 小结

MicroClaw 的后台任务系统已经超出了“简单定时提醒”的范畴。它通过 Scheduler、Reflector 和 subagents 逐步构建出一个可持久化、可回传、可补偿、可观测的后台执行平面。这正是聊天型 Agent Runtime 向真正运行时系统演进的关键一步。

下一章我们转向 Web 与 API。因为一旦系统具备后台执行能力，团队就几乎一定会希望通过可视化控制面和程序接口去观察、配置和管理它。

# Chapter 10 调度与后台任务

## Agent 如何从"收到消息才回答"升级为"具备时间维度的执行体"？

一个真正可用的 Agent Runtime 不能只会"收到消息再回答"。用户很快会提出更复杂的要求：

- 明天上午提醒我跟进这件事。
- 每天早上汇总一次仓库风险。
- 后台起一个子代理去调研这个问题，完成后通知我。
- 同时编排三个子代理，分别做代码审查、测试和文档。

这就把系统从"交互式问答"推进到"具备时间维度、并行执行和生命周期管理能力的 runtime"。MicroClaw v0.1.38 在这个方向上有非常显著的进展：Scheduler 增加了 rate-limit-aware 指数退避重试，但更大的变化来自 session-native subagents——一套完整的子代理生命周期管理系统，包含 spawn、orchestrate、cancel、announce 回传、restricted tool registry 和 ACP 外部执行。

这一章读完后，你应该理解：

1. Scheduler 如何从持久化数据变成实际执行，以及 rate-limit 退避为什么是生产必需品。
2. Session-native subagents 的完整生命周期：accepted → queued → running → completed/failed/timed_out/cancelled。
3. 子代理的受限工具注册表和 token budget 为什么不能省。
4. Announce 回传如何在异步完成后把结果送回父会话。
5. 后台 lane 不只有 cron，还有 Reflector、subagents 和 announce relay。

## Scheduler 的主循环：先保证"会执行"，再谈花样

`src/scheduler.rs` 的 `spawn_scheduler` 体现了一条很典型的工程原则：后台任务首先要保证可靠触发。

当前实现做了几件很重要的事：

1. **启动恢复**：`recover_running_tasks` 把上次进程异常退出时仍处于 running 状态的任务恢复为可执行状态。
2. **立即执行**：启动后立刻执行一次 `run_due_tasks`，避免过期任务等到下一分钟。
3. **Wall-clock 对齐**：用 `Utc::now().timestamp().rem_euclid(60)` 计算到下一分钟边界的偏移，让"每分钟"调度更加稳定。
4. **Skip 策略**：使用 `MissedTickBehavior::Skip`，避免系统落后后突发补跑。

这些细节说明 MicroClaw 并没有把 Scheduler 当成"简单 while loop + sleep 60s"。它把对齐、恢复和补偿都纳入了第一版语义。

## Rate-Limit-Aware 指数退避

v0.1.38 的 Scheduler 增加了一个非常务实的能力：`deliver_scheduler_message_with_backoff`。当消息投递遇到 rate limit（429、"too many requests"、"频控"等），系统不会直接放弃，而是用指数退避重试。

`is_retryable_delivery_rate_limit` 识别的错误模式包括：

- HTTP 429 / rate limit
- "too many requests" / "too many"
- 中文的"频控"、"限流"、"请求过于频繁"

退避策略是 `2^attempt` 秒，最多 3 次尝试。这个设计看起来简单，但它解决了一个在多渠道系统中非常常见的问题：当 Scheduler 批量执行多个到期任务时，短时间内向同一个渠道发送大量消息，很容易触发平台的 rate limit。没有退避的系统会在这种情况下丢失消息或报告虚假失败。

### 任务模型为什么首先要把 `next_run` 和时区写清楚？

调度系统最容易犯的错误不是"不会执行"，而是"在错误的时间执行"。只要 `next_run` 和 `timezone` 没有成为任务的一等字段，调度器迟早会在跨时区或重启恢复时出现语义漂移。

Rust 版本把这两个时间语义直接放进 `ScheduledTask`。

```rust
struct ScheduledTask {
    id: i64,
    prompt: String,
    schedule_type: String,  // "cron" | "once"
    schedule_value: String, // cron 表达式或 ISO 时间
    timezone: String,
    next_run: i64,
    chat_id: i64,
}
```

Python 版本用 `@dataclass` 保持同样的任务事实。

```python
from dataclasses import dataclass


@dataclass
class ScheduledTask:
    id: int
    prompt: str
    schedule_type: str
    schedule_value: str
    timezone: str
    next_run: int
    chat_id: int
```

## 调度任务如何复用统一循环

有了可靠的触发和退避机制之后，接下来的关键问题是：到期任务该走哪条执行路径？如果为 Scheduler 单独写一套推理逻辑，会迅速和交互式消息的行为分叉——session 恢复、记忆注入、工具调用、权限边界都需要重复实现。MicroClaw 选择了更稳妥的做法。

`run_due_tasks` 中最关键的一步，是当任务到期后，它直接调用 `process_with_agent`，而不是走一条"专门给 Scheduler 写的推理路径"。

```rust
let (success, result_summary) = match process_with_agent(
    state,
    AgentRequestContext {
        caller_channel: &routing.channel_name,
        chat_id: task.chat_id,
        chat_type: routing.conversation.as_agent_chat_type(),
    },
    Some(&task.prompt),
    None,
).await { ... };
```

路由信息通过 `get_chat_routing` 从数据库恢复。这意味着 Scheduler 不需要自己记住"这个任务应该回复到哪个渠道"，而是复用和交互式消息完全相同的路由逻辑。

这个设计至关重要，因为它保证了以下行为在定时任务和即时消息之间保持一致：session 恢复、记忆注入、工具调用、权限边界、最终消息回投。

## 任务回传与失败补偿

### 成功路径

如果 `process_with_agent` 返回非空文本，Scheduler 通过 `deliver_scheduler_message_with_backoff` 把结果投递回原 chat，并同时记录 task run log。

### 失败路径

如果运行报错，系统也会给目标 chat 发送一条失败说明。同时，失败任务会被写入 DLQ（`insert_scheduled_task_dlq`）。

DLQ 配合 `list_scheduled_task_dlq` 和 `replay_scheduled_task_dlq` 工具，让运维能够：

- 批量查看失败面
- 定向重放某个 task
- 把 replay 行为本身纳入审计

## Session-Native Subagents：RFC 0005 的实现

v0.1.38 最大的后台执行进展来自 `src/tools/subagents.rs`。这不是简单的"在工具调用里嵌套一次 LLM 请求"，而是一套完整的 session-native 异步任务系统。

### 核心设计：立即返回 + 异步执行 + 结果回传

`sessions_spawn` 工具不会阻塞父会话。它：

1. 立即创建一个 `run_id`。
2. 在数据库中记录 subagent run（`CreateSubagentRunParams`），状态为 accepted。
3. 获取 `SubagentRuntime` 的信号量 permit（并发控制）。
4. 状态流转：accepted → queued → running。
5. 在后台 tokio 任务中执行子代理循环。
6. 执行完毕后状态变为 completed / failed / timed_out / cancelled。
7. 通过 announce 机制把结果送回父会话。

这意味着父会话在 spawn 之后立即恢复可交互状态。用户可以继续发送消息、查看子代理状态、甚至 kill 某个子代理。

### 生命周期状态机

```
accepted → queued → running → completed
                             → failed
                             → timed_out
                             → cancelled
```

每个状态转换都持久化到数据库。这让系统在进程重启后能够恢复子代理的状态，也让 Web 面板和 CLI 工具能够实时展示子代理进度。

### Restricted Tool Registry

子代理不能拥有和主会话同等的无界能力。`ToolRegistry::new_sub_agent` 创建了一个受限的工具注册表：

```rust
let tools = ToolRegistry::new_sub_agent(
    &config,
    db.clone(),
    Some(channel_registry),
    allow_session_tools,  // depth < max_spawn_depth
);
```

`allow_session_tools` 参数由嵌套深度决定：只有当 `depth < config.subagents.max_spawn_depth` 时，子代理才能继续 spawn 自己的子代理。这防止了递归 spawn 导致的资源耗尽。

### Token Budget 传播

子代理还有一个 token budget 机制。`compute_child_token_budget` 函数根据父代理的剩余 budget 计算子代理的配额：

```rust
fn compute_child_token_budget(
    requested_budget: Option<i64>,
    parent_budget_remaining: Option<i64>,
    configured_max: i64,
) -> Result<i64, String> {
    // ...
    if let Some(parent_remaining) = parent_budget_remaining {
        if parent_remaining < 2_000 {
            return Err("subagent budget exhausted".into());
        }
        let desired = requested_budget.unwrap_or((parent_remaining / 2).max(2_000));
        return Ok(desired.clamp(2_000, parent_remaining.min(configured_max)));
    }
    // ...
}
```

默认策略是把父代理剩余 budget 的一半分配给子代理。这个设计防止了一个常见的失控场景：子代理消耗远超预期的 token，导致整个系统的 LLM 成本失控。

### 子代理 Orchestration

`subagents_orchestrate` 工具允许一次性 spawn 多个子代理：

```rust
// 伪代码示意
let results = orchestrate(vec![
    ("reviewer", "Review the PR for security issues"),
    ("tester", "Write integration tests for the new API"),
    ("docs", "Update API documentation"),
]);
```

orchestration 的关键在于它不只是并行 spawn——它还会等待所有子代理完成，并收集 artifacts 合并返回。这让 Agent 能够表达"把这个大任务分成三部分并行做"的意图。

### 可取消与可查询

子代理的完整管理工具集包括：

- `subagents_list` / `subagents_info`：查看状态
- `subagents_kill`：请求取消
- `subagents_log`：查看执行日志
- `subagents_focus` / `subagents_unfocus` / `subagents_focused`：UI 焦点管理
- `subagents_send`：向运行中的子代理发送消息
- `subagents_retry_announces`：重放未成功投递的 announce

取消机制使用双层检测：内存中的 `AtomicBool`（`local_cancel`）和数据库中的 `is_subagent_cancel_requested`。这保证了即使主进程重启，取消请求也不会丢失。

## Announce 回传机制

子代理完成后，结果需要送回父会话。这通过 announce 机制实现。

`runtime.rs` 中可以看到，当 `config.subagents.announce_to_chat` 为 true 时，系统会启动一个 `subagents_announce_relay` 后台任务：

```rust
if state.config.subagents.announce_to_chat {
    spawn_guarded("subagents_announce_relay".to_string(), async move {
        // 启动时先恢复 pending announces
        let first_processed = flush_pending_announces_once(...).await;
        // 然后按 interval 轮询
        loop {
            ticker.tick().await;
            flush_pending_announces_once(...).await;
        }
    });
}
```

这个 relay 做两件事：

1. **启动恢复**：进程重启后，先把之前未投递的 pending announces 恢复出来。
2. **定期轮询**：按配置的 `announce_relay_interval_secs` 检查是否有新的 announce 需要投递。

Announce 的 artifact payload 使用 `normalize_subagent_artifact_payload` 统一格式化，包含 summary、findings、artifacts、next_actions 和 final_answer。这让父会话能够结构化地理解子代理的输出，而不是拿到一大段原始文本。

## ACP Subagent Runtime

子代理不仅可以在本地执行（native runtime），还可以通过 ACP 协议委托给外部进程执行。`SubagentExecutionRuntime` 区分了两种模式：

```rust
enum SubagentExecutionRuntime {
    Native,
    Acp,
}
```

当 `runtime = "acp"` 时，子代理的执行被委托给 `crate::acp_subagent::run_acp_subagent_task`。这意味着一个 MicroClaw 实例可以把繁重的子任务分发给其他 Agent 进程，实现真正的分布式执行。

## Reflector：第二条后台 lane

在 MicroClaw 中，后台执行不只属于 Scheduler 和 subagents。Reflector 同样是一条重要的后台 lane。`spawn_reflector` 按固定周期（`reflector_interval_mins`）扫描活跃对话、调用 LLM 提取 durable memory、应用去重和 supersede 规则。

v0.1.38 的 Reflector 有几个关键改进：

1. **增量反射**：使用 `get_reflector_cursor` 和 `set_reflector_cursor` 实现增量处理。已经反射过的消息不会重复处理。
2. **内存投毒防护**：`REFLECTOR_SYSTEM_PROMPT` 明确要求"NEVER describe broken behavior as a fact"，防止错误行为被记忆化后导致 Agent 重复犯错。
3. **外部记忆提供者健康检查**：`should_pause_reflector_writes` 会在外部记忆提供者不健康时暂停写入，避免丢失提取结果。
4. **Embedding 回填**：`backfill_embeddings` 为缺少 embedding 的记忆补充向量索引。
5. **陈旧记忆归档**：`archive_stale_memories(30)` 自动归档 30 天未更新的记忆。

## 四条后台 Lane 的统一视角

从架构视角看，MicroClaw v0.1.38 的后台执行平面由四条 lane 构成：

| Lane | 触发方式 | 持久化 | 可取消 | 结果回传 |
|------|---------|--------|--------|---------|
| Scheduler | 时间驱动（cron/once） | 是 | 通过删除任务 | 消息投递到 chat |
| Reflector | 周期驱动 | 是（cursor） | 通过 config 禁用 | 写入记忆系统 |
| Subagents | 工作驱动（spawn） | 是 | kill + cancel flag | announce 回传 |
| Announce Relay | 周期驱动 | 是（pending state） | 自动 | 消息投递到 chat |

它们共享同样的后台系统原则：持久化生命周期、可观测、有结果回传、不绕过主系统权限边界。

## 示例代码：子代理生命周期为什么必须持久化？

子代理的关键设计问题不是"怎么 spawn"，而是"进程重启后子代理状态是否还在"。只有把生命周期写入持久化存储，announce relay 才能在重启后恢复 pending 投递，Web 面板才能展示准确状态。

Rust 版本把子代理的核心状态机和取消机制拆成两层：内存中的 `AtomicBool` 提供低延迟取消检测，数据库中的 `is_subagent_cancel_requested` 提供持久化保证。

```rust
use std::sync::atomic::{AtomicBool, Ordering};
use std::sync::Arc;

struct SubagentRun {
    run_id: String,
    status: String,      // accepted, queued, running, completed, failed, timed_out, cancelled
    parent_chat_id: i64,
    task: String,
    token_budget: i64,
    cancel_flag: Arc<AtomicBool>,
}

impl SubagentRun {
    fn request_cancel(&self) {
        self.cancel_flag.store(true, Ordering::Relaxed);
    }

    fn is_cancelled(&self) -> bool {
        self.cancel_flag.load(Ordering::Relaxed)
    }
}
```

Python 版本保持同样的双层设计，用 `threading.Event` 替代 `AtomicBool`。

```python
from dataclasses import dataclass, field
from threading import Event


@dataclass
class SubagentRun:
    run_id: str
    status: str  # accepted, queued, running, completed, failed, timed_out, cancelled
    parent_chat_id: int
    task: str
    token_budget: int
    cancel_event: Event = field(default_factory=Event)

    def request_cancel(self) -> None:
        self.cancel_event.set()

    def is_cancelled(self) -> bool:
        return self.cancel_event.is_set()
```

## 关键权衡

### 决策一：Scheduler 增加 rate-limit 退避

优点是不会因为批量执行触发平台限流而丢失消息。代价是单次任务执行时间可能因重试而延长。

### 决策二：子代理使用 session-native 异步模型

优点是父会话不被阻塞，子代理有完整生命周期管理。代价是系统复杂度显著增加——需要维护状态机、announce relay、取消机制和 artifact 格式。

### 决策三：子代理使用受限工具注册表

优点是防止子代理拥有无界能力。代价是某些工具在子代理中不可用，可能限制子代理的能力范围。

### 决策四：Token budget 传播

优点是防止子代理消耗失控。代价是深度嵌套的子代理可用 budget 会指数衰减，需要合理规划最大深度。

### 决策五：Announce relay 作为独立后台 lane

优点是 announce 投递和子代理执行解耦，重启后可恢复。代价是引入了额外的异步窗口，用户看到结果会有一定延迟。

## 容易走错的地方

### 失败模式 1：给 Scheduler 单独写一套执行逻辑

这样做会让定时任务和即时对话行为快速分叉。MicroClaw 通过复用 `process_with_agent` 避免了这个问题。

### 失败模式 2：子代理没有 token budget

没有 budget 控制的子代理编排，一次 orchestrate 可能消耗掉一个月的 LLM 配额。

### 失败模式 3：子代理状态只存内存

进程重启后所有子代理状态丢失，announce 无法投递，Web 面板显示空白。

### 失败模式 4：忽略 rate-limit 退避

多个到期任务同时执行时，批量消息投递很容易触发平台限流，导致大量任务被标记为失败。

### 失败模式 5：Reflector 写入记忆时不检查外部提供者健康

如果外部记忆提供者暂时不可用，Reflector 仍然继续写入，会导致提取结果丢失。

## 小结

MicroClaw v0.1.38 的后台任务系统已经从 v0.1.16 的"Scheduler + Reflector + subagents 雏形"演进为一个完整的后台执行平面。Scheduler 增加了 rate-limit 退避，让批量执行更可靠。Session-native subagents 提供了完整的生命周期管理，包括并发控制、受限工具注册表、token budget、可取消、可查询和 announce 回传。四条后台 lane（Scheduler、Reflector、Subagents、Announce Relay）共同构成了一个可持久化、可恢复、可观测的后台执行平面。

下一章我们转向 Web 与 API。因为一旦系统具备后台执行能力，团队就几乎一定会希望通过可视化控制面和程序接口去观察、配置和管理它。

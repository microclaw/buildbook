# Chapter 8 记忆系统

## 这一章要回答什么问题

如果一个 Agent 每次都从零开始，它就永远只是一个“即时应答器”。真正能长期工作的 runtime，必须能记住用户是谁、上下文是什么、哪些事实需要持续生效、哪些旧信息已经过时。MicroClaw 在这件事上的设计很有代表性，因为它没有把记忆简单做成一个向量库，也没有只靠一个 `AGENTS.md` 文件硬撑，而是做成了双层系统。

这一章读完后，你应该理解：

1. 文件记忆和结构化记忆为什么必须分层。
2. 显式记忆、自动提取、检索注入为什么要走不同路径。
3. 记忆质量治理为什么比“记得更多”更重要。
4. 为什么记忆系统必须有审计、观测和生命周期，而不是只做 append。

## 第一层记忆：`AGENTS.md` 解决可读与可编辑

MicroClaw 在 README 中把持久记忆描述为多层 `AGENTS.md` 文件：全局、bot/account、chat 作用域都会被装载到每次请求里。这类文件记忆的优势非常明确。

第一，它对人类友好。运维人员或用户可以直接打开文件修改，不需要理解数据库 schema 或 API。

第二，它适合放高层规则和稳定背景。例如：

- 团队协作约定
- 仓库工作目录说明
- 长期有效的风格要求
- 某个 chat 的特殊使用习惯

第三，它天然可审计。文件在版本控制和本地备份里都更容易跟踪。

但文件记忆也有明显边界：

- 难以做细粒度检索。
- 不适合维护大量短事实。
- 缺乏结构化置信度、来源和 supersede 关系。

因此，MicroClaw 没有试图让 `AGENTS.md` 独自承担全部记忆职责。

### 小例子：为什么记忆要先按作用域装载，再决定注入多少？

文件记忆和结构化记忆都不是“越多越好”，而是先分清作用域，再决定哪些内容值得进入当前轮次。否则全局规则、账号偏好和 chat 私有事实会在同一层里互相污染。

Rust 版本先定义 `MemoryScope`，再让加载器按作用域返回内容。这样后面的预算控制和相关性筛选都有了清晰起点。

```rust
#[derive(Debug, Clone, Copy)]
enum MemoryScope {
    Global,
    Account,
    Chat,
}

#[async_trait::async_trait]
trait MemoryLoader {
    async fn load_scope(&self, scope: MemoryScope, chat_id: i64) -> anyhow::Result<Vec<String>>;
}
```

Python 版本用 `Enum + Protocol` 保持同样的表达。这样读者会更容易理解“记忆系统”首先是层次系统，而不是单一存储桶。

```python
from enum import Enum
from typing import Protocol


class MemoryScope(str, Enum):
    GLOBAL = "global"
    ACCOUNT = "account"
    CHAT = "chat"


class MemoryLoader(Protocol):
    async def load_scope(self, scope: MemoryScope, chat_id: int) -> list[str]: ...
```

## 第二层记忆：结构化记忆解决检索与生命周期

真正支撑“长期运行中持续积累事实”的，是 `microclaw-storage` 里的结构化记忆表。`crates/microclaw-storage/src/db.rs` 暴露的 `Memory` 结构已经明确包含：

- `content`
- `category`
- `confidence`
- `source`
- `last_seen_at`
- `is_archived`
- `archived_at`
- 可选 `embedding_model`

这些字段说明结构化记忆不是一个“只会塞文本”的黑箱，而是有完整元数据的事实对象。

这会带来几个直接能力：

- 可以按 chat 或全局范围检索。
- 可以区分 PROFILE / KNOWLEDGE / EVENT 等类别。
- 可以根据置信度和最近使用情况做淘汰与归档。
- 可以记录事实是如何进入系统的，是 explicit、reflector 还是其他来源。

对于 Agent Runtime 来说，这种结构化能力非常关键。因为长期运行时最难的不是“记住”，而是“知道记住的东西还值不值得继续注入”。

## 为什么必须双层：文件记忆与结构化记忆不是替代关系

很多系统在记忆上会走极端：

- 要么一切都写进 Markdown 文件。
- 要么一切都塞进向量库或数据库。

MicroClaw 的做法更务实。它让两层记忆各自承担不同职责。

### 文件记忆适合声明性上下文

它适合那些人类希望直接维护、且持续有效的规则与背景，例如：

- “这个仓库默认用 `rg` 查文件。”
- “优先给出工程化答案，不要只讲概念。”
- “生产环境禁止直接修改某个目录。”

### 结构化记忆适合事实性增量

它适合那些从对话中逐渐积累、并需要检索与生命周期管理的内容，例如：

- 用户偏好某个渠道输出简洁。
- 某个 chat 对应一个具体项目目录。
- 最近一次部署失败和某个配置项有关。

把这两层混在一起，会同时失去可编辑性和结构化治理。MicroClaw 选择分层，是因为两者本来就服务不同场景。

## 显式记忆 fast-path：明确意图不必交给模型猜

`memory_service.rs` 的 `maybe_handle_explicit_memory_command` 展示了当前系统非常成熟的一点：当用户明确要求“记住某件事”时，系统不必再把这件事交给完整 agent loop 重新解释。

这条 fast-path 会做几步关键工作：

1. 检测最近用户消息是否属于显式记忆命令。
2. 执行质量校验，过滤过于模糊的内容。
3. 与现有记忆做去重。
4. 如果同一 topic 发生冲突，则走 supersede。
5. 以较高置信度写入结构化记忆。

这比“让模型决定要不要记”稳得多。因为显式记忆本质上是用户明确授权的事实录入，不应该被模型风格、当轮提示词或上下文噪声影响。

## 检索注入：不是把所有记忆都塞回 prompt

记忆系统真正影响模型质量的环节，不是写入，而是注入。MicroClaw 当前至少在三个维度上避免了“全量灌回上下文”的粗暴做法。

### 第一，只注入相关记忆

`memory_service.rs` 里有针对相关性的 token 化和打分逻辑，包括对 CJK 文本的处理。这说明检索不是单纯按字符串包含，而是尽量估计当前请求和历史事实的关联度。

### 第二，只注入预算内的记忆

`config.rs` 中有 `memory_token_budget` 这类配置，说明记忆注入与上下文窗口共享同一份成本预算。系统必须平衡：

- 当前消息历史
- 技能上下文
- 工具 schema
- 结构化记忆

如果不做预算控制，记忆系统越“丰富”，模型输入反而越混乱。

### 第三，注入行为本身可观测

存储层暴露了 `MemoryInjectionLog` 和 `MemoryObservabilitySummary`。这意味着系统不仅记录“有什么记忆”，还记录：

- 检索时选了多少候选
- 最终注入了多少
- 采用什么 retrieval method
- 过去 24 小时注入情况如何

这一步非常关键。没有注入观测，团队只能凭感觉判断“记忆有没有用”；有了这些日志，你才能真正调优检索口径和预算。

## Reflector：自动提取记忆，但不让它失控

如果只有显式记忆，系统的长期积累速度会很慢。所以 MicroClaw 还提供了 Reflector，也就是后台记忆提取器。`scheduler.rs` 中的 `spawn_reflector` 会按配置周期运行，默认间隔是 `15` 分钟。

Reflector 的系统提示词非常值得仔细读，因为它揭示了这套系统最成熟的一面：它不仅告诉模型“提取事实”，还明确约束它不要把 bug 描述成事实记忆。

例如文档中给出的坏例子是：

- “tool calls were broken”

而更合适的写法是：

- “TODO: ensure tool calls always execute via tool system”

这就是典型的记忆污染防护。因为一旦把错误行为当作事实长期注入，Agent 可能会在后续对话中不断强化错误模式。MicroClaw 已经显式把这种风险纳入设计。

### Reflector 为什么必须和质量门控绑定

自动提取的价值很高，但风险也更高。它可能产生：

- 模糊记忆
- 重复记忆
- 过时记忆
- 误把问题描述成行为准则

因此 Reflector 不是“抽到就存”，而是要经过：

- JSON 解析
- 去重
- 类别判断
- supersede 处理
- 记忆质量与 poisoning 风险检查

这就是为什么记忆系统必须有独立的 `memory_quality` 逻辑。没有质量门控，自动记忆迟早会把系统变成噪声放大器。

## `sqlite-vec`：向量检索是可选增强，而不是设计前提

MicroClaw 支持 `sqlite-vec` feature，并在有 embedding provider 时为记忆补齐向量索引。但从当前设计看，这项能力是增强层，而不是前提层。

这说明它在记忆设计上很克制：

- 没有把“必须先建向量检索”作为启动门槛。
- 没有让语义检索成为唯一检索路径。
- 允许本地 SQLite 方案先工作，再逐步打开 semantic memory。

这类策略对工程项目很重要。因为很多系统在“记忆”上过早依赖 embedding 和外部向量数据库，结果基础检索、生命周期和可观测都没做好。MicroClaw 的顺序是反过来的：先把结构化治理做好，再加语义增强。

## 生命周期管理：记忆不是永生的

从 `Memory` 结构和相关 observability 字段能看出，MicroClaw 已经明确把记忆分成活跃和归档状态。这是一个经常被低估的设计点。

如果所有记忆都永久生效，系统会出现两个问题：

- 旧事实压制新事实。
- prompt 注入越来越臃肿。

因此，真正可用的记忆系统必须支持：

- 更新
- supersede
- 归档
- 删除
- 最近使用时间更新

这也是为什么 MicroClaw 提供了结构化记忆搜索、删除、更新工具。它把记忆视为可治理数据，而不是神圣不可碰的“AI 回忆”。

## 示例代码：为什么显式 remember 应该走受控写入，而不是交给模型猜？

对“remember ……”这类明确意图，最稳妥的处理通常不是让模型自由发挥，而是走一条带来源、分类和置信度的受控路径。这样后续检索、清理和审计时，系统知道这条记忆为什么会存在。

Rust 版本把记忆写入能力抽成 trait，再用专门的 writer struct 封装 fast-path。这样显式记忆不是某次 prompt 巧合写出来的副作用，而是一条可以稳定复用的运行时机制。

```rust
#[async_trait::async_trait]
trait MemoryStore {
    async fn upsert_memory(
        &self,
        chat_id: i64,
        content: &str,
        category: &str,
        confidence: f32,
        source: &str,
    ) -> anyhow::Result<()>;
}

struct ExplicitMemoryWriter<S> {
    store: S,
}
```

这段先把显式记忆写入依赖的最小边界钉住。先理解“谁能写、写入需要哪些元数据”，再看 `remember` 这个快捷入口，逻辑会更稳。

```rust

impl<S: MemoryStore> ExplicitMemoryWriter<S> {
    async fn remember(&self, chat_id: i64, content: &str) -> anyhow::Result<()> {
        self.store
            .upsert_memory(chat_id, content, "KNOWLEDGE", 0.95, "explicit")
            .await
    }
}
```

Python 版本保留同样的写入口径，但用 `Protocol` 和 `@dataclass` 把依赖关系写明。这样读者能直接看出：显式记忆是一个工程化接口，不是“让模型顺手帮我记一下”。

```python
from dataclasses import dataclass
from typing import Protocol


class MemoryStore(Protocol):
    async def upsert_memory(
        self,
        chat_id: int,
        content: str,
        category: str,
        confidence: float,
        source: str,
    ) -> None: ...


@dataclass
class ExplicitMemoryWriter:
    store: MemoryStore
```

Python 也先让写入器对象单独站住，再进入实际写入动作。这样示例更像一个受控接口，而不是“顺手向存储里写几行数据”。

```python

    async def remember(self, chat_id: int, content: str) -> None:
        await self.store.upsert_memory(
            chat_id=chat_id,
            content=content,
            category="KNOWLEDGE",
            confidence=0.95,
            source="explicit",
        )
```

## 关键权衡

### 决策一：同时保留文件记忆与结构化记忆

优点是兼顾人类可编辑性和运行时可检索性。代价是两层系统需要持续解释职责边界。

### 决策二：显式记忆走 fast-path

优点是可靠性更高、行为更可预测。代价是需要额外维护一条非通用的记忆写入路径。

### 决策三：Reflector 默认开启，但必须带质量门控

优点是系统能自动积累事实。代价是需要投入更多规则与观测来防止记忆污染。

### 决策四：把向量检索作为增强层，而不是强依赖

优点是启动更简单、默认行为更稳。代价是纯本地模式下的语义召回能力有限。

## 容易走错的地方

### 失败模式 1：把所有长期信息都塞进 `AGENTS.md`

这样做短期简单，但后续很难做去重、检索和生命周期管理。

### 失败模式 2：把结构化记忆当成无上限日志

没有置信度、归档和 supersede，记忆越多，系统反而越难用。

### 失败模式 3：自动提取记忆时忽视污染风险

一旦把错误行为、临时抱怨或模糊指令写成长期事实，系统会不断自我强化偏差。

## 小结

MicroClaw 的记忆系统之所以值得学习，不在于它“支持长期记忆”，而在于它把记忆做成了一个可编辑、可检索、可观测、可归档、可纠错的双层系统。这样的设计，才能让 Agent 的经验真正积累为工程资产，而不是变成另一个不透明黑箱。

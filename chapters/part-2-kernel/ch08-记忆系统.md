# Chapter 8 记忆系统

## Agent 怎样才能不每次都从零开始？

如果一个 Agent 每次都从零开始，它就永远只是一个"即时应答器"。真正能长期工作的 runtime，必须记住用户是谁、上下文是什么、哪些事实需要持续生效、哪些旧信息已经过时。

MicroClaw 的记忆系统是一个双层架构：文件记忆（`AGENTS.md`）解决可读与可编辑，结构化记忆（SQLite + 可选 `sqlite-vec` 向量索引）解决检索与生命周期。在此之上，`MemoryBackend` 抽象支持本地 SQLite 或 MCP 外接，Reflector 提取器负责自动记忆积累，`memory_quality` 模块提供质量门控，注入审计记录每次记忆使用。

这一章读完后，你应该理解：

1. 文件记忆和结构化记忆为什么必须分层，以及各自的边界。
2. 显式记忆 fast-path、自动提取、检索注入三条路径的设计意图。
3. `MemoryBackend` 如何抽象后端选择，以及 `sqlite-vec` 如何作为可选增强层。
4. 记忆质量治理为什么比"记得更多"更重要。
5. 记忆注入如何在 token 预算下做相关性排序。

## 第一层记忆：`AGENTS.md` 解决可读与可编辑

MicroClaw 的文件记忆系统把持久记忆描述为多层 `AGENTS.md` 文件：全局、bot/account、chat 作用域都会被装载到每次请求里。`MemoryManager` 通过 `build_memory_context` 按作用域拼装文件记忆内容。

文件记忆的优势明确：

**对人类友好。** 运维人员或用户可以直接打开文件修改，不需要理解数据库 schema 或 API。

**适合放高层规则和稳定背景。** 例如团队协作约定、仓库工作目录说明、长期有效的风格要求、某个 chat 的特殊使用习惯。

**天然可审计。** 文件在版本控制和本地备份里都更容易跟踪。

但文件记忆也有明显边界：难以做细粒度检索、不适合维护大量短事实、缺乏结构化置信度和 supersede 关系。因此，MicroClaw 没有试图让 `AGENTS.md` 独自承担全部记忆职责。

### 记忆为什么要先按作用域装载，再决定注入多少？

文件记忆和结构化记忆都不是"越多越好"，而是先分清作用域，再决定哪些内容值得进入当前轮次。

Rust 版本先定义 `MemoryScope`，再让加载器按作用域返回内容：

```rust
#[derive(Debug, Clone, Copy)]
enum MemoryScope {
    Global,
    Account,
    Chat,
}

#[async_trait::async_trait]
trait MemoryLoader {
    async fn load_scope(&self, scope: MemoryScope, chat_id: i64) -> anyhow::Result<Vec<String>>;
}
```

Python 版本用 `Enum + Protocol` 保持同样的表达：

```python
from enum import Enum
from typing import Protocol


class MemoryScope(str, Enum):
    GLOBAL = "global"
    ACCOUNT = "account"
    CHAT = "chat"


class MemoryLoader(Protocol):
    async def load_scope(self, scope: MemoryScope, chat_id: int) -> list[str]: ...
```

## 第二层记忆：结构化记忆与 `MemoryBackend`

### 结构化记忆的数据模型

`microclaw-storage` 的 `Memory` 结构包含完整元数据：

- `id`：唯一标识
- `chat_id`：关联 chat（None 表示全局）
- `content`：记忆内容
- `category`：PROFILE / KNOWLEDGE / EVENT
- `confidence`：置信度（0.0–1.0）
- `source`：来源（explicit / reflector / tool 等）
- `last_seen_at`：最近使用时间
- `is_archived`：是否已归档
- `archived_at`：归档时间
- `embedding_model`：用于生成向量的模型名

这些字段说明结构化记忆不是一个"只会塞文本"的黑箱，而是有完整元数据的事实对象。

### `MemoryBackend` 抽象

`MemoryBackend`（`src/memory_backend.rs`）抽象了记忆存储后端。默认使用 `local_only` 模式（直接操作 SQLite），但也支持通过 MCP 协议外接记忆服务器。`MemoryMcpClient` 会自动发现提供 `memory_query` 和 `memory_upsert` 工具的 MCP 服务器。

`MemoryBackend` 暴露的核心接口包括：

- `get_all_memories_for_chat`：获取某 chat 的所有活跃记忆
- `get_memories_for_context`：获取用于注入的记忆（带数量限制）
- `insert_memory_with_metadata`：带元数据插入
- `update_memory_with_metadata`：带元数据更新
- `supersede_memory`：supersede 操作（归档旧记忆，创建新记忆）

这个抽象让记忆系统可以从"嵌入式 SQLite"平滑迁移到"外部记忆服务"，而上层的 Reflector、显式记忆、检索注入代码不需要改变。

## 为什么必须双层：文件记忆与结构化记忆不是替代关系

很多系统在记忆上走极端：要么一切写进 Markdown 文件，要么一切塞进向量库。MicroClaw 的做法更务实——两层各自承担不同职责。

| 维度 | 文件记忆（AGENTS.md） | 结构化记忆（SQLite） |
| --- | --- | --- |
| 适合内容 | 声明性规则、稳定背景 | 事实性增量、对话积累 |
| 典型例子 | "优先给出工程化答案" | "用户偏好简洁输出" |
| 维护方式 | 人类直接编辑文件 | Agent 或 Reflector 自动管理 |
| 检索方式 | 按作用域全量装载 | 关键词/KNN 相关性排序 |
| 生命周期 | 手动维护 | 置信度、归档、supersede |

把这两层混在一起，会同时失去可编辑性和结构化治理。

## 显式记忆 fast-path：明确意图不必交给模型猜

`memory_service.rs` 的 `maybe_handle_explicit_memory_command` 在进入完整 agent loop 之前执行。如果用户的最新消息是显式记忆命令（如"记住 X"、"remember X"），这条 fast-path 会：

1. **内容提取**：通过 `memory_quality::extract_explicit_memory_command` 解析。
2. **质量检查**：通过 `memory_quality::memory_quality_ok` 过滤过于模糊的内容。
3. **去重**：与现有记忆做 Jaccard 相似度比较（阈值 0.55）。如果找到高度相似的现有记忆，走更新而不是插入。
4. **Topic 冲突检测**：通过 `memory_quality::memory_topic_key` 生成 topic 关键字，如果同一 topic 存在不同内容，走 supersede。
5. **写入**：以 `confidence=0.95`、`source="explicit"` 写入结构化记忆。
6. **向量索引**：如果启用了 `sqlite-vec`，立即生成 embedding。

这比"让模型决定要不要记"稳得多。显式记忆本质上是用户明确授权的事实录入，不应该被模型风格、当轮提示词或上下文噪声影响。

## 检索注入：不是把所有记忆都塞回 prompt

`build_db_memory_context` 是记忆系统真正影响模型质量的环节。它在三个维度上避免了"全量灌回上下文"：

### 第一，相关性排序

检索分为三条路径，按优先级递降：

```
sqlite-vec + embedding 可用？ ──是──→ KNN 语义检索
       │
       否
       │
sqlite-vec 可用但 provider 不可用？ ──是──→ 本地语义排序
       │
       否
       │
       └──→ 关键词打分（支持 CJK bigram）
```

1. **KNN 语义检索**（`sqlite-vec` + embedding provider 可用时）：把当前查询向量化，用 `db.knn_memories` 做 k-nearest-neighbor 检索。
2. **本地语义排序**（`sqlite-vec` 可用但 provider 不可用时）：降级到关键词匹配。
3. **关键词打分**（默认路径）：使用 `tokenize_for_relevance` 分词（支持 CJK bigram），然后按 token 交集打分。

`tokenize_for_relevance` 支持 CJK：它把中文字符按 bigram 切分，确保中文记忆也能被合理召回，而不是只依赖空格分词。

### 第二，Token 预算控制

`memory_token_budget`（默认 1500）限制了注入的总 token 数。系统按每 4 个字符估算一个 token，逐条累加，超过预算立即停止。被截断的记忆数量会被记录为 `omitted`。

注入的格式是结构化 XML：

```xml
<structured_memories>
[KNOWLEDGE] [chat] User prefers concise responses
[PROFILE] [global] Team uses TypeScript for frontend
(+3 memories omitted)
</structured_memories>
```

### 第三，注入审计

每次注入都会通过 `db.log_memory_injection` 记录：

- `retrieval_method`：knn / keyword / provider
- `candidate_count`：候选记忆总数
- `selected_count`：最终注入数
- `omitted`：被截断数
- `used_tokens`：估算使用 token 数

这意味着系统不仅记录"有什么记忆"，还记录"每次注入了多少、用了什么方法"。没有注入审计，团队只能凭感觉判断"记忆有没有用"；有了这些日志，才能真正调优检索口径和预算。

## `sqlite-vec`：向量检索是可选增强，而不是设计前提

MicroClaw 通过 `sqlite-vec` feature flag 控制向量检索能力。当启用时：

1. `runtime::run` 在启动时调用 `db.prepare_vector_index` 创建向量索引。
2. `build_db_memory_context` 优先使用 KNN 检索。
3. 显式记忆和 Reflector 写入时自动生成 embedding。
4. `microclaw reembed` 命令可以批量重新生成所有活跃记忆的向量。

当未启用时，系统完全正常工作——只是退化到关键词打分排序。这种设计的关键价值在于：

- 没有把"必须先建向量检索"作为启动门槛。
- 没有让语义检索成为唯一检索路径。
- 允许本地 SQLite 方案先工作，再逐步打开 semantic memory。

`reembed` 命令允许在更换 embedding model 后重建所有向量索引，而不需要重新触发 Reflector 提取。

## Reflector：自动提取记忆，但不让它失控

Reflector 是后台记忆提取器，由 `scheduler.rs` 中的 `spawn_reflector` 按配置周期运行（默认间隔 15 分钟，`reflector_interval_mins`）。

### 提取流程

Reflector 读取最近对话历史，调用 LLM 提取结构化记忆。提取结果是 JSON 数组，每条包含 `content`、`category` 和可选的 `supersedes_id`。

### `apply_reflector_extractions`：质量门控链

`memory_service.rs` 的 `apply_reflector_extractions` 对每条提取的记忆执行完整的质量门控：

```
JSON 解析 → 类别校验 → 内容标准化 → 毒性检测 → 质量检查
  → Supersede 处理 → Topic 冲突 → 去重（Jaccard/语义）→ 合并决策
```

1. **JSON 解析**：必须有 `content` 字段。
2. **类别校验**：只接受 PROFILE / KNOWLEDGE / EVENT。
3. **内容标准化**：通过 `memory_quality::normalize_memory_content` 处理（截断到 180 字符）。
4. **毒性检测**：`should_skip_memory_poisoning_risk` 检查是否是"把错误行为描述成事实"的有毒记忆。
5. **质量检查**：`memory_quality::memory_quality_ok` 过滤模糊内容。
6. **Supersede 处理**：如果提取结果指定了 `supersedes_id` 且该 ID 存在，更新而不是插入。
7. **Topic 冲突**：通过 topic key 检测同主题冲突，走 supersede。
8. **去重**：Jaccard 相似度 ≥ 0.85 的内容视为重复。
9. **合并决策**：`should_merge_duplicate` 判断是更新现有记忆还是跳过。

### 毒性检测的设计细节

`should_skip_memory_poisoning_risk` 检查内容是否包含 "tool calls were broken"、"authentication error"、"failed" 等破坏性描述词，但排除 corrective action items（以 "TODO:" 开头或包含 "ensure" 的内容）。

这意味着：

- "tool calls were broken" → 被跳过（有毒记忆）
- "TODO: ensure tool calls always execute via tool system" → 被保留（纠正性行动项）

这就是典型的记忆污染防护。一旦把错误行为当作事实长期注入，Agent 可能在后续对话中不断强化错误模式。

### 去重方法选择

`apply_reflector_extractions` 在 `sqlite-vec` + embedding 可用时使用 `"semantic"` 去重方法，否则使用 `"jaccard"` 方法。这个选择会被记录在 `dedup_method` 字段中，方便后续分析哪种去重方法更有效。

## 生命周期管理：记忆不是永生的

从 `Memory` 结构和相关操作可以看出，MicroClaw 把记忆明确分为活跃和归档状态。

如果所有记忆都永久生效，系统会出现两个问题：旧事实压制新事实、prompt 注入越来越臃肿。

因此，MicroClaw 支持完整的生命周期操作：

- **更新**：`update_memory_with_metadata` 修改内容、类别、置信度和来源。
- **Supersede**：`supersede_memory` 归档旧记忆，创建新记忆，保留关联关系。
- **归档**：设置 `is_archived=true` 和 `archived_at`。
- **删除**：通过 `structured_memory_delete` 工具。
- **最近使用时间**：`last_seen_at` 在注入时更新。

Agent 可以通过 `structured_memory_search`、`structured_memory_update`、`structured_memory_delete` 三个工具主动管理记忆，而不只是被动接受。

## 示例代码：显式 remember 为什么应该走受控写入？

对"remember……"这类明确意图，最稳妥的处理是走一条带来源、分类和置信度的受控路径。这样后续检索、清理和审计时，系统知道这条记忆为什么存在。

Rust 版本把记忆写入能力抽成 trait，再用专门的 writer struct 封装 fast-path。显式记忆不是 prompt 巧合写出来的副作用，而是可以稳定复用的运行时机制。

```rust
#[async_trait::async_trait]
trait MemoryStore {
    async fn insert_memory(
        &self,
        chat_id: i64,
        content: &str,
        category: &str,
        confidence: f32,
        source: &str,
    ) -> anyhow::Result<i64>;

    async fn update_memory(&self, memory_id: i64, content: &str) -> anyhow::Result<()>;

    async fn find_similar(&self, chat_id: i64, content: &str, threshold: f64)
        -> anyhow::Result<Option<i64>>;
}

struct ExplicitMemoryWriter<S> {
    store: S,
}
```

先把显式记忆写入依赖的最小边界钉住。先理解"谁能写、写入需要哪些元数据"，再看 `remember` 入口。

```rust

impl<S: MemoryStore> ExplicitMemoryWriter<S> {
    async fn remember(&self, chat_id: i64, content: &str) -> anyhow::Result<String> {
        if !memory_quality_ok(content) {
            return Ok("Skipped: content too vague.".into());
        }
        if let Some(existing_id) = self.store.find_similar(chat_id, content, 0.55).await? {
            self.store.update_memory(existing_id, content).await?;
            return Ok(format!("Updated memory #{existing_id}"));
        }
        let id = self.store.insert_memory(chat_id, content, "KNOWLEDGE", 0.95, "explicit").await?;
        Ok(format!("Saved memory #{id}"))
    }
}
```

Python 版本保留同样的写入口径：

```python

    async def find_similar(
        self, chat_id: int, content: str, threshold: float,
    ) -> int | None: ...


@dataclass
class ExplicitMemoryWriter:
    store: MemoryStore
```

## 关键权衡

### 决策一：同时保留文件记忆与结构化记忆

优点是兼顾人类可编辑性和运行时可检索性。代价是两层系统需要持续解释职责边界。

### 决策二：`MemoryBackend` 抽象支持 MCP 外接

优点是记忆存储可以演进到外部服务而不改上层代码。代价是本地模式和 MCP 模式的行为需要保持一致，增加了测试矩阵。

### 决策三：显式记忆走 fast-path 并绕过完整循环

优点是可靠性更高、行为更可预测、成本更低（不需要调 LLM）。代价是需要额外维护一条非通用的记忆写入路径。

### 决策四：Reflector 默认开启，但必须带多层质量门控

优点是系统能自动积累事实。代价是毒性检测、质量检查、去重和 supersede 逻辑需要持续维护。但不做这些，自动记忆迟早会把系统变成噪声放大器。

### 决策五：把向量检索作为增强层，而不是强依赖

优点是启动更简单、默认行为更稳。代价是纯本地模式下的语义召回能力有限。但 MicroClaw 的顺序是反过来的：先把结构化治理做好，再加语义增强。

## 容易走错的地方

### 失败模式 1：把所有长期信息都塞进 `AGENTS.md`

短期简单，但后续很难做去重、检索和生命周期管理。文件记忆适合少量高层规则，不适合大量细粒度事实。

### 失败模式 2：把结构化记忆当成无上限日志

没有置信度、归档和 supersede，记忆越多，系统反而越难用——旧事实不断稀释新事实的注入机会。

### 失败模式 3：自动提取记忆时忽视污染风险

一旦把错误行为、临时抱怨或模糊指令写成长期事实，系统会不断自我强化偏差。`should_skip_memory_poisoning_risk` 的存在就是为了防止这种退化。

### 失败模式 4：没有注入审计就调优检索策略

不知道每次注入了多少记忆、用了什么方法、截断了多少，就无法判断记忆系统是否真正帮助了模型。`log_memory_injection` 让调优有据可依。

### 失败模式 5：过早依赖 embedding 而基础检索没做好

一个典型的反模式是：团队花两周接入了外部向量数据库和 embedding 模型，但 Reflector 的毒性检测还没实现、记忆归档策略还没定义、注入审计日志还不存在。结果系统确实能做语义检索，但写进去的全是低质量或过时的记忆，检索出来的结果反而让模型表现更差。MicroClaw 的做法是反过来的：先把关键词打分检索、Jaccard 去重、supersede 关系和质量门控做扎实，让没有 embedding 的系统也能正常工作，再用 `sqlite-vec` 作为可选增强层叠加语义能力。

- 核心源码路径：`src/memory_service.rs`（显式记忆 fast-path、build_db_memory_context、apply_reflector_extractions）、`src/memory_backend.rs`（MemoryBackend 抽象、MemoryMcpClient）、`crates/microclaw-storage/src/db.rs`（Memory 结构、log_memory_injection、knn_memories）、`crates/microclaw-storage/src/memory_quality.rs`（质量门控、毒性检测）
- 工具源码：`src/tools/memory.rs`（read/write_memory）、`src/tools/structured_memory.rs`（search/update/delete）
- 关键配置项：`memory_token_budget=1500`、`reflector_enabled=true`、`reflector_interval_mins=15`、`embedding_provider`（可选）、`embedding_dim`（可选）

## 小结

MicroClaw 的记忆系统之所以值得学习，不在于它"支持长期记忆"，而在于它把记忆做成了一个可编辑、可检索、可观测、可归档、可纠错的双层系统。`MemoryBackend` 抽象让后端可以从嵌入式 SQLite 演进到 MCP 外接而不影响上层逻辑。`sqlite-vec` 作为可选增强而不是强依赖，让系统可以在没有 embedding 基础设施时也能正常工作。毒性检测和质量门控确保了自动提取的记忆不会成为系统自我污染的来源。

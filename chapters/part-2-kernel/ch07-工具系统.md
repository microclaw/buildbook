# Chapter 7 工具系统

## 44 个工具如何变成一套可治理的执行系统？

Agent 能不能真正完成任务，最终取决于工具系统。没有工具，模型最多是一个会说话的建议器；有了工具，它才可能变成一个能读文件、改代码、查网页、发消息、排任务、编排子代理的执行体。

但工具系统越强，风险也越大。MicroClaw 内置了 44 个工具（不含 MCP 动态注入的），分布在 22 个模块文件中，配合 wave-based 并行执行引擎（`tool_executor.rs`，877 行）和基于 `ToolConcurrencyClass` 的三级并发分类。这不是简单的"工具越多越好"，而是把工具做成了一套可以治理、可以并行、可以审计的运行时系统。

这一章读完后，你应该能理解五件事：

1. `Tool` 抽象为什么必须足够统一。
2. 44 个工具如何按功能和风险分类。
3. wave-based 并行执行如何工作，三级 concurrency class 如何决定执行拓扑。
4. `ToolRegistry` 如何自动注入 chat_id、sandbox、授权上下文。
5. Hook 如何在工具执行前后插入策略控制。

## 44 个工具的全景分类

在讲抽象之前，先看清工具全景。按功能和风险，可以分为以下几类：

### 文件操作（6 个）

| 工具 | 风险 | 并发类别 |
| --- | --- | --- |
| `read_file` | 低 | ReadOnly |
| `glob` | 低 | ReadOnly |
| `grep` | 低 | ReadOnly |
| `write_file` | 中 | SideEffect |
| `edit_file` | 中 | SideEffect |
| `browser` | 低 | ReadOnly |

这组工具支持 `working_dir_isolation`（默认 Chat 级），意味着不同 chat 的文件操作在不同子目录下进行，天然隔离。

### 命令执行（1 个）

| 工具 | 风险 | 并发类别 |

| `bash` | 高 | Exclusive |

## `Tool` 抽象：先统一契约，再谈能力扩展

上面 44 个工具跨越了文件、命令、网络、记忆、调度、子代理、时间、技能和协作等九个功能域。如果每个工具自行定义输入输出格式，核心循环就无法统一处理成功/失败、审计、重试和权限。因此，真正的抽象核心在 `microclaw-tools::runtime` 的 `Tool` trait。一个工具至少要回答三个问题：名字是什么、输入 schema 是什么、如何执行。`ToolResult` 是统一返回结构，包含 `content`、`is_error`、`error_type`、`status_code`、`bytes` 和 `metadata`。

工具系统的第一原则从来不是"工具够多"，而是"工具足够像同一种东西"。

### 工具 schema 和实现为什么要一起注册？

Rust 版本用 `ToolSpec` 描述输入契约，用 `Tool` trait 持有实现。`ToolRegistry` 在 `new` 阶段把所有工具实例化并持有。

```rust
struct ToolSpec {
    name: String,
    description: String,
    input_schema: serde_json::Value,
}

#[async_trait::async_trait]
trait Tool: Send + Sync {
    fn spec(&self) -> ToolSpec;
    async fn execute(&self, input: serde_json::Value) -> ToolResult;
}
```

Python 版本用 `Protocol` 和 `@dataclass`：

```python
from dataclasses import dataclass
from typing import Any, Protocol


@dataclass
class ToolSpec:
    name: str
    description: str
    input_schema: dict[str, Any]


class Tool(Protocol):
    def spec(self) -> ToolSpec: ...
    async def execute(self, input: dict[str, Any]) -> ToolResult: ...
```

## `ToolRegistry`：执行能力的总装配点

`src/tools/mod.rs` 中的 `ToolRegistry::new` 接收 `Config`、`ChannelRegistry`、`Database` 和 `MemoryBackend`，一次性实例化所有 44 个工具。这个注册器的重要性不在于"把工具都 new 一遍"，而在于它是能力治理的唯一入口。

### 自动注入 chat_id

`ToolRegistry` 有一个很关键的函数 `inject_default_chat_id_if_missing`。当模型调用 `write_memory`、`send_message`、`sessions_spawn` 等需要 chat_id 的工具时，如果模型没有传 chat_id，registry 会自动从 `ToolAuthContext.caller_chat_id` 注入。

这解决了一个根本问题：模型不能被信任来声明权限边界。如果让模型自己传 chat_id，它生成错误参数时就可能越权访问别的会话。

### 授权上下文注入

`ToolAuthContext` 包含 `caller_channel`、`caller_chat_id`、`control_chat_ids` 和 `env_files`。每次工具执行前，registry 把这些信息注入到工具输入的 `__microclaw_auth` 字段中。工具内部通过 `auth_context_from_input` 解析授权信息，通过 `authorize_chat_access` 做跨 chat 权限判定。

权限来自运行时事实，而不是来自模型自述。

### Sandbox 路由

`SandboxRouter` 在 registry 创建时初始化，传给 `BashTool`。它决定命令是在宿主机执行还是在容器内执行，并管理安全 profile（hardened/standard/privileged）、挂载校验和敏感路径阻断。

### 子代理受限工具集

`ToolRegistry::new_sub_agent` 提供受限版本：不包含 `write_memory`、`send_message`、`schedule_*`、`todo_write`、`a2a_*` 等工具。是否暴露编排工具取决于 `allow_session_tools` 参数（受 `subagent_max_spawn_depth` 约束）。这防止了子代理无限递归创建子代理。

## Wave-Based 并行执行

`tool_executor.rs` 是一个 877 行的并行执行引擎。当模型在一次响应中返回多个 `tool_use` 块时，这些工具不再简单顺序执行，而是被分割成 wave 并行处理。

### 三级并发分类

`ToolConcurrencyClass` 定义了三个级别：

```rust
pub enum ToolConcurrencyClass {
    ReadOnly,    // 可以和其他 ReadOnly 并行
    SideEffect,  // 必须和其他 SideEffect/Exclusive 串行
    Exclusive,   // 必须独占执行
}
```

每个内置工具都有默认分类（在 `tool_concurrency_class` 函数中硬编码），MCP 工具默认为 `SideEffect`，未知工具保守默认为 `SideEffect`。配置中的 `tool_concurrency_overrides` 可以覆盖任意工具的分类。

### 分区算法

`partition_into_waves` 的规则很清晰：

1. 如果只有一个工具调用——不分区，直接执行。
2. 收集所有 ReadOnly 工具——放入第一个 wave，并行执行。
3. 收集所有 SideEffect 工具——每个单独一个 wave，顺序执行。
4. 收集所有 Exclusive 工具——每个单独一个 wave，顺序执行。

例如，如果模型同时请求 `read_file`、`grep`、`web_fetch`、`write_file`、`bash`，分区结果是：

- Wave 1：`[read_file, grep, web_fetch]`（ReadOnly，并行）
- Wave 2：`[write_file]`（SideEffect，独占）
- Wave 3：`[bash]`（Exclusive，独占）

### 执行机制

`execute_tool_batch` 是入口。对于每个 wave：

1. 发送 `AgentEvent::ToolWaveStart`。
2. 如果 wave 只有一个工具，内联执行（避免 spawn 开销）。
3. 如果 wave 有多个工具，使用 `tokio::JoinSet` 并行执行，受 `parallel_tool_max_concurrency`（默认 8）限制。
4. 发送 `AgentEvent::ToolWaveComplete`。

结果按原始索引排序，确保返回给模型的 `ToolResult` 顺序与请求一致。

### 单个工具执行的完整流程

### 单工具执行的完整链路

```
名字检查 → send_message 限制 → Feishu 文本限制 → BeforeToolCall hook
  → ToolStart 事件 → 执行工具 → OTLP trace
    → approval_required 自动重试 → activate_skill metadata
      → AfterToolCall hook → ToolResult 事件 → 更新错误跟踪
```

`execute_single_tool` 对每个工具执行：

1. **名字检查**：空名字直接返回错误。
2. **send_message 连续调用限制**：连续 3 次后阻断。
3. **Feishu 文本限制**：Feishu 渠道禁止纯文本 send_message（必须用附件）。
4. **BeforeToolCall hook**：可 block 或 modify input。
5. **ToolStart 事件**。
6. **执行工具**：通过 `tools.execute_with_auth`。
7. **OTLP trace span**。
8. **approval_required 自动重试**：如果工具返回需要审批，根据配置和用户最新消息判断是否可自动重试。
9. **activate_skill metadata 处理**：提取 `skill_env_file` 并持久化。
10. **AfterToolCall hook**：可 block（标记错误）或 modify（修改 content/is_error/error_type）。
11. **ToolResult 事件**。
12. **更新错误跟踪**。

这 12 步中，hook 和审批逻辑是最容易被低估的。它们确保了"工具执行"不是简单的函数调用，而是一个有策略控制的运行时行为。

## 示例代码：wave-based 并行执行的核心结构

Rust 版本展示了分区和执行的核心思路。先按并发类别分区，再逐 wave 执行。

```rust
#[derive(Clone, Copy, PartialEq)]
enum ConcurrencyClass {
    ReadOnly,
    SideEffect,
    Exclusive,
}

fn partition_into_waves(
    calls: &[ToolCall],
    classify: impl Fn(&str) -> ConcurrencyClass,
) -> Vec<Vec<usize>> {
    let mut readonly = Vec::new();
    let mut side_effect = Vec::new();
    let mut exclusive = Vec::new();
    for (i, call) in calls.iter().enumerate() {
        match classify(&call.name) {
            ConcurrencyClass::ReadOnly => readonly.push(i),
            ConcurrencyClass::SideEffect => side_effect.push(i),
            ConcurrencyClass::Exclusive => exclusive.push(i),
        }
    }
    let mut waves = Vec::new();
    if !readonly.is_empty() { waves.push(readonly); }
    for idx in side_effect { waves.push(vec![idx]); }
    for idx in exclusive { waves.push(vec![idx]); }
    waves
}
```

```rust

async fn execute_batch(tools: &ToolRegistry, calls: &[ToolCall]) -> Vec<ToolResult> {
    let waves = partition_into_waves(calls, |name| tool_concurrency_class(name));
    let mut all_results = Vec::new();
    for wave in &waves {
        if wave.len() == 1 {
            let result = tools.execute(&calls[wave[0]]).await;
            all_results.push((wave[0], result));
        } else {
            let mut join_set = tokio::task::JoinSet::new();
            for &idx in wave {
                let call = calls[idx].clone();
                join_set.spawn(async move { (idx, tools.execute(&call).await) });
            }
            while let Some(Ok(result)) = join_set.join_next().await {
                all_results.push(result);
            }
        }
    }
    all_results.sort_by_key(|(idx, _)| *idx);
    all_results.into_iter().map(|(_, r)| r).collect()
}
```

Python 版本用 `asyncio.gather` 替代 JoinSet：

```python
from collections.abc import Callable
from enum import Enum


class ConcurrencyClass(Enum):
    READ_ONLY = "read_only"
    SIDE_EFFECT = "side_effect"
    EXCLUSIVE = "exclusive"


def partition_into_waves(
    calls: list["ToolCall"],
    classify: Callable[[str], ConcurrencyClass],
) -> list[list[int]]:
    readonly, side_effect, exclusive = [], [], []
    for i, call in enumerate(calls):
        cls = classify(call.name)
        if cls == ConcurrencyClass.READ_ONLY:
            readonly.append(i)
        elif cls == ConcurrencyClass.SIDE_EFFECT:
            side_effect.append(i)
        else:
            exclusive.append(i)
    waves = []
    if readonly:
        waves.append(readonly)
    waves.extend([idx] for idx in side_effect)
    waves.extend([idx] for idx in exclusive)
    return waves
```

```python

async def execute_batch(tools: "ToolRegistry", calls: list["ToolCall"]) -> list["ToolResult"]:
    import asyncio

    waves = partition_into_waves(calls, tool_concurrency_class)
    all_results: list[tuple[int, "ToolResult"]] = []
    for wave in waves:
        if len(wave) == 1:
            result = await tools.execute(calls[wave[0]])
            all_results.append((wave[0], result))
        else:
            tasks = [tools.execute(calls[idx]) for idx in wave]
            results = await asyncio.gather(*tasks)
            all_results.extend(zip(wave, results))
    all_results.sort(key=lambda x: x[0])
    return [r for _, r in all_results]
```

## Hook：工具治理的可插拔层

Hook 系统支持三个事件：

- `BeforeLLMCall`：可 block 整个请求或修改 system prompt。
- `BeforeToolCall`：可 block 工具调用或修改工具输入。
- `AfterToolCall`：可 block 工具结果（标记错误）或修改输出 content/is_error/error_type/status_code。

Hook 用 `HookOutcome` 表达结果：

```rust
pub enum HookOutcome {
    Allow { patches: Vec<serde_json::Value> },
    Block { reason: String },
}
```

Hook 定义通过 frontmatter 格式的脚本文件，支持 `enabled` 开关和 `priority` 排序。`HookManager` 管理 Hook 的发现、加载、状态持久化和执行。

### 为什么 Hook 有治理价值

很多策略不适合写死在工具实现里。例如：

- 某个团队要求所有 bash 操作都附带风险摘要。
- 某个部署环境临时禁用某个工具。
- 某个工具结果需要做脱敏。

Hook 的位置刚好在中间：足够靠近执行点（可以真正阻断或修改），又不必污染每个工具实现。

## 新增工具时的质量标准

一个合格的工具，至少要满足：

1. 输入 schema 足够清晰，模型不容易误用。
2. 成功和失败结果都能被 `ToolResult` 统一表达。
3. 风险级别明确，是否需要 `require_high_risk_approval`。
4. 正确的 `ToolConcurrencyClass` 分类。
5. 能接受 `ToolAuthContext` 注入的授权上下文。
6. 有明确的超时（通过 `config.tool_timeout_secs`）。
7. 如果涉及文件操作，支持 `working_dir_isolation`。

## 关键权衡

### 决策一：wave-based 分区而非完全并行

优点是安全可控——ReadOnly 工具并行，SideEffect 串行，Exclusive 独占。代价是无法做更细粒度的依赖分析（比如两个写入不同文件的 write_file 理论上可以并行）。但简单规则比复杂分析更可靠。

### 决策二：concurrency class 硬编码 + 可配置覆盖

优点是默认行为安全，不需要用户理解并发模型。`tool_concurrency_overrides` 给高级用户提供了逃逸口。代价是新工具需要手动在 `tool_concurrency_class` 中添加分类。

### 决策三：授权上下文由 runtime 注入，而不是由模型声明

优点是权限边界更可信。代价是工具接口设计稍复杂。

### 决策四：Hook 承载部署级策略

优点是治理逻辑不污染工具实现。代价是 Hook 需要自己的文档、测试和故障隔离（timeout、error handling）。

### 决策五：子代理使用受限工具集

优点是防止递归失控。代价是子代理能力受限，某些场景下可能不够灵活。`allow_session_tools` 参数提供了一定程度的可配置性。

## 容易走错的地方

### 失败模式 1：把工具当成简单函数调用

忽视 schema、结果结构、审计、权限、超时、concurrency class 和 hook 等系统性需求。

### 失败模式 2：相信模型参数天然可信

模型生成的 chat_id、路径或命令都不应直接当成事实使用。`inject_default_chat_id_if_missing` 和 `authorize_chat_access` 存在的意义正在于此。

### 失败模式 3：忽视并行执行的正确性

如果把所有工具都标记为 ReadOnly，write_file 和 bash 可能并发导致文件损坏。concurrency class 看似保守，但保守是并行安全的代价。

### 失败模式 4：把 sandbox 当成唯一安全手段

没有审批、路径限制、授权上下文和 Hook，单靠 sandbox 仍然会留下大量治理盲区。

## 小结

MicroClaw 的工具系统真正有价值的地方，不是它"内置了 44 个工具"，而是它把工具做成了一套可以统一授权、并行执行、审批隔离、审计和策略扩展的能力层。wave-based 执行引擎让"读取类工具并行、写入类工具串行、高危工具独占"成为可编程的调度策略，而不是靠开发者记住"这两个工具不能同时跑"。

下一章我们继续沿着这条主链路前进，讨论工具系统经常打交道的另一层核心能力：记忆。对长期运行的 Agent 来说，记忆不是锦上添花，而是决定执行质量能否积累的基础设施。

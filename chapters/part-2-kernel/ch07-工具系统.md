# Chapter 7 工具系统

## 这一章要回答什么问题

Agent 能不能真正完成任务，最终取决于工具系统。没有工具，模型最多是一个会说话的建议器；有了工具，它才可能变成一个能读文件、改代码、查网页、发消息、排任务的执行体。

但工具系统越强，风险也越大。命令执行、文件写入、跨 chat 操作、网络抓取、插件注入，每一项都是能力和风险同时增长的地方。MicroClaw 在这里的价值，不只是“内置很多工具”，而是把工具做成了一套可以治理的运行时系统。

这一章读完后，你应该能理解四件事：

1. `Tool` 抽象为什么必须足够统一。
2. 工具注册和执行上下文如何被 runtime 控制。
3. 权限、审批、sandbox、path guard 如何形成多层防线。
4. 为什么 Hook 是工具治理，而不是简单的扩展点。

## `Tool` 抽象：先统一契约，再谈能力扩展

MicroClaw 没有把工具系统设计成“到处塞函数调用”。真正的抽象核心在 `microclaw-tools::runtime`，而顶层 `src/tools/mod.rs` 主要负责把这些工具装配成 `ToolRegistry`。

从接口层面看，一个工具至少要回答三个问题：

1. 它叫什么。
2. 它向模型暴露什么输入 schema。
3. 它收到输入后如何执行，并以统一结果结构返回。

这类抽象听起来很普通，但它决定了后面的所有治理能否成立。

如果不同工具各自返回不同格式，核心循环就无法统一处理成功/失败、日志、审计、重试和用户提示。如果输入 schema 不统一，Provider 适配层也无法稳定把工具定义翻译给模型。所以，工具系统的第一原则从来不是“工具够多”，而是“工具足够像同一种东西”。

### 小例子：为什么工具 schema 和实现要一起注册？

如果注册表只知道“有个函数能跑”，却不知道它暴露给模型的输入长什么样，工具系统很快就会在文档、调用和治理之间分裂。把 schema 和实现绑在一起，正是为了避免这种分裂。

Rust 版本用 `ToolSpec` 描述输入契约，再用 `RegisteredTool` 把契约和实现合到同一个对象里。这样注册表管理的就不是裸实现，而是“可被模型理解、可被 runtime 执行”的完整工具单元。

```rust
struct ToolSpec {
    name: String,
    description: String,
}

#[async_trait::async_trait]
trait Tool {
    async fn execute(&self, input: serde_json::Value) -> anyhow::Result<String>;
}

struct RegisteredTool<T: Tool> {
    spec: ToolSpec,
    implementation: T,
}
```

Python 版本用 `@dataclass` 和 `Protocol` 维持同样的结构。动态语言里这一步尤其重要，因为否则 schema 很容易退化成散落在注释和 prompt 里的隐式知识。

```python
from dataclasses import dataclass
from typing import Any, Protocol


@dataclass
class ToolSpec:
    name: str
    description: str


class Tool(Protocol):
    async def execute(self, input: dict[str, Any]) -> str: ...


@dataclass
class RegisteredTool:
    spec: ToolSpec
    implementation: Tool
```

## `ToolRegistry`：执行能力的总装配点

`src/tools/mod.rs` 中的 `ToolRegistry::new` 把当前内置工具组装到一起。现在已经包括多类能力：

- Shell 与文件：`bash`、`read_file`、`write_file`、`edit_file`、`glob`、`grep`
- 网络：`browser`、`web_fetch`、`web_search`
- 记忆：`read_memory`、`write_memory`、结构化记忆搜索/更新/删除
- 调度：创建、列出、暂停、恢复、取消任务，以及 DLQ 回放
- 协作：`send_message`
- 子代理：spawn、list、info、kill、focus、send、log、retry announces
- 扩展：`activate_skill`、`sync_skills`
- 计划：`todo_read`、`todo_write`
- 生态：A2A、ClawHub 工具

这个注册器的重要性不在于“把工具都 new 一遍”，而在于它是能力治理的唯一入口。

例如：

- 默认工作目录在哪里。
- 是否启用 working directory isolation。
- 某个工具的默认超时是多少。
- sandbox 路由如何注入。
- channel registry 和 database 如何注入。

这些都发生在注册阶段，而不是分散在循环内部。结果就是：当你要新增工具时，只要它满足既有契约，就可以接入统一治理链。

## 为什么工具需要显式执行上下文

单看模型返回的 JSON，工具好像只需要输入参数。但在真实系统里，这远远不够。MicroClaw 明确把授权信息注入到工具输入中，例如 `__microclaw_auth`，并通过辅助函数解析：

- 调用方 chat_id
- 跨 chat 是否允许
- 是否属于 control chat

这一步解决了 Agent 系统里一个非常容易被忽视的问题：模型不能被信任来声明权限边界。

如果让模型自己传“我要操作哪个 chat_id”，那它生成错误参数时，就可能越权访问别的会话。MicroClaw 的做法是：

- 模型只负责表达意图。
- runtime 负责补充真实的调用者上下文。
- 工具执行前统一做授权判定。

这使得工具权限来自运行时事实，而不是来自模型自述。

## 工具分层：能力不是一个平面

要治理工具，首先得承认不同工具风险并不相同。MicroClaw 至少把工具分成了几类不同性质。

### 第一类：纯读取或低副作用工具

例如 `read_file`、`glob`、`grep`、`web_search`。这类工具通常主要产生信息增量，副作用较弱，适合频繁使用。

### 第二类：本地状态修改工具

例如 `write_file`、`edit_file`、`write_memory`、`todo_write`。它们会改变系统状态，但范围相对局部，可通过路径限制、scope 限制和审计进行约束。

### 第三类：高风险执行工具

`bash` 是最典型的代表。它既可能完成大量有价值工作，也最容易扩大失控面。因此它需要额外的超时、审批和 sandbox 配合。

### 第四类：跨会话与主动外发工具

例如 `send_message`、调度工具、子代理控制工具。这类工具的风险不只来自本地副作用，还来自跨 chat 影响和主动发送行为，因此必须绑定权限模型。

把工具按这种风险结构理解，你就不会再问“为什么有些工具要审批，有些不用”。这不是体验不一致，而是风险治理本来就应该分层。

## 高风险审批、执行策略与 sandbox

### 审批门：把危险动作从“默认可执行”改成“默认需要确认”

在统一循环里，当工具返回 `approval_required` 时，系统会结合配置和最近用户文本做下一步决策。这里最重要的默认配置是：

- `high_risk_tool_user_confirmation_required = true`

这意味着高风险动作不再是“只要模型想做就做”，而是“必须满足明确授权条件才做”。从工程视角看，这比单纯在 prompt 里写“危险操作前要确认”强得多，因为真正的控制点在 runtime，而不是在模型服从性上。

### 执行策略：`host_only`、`sandbox_only`、`dual`

工具和插件执行时会带执行策略元数据。`docs/security/execution-model.md` 描述了当前策略基线：

- `bash` 是 `dual`
- 写文件和编辑文件当前仍偏 `host-only`
- 其它大多按当前实现走 `host-only`

这个语义的价值，不在于把所有工具都塞进容器，而在于允许系统表达：

- 某类操作必须在 sandbox 执行。
- 某类操作只能在宿主机执行。
- 某类操作可根据环境降级。

这是后续做安全 profile、平台兼容和调试模式时非常重要的基础。

### Sandbox 不是万能防线，但它是关键防线

`ToolRegistry` 在创建时会初始化 `SandboxRouter`。配合 `docs/security/execution-model.md` 可以看到，当前 sandbox 能力已经覆盖：

- backend 选择：`auto` / `docker` / `podman`
- 安全 profile：`hardened` / `standard` / `privileged`
- 挂载校验
- 敏感路径阻断
- symlink 验证
- allowlist 扩展

这说明 MicroClaw 的安全哲学不是“完全信任容器”，也不是“完全依赖 prompt”。它把 sandbox 放在多层防线中的中间位置：重要，但不是唯一。

## Path Guard 与网络校验：两个最容易被低估的细节

### 文件路径限制

本地文件工具真正危险的地方，不是“会写文件”，而是“会写到哪里”。`microclaw-tools` 把路径解析、工作目录隔离和敏感路径限制下沉成通用能力，这比在每个文件工具里各写一次 if 判断更可靠。

### `web_fetch` 的内容与 URL 双重校验

`docs/security/execution-model.md` 和运行手册都强调了 `web_fetch` 的两层控制：

1. URL 策略校验
2. 抓取后内容校验

这是一个非常好的设计细节。因为对 Agent 来说，网页不仅是数据源，也可能是提示注入源。只防 SSRF 不够，只做内容清洗也不够，必须两层同时存在。

## Hook：工具治理的可插拔层

MicroClaw 的 Hook 不是另一个“技能系统变种”，它更接近策略注入器。`docs/hooks/HOOK.md` 规定了三个关键事件：

- `BeforeLLMCall`
- `BeforeToolCall`
- `AfterToolCall`

其中和工具系统直接相关的是后两者。

### 为什么 Hook 有治理价值

因为很多策略不适合写死在工具实现里。例如：

- 某个团队要求所有 shell 操作都附带风险摘要。
- 某个部署环境临时禁用某个工具。
- 某个工具结果需要做脱敏或补充标记。

如果这些要求都写进工具本身，工具很快就会被部署环境污染；如果完全放在提示词里，又没有执行保证。Hook 的位置刚好在中间：

- 足够靠近执行点，可以真正阻断或修改。
- 又不必污染每个工具实现。

这类机制的价值，会在团队规模和部署环境增加时变得越来越明显。

## 新增工具时，应该怎么判断“算不算好工具”

一个合格的工具，不只是“能跑通功能”。至少要满足下面几条：

1. 输入 schema 足够清晰，模型不容易误用。
2. 成功和失败结果都能被统一处理。
3. 风险级别明确，知道是否需要审批。
4. 能接受 runtime 注入的授权上下文，而不是自信地相信模型参数。
5. 有明确的超时、路径或执行环境边界。

如果这些条件缺失，工具表面上接进了系统，实际上会增加大量不可观测和不可治理的灰区。

## 示例代码：为什么所有工具都要服从同一份契约？

工具系统最核心的约束，不是“能不能跑”，而是“能不能被统一治理”。一旦某个工具绕开统一描述和上下文注入，它表面上是一个能力，实际上是一个治理缺口。

Rust 版本让工具 trait 明确接收执行上下文，再由注册表统一调度。这样授权信息、风险级别和超时预算都有地方进入，而不会被偷偷埋在工具实现细节里。

```rust
#[derive(Clone)]
struct ToolContext {
    working_dir: String,
    approved: bool,
}

#[async_trait::async_trait]
trait Tool {
    fn name(&self) -> &str;
    async fn execute(&self, input: serde_json::Value, ctx: &ToolContext) -> anyhow::Result<String>;
}

struct ReadFileTool;
```

先把上下文和契约单独拿出来，能更直接说明工具为什么必须接受统一治理。没有这层边界，具体实现再简洁，也只是绕开 runtime 的局部特例。

```rust

#[async_trait::async_trait]
impl Tool for ReadFileTool {
    fn name(&self) -> &str {
        "read_file"
    }

    async fn execute(&self, input: serde_json::Value, ctx: &ToolContext) -> anyhow::Result<String> {
        let relative = input["path"].as_str().unwrap_or_default();
        let path = format!("{}/{}", ctx.working_dir, relative);
        Ok(std::fs::read_to_string(path)?)
    }
}
```

Python 版本用 `Protocol` 标出同样的接口，再用 `@dataclass` 承载上下文。这样动态语言里的示例也能清楚表达一个事实：真正被 runtime 管理的是“工具 + 执行条件”，不是裸函数本身。

```python
from dataclasses import dataclass
from typing import Any, Protocol


@dataclass
class ToolContext:
    working_dir: str
    approved: bool


class Tool(Protocol):
    name: str

    async def execute(self, input: dict[str, Any], ctx: ToolContext) -> str: ...


class ReadFileTool:
    name = "read_file"
```

Python 同样先定义上下文和接口，再落具体工具。这样读者会先看到“工具受什么约束”，再看到“单个工具如何工作”，章节主旨更清楚。

```python

    async def execute(self, input: dict[str, Any], ctx: ToolContext) -> str:
        path = f"{ctx.working_dir}/{input['path']}"
        with open(path, "r", encoding="utf-8") as f:
            return f.read()
```

## 关键权衡

### 决策一：所有工具都遵循统一 trait 和结果结构

优点是核心循环、日志、审计、Provider 暴露都能统一处理。代价是工具实现需要服从更多约束，自由度降低。

### 决策二：把授权上下文由 runtime 注入，而不是由模型声明

优点是权限边界更可信。代价是工具接口设计稍复杂，不能只看模型生成的参数。

### 决策三：审批、sandbox、path guard 叠加使用

优点是形成多层防线。代价是系统行为不再是“有一个大开关”，排查时需要理解多层策略交互。

### 决策四：用 Hook 承载部署级策略

优点是治理逻辑不污染工具实现。代价是 Hook 也需要自己的文档、测试和故障隔离。

## 容易走错的地方

### 失败模式 1：把工具当成简单函数调用

这样做会忽视 schema、结果结构、审计、权限和超时等系统性需求。

### 失败模式 2：相信模型参数天然可信

模型生成的 `chat_id`、路径或命令都不应直接当成事实使用。

### 失败模式 3：把 sandbox 当成唯一安全手段

没有审批、路径限制和授权上下文，单靠 sandbox 仍然会留下大量治理盲区。

## 小结

MicroClaw 的工具系统真正有价值的地方，不是它“内置了很多工具”，而是它把工具做成了 runtime 可以统一授权、审批、隔离、审计和扩展的能力层。只有在这样的前提下，Agent 才配得上“可执行系统”这个称呼。

下一章我们继续沿着这条主链路前进，讨论工具系统经常打交道的另一层核心能力：记忆。对长期运行的 Agent 来说，记忆不是锦上添花，而是决定执行质量能否积累的基础设施。

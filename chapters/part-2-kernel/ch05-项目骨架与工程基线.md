# Chapter 5 项目骨架与工程基线

## 这一章要回答什么问题

技术选型回答了“为什么这样选”，但还没有回答“代码到底怎么组织”。这一步非常关键。很多 Agent 项目死得并不突然，而是随着功能增加逐渐失去可读性：入口越来越多，配置越来越散，状态在不同模块里重复定义，最后谁也说不清系统是怎么启动的。

MicroClaw 当前的仓库结构，正好处在一个很有代表性的阶段：它已经超过“玩具项目”的复杂度，但仍然努力维持单进程 runtime 的清晰主链路。这一章读完后，你应该已经掌握三件事：

1. 从 CLI 到运行时启动的完整路径。
2. workspace 中各个 crate 的职责边界。
3. 为什么 `doctor`、自检、日志、默认配置这些看似琐碎的部分，其实就是工程基线。

## 从 `main.rs` 到 `runtime::run`：启动链路如何收口

MicroClaw 的运行入口在 `src/main.rs`。这个文件不是简单的参数转发器，而是整个系统的“启动编排器”。当前 CLI 已经包含：

- `microclaw start`
- `microclaw acp`
- `microclaw setup`
- `microclaw doctor`
- `microclaw gateway`
- `microclaw skill`
- `microclaw hooks`
- `microclaw web`
- `microclaw reembed`
- `microclaw upgrade`

这说明入口层承担的职责，已经不只是“启动 bot”，而是管理整个 runtime 生命周期。

对书里的主线来说，最重要的是 `start` 路径。它大致会完成几步：

1. 读取配置并解析默认值。
2. 初始化运行时数据目录。
3. 打开数据库并准备 schema。
4. 初始化记忆、技能、MCP、日志等组件。
5. 进入 `runtime::run` 完成渠道、工具、Web、后台任务的装配。

这个链路的关键好处，是它把“系统何时进入可服务状态”压缩到单个显式入口。如果你以后要加一个新能力，例如新的 observability exporter 或新的后台守护任务，就能比较清楚地知道它应该挂在启动链的哪一层，而不是散落在多个渠道适配器里偷偷初始化。

### 小例子：为什么启动链需要一个显式 bootstrap？

很多项目的启动逻辑失败，不是因为少了某个功能，而是因为初始化顺序散落在多个入口里。把启动前动作收进一个 bootstrap 对象，可以让“准备环境”和“进入服务状态”成为两件可审计的事。

Rust 版本用 `Bootstrap` 显式表示启动前阶段。这样数据目录、数据库和配置检查都可以在进入 runtime 之前完成，而不是等某个请求第一次打进来才暴露问题。

```rust
struct Bootstrap {
    config: Config,
}

impl Bootstrap {
    async fn prepare(&self) -> anyhow::Result<()> {
        ensure_data_dir(&self.config).await?;
        ensure_database(&self.config).await?;
        validate_config(&self.config)?;
        Ok(())
    }
}
```

Python 版本用 `@dataclass` 保持同一思路。这样即使语言允许更随意的初始化方式，示例也仍然强调启动链应该是一个显式阶段，而不是一串隐藏副作用。

```python
from dataclasses import dataclass


@dataclass
class Bootstrap:
    config: Config

    async def prepare(self) -> None:
        await ensure_data_dir(self.config)
        await ensure_database(self.config)
        validate_config(self.config)
```

## Workspace 的真实职责边界

从根 `Cargo.toml` 看，MicroClaw 已经把通用能力抽到多个 crate。理解这些边界，比死记目录树更重要。

### `microclaw-core`：共享类型和错误语义

`crates/microclaw-core` 主要承载跨模块共享、且不依赖具体运行时环境的内容，例如：

- 错误类型
- 文本处理工具
- LLM 消息和工具调用相关数据结构

这类内容如果仍然放在顶层应用里，很快会造成循环依赖。抽到 core 的意义不是“形式上更优雅”，而是让上层模块可以共享同一套领域语言。

### `microclaw-storage`：统一状态持久化

`crates/microclaw-storage/src/db.rs` 是全书最值得反复阅读的文件之一。它把消息、会话、任务、记忆、指标、审计、子代理等状态统一管理，并通过 `call_blocking` 把同步 SQLite 操作包装进异步运行时。

这层的价值是双重的：

- 统一数据库语义，避免每个模块自行拼 SQL。
- 给上层暴露稳定 API，降低 schema 变更扩散面。

### `microclaw-tools`：执行时风险控制内核

`crates/microclaw-tools` 并不只是“工具函数集合”。它沉淀的是工具系统的底层规则：

- `Tool` trait 运行时契约
- 路径守卫（path guard）
- 命令执行器
- sandbox 路由
- web 内容校验
- 工具风险和执行策略语义

也就是说，真正决定“工具能不能安全执行”的共性逻辑，并不在某个具体 `bash` 工具文件里，而是在这个 crate 里。

### `microclaw-channels`：通道抽象边界

`crates/microclaw-channels/src/channel_adapter.rs` 定义了 `ChannelAdapter` trait 和 `ChannelRegistry`。顶层 `src/channels/*.rs` 则负责具体平台的 runtime context 和适配器实现。

这种拆法很合理：

- trait 和路由属于稳定抽象，适合下沉。
- 平台细节变化快，保留在应用层更灵活。

### `microclaw-observability`：把采集与导出剥离

随着 Metrics、Traces、Logs 增加，如果仍全部写在顶层应用里，业务代码会不断和 OTLP、平台适配、资源语义耦合。`crates/microclaw-observability` 把这些导出能力单独封装后，`runtime.rs` 只需要决定“要不要启用 exporter”，而不是直接理解每种导出协议。

### 顶层 `src/`：高层编排，不是杂物间

理解了上面四层后，顶层 `src/` 的定位就清楚了。它保留的是那些必须从全局视角编排的模块：

- `main.rs`：CLI 入口
- `runtime.rs`：运行时装配
- `agent_engine.rs`：统一循环
- `llm.rs`：Provider 适配
- `scheduler.rs`：后台任务与反射器
- `web.rs`：Web 控制面

这是一种很成熟的骨架：把“基础能力”做成 crate，把“主链路协调”留在应用层。

## `AppState`：为什么统一装配层是核心

`runtime.rs` 中的 `AppState` 可以看作全系统的依赖中枢。它集中持有：

- `Config`
- `ChannelRegistry`
- `Database`
- `MemoryManager`
- `SkillManager`
- `HookManager`
- `LlmProvider`
- provider/model override
- `MemoryBackend`
- `ToolRegistry`
- Metrics / Traces / Logs exporter

如果没有这层，渠道、Web、Scheduler、Subagent、工具系统很快就会各自维护一套依赖组合，最后形成重复初始化和行为不一致。

`AppState` 的真正价值不是“集中放东西”，而是把系统装配变成显式行为。这样一来：

- 你知道某个能力在启动时是否已准备好。
- 你可以从类型层面追踪谁依赖谁。
- 你可以在测试里构造最小运行时状态。

对一本讲“构建 runtime”的书来说，这一点比任何设计图都更重要。真正能让系统长期演进的，往往就是这种朴素而严格的装配层。

## 配置系统与默认值策略

`src/config.rs` 很长，但它体现了一个清晰的原则：默认值不是附属品，而是产品能力的一部分。

以几个典型配置为例：

- `max_tool_iterations` 默认 `100`
- `max_session_messages` 默认 `40`
- `compact_keep_recent` 默认 `20`
- `default_tool_timeout_secs` 默认 `30`
- `high_risk_tool_user_confirmation_required` 默认 `true`
- `web_enabled` 默认 `true`
- `web_port` 默认 `10961`
- `subagent_max_concurrent` 默认 `4`

这些值告诉我们两件事。

第一，MicroClaw 不是“你自己从零组装所有策略”的裸框架。它试图给出一套可以直接运行的缺省姿态。

第二，默认值本身就在表达产品立场。例如高风险工具确认默认开启，说明它把执行安全放在默认路径，而不是专家路径；Web 控制面默认开启，则说明本地可观测和控制面是核心功能，不是附加组件。

更进一步，配置系统还承担了 profile 和兼容层的职责。Provider preset、channel model override、sandbox 配置、observability 配置、plugins 配置等，都是通过统一的配置结构暴露给运行时。这让“新功能是否进入系统”不依赖硬编码开关，而依赖一致的配置装配机制。

## 日志、`doctor` 与自检为什么属于工程基线

很多团队把这些内容放到项目后期再补，但 MicroClaw 的 CLI 从一开始就把它们做成正式能力。

### `doctor` 不是锦上添花，而是启动前门槛

README 和 `DEVELOP.md` 都把 `microclaw doctor` 放在高优先级位置。它会检查：

- PATH 和 shell 环境
- sandbox 运行时
- MCP 依赖
- 平台前置条件

这类检查的核心价值，是把“运行失败”尽量提前到启动前，而不是让用户在第一轮消息处理时才发现环境缺失。

### 自检接口把策略问题显式化

`web` 控制面提供配置自检、风险等级和告警信息。也就是说，系统不只允许配置，还会对配置进行评估。这和单纯提供一个 YAML 文件完全不同。

对 Agent Runtime 来说，危险从来不只来自代码 bug，也来自误配置。把自检做成一等能力，等于把“可托管性”前移到了工程骨架层。

### 日志和追踪不是后期补丁

当 `runtime.rs` 在装配 `OtlpMetricExporter`、`OtlpTraceExporter`、`OtlpLogExporter` 时，其实已经表达了一个很明确的态度：可观测性属于主链路，不属于“有空再接”。

这类设计会直接影响后续写代码的方式。因为一旦你知道某个模块最终要进入统一日志和追踪体系，你在接口设计时就更不容易偷懒。

## 工程基线应该长什么样

如果你要按 MicroClaw 的思路搭自己的项目，至少应该固化下面这条基线：

1. 一个稳定的 CLI 入口，能把 setup、start、doctor、运维管理统一进来。
2. 一个统一的 `AppState` 或等价装配层。
3. 一个独立的存储层 crate 或模块，禁止在业务代码里散落 SQL。
4. 一个能显式表达默认值和风险开关的配置系统。
5. 启动前诊断、自检接口和最基本的日志/追踪导出能力。

只有有了这条基线，后面加渠道、加插件、加调度、加记忆时，系统才不会立刻变形。

## 示例代码：为什么 `AppState` 必须成为唯一装配入口？

这一组代码故意只保留最少字段，因为关键不在“塞多少依赖”，而在“所有依赖是不是从同一处进入系统”。只要装配入口分裂，启动链路、测试替身和运维自检都会迅速发散。

Rust 版本让 `AppState` 直接持有核心依赖，再用一个异步预热方法表达启动时必须完成的动作。这样“对象构造”和“运行前准备”是分开的，工程基线就不会被随手写进 `main.rs`。

```rust
#[async_trait::async_trait]
trait LlmProvider {
    async fn warm_up(&self) -> anyhow::Result<()>;
}

struct AppState {
    db: Database,
    llm: Box<dyn LlmProvider>,
    tools: ToolRegistry,
    config: Config,
}
```

这一段先把装配入口自身的形状固定下来。只有先看清 `AppState` 真正持有哪些依赖，读者才会明白为什么启动基线必须从同一个入口进入系统。

```rust

impl AppState {
    fn new(config: Config) -> Self {
        let db = Database::open(&config.data_dir);
        let llm = create_provider(&config);
        let tools = ToolRegistry::new(&config);
        Self { db, llm, tools, config }
    }

    async fn warm_up(&self) -> anyhow::Result<()> {
        self.db.ping().await?;
        self.llm.warm_up().await?;
        self.tools.self_check().await?;
        Ok(())
    }
}
```

```{=typst}
#pagebreak(weak: true)
```

Python 版本用 `Protocol` 和 `@dataclass` 保持同一层次关系。这样即使实现语言更宽松，装配层仍然是一个明确对象，而不是启动脚本里一串彼此共享局部变量的初始化语句。

```python
from dataclasses import dataclass
from typing import Protocol


class LlmProvider(Protocol):
    async def warm_up(self) -> None: ...


@dataclass
class AppState:
    config: Config
    db: Database
    llm: LlmProvider
    tools: ToolRegistry
```

Python 也先放装配对象，再放构造和预热流程。这样“对象边界”和“启动动作”不会糊在一起，更适合排版，也更适合工程读者顺着职责理解。

```python

    @classmethod
    def build(cls, config: Config) -> "AppState":
        return cls(
            config=config,
            db=Database.open(config.data_dir),
            llm=create_provider(config),
            tools=ToolRegistry(config),
        )

    async def warm_up(self) -> None:
        await self.db.ping()
        await self.llm.warm_up()
        await self.tools.self_check()
```

## 关键权衡

### 决策一：保留单二进制启动路径

优点是部署和调试简单，状态边界集中。代价是单进程内要同时处理更多职责，必须严控模块边界。

### 决策二：通过 workspace 拆 crate，而不是拆服务

优点是可以先治理代码复杂度，再决定是否服务化。代价是 crate 之间的接口设计需要更自觉，否则只会把复杂度从目录树挪到依赖关系里。

### 决策三：让配置默认值承载产品立场

优点是上手路径更顺滑，也更安全。代价是默认值需要持续维护，一旦过时就会误导用户。

### 决策四：把 `doctor`、自检、日志纳入主干

优点是运维能力从 Day 1 就存在。代价是前期开发必须投入更多“看起来不直接产出功能”的工作。

## 容易走错的地方

### 失败模式 1：把骨架理解成目录美化

目录好看没有价值，真正重要的是依赖方向、状态归属和启动路径是否清楚。

### 失败模式 2：让每个模块自行初始化依赖

一旦数据库、配置、工具、日志在多个入口分别初始化，行为一致性很快就会消失。

### 失败模式 3：只做功能，不做诊断

没有 `doctor` 和自检，很多问题会被拖到线上或用户首次运行时暴露，排查成本会急剧上升。

## 读到这里，你应该能回答

- 你是否能从 `main.rs` 说清 `start` 到 `runtime::run` 的启动链？
- 你是否理解每个 crate 的职责边界，而不是只会背目录名？
- 你是否有统一装配层，而不是让依赖在多个入口里自行生长？
- 你是否把默认值、诊断、自检和日志视为工程基线，而不是“后面再补”的辅助项？

## 小结

MicroClaw 的工程骨架之所以值得学习，不在于它用了 workspace，而在于它把“运行时装配、状态存储、风险控制、可观测、诊断入口”这些基础设施在很早阶段就固化下来。这样一来，后面的所有能力扩展，都能围绕一条相对稳定的主链路生长。

下一章我们进入这条主链路的核心，也就是 `agent_engine.rs` 里的统一循环。到了那里，你会看到仓库骨架是如何真正支撑一次消息处理闭环的。

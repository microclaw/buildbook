# Chapter 5 项目骨架与工程基线

## 启动链路

```
CLI 参数 → Config 加载/校验 → 数据目录准备 → 日志初始化
  → Database → MemoryManager → SkillManager → McpManager
    → runtime::run（AppState 装配 + 渠道启动 + 后台任务）
```

`Config::load()` 失败时自动启动 `setup::run_setup_wizard()`。`acp` 路径共享同一套准备链，仅最后入口不同。

```rust
struct Bootstrap {
    config: Config,
}

impl Bootstrap {
    async fn prepare(&self) -> anyhow::Result<()> {
        ensure_data_dirs(&self.config).await?;
        let db = Database::new(&self.config.runtime_data_dir())?;
        let memory = MemoryManager::new(&self.config.runtime_data_dir());
        let skills = SkillManager::from_skills_and_runtime(
            &self.config.skills_data_dir(),
            &self.config.runtime_data_dir(),
        );
        validate_config(&self.config)?;
        Ok(())
    }
}
```

## 8 个 Crate 的职责边界

| Crate | 职责 | 关键类型 |
| --- | --- | --- |
| `microclaw-core` | 共享类型、错误语义 | `Message`、`ContentBlock`、`ToolDefinition` |
| `microclaw-storage` | 统一持久化 | `Database`、`Memory`、`SessionSettings` |
| `microclaw-tools` | 执行时风险控制内核 | `Tool` trait、`ToolConcurrencyClass`、`SandboxRouter` |
| `microclaw-channels` | 通道抽象 | `ChannelAdapter` trait、`ChannelRegistry` |
| `microclaw-clawhub` | 技能市场客户端 | `clawhub_search`、`clawhub_install` |
| `microclaw-app` | 应用级辅助 | 内置 skills、日志初始化、语音转录 |
| `microclaw-observability` | 采集与导出 | `OtlpMetricExporter`、`OtlpTraceExporter` |
| 顶层 `src/` | 高层编排 | `AppState`、`agent_engine`、`runtime` |

依赖方向单向：`microclaw-tools` 不依赖 `agent_engine`，`microclaw-storage` 不依赖渠道层。

## `AppState`：统一装配层

```rust
pub struct AppState {
    pub config: Config,
    pub channel_registry: Arc<ChannelRegistry>,
    pub db: Arc<Database>,
    pub memory: MemoryManager,
    pub skills: SkillManager,
    pub hooks: Arc<HookManager>,
    pub llm: Box<dyn LlmProvider>,
    pub llm_provider_overrides: Arc<RwLock<HashMap<String, String>>>,
    pub llm_model_overrides: Arc<RwLock<HashMap<String, String>>>,
    pub embedding: Option<Arc<dyn EmbeddingProvider>>,
    pub memory_backend: Arc<MemoryBackend>,
    pub tools: ToolRegistry,
    pub chat_turn_queue: Arc<ChatTurnQueue>,
    pub metric_exporter: Option<Arc<OtlpMetricExporter>>,
    pub trace_exporter: Option<Arc<OtlpTraceExporter>>,
    pub log_exporter: Option<Arc<OtlpLogExporter>>,
}
```

| 字段 | 设计意图 |
| --- | --- |
| `chat_turn_queue` | 同一 chat 同时只有一个 agent run |
| `memory_backend` | 支持本地 SQLite 或 MCP 外接 |
| `llm_provider_overrides` / `llm_model_overrides` | 不同渠道使用不同模型 |
| `tools` | 统一管理 44 个内置工具 |

`runtime::run` 构造 `AppState` 后通过 `spawn_guarded` 启动各渠道（catch panic，渠道崩溃不拖垮其他渠道）。

## 配置默认值

| 配置项 | 默认值 | 含义 |
| --- | --- | --- |
| `max_tool_iterations` | 100 | 单次 agent 运行最大工具循环次数 |
| `max_session_messages` | 40 | 触发 compaction 的消息阈值 |
| `parallel_tool_max_concurrency` | 8 | 并行工具执行最大并发数 |
| `high_risk_tool_user_confirmation_required` | true | 高风险工具需要用户确认 |
| `working_dir_isolation` | Chat | 工作目录隔离粒度 |
| `subagent_max_spawn_depth` | 1 | 子代理最大嵌套深度 |
| `reflector_interval_mins` | 15 | 记忆反射器运行间隔 |
| `memory_token_budget` | 1500 | 记忆注入 token 预算 |

默认值表达产品立场：高风险确认默认开启 = 安全优先；Chat 级隔离 = 多会话安全是默认行为。

## 工程基线三件套

- **`doctor`**：启动前检查 PATH、sandbox、MCP 依赖。
- **`setup`**：全屏引导向导，`Config::load()` 失败时自动回退。
- **`gateway`**：`install/start/stop/status/logs`，系统服务化运行。
- **可观测性**：OTLP exporter 在 `runtime.rs` 装配，属于主链路。

impl AppState {
    fn new(config: Config) -> Self {
        let db = Arc::new(Database::open(&config.data_dir));
        let llm = create_provider(&config);
        let tools = ToolRegistry::new(&config);
        let chat_turn_queue = Arc::new(ChatTurnQueue::new(config.chat_turn_queue_max_pending));
        let memory_backend = Arc::new(MemoryBackend::local_only(db.clone()));
        Self { db, llm, tools, chat_turn_queue, memory_backend, config }
    }

    async fn warm_up(&self) -> anyhow::Result<()> {
        self.db.ping().await?;
        self.llm.warm_up().await?;
        self.tools.self_check().await?;
        Ok(())
    }
}

## 关键权衡

| 决策 | 优点 | 代价 |
| --- | --- | --- |
| 单二进制启动路径 | 部署简单、状态集中 | 单进程内职责多，需 `spawn_guarded` 隔离 |
| workspace 拆 crate 而非拆服务 | 先治代码复杂度，再决定是否服务化 | crate 接口设计需更自觉 |
| 默认值承载产品立场 | 上手顺滑、更安全 | 默认值需持续维护 |
| `doctor`/`setup`/`gateway` 纳入主干 | 运维能力从 Day 1 存在 | 前期需投入更多非功能工作 |

## 容易走错的地方

1. **把骨架理解成目录美化**：crate 之间依赖方向的单向性才是工程价值。只模仿目录布局却让 crate 互相 import，结果比单文件更糟。
2. **让每个模块自行初始化依赖**：`start` 和 `acp` 共享同一套初始化链就是正面案例。
3. **只做功能不做诊断**：`microclaw doctor sandbox` 能在启动前告诉你容器运行时是否可用。
4. **忽视遗留布局迁移**：`migrate_legacy_runtime_layout` 保证跨版本升级时数据不丢失。

## 小结

MicroClaw 把运行时装配、状态存储、并发控制、风险治理、可观测和诊断在早期固化。8 个 crate 让核心契约可被严格管理。

# Chapter 5 项目骨架与工程基线

## 代码到底怎么组织，系统又是怎么启动的？

技术选型回答了"为什么这样选"，但还没有回答"代码到底怎么组织"。很多 Agent 项目死得并不突然，而是随着功能增加逐渐失去可读性：入口越来越多，配置越来越散，状态在不同模块里重复定义，最后谁也说不清系统是怎么启动的。

MicroClaw v0.1.38 的仓库已经是一个相当成熟的 Rust workspace：8 个 crate、12 个 CLI 子命令、覆盖 16 个渠道适配器。这种规模正好处在一个非常有代表性的阶段——已经远超玩具项目，但仍然坚持单进程 runtime 的清晰主链路。

这一章读完后，你应该掌握三件事：

1. 从 CLI 到运行时启动的完整路径，包括每条子命令承担的职责。
2. 8 个 workspace crate 的职责边界与依赖方向。
3. 为什么 `doctor`、`setup`、`gateway`、日志、默认配置这些看似琐碎的部分，其实就是工程基线。

## 从 `main.rs` 到 `runtime::run`：启动链路如何收口

MicroClaw 的运行入口在 `src/main.rs`。这个文件不是简单的参数转发器，而是整个系统的"启动编排器"。v0.1.38 的 CLI 已经包含 12 个子命令：

- `microclaw start` — 启动 runtime，进入正常服务
- `microclaw acp` — 以 Agent Client Protocol 模式通过 stdio 服务
- `microclaw setup` — 全屏引导向导（或 `setup --enable-sandbox`）
- `microclaw doctor` — 启动前诊断
- `microclaw gateway` — 管理系统服务（install/start/stop/status/logs）
- `microclaw skill` — 管理 ClawHub 技能（search/install/list/inspect）
- `microclaw hooks` — 管理运行时 Hook（list/info/enable/disable）
- `microclaw weixin` — 管理微信原生登录状态（login/status/logout）
- `microclaw web` — 管理 Web UI 配置（password/password-generate/password-clear）
- `microclaw reembed` — 重新生成活跃记忆的向量索引
- `microclaw upgrade` — 升级到最新版本
- `microclaw version` — 打印版本号

这说明入口层承担的职责，已经不只是"启动 bot"，而是管理整个 runtime 生命周期加上运维周边能力。

对本书主线来说，最重要的是 `start` 路径。它大致完成以下几步：

1. 解析 CLI 参数，如有 `--config` 则覆盖默认配置路径。
2. 加载并校验 `Config`。如果配置不存在或无效，自动启动 `setup` 向导。
3. 确定 `data_root_dir`、`runtime_data_dir`、`skills_data_dir`，执行遗留布局迁移。
4. 根据是否 gateway 模式选择日志初始化策略（文件日志 vs 控制台日志）。
5. 确保内置 skills 存在。
6. 打开 `Database`（SQLite）。
7. 初始化 `MemoryManager`（文件记忆）。
8. 初始化 `SkillManager`，扫描可用技能。
9. 从 `mcp.json` + `mcp.d/*.json` 初始化 `McpManager`。
10. 进入 `runtime::run` 完成渠道、工具、Web、后台任务的装配。

`acp` 路径走的是几乎相同的准备链，只是最后进入 `acp::serve` 而不是 `runtime::run`。这意味着 ACP 和 start 共享同一套基础设施，差别仅在入口协议。

### 启动链为什么需要显式 bootstrap？

很多项目的启动逻辑失败，不是因为少了某个功能，而是因为初始化顺序散落在多个入口里。`main.rs` 把启动前动作收进一段线性流程，让"准备环境"和"进入服务状态"成为两件可审计的事。

Rust 版本用一个线性 bootstrap 区段表示启动前阶段。数据目录、数据库、记忆管理器和技能管理器都在进入 runtime 之前完成，而不是等某个请求第一次打进来才暴露问题。

```rust
struct Bootstrap {
    config: Config,
}

impl Bootstrap {
    async fn prepare(&self) -> anyhow::Result<()> {
        ensure_data_dirs(&self.config).await?;
        let db = Database::new(&self.config.runtime_data_dir())?;
        let memory = MemoryManager::new(&self.config.runtime_data_dir());
        let skills = SkillManager::from_skills_and_runtime(
            &self.config.skills_data_dir(),
            &self.config.runtime_data_dir(),
        );
        validate_config(&self.config)?;
        Ok(())
    }
}
```

Python 版本用 `@dataclass` 保持同一思路。这样即使语言允许更随意的初始化方式，示例也仍然强调启动链应该是一个显式阶段，而不是一串隐藏副作用。

```python
from dataclasses import dataclass


@dataclass
class Bootstrap:
    config: Config

    async def prepare(self) -> None:
        await ensure_data_dirs(self.config)
        db = Database.open(self.config.runtime_data_dir)
        memory = MemoryManager(self.config.runtime_data_dir)
        skills = SkillManager(self.config.skills_data_dir, self.config.runtime_data_dir)
        validate_config(self.config)
```

### Config::load() 失败时如何回退？

`main.rs` 中有一个关键设计：当 `Config::load()` 返回配置无效错误时，系统不是直接 panic，而是自动启动 `setup::run_setup_wizard()`。这意味着对第一次使用的用户，系统会主动引导，而不是给出一段晦涩的错误消息。这种"启动链渐进回退"在工程上比看起来更重要——它决定了新用户的第一印象。

## 8 个 Crate 各自负责什么？

从根 `Cargo.toml` 看，MicroClaw v0.1.38 的 workspace 包含 7 个 library crate 加主二进制，共 8 个单元。理解这些边界，比死记目录树更重要。

### `microclaw-core`：共享类型和错误语义

`crates/microclaw-core` 主要承载跨模块共享、且不依赖具体运行时环境的内容：

- 错误类型（`MicroClawError`）
- 文本处理工具（`floor_char_boundary` 等）
- LLM 消息和工具调用相关数据结构（`Message`、`ContentBlock`、`ImageSource`、`ToolDefinition`、`ResponseContentBlock`）

这类内容如果仍然放在顶层应用里，很快会造成循环依赖。抽到 core 的意义不是追求形式上的整洁，而是让上层模块可以共享同一套领域语言。特别是 `ContentBlock` 和 `ResponseContentBlock` 这对类型，它们定义了"请求侧消息结构"和"响应侧消息结构"的区别——前者用于构造发给模型的消息，后者用于解析模型返回。这种区分在工具循环中至关重要。

### `microclaw-storage`：统一状态持久化

`crates/microclaw-storage/src/db.rs` 是全书最值得反复阅读的文件之一。它把消息、会话、任务、记忆、指标、审计、子代理等状态统一管理，并通过 `call_blocking` 把同步 SQLite 操作包装进异步运行时。

这层的价值是双重的：

- 统一数据库语义，避免每个模块自行拼 SQL。
- 给上层暴露稳定 API，降低 schema 变更扩散面。

v0.1.38 新增了 `SessionSettings`（per-chat 会话配置）、`MemoryInjectionLog`（记忆注入审计）和 `memory_quality` 模块（记忆质量门控），这些都通过 storage crate 统一暴露。

### `microclaw-tools`：执行时风险控制内核

`crates/microclaw-tools` 不只是"工具函数集合"。它沉淀的是工具系统的底层规则：

- `Tool` trait 运行时契约
- `ToolConcurrencyClass` 三级并发分类（ReadOnly / SideEffect / Exclusive）
- `tool_concurrency_class()` 为每个内置工具分配默认并发类别
- 路径守卫（path guard）
- 命令执行器和 sandbox 路由（`SandboxRouter`、`SandboxMode`、`SecurityProfile`）
- web 内容校验（`WebContentValidationConfig`、`WebFetchUrlValidationConfig`）
- 工具风险和执行策略语义（`ToolRisk`、`tool_risk()`、`tool_execution_policy()`）

也就是说，真正决定"工具能不能安全执行"和"工具能不能并行执行"的共性逻辑，并不在某个具体工具文件里，而是在这个 crate 里。

### `microclaw-channels`：通道抽象边界

`crates/microclaw-channels/src/channel_adapter.rs` 定义了 `ChannelAdapter` trait 和 `ChannelRegistry`。顶层 `src/channels/*.rs` 则负责具体平台的 runtime context 和适配器实现——目前包括 Telegram、Discord、Slack、Feishu/Lark、DingTalk、Signal、WhatsApp、iMessage、Email、QQ、Nostr、IRC、Matrix（feature-gated）和微信原生。

这种拆法很合理：trait 和路由属于稳定抽象，适合下沉；平台细节变化快，保留在应用层更灵活。

### `microclaw-clawhub`：技能市场客户端

`crates/microclaw-clawhub` 封装了与 ClawHub 注册中心的交互能力：搜索、安装、校验、lockfile 管理。它让 `microclaw skill` 子命令和 agent 内置工具（`clawhub_search`、`clawhub_install`）共享同一套技能管理逻辑。

### `microclaw-app`：应用级辅助

`crates/microclaw-app` 包含内置 skills 管理、日志初始化和语音转录等应用级辅助能力。这些功能需要依赖较多外部库，但又不属于核心运行时路径，放在独立 crate 里可以避免主二进制编译链的膨胀。

### `microclaw-observability`：把采集与导出剥离

`crates/microclaw-observability` 封装了 OTLP 导出能力，包括 `OtlpMetricExporter`、`OtlpTraceExporter`、`OtlpLogExporter`。业务代码只需要决定"要不要启用 exporter"，而不是直接理解每种导出协议。从 `SpanData` 的定义可以看到，它已经包含 trace_id、span_id、parent_span_id、attributes、status——这是一套完整的分布式追踪语义。

### 顶层 `src/`：高层编排，不是杂物间

理解了上面七层后，顶层 `src/` 的定位就清楚了。它保留的是那些必须从全局视角编排的模块：

- `main.rs`：CLI 入口和启动编排
- `runtime.rs`：运行时装配，构造 `AppState`，启动渠道和后台任务
- `agent_engine.rs`：3541 行，统一循环核心
- `tool_executor.rs`：877 行，并行工具执行引擎
- `llm.rs`：Provider 适配
- `config.rs`：配置系统
- `hooks.rs`：Hook 管理器
- `run_control.rs`：运行取消与中止
- `chat_turn_queue.rs`：per-chat 轮次串行化
- `memory_service.rs`：记忆服务（显式记忆、检索注入、reflector）
- `memory_backend.rs`：记忆后端抽象（本地 SQLite 或 MCP 外接）
- `scheduler.rs`：调度器与 reflector
- `web.rs`：Web 控制面
- `tools/`：22 个工具模块文件

这是一种很成熟的骨架：把"基础能力"做成 crate，把"主链路协调"留在应用层。

## `AppState`：统一装配层为什么是核心？

`runtime.rs` 中的 `AppState` 是全系统的依赖中枢。它的字段列表值得完整展示：

```rust
pub struct AppState {
    pub config: Config,
    pub channel_registry: Arc<ChannelRegistry>,
    pub db: Arc<Database>,
    pub memory: MemoryManager,
    pub skills: SkillManager,
    pub hooks: Arc<HookManager>,
    pub llm: Box<dyn LlmProvider>,
    pub llm_provider_overrides: Arc<RwLock<HashMap<String, String>>>,
    pub llm_model_overrides: Arc<RwLock<HashMap<String, String>>>,
    pub embedding: Option<Arc<dyn EmbeddingProvider>>,
    pub memory_backend: Arc<MemoryBackend>,
    pub tools: ToolRegistry,
    pub chat_turn_queue: Arc<ChatTurnQueue>,
    pub metric_exporter: Option<Arc<OtlpMetricExporter>>,
    pub trace_exporter: Option<Arc<OtlpTraceExporter>>,
    pub log_exporter: Option<Arc<OtlpLogExporter>>,
}
```

相比 v0.1.16，v0.1.38 新增了几个关键字段：

- `chat_turn_queue`：per-chat 轮次串行化，确保同一个 chat 同时只有一个 agent run。
- `embedding`：可选的 embedding provider，用于向量化记忆检索。
- `memory_backend`：记忆后端抽象，支持本地 SQLite 或 MCP 外接。
- `llm_provider_overrides` / `llm_model_overrides`：per-channel 的 provider 和 model 覆盖，支持不同渠道使用不同模型。

`AppState` 的真正价值不是"集中放东西"，而是把系统装配变成显式行为。这样一来：

- 你知道某个能力在启动时是否已准备好。
- 你可以从类型层面追踪谁依赖谁。
- 你可以在测试里构造最小运行时状态。

`runtime::run` 函数负责构造 `AppState`，然后通过 `spawn_guarded` 启动各渠道 runtime。`spawn_guarded` 会 catch panic，确保一个渠道的崩溃不会拖垮其他渠道——这是单进程多渠道架构里非常关键的隔离策略。

## 配置系统与默认值策略

`src/config.rs` 超过 1000 行，但它体现了一个清晰的原则：默认值不是附属品，而是产品能力的一部分。

以 v0.1.38 中的典型配置为例：

| 配置项 | 默认值 | 含义 |
| --- | --- | --- |
| `max_tool_iterations` | 100 | 单次 agent 运行最大工具循环次数 |
| `max_session_messages` | 40 | 触发 compaction 的消息阈值 |
| `compact_keep_recent` | 20 | compaction 后保留最近消息数 |
| `default_tool_timeout_secs` | 30 | 工具默认超时 |
| `parallel_tool_max_concurrency` | 8 | 并行工具执行最大并发数 |
| `high_risk_tool_user_confirmation_required` | true | 高风险工具需要用户确认 |
| `web_enabled` | true | Web 控制面默认开启 |
| `web_port` | 10961 | Web 端口 |
| `subagent_max_concurrent` | 4 | 子代理最大并发数 |
| `subagent_max_spawn_depth` | 1 | 子代理最大嵌套深度 |
| `reflector_enabled` | true | 记忆反射器默认开启 |
| `reflector_interval_mins` | 15 | 反射器运行间隔 |
| `memory_token_budget` | 1500 | 记忆注入 token 预算 |
| `chat_turn_queue_max_pending` | 20 | per-chat 排队消息上限 |
| `working_dir_isolation` | Chat | 工作目录隔离粒度 |

这些值告诉我们两件事。

第一，MicroClaw 不是"你自己从零组装所有策略"的裸框架。它给出了一套可以直接运行的缺省姿态。

第二，默认值本身在表达产品立场。高风险工具确认默认开启，说明它把执行安全放在默认路径；Web 控制面默认开启，说明本地可观测和控制面是核心功能；工作目录 Chat 级隔离默认开启，说明多会话安全是默认行为，不是可选配置。

v0.1.38 还引入了 `LlmProviderProfile`（多 provider 配置）、`SubagentAcpConfig`（ACP 子代理目标配置）和 `tool_concurrency_overrides`（并发类别覆盖），这些都通过统一的配置结构暴露给运行时。

## `doctor`、`setup`、`gateway`：工程基线三件套

### `doctor` 不是锦上添花，而是启动前门槛

`microclaw doctor` 会检查 PATH 和 shell 环境、sandbox 运行时（Docker/Podman）、MCP 依赖和平台前置条件。这类检查的核心价值是把"运行失败"尽量提前到启动前。

### `setup` 让初始化成为正式流程

`microclaw setup` 不只是生成一个配置文件。它是一个全屏引导向导，帮助用户选择 provider、设置 API key、配置渠道。`setup --enable-sandbox` 则专门处理 sandbox 启用。当 `Config::load()` 失败时，`main.rs` 会自动回退到 setup——这意味着即使用户什么都不配，系统也不会以不可理解的方式失败。

### `gateway` 把服务化运维纳入 CLI

`microclaw gateway install/start/stop/status/logs` 让系统可以作为系统服务运行，而不是只能在前台 terminal 里跑。这对生产部署至关重要。

### 日志和追踪不是后期补丁

当 `runtime.rs` 在装配 `OtlpMetricExporter`、`OtlpTraceExporter`、`OtlpLogExporter` 时，它已经在说：可观测性属于主链路。`main.rs` 根据是否 gateway 模式选择不同的日志初始化策略（`init_logging` 写文件 vs `init_console_logging` 输出终端），这也说明日志系统从 Day 1 就考虑了多种部署形态。

## 示例代码：`AppState` 为什么必须成为唯一装配入口？

这一组代码故意只保留最少字段，因为关键不在"塞多少依赖"，而在"所有依赖是不是从同一处进入系统"。只要装配入口分裂，启动链路、测试替身和运维自检都会迅速发散。

Rust 版本让 `AppState` 直接持有核心依赖，包括 v0.1.38 新增的 `chat_turn_queue` 和 `memory_backend`。异步预热方法表达启动时必须完成的动作。

```rust
#[async_trait::async_trait]
trait LlmProvider {
    async fn warm_up(&self) -> anyhow::Result<()>;
}

struct AppState {
    db: Arc<Database>,
    llm: Box<dyn LlmProvider>,
    tools: ToolRegistry,
    chat_turn_queue: Arc<ChatTurnQueue>,
    memory_backend: Arc<MemoryBackend>,
    config: Config,
}
```

先把装配入口自身的形状固定下来。只有先看清 `AppState` 真正持有哪些依赖，读者才会明白为什么启动基线必须从同一个入口进入系统。

```rust

impl AppState {
    fn new(config: Config) -> Self {
        let db = Arc::new(Database::open(&config.data_dir));
        let llm = create_provider(&config);
        let tools = ToolRegistry::new(&config);
        let chat_turn_queue = Arc::new(ChatTurnQueue::new(config.chat_turn_queue_max_pending));
        let memory_backend = Arc::new(MemoryBackend::local_only(db.clone()));
        Self { db, llm, tools, chat_turn_queue, memory_backend, config }
    }

    async fn warm_up(&self) -> anyhow::Result<()> {
        self.db.ping().await?;
        self.llm.warm_up().await?;
        self.tools.self_check().await?;
        Ok(())
    }
}
```

## 关键权衡

### 决策一：保留单二进制启动路径

优点是部署和调试简单，状态边界集中。代价是单进程内要同时处理更多职责，必须严控模块边界。`spawn_guarded` 的 panic 隔离策略是这个决策的直接产物。

### 决策二：通过 workspace 拆 crate，而不是拆服务

优点是可以先治理代码复杂度，再决定是否服务化。代价是 crate 之间的接口设计需要更自觉，否则只会把复杂度从目录树挪到依赖关系里。v0.1.38 新增的 `microclaw-clawhub` 和 `microclaw-app` 正是这种演进的实例。

### 决策三：让配置默认值承载产品立场

优点是上手路径更顺滑，也更安全。代价是默认值需要持续维护。v0.1.38 把 `parallel_tool_max_concurrency` 默认设为 8、`working_dir_isolation` 默认设为 Chat 级，都是基于真实使用反馈做出的产品判断。

### 决策四：把 `doctor`、`setup`、`gateway` 纳入主干

优点是运维能力从 Day 1 就存在。代价是前期开发必须投入更多"看起来不直接产出功能"的工作。但回报是：用户第一次 `microclaw start` 就能获得引导而不是 panic。

## 容易走错的地方

### 失败模式 1：把骨架理解成目录美化

一个常见的误区是花大量时间调整目录结构和命名规范，却不去追问依赖方向是否正确、状态归属是否唯一、启动路径是否线性。MicroClaw 的 8-crate 划分之所以有效，不是因为目录好看，而是因为 `microclaw-tools` 不依赖 `agent_engine`、`microclaw-storage` 不依赖渠道层——这些依赖方向的单向性才是真正的工程价值。如果你只模仿目录布局，却让 crate 之间随意互相 import，结果会比单文件更糟。

### 失败模式 2：让每个模块自行初始化依赖

一旦数据库、配置、工具、日志在多个入口分别初始化，行为一致性很快就会消失。v0.1.38 中 `start` 和 `acp` 共享同一套初始化链就是正面案例。

### 失败模式 3：只做功能，不做诊断

没有 `doctor` 和自检，很多问题会被拖到用户首次运行时暴露。`microclaw doctor sandbox` 能在启动前告诉你容器运行时是否可用，这比"第一次执行 bash 工具时才发现 sandbox 不工作"省下大量排查时间。

### 失败模式 4：忽视遗留布局迁移

`main.rs` 中的 `migrate_legacy_runtime_layout` 和 `migrate_legacy_skills_dir` 看起来很琐碎，但它们保证了跨版本升级时数据不会丢失。这种"显式迁移"比"用户自己改目录"可靠得多。

## 读到这里，你应该能回答

- 你是否能从 `main.rs` 说清 `start` 到 `runtime::run` 的启动链？
- 你是否理解 8 个 crate 的职责边界，而不是只会背目录名？
- 你是否有统一装配层（`AppState`），而不是让依赖在多个入口里自行生长？
- 你是否把默认值、诊断、自检和日志视为工程基线，而不是"后面再补"的辅助项？
- 你是否理解 `start` 和 `acp` 为什么能共享同一套基础设施？

## 证据来源（v0.1.38）

- 核心源码路径：`src/main.rs`（CLI 入口、启动编排、遗留迁移）、`src/runtime.rs`（`AppState` 定义、`run` 函数、`spawn_guarded`）、`src/config.rs`（配置结构、默认值、provider profile）
- Crate 边界：`crates/microclaw-core`（共享类型）、`crates/microclaw-storage/src/db.rs`（统一持久化）、`crates/microclaw-tools/src/runtime.rs`（Tool trait、ToolConcurrencyClass）、`crates/microclaw-channels/src/channel_adapter.rs`（渠道抽象）、`crates/microclaw-clawhub`（ClawHub 客户端）、`crates/microclaw-app`（应用辅助）、`crates/microclaw-observability`（OTLP 导出）
- 关键配置项：`src/config.rs` 中 `parallel_tool_max_concurrency=8`、`working_dir_isolation=Chat`、`chat_turn_queue_max_pending=20`、`subagent_max_spawn_depth=1` 等默认值

## 小结

MicroClaw v0.1.38 的工程骨架之所以值得学习，不在于它用了 workspace，而在于它把"运行时装配、状态存储、并发控制、风险治理、可观测、诊断入口"这些基础设施在很早阶段就固化下来。8 个 crate 的划分不是形式主义，而是让 `Tool` trait、`ChannelAdapter` trait、`ToolConcurrencyClass` 等核心契约可以被严格管理。

下一章我们进入这条主链路的核心，也就是 `agent_engine.rs` 里的统一循环。到了那里，你会看到仓库骨架是如何真正支撑一次消息处理闭环的。

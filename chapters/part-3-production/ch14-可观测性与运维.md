# Chapter 14 可观测性与运维

## 这一章要回答什么问题

Agent Runtime 最让人头疼的地方，不是它会报错，而是它经常“看起来在运行，却不知道正在做什么”。一次请求可能跨越模型调用、工具执行、消息投递、后台任务和记忆提取；如果没有统一观测，你能看到的往往只有最后一句失败提示。

MicroClaw 当前的可观测设计已经相当完整：指标、追踪、日志、SLO、历史持久化、配置自检和运维手册都已经进入主干。这一章读完后，你应该理解：

1. MicroClaw 观察的不是服务器，而是运行时行为。
2. 为什么 Metrics、Traces、Logs 必须共用同一套语义。
3. 什么样的 SLO 才适合聊天型 Agent Runtime。
4. 运维体系为什么必须包括 runbook、告警阈值和回滚标准。

## 可观测性要观察什么

传统 Web 服务主要观察吞吐、错误率和资源占用。但对 Agent Runtime 来说，真正的风险还包括：

- 工具成功率是否下降
- MCP 是否被 rate limit 或熔断
- 会话是否在并发下失控
- Scheduler 是否在重启后正确恢复
- 高风险审批是否异常积压

因此，MicroClaw 的观察对象从一开始就不是“CPU 和内存”，而是运行时行为本身。

`docs/observability/architecture.md` 把这种设计分成三层：

- Metrics
- Traces
- Logs

并通过 `microclaw-observability` crate 统一导出。

## 统一三类信号：不要各说各话

`docs/observability/architecture.md` 最值得重视的一点，是它强调三类信号共享同一套资源语义和配置上下文。这并不是文档上的整洁，而是生产上的必要条件。

如果指标、追踪和日志分别用不同的 service name、不同的属性命名、不同的导出逻辑，运维团队最终会得到三套无法对齐的视图。MicroClaw 通过 `sdk.rs`、`metrics.rs`、`traces.rs`、`logs.rs` 把这些信号统一进一个 observability crate，目的就是避免这种碎片化。

### Metrics 适合回答什么

Metrics 负责高层趋势和健康度判断，例如：

- 请求成功率
- LLM tokens 输入输出
- 工具执行次数与失败率
- MCP 拒绝计数
- 活跃 session 数

### Traces 适合回答什么

Traces 负责一次请求的结构化路径。例如：

- 整个 `agent_run`
- 中间的 `llm_generation`
- 各次 `tool_execution`

这让团队能看到单次请求究竟卡在了哪里。

### Logs 适合回答什么

Logs 负责补充诊断细节，尤其是在失败、降级、超时、fallback 等场景下记录更细颗粒的解释。

三者合起来，才构成一个真正能用的观测面。

### 小例子：为什么 metric、trace、log 要共享同一组运行字段？

如果一次运行在 metrics 里叫 `agent_run`，在 trace 里又换一个名字，在日志里再少一个 `run_id`，后续排障几乎一定会失真。统一字段看起来琐碎，实际上是可观测体系能否闭环的前提。

Rust 版本用 `RunObservation` 把几种信号共用的标签收进一个 `struct`。这样无论发指标还是打日志，调用方都能围绕同一份上下文工作。

```rust
struct RunObservation {
    run_id: String,
    channel: String,
    chat_id: i64,
}

impl RunObservation {
    fn metric_key(&self, name: &str) -> String {
        format!("{name}:{}:{}", self.channel, self.chat_id)
    }
}
```

Python 版本用 `@dataclass` 表达同样的思路。示例的重点不是字符串拼接，而是告诉读者：统一语义必须先体现在运行上下文对象里。

```python
from dataclasses import dataclass


@dataclass
class RunObservation:
    run_id: str
    channel: str
    chat_id: int

    def metric_key(self, name: str) -> str:
        return f"{name}:{self.channel}:{self.chat_id}"
```

## 指标设计：围绕运行时，而不是围绕机器

`docs/observability/metrics.md` 和 `web.rs` 当前暴露的指标，已经体现出 MicroClaw 的重点关注面：

- `http_requests`
- `request_ok`
- `request_error`
- `llm_completions`
- `llm_input_tokens`
- `llm_output_tokens`
- `tool_executions`
- `tool_success`
- `tool_error`
- `tool_policy_blocks`
- `mcp_calls`
- `mcp_rate_limited_rejections`
- `mcp_bulkhead_rejections`
- `mcp_circuit_open_rejections`
- `active_sessions`

这组指标有一个很强的特点：它们不是基础设施视角，而是 runtime 视角。也就是说，系统更关心“Agent 有没有可靠完成工作”，而不是“机器此刻 CPU 百分之多少”。

这对 Agent 系统尤其重要，因为很多核心故障和资源占用没有直接线性关系。真正的生产问题往往发生在：

- 工具失败率持续抬高
- 某个 MCP server 反复熔断
- Web 控制面并发队列积压
- Scheduler 在重启后恢复失败

## SLO：系统到底什么算健康

`/api/metrics/summary` 暴露的 SLO 合约，是当前运维体系最成熟的部分之一。它至少定义了四个面向用户价值的目标：

- request success rate
- end-to-end latency P95
- tool reliability
- scheduler recoverability

这四个目标选得非常好，因为它们分别覆盖了：

- 对话可用性
- 交互体验
- 执行动作质量
- 后台任务可靠性

尤其是 `tool_reliability` 把 `approval_required` 和 `execution_policy_blocked` 排除在失败分母外，这个细节很重要。因为被策略主动阻止的操作，不应被算成“系统故障”。

这说明 MicroClaw 的 SLO 设计已经不只是计数，而是在表达产品立场。

## 指标历史持久化：观测不能只看当前瞬时值

MicroClaw 不只是提供 `/api/metrics` 快照，还会把 minute bucket 写入 SQLite 的 `metrics_history` 表。这样做有三个明显好处。

第一，可以查询过去 60 分钟或 24 小时趋势，而不依赖外部监控系统才能做最基本的回看。

第二，Web 控制面可以直接展示历史走势，而不用自己维护额外时序存储。

第三，即使外部 OTLP 或 Grafana 没配好，本地 runtime 仍有最基本的观测留痕。

这再次体现了 MicroClaw 的思路：先让本地自治能力成立，再逐步把信号导出到更大平台。

## Traces：把一次请求拆开看

`docs/observability/metrics.md` 已经列出了 MicroClaw 生成的关键 span：

- `agent_run`
- `llm_generation`
- `tool_execution`

这些 span 的价值非常直接。一次“回答慢了”的请求，在 trace 中就能被拆开成：

- 等模型太久
- 某个工具超时
- 工具很多轮才收敛
- 输出流式回传很快但最终消息存储慢

没有 trace，团队只能从总耗时猜；有 trace，团队可以围绕阶段做优化。

## Logs：记录降级与异常分类

在运行手册和代码里，MicroClaw 对很多异常都采用了分类处理，而不是只打一条模糊错误：

- MCP rate limited
- MCP bulkhead rejected
- circuit open
- approval required
- execution policy blocked
- timeout
- invalid payload

这类稳定分类对运维很重要。因为日志的核心任务不是“把原始错误堆出来”，而是让值班人员快速知道这是：

- 依赖抖动
- 策略阻断
- 配置问题
- 用户输入问题
- 系统 bug

分类稳定，Runbook 才写得出来，自动告警才有意义。

## 运维体系：Runbook、稳定性计划和回滚

MicroClaw 当前不是只有监控接口，还有成体系的运行文档：

- `docs/operations/runbook.md`
- `docs/operations/stability-plan-2026-q1.md`
- `docs/releases/upgrade-guide.md`

这些文档共同定义了三个关键运维动作。

### 发现问题

通过 `/api/metrics`、`/api/metrics/summary`、`/api/metrics/history` 以及控制面自检发现风险。

### 定位问题

通过 trace、日志、hook 状态、session tree、scheduler DLQ 等视图定位故障根因。

### 恢复问题

通过 replay DLQ、冻结 feature merge、回滚二进制和恢复 SQLite 备份完成恢复。

也就是说，运维体系不是“有 dashboard 就够了”，而是“有仪表盘，还要知道当它变红时具体怎么做”。

## 外部导出：OTLP 不是装饰功能

MicroClaw 当前支持 OTLP Metrics、Traces 和 Logs 导出，并且已经为 Langfuse、AgentOps 做了 adapter。这个设计的重要性在于，它没有把业务采集逻辑和 vendor 绑定死。

`docs/observability/architecture.md` 明确要求：

- adapter 主要负责 endpoint 和认证头注入
- 业务信号映射逻辑留在核心 exporter

这样做的收益是，接新平台时不会把主逻辑再复制一遍。对于后续接更多观测后端，这种边界非常值钱。

## 常见事故场景应该怎样被观察

一个成熟的可观测体系，不只是字段完整，还要能覆盖常见事故模式。以 MicroClaw 为例，至少要回答下面几类问题：

- 请求成功率突然下降，是模型错误、工具错误还是控制面限流？
- 某次对话为什么异常慢，是 `llm_generation` 还是 `tool_execution` 阶段拖长？
- MCP 异常是远端服务不可用，还是本地并发 / 限流保护触发？
- Scheduler 为什么看似没工作，是任务没到期、claim 失败，还是执行后进入 DLQ？

只要你的指标、trace 和日志能稳定回答这些问题，运维体系才算真正可用。

## 示例代码：为什么指标应该围绕运行事件，而不是围绕机器资源？

可观测性的第一步，通常不是接入多复杂的平台，而是先把关键运行事件变成稳定信号。只有事件口径稳定了，后面的 metrics、trace 和 runbook 才有共同语言。

Rust 版本把信号上报抽成 trait，再由一个观察器 struct 在工具执行结束时发出统一指标。这样业务代码只关心“发生了什么”，而不是把导出细节散落在各个调用点里。

```rust
#[async_trait::async_trait]
trait MetricsSink {
    async fn increment(&self, key: &str) -> anyhow::Result<()>;
}

struct ToolObserver<S> {
    sink: S,
}

impl<S: MetricsSink> ToolObserver<S> {
    async fn record_tool_result(&self, ok: bool) -> anyhow::Result<()> {
        self.sink.increment("tool_executions").await?;
        let key = if ok { "tool_success" } else { "tool_error" };
        self.sink.increment(key).await
    }
}
```

Python 版本用 `Protocol` 保持上报接口稳定，再用 `@dataclass` 把观察器和业务事件绑定起来。这样示例会更贴近真实运维代码：采集逻辑集中，事件语义稳定，导出后端可替换。

```python
from dataclasses import dataclass
from typing import Protocol


class MetricsSink(Protocol):
    async def increment(self, key: str) -> None: ...


@dataclass
class ToolObserver:
    sink: MetricsSink

    async def record_tool_result(self, ok: bool) -> None:
        await self.sink.increment("tool_executions")
        key = "tool_success" if ok else "tool_error"
        await self.sink.increment(key)
```

## 关键权衡

### 决策一：把观测信号抽到独立 crate

优点是业务代码与导出逻辑分离。代价是 observability 也要维护自己的接口稳定性。

### 决策二：SLO 面向运行时行为而不是机器资源

优点是更贴近用户体验。代价是需要更深入理解 Agent 系统的真正故障形态。

### 决策三：本地持久化指标历史

优点是即使没有外部平台也能回看趋势。代价是需要管理 SQLite 中观测数据的保留策略。

### 决策四：Runbook 与升级回滚文档并入主干

优点是运维知识不再只存在于个人脑中。代价是文档必须持续跟上实现演进。

## 容易走错的地方

### 失败模式 1：只看基础设施指标，不看 runtime 指标

这样常常会错过真正影响用户体验的执行失败和策略阻断。

### 失败模式 2：有 metrics，没有 trace 和日志分类

你能知道系统不好，却不知道到底坏在哪里。

### 失败模式 3：有 dashboard，没有 runbook 和回滚标准

事故一来，团队只能临场猜测操作步骤。

## 小结

MicroClaw 的可观测性设计说明了一个成熟 runtime 的基本标准：系统不仅要工作，还要能解释自己在做什么、做得好不好、出了问题该怎么办。只有把指标、追踪、日志和运维手册连成闭环，Agent Runtime 才真正具备生产价值。

下一章我们进入测试策略。因为再好的观测体系，也只能帮助你发现问题；要想减少问题进入生产，还必须靠系统化的测试和回归门禁。

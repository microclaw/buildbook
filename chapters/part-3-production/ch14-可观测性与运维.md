# Chapter 14 可观测性与运维

## 观测体系概览

Agent Runtime 经常"看起来在运行，却不知道正在做什么"。MicroClaw 通过 `microclaw-observability` 独立 crate 统一管理 metrics、traces、logs，并支持 OTLP 导出、Langfuse/AgentOps adapter、ToolWave 事件和子代理运行追踪。

## microclaw-observability：独立 crate

```
crates/microclaw-observability/src/
├── lib.rs          # 模块入口
├── sdk.rs          # 共享配置（endpoint、headers、开关）
├── metrics.rs      # OTLP Metrics 导出
├── traces.rs       # OTLP Traces 导出
├── logs.rs         # OTLP Logs 导出
└── adapters/
    ├── langfuse.rs # Langfuse endpoint/header 注入
    └── agentops.rs # AgentOps endpoint/header 注入
```

独立的收益：业务 crate 不需要依赖 OpenTelemetry 类型体系；adapter 只负责 endpoint 注入，不改信号语义；配置通过 `OTelSdkContext` 集中管理。

### Adapter 只注入 endpoint，不改写信号

```rust
pub struct TraceTargetConfig {
    pub endpoint: Option<String>,
    pub headers: HashMap<String, String>,
}

// langfuse adapter: 仅注入 endpoint 和 auth header
pub fn apply(config: &YamlMapping, target: &mut TraceTargetConfig) {
    if let Some(host) = get_trimmed(config, "langfuse_host") {
        target.endpoint = Some(format!("{}/api/public/otel/v1/traces", host));
    }
}
```

```python
from dataclasses import dataclass, field


@dataclass
class TraceTargetConfig:
    endpoint: str | None = None
    headers: dict[str, str] = field(default_factory=dict)

def apply_langfuse(config: dict, target: TraceTargetConfig) -> None:
    host = config.get("langfuse_host")
    if host:
        target.endpoint = f"{host}/api/public/otel/v1/traces"
```

LLM 返回多个工具调用时，`partition_into_waves` 按 concurrency class 分组执行。Web UI 通过 SSE 实时展示 wave 级执行进度。

### OTLP span 层级

```
agent_run (root span)
  ├── llm_generation (span)
  ├── tool_wave_1 (span)
  │   ├── read_file (span, parallel)
  │   ├── grep (span, parallel)
  │   └── glob (span, parallel)
  ├── tool_wave_2 (span)
  │   └── bash (span, sequential)
  └── llm_generation (span)
```

精确定位：模型推理慢、工具执行慢、还是某个 wave 中某工具超时。

## 子代理运行追踪

`subagent_runs` 表记录每次子代理的 `run_id`、`parent_chat_id`、`status`、`task`、`output_summary`、`token_usage`、`cost_estimate`、时间区间。配合 `model_prices` 可直接查询汇总成本。

## RunHub 与 SSE Replay

`RunHub` 维护 bounded 运行历史（默认 512 条），支持实时 SSE、历史 replay、状态查询。`web_run_history_limit` 控制历史深度。

## 三类信号

| 信号类型 | 回答的问题 | 代表内容 |
|---------|-----------|---------|
| Metrics | 宏观健不健康？ | `http_requests`、`llm_completions`、`tool_executions`、`active_sessions` |
| Traces | 一次请求哪里慢？ | wave 级并行细节、每个工具独立 span |
| Logs | 异常的具体原因？ | MCP rate limited、circuit open、approval required、timeout |

三类信号通过 `OTelSdkContext` 共享同一套配置，避免 service name、属性命名和导出逻辑不一致。

## 指标历史持久化

Minute bucket 写入 SQLite `metrics_history` 表。即使 OTLP 没配好，本地仍有观测留痕。

## SLO

`/api/metrics/summary` 覆盖四个维度：request success rate、end-to-end latency P95、tool reliability、scheduler recoverability。

## 运维动作闭环

```
发现 ──────────────► 定位 ──────────────► 恢复
  │                    │                    │
  ├ /api/metrics       ├ OTLP trace        ├ replay DLQ
  ├ /api/metrics/      │ (wave 级粒度)     ├ 冻结 feature merge
  │   summary          ├ 日志分类          ├ 回滚二进制
  ├ ToolWave 事件      ├ hook 状态         └ 恢复 SQLite 备份
  ├ subagent_runs      ├ session tree
  └ 控制面自检         └ scheduler DLQ
```

## 示例代码：wave 级事件

```rust
enum AgentEvent {
    ToolWaveStart { wave: usize, tool_count: usize },
    ToolWaveComplete { wave: usize },
}

fn emit_wave_events(wave: usize, tools: &[String], start: std::time::Instant) {
    println!("Wave {wave}: {count} tools starting", count = tools.len());
    let elapsed_ms = start.elapsed().as_millis();
    println!("Wave {wave}: completed in {elapsed_ms}ms");
}
```

```python
from dataclasses import dataclass
from time import monotonic


@dataclass
class WaveStartEvent:
    wave: int
    tool_count: int

@dataclass
class WaveCompleteEvent:
    wave: int

def execute_wave(wave: int, tools: list[str]) -> WaveCompleteEvent:
    start = monotonic()
    elapsed_ms = int((monotonic() - start) * 1000)
    print(f"Wave {wave}: {len(tools)} tools completed in {elapsed_ms}ms")
    return WaveCompleteEvent(wave=wave)
```

## 容易走错的地方

### 失败模式 1：只看基础设施指标，不看 wave 指标

会错过并行执行中的单点瓶颈。

### 失败模式 2：子代理运行不追踪

无法回答"子代理花了多少钱"。

### 失败模式 3：观测平台和业务代码耦合

Langfuse API 散落在 agent_engine 中，换平台需大规模重构。

### 失败模式 4：有 dashboard，没有 runbook

事故一来，团队只能临场猜测。

# Chapter 14 可观测性与运维

## 观测体系概览

Agent Runtime 的最大问题是「看起来在运行，却不知道正在做什么」。MicroClaw v0.1.57 通过独立 crate `microclaw-observability` 把 metrics、traces、logs 聚到同一套 OTLP 出口，配合 SQLite 的 `metrics_history`、`audit_logs` 与 `subagent_runs` 表，让本地无 OTLP 时也保留可追溯证据。

## microclaw-observability crate 结构

```
crates/microclaw-observability/src/
├── lib.rs          # 模块出口
├── sdk.rs          # OTelSdkContext（endpoint / headers / 共享配置）
├── metrics.rs      # OtlpMetricExporter
├── traces.rs       # OtlpTraceExporter
├── logs.rs         # OtlpLogExporter
└── adapters/
    ├── langfuse.rs # Langfuse endpoint/header 注入
    └── agentops.rs # AgentOps endpoint/header 注入
```

业务 crate（`microclaw-tools`、`microclaw-storage` 等）不依赖 OpenTelemetry 类型体系，只通过共享 `Config` 暴露字段；adapter 只做端点与认证 header 注入，不改变信号语义。

## 三个 Exporter 的注入点

`AppState`（`src/runtime.rs`）持有三个 Optional Exporter：

```rust
pub struct AppState {
    // ... 其它字段
    pub metric_exporter: Option<Arc<OtlpMetricExporter>>,
    pub trace_exporter: Option<Arc<OtlpTraceExporter>>,
    pub log_exporter: Option<Arc<OtlpLogExporter>>,
}
```

启动时由 `Config::observability` 决定是否实例化：

```rust
let metric_exporter = OtlpMetricExporter::from_observability(config.observability.as_ref());
let trace_exporter = OtlpTraceExporter::from_observability(config.observability.as_ref());
let log_exporter   = OtlpLogExporter::from_observability(config.observability.as_ref());
```

`from_observability` 内部读取 endpoint、headers 与开关，没有配置就返回 `None`，业务代码只需写：

```rust
if let Some(exp) = &state.trace_exporter {
    exp.record_tool_span(name, duration_ms, status);
}
```

不打开 OTLP 时这部分代码是空操作，不需要任何 feature flag。

## 适配 Langfuse 与 AgentOps

Langfuse / AgentOps 都接受 OTLP 协议。adapter 把它们当成预设端点：

```rust
pub struct TraceTargetConfig {
    pub endpoint: Option<String>,
    pub headers: HashMap<String, String>,
}

// adapters/langfuse.rs
pub fn apply(config: &YamlMapping, target: &mut TraceTargetConfig) {
    if let Some(host) = get_trimmed(config, "langfuse_host") {
        target.endpoint = Some(format!("{}/api/public/otel/v1/traces", host));
    }
    if let (Some(pk), Some(sk)) = (get_trimmed(config, "langfuse_public_key"),
                                    get_trimmed(config, "langfuse_secret_key")) {
        let basic = base64_basic(&pk, &sk);
        target.headers.insert("Authorization".into(), format!("Basic {basic}"));
    }
}
```

换平台只换 adapter，业务代码完全不动。

## OTLP span 层级

`AgentEvent`（`src/agent_engine.rs`）是统一事件流，`tool_executor` 把每个 wave 包成一个 trace span：

```
agent_run (root span)
  ├── llm_generation
  ├── tool_wave_1
  │   ├── read_file (parallel)
  │   ├── grep      (parallel)
  │   └── glob      (parallel)
  ├── tool_wave_2
  │   └── bash      (Exclusive，独占 wave)
  └── llm_generation
```

排查时直接看哪个 span 长：模型推理慢、还是某个 wave 中某工具超时一目了然。

## 历史指标持久化

`metrics_history` 表（schema v7+，逐版本扩列至 v10）按分钟桶聚合，关键列：

```sql
CREATE TABLE metrics_history (
    timestamp_ms                 INTEGER PRIMARY KEY,
    llm_completions              INTEGER NOT NULL DEFAULT 0,
    llm_input_tokens             INTEGER NOT NULL DEFAULT 0,
    llm_output_tokens            INTEGER NOT NULL DEFAULT 0,
    http_requests                INTEGER NOT NULL DEFAULT 0,
    tool_executions              INTEGER NOT NULL DEFAULT 0,
    mcp_calls                    INTEGER NOT NULL DEFAULT 0,
    mcp_rate_limited_rejections  INTEGER NOT NULL DEFAULT 0,
    mcp_bulkhead_rejections      INTEGER NOT NULL DEFAULT 0,
    mcp_circuit_open_rejections  INTEGER NOT NULL DEFAULT 0,
    active_sessions              INTEGER NOT NULL DEFAULT 0
);
CREATE INDEX idx_metrics_history_ts ON metrics_history(timestamp_ms);
```

设计要点：

- 即使 OTLP 没接通，本地仍有按分钟可回放的数字证据。
- 三个独立的 `mcp_*_rejections` 列让「外部依赖被治理拦下」与「外部依赖真正失败」可以区分。

## audit_logs：写操作的法定证据链

```sql
CREATE TABLE audit_logs (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    kind        TEXT NOT NULL,
    actor       TEXT NOT NULL,
    action      TEXT NOT NULL,
    target      TEXT,
    status      TEXT NOT NULL,
    detail      TEXT,
    created_at  TEXT NOT NULL
);
```

控制面所有「会改世界」的操作（设密码、创建/撤销/轮换 API key、暂停调度任务、批准高风险工具）都过 `audit_log(actor, action, target, status, detail)`，写入异步执行不阻塞响应路径。

## 子代理运行追踪

`subagent_runs` 表存每次子代理的 `run_id`、`parent_chat_id`、`status`（`running` / `completed` / `timeout` / `cancelled` / `failed`）、`task`、`output_summary`、`token_usage`、`cost_estimate`、起止时间。结合 `model_prices` 配置可直接 SQL 出每月子代理花了多少钱：

```rust
struct SubagentSummary {
    status: String,
    count: i64,
    total_cost_usd: f64,
    total_tokens: i64,
}
```

## SLO 与运维动作闭环

`/api/metrics/summary` 给出四个核心 SLO：

| 信号 | 含义 | 失败现象 |
|------|------|---------|
| 消息处理延迟 P95 | LLM + 工具 + 渠道往返耗时 | 用户感知卡顿 |
| 工具失败率 | `tool_executions` 中 `status=error` 占比 | 模型不停重试，token 飙升 |
| 调度命中率 | `scheduled_tasks` 在期望时间窗内完成 | 定时作业悄悄漂移 |
| MCP 拒绝率 | 三类 `*_rejections` 之和 / `mcp_calls` | 上游过载或下游堵 |

```
发现 ──────────────► 定位 ─────────────► 恢复
  │                   │                   │
  ├ /api/metrics/     ├ OTLP trace        ├ replay scheduled_task_dlq
  │   summary         │ (wave 级粒度)     ├ pause_scheduled_task / cancel_scheduled_task
  ├ ToolWave 事件     ├ AgentEvent 流     ├ 回滚二进制（gateway）
  ├ subagent_runs     ├ audit_logs 上下文 └ SQLite 备份恢复
  └ doctor 自检       └ doctor --json
```

## Runbook 与回滚路径

**Gateway 子命令**（`src/gateway.rs`）封装 systemd / launchd / Windows Service：

```bash
microclaw gateway install        # 安装服务（按平台生成 plist / unit / service）
microclaw gateway start          # 启动
microclaw gateway stop           # 停止
microclaw gateway restart        # 重启
microclaw gateway status --json  # 机器可读状态
microclaw gateway status --deep  # 通过 WebSocket RPC 深度健康检查
microclaw gateway logs 500       # 拉最近 500 行日志
```

**Doctor 自检**（`src/doctor.rs`）按检查项产出 `PASS / WARN / FAIL`，覆盖：

- 配置文件位置与字段合法性
- `web_fetch` 内容与 URL 校验是否启用、是否 strict mode
- Sandbox 配置一致性（mode + runtime + image + 挂载白名单）
- ACP / 子代理预算是否合理
- bash 危险命令模式
- MCP server 依赖（命令是否可执行、超时是否过大）
- 数据库 schema 版本

**Upgrade**（`microclaw upgrade`）下载新二进制后先备份 `<data_dir>/microclaw.db`，启动失败可立即回滚到上一份二进制。

## 日志分类

OTLP `OtlpLogExporter` 把结构化日志直传后端，本地无 OTLP 时回落到 stdout 的 JSON 行格式。一些值得专门告警的事件类别：

- `mcp.rate_limited`、`mcp.circuit_open`、`mcp.bulkhead_rejected`
- `tool.timeout`、`tool.exclusive_violated`
- `auth.login.failed`（连续失败触发 60s 内 5 次的限流）
- `subagent.timeout`、`subagent.token_budget_exhausted`

## 容易走错的地方

**只看基础设施指标，不看 wave 指标**：单工具延迟、单 LLM 延迟都正常，但某个 wave 因为最慢的工具拖了 10 秒——只有在 wave 级 span 上才能看见。

**子代理运行不追踪**：没有 `subagent_runs`，回答不了「上周子代理花了多少钱」。

**观测平台与业务耦合**：把 Langfuse 调用散在 `agent_engine` 里，换平台需要大规模重构。`microclaw-observability` 的 adapter 模型避免了这一点。

**有 dashboard，没有 runbook**：事故来时团队只能临场猜。Runbook 至少包含：哪些 metric 触发哪条告警 → 第一步看哪张 dashboard → 然后查哪张表 → 最后怎么回滚。

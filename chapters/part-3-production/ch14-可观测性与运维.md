# Chapter 14 可观测性与运维

## 系统"看起来在运行，却不知道正在做什么"怎么办？

Agent Runtime 最让人头疼的地方不是它会报错，而是它经常"看起来在运行，却不知道正在做什么"。一次请求可能跨越模型调用、并行工具执行波次、消息投递、后台任务、子代理运行和记忆提取。如果没有统一观测，你能看到的往往只有最后一句失败提示。

MicroClaw 的可观测体系包含以下核心能力：

- `microclaw-observability` 独立 crate 统一管理 metrics、traces、logs
- OTLP HTTP/proto 导出到任意 OpenTelemetry 后端
- Langfuse 和 AgentOps adapter 无需改主逻辑即可接入
- `ToolWaveStart` / `ToolWaveComplete` 事件让并行工具执行过程可见
- `subagent_runs` 表追踪子代理运行全生命周期
- `RunHub` 支持 run history 和 SSE replay

这一章读完后，你应该理解：

1. 为什么 `microclaw-observability` 要作为独立 crate 存在。
2. ToolWave 事件如何让并行执行变得可观测。
3. 子代理运行追踪如何闭合成本和可靠性视图。
4. OTLP 导出与 adapter 分离为什么是正确的分层。

## microclaw-observability：为什么观测要独立成 crate

8-crate workspace 中，`microclaw-observability` 负责所有信号的采集、格式化和导出。它的内部结构是：

```
crates/microclaw-observability/src/
├── lib.rs          # 模块入口
├── sdk.rs          # 共享配置解析（endpoint、headers、开关）
├── metrics.rs      # OTLP Metrics 导出
├── traces.rs       # OTLP Traces 导出
├── logs.rs         # OTLP Logs 导出
└── adapters/
    ├── mod.rs
    ├── langfuse.rs # Langfuse endpoint/header 注入
    └── agentops.rs # AgentOps endpoint/header 注入
```

这种分离有三个关键收益：

第一，业务 crate（`microclaw-tools`、`microclaw-storage`）不需要直接依赖 OpenTelemetry 的复杂类型体系。它们只需要发出结构化事件，导出由 observability crate 统一处理。

第二，adapter 只负责 endpoint 和认证头注入，不修改信号语义。这意味着接入新的观测平台时，不需要改动任何业务信号映射逻辑。

第三，观测配置可以在 YAML 中集中管理，通过 `sdk.rs` 的 `OTelSdkContext` 统一解析 `otlp_tracing_endpoint`、`otlp_tracing_enabled`、headers 等字段。

### Adapter 为什么应该只注入 endpoint，不改写信号？

很多团队在接入 Langfuse 或其他 LLM 观测平台时，在业务代码中散落平台特定的 API 调用。后果是：换平台时需要到处改代码，多平台并存时信号口径不一致。

Rust 版本展示了 MicroClaw 的 adapter 设计：adapter 只修改 trace exporter 的目标配置，不接触 span 内容。

```rust
pub struct TraceTargetConfig {
    pub endpoint: Option<String>,
    pub headers: HashMap<String, String>,
}

// langfuse adapter: 仅注入 endpoint 和 auth header
pub fn apply(config: &YamlMapping, target: &mut TraceTargetConfig) {
    if let Some(host) = get_trimmed(config, "langfuse_host") {
        target.endpoint = Some(format!("{}/api/public/otel/v1/traces", host));
        // 注入 Basic auth header
    }
}
```

Python 版本用同样的结构表达。重点不是 HTTP 细节，而是"adapter 的职责边界"：只负责告诉 exporter 往哪里发、带什么认证，不负责信号本身怎么组织。

```python
from dataclasses import dataclass, field


@dataclass
class TraceTargetConfig:
    endpoint: str | None = None
    headers: dict[str, str] = field(default_factory=dict)

def apply_langfuse(config: dict, target: TraceTargetConfig) -> None:
    host = config.get("langfuse_host")
    if host:
        target.endpoint = f"{host}/api/public/otel/v1/traces"
        # 注入 Basic auth header
```

## ToolWave 事件：让并行执行变得可观测

有了独立的 observability crate 之后，下一个问题是：系统应该发出什么信号？传统的请求级 metrics 和 trace 已经不够——MicroClaw 支持并行工具执行，如果观测粒度还停在"一次请求用了多少毫秒"，你就无法判断是某个 wave 里的某个工具拖了后腿，还是整体都慢。因此，系统为并行工具执行引入了 wave 级别的事件。

`tool_executor.rs` 在执行每个 wave 前后分别发出：

```rust
// agent_engine.rs 中定义
pub enum AgentEvent {
    // ...
    ToolWaveStart {
        wave: usize,
        tool_count: usize,
    },
    ToolWaveComplete {
        wave: usize,
    },
    // ...
}
```

当 LLM 一次返回 5 个工具调用时，`partition_into_waves` 会按 concurrency class 分组：

1. Wave 1：所有 `ReadOnly` 工具并行执行
2. Wave 2+：`SideEffect` 工具逐个顺序执行
3. `Exclusive` 工具独占一个 wave

Web UI 通过 SSE 接收这些事件后，可以实时展示"Wave 1: 3 tools parallel → Wave 2: bash sequential"这样的执行进度。这让用户和运维人员能清楚看到：

- 并行是否真的在发生
- 某个 wave 是否异常慢
- 工具分组是否符合预期

### OTLP span 层级

`tool_executor.rs` 为每次工具执行创建独立的 OTLP span，带有 trace_id 和 parent_span_id。这意味着在 Jaeger 或 Langfuse 的 trace 视图中，一次 agent run 会呈现为：

```
agent_run (root span)
  ├── llm_generation (span)
  ├── tool_wave_1 (span)
  │   ├── read_file (span, parallel)
  │   ├── grep (span, parallel)
  │   └── glob (span, parallel)
  ├── tool_wave_2 (span)
  │   └── bash (span, sequential)
  └── llm_generation (span)
```

这种层级结构让团队能精确定位：一次"回答慢了"到底是模型推理慢、工具执行慢，还是某个特定 wave 中的某个工具超时。

## 子代理运行追踪

MicroClaw 在数据库中提供了 `subagent_runs` 表，记录每次子代理运行的完整生命周期：

- `run_id`：唯一标识
- `parent_chat_id`：触发该子代理的父 chat
- `status`：running / completed / failed / timeout
- `task`：子代理被分配的任务描述
- `output_summary`：运行结果摘要
- `token_usage`：消耗的 token 数量
- `cost_estimate`：成本估算
- `started_at` / `completed_at`：时间区间

这个表的价值不只是事后审计，更重要的是它闭合了子代理的成本视图。当 `model_prices` 配置了各模型的 per-million-token 价格时，每次子代理运行的成本可以直接从 `subagent_runs` 表中查询和汇总。

## RunHub 与 SSE Replay

`web.rs` 中的 `RunHub` 是运行时观测中枢。它维护了一个 bounded 的运行历史（默认 512 条），支持：

- **实时 SSE**：Web UI 通过 Server-Sent Events 实时接收 `AgentEvent` 流，包括 `ToolWaveStart`、`ToolWaveComplete`、文本生成、工具结果等
- **历史 replay**：对于已完成的 run，可以通过 API 回放完整事件序列
- **状态查询**：查询任意 run 的当前状态（running / completed / failed）

`web_run_history_limit` 配置项控制历史深度，防止内存无限增长。

## 统一三类信号：不要各说各话

`microclaw-observability` 最值得重视的一点，是三类信号通过 `sdk.rs` 共享同一套配置上下文。`OTelSdkContext` 统一管理 endpoint、headers、enabled 开关、队列参数。这不是文档上的整洁，而是生产上的必要条件。

如果指标、追踪和日志分别用不同的 service name、不同的属性命名、不同的导出逻辑，运维团队最终会得到三套无法对齐的视图。

### 三类信号各自回答什么

| 信号类型 | 回答的问题 | 代表指标 / 内容 |
|---------|-----------|---------------|
| Metrics | 系统在宏观上健不健康？ | `http_requests`、`llm_completions`、`tool_executions`、`tool_policy_blocks`、`mcp_calls`、`active_sessions` |
| Traces | 一次请求的路径在哪里慢？ | 结构化路径，含 wave 级并行执行细节和每个工具的独立 span |
| Logs | 降级或异常的具体原因？ | 稳定分类：MCP rate limited、circuit open、approval required、execution policy blocked、timeout、invalid payload |

## 指标历史持久化

MicroClaw 不只是提供 `/api/metrics` 快照，还会把 minute bucket 写入 SQLite 的 `metrics_history` 表。这样做有三个好处：

1. 可以查询过去 60 分钟或 24 小时趋势，不依赖外部监控
2. Web 控制面可以直接展示历史走势
3. 即使 OTLP 没配好，本地仍有最基本的观测留痕

## SLO：系统到底什么算健康

`/api/metrics/summary` 暴露的 SLO 合约覆盖四个维度：

- request success rate
- end-to-end latency P95
- tool reliability（把 `approval_required` 和 `execution_policy_blocked` 排除在失败分母外）
- scheduler recoverability

这四个目标分别覆盖了对话可用性、交互体验、执行动作质量和后台任务可靠性。

## 外部导出：OTLP 不是装饰功能

`traces.rs` 中的 `OtlpTraceExporter` 使用 OpenTelemetry SDK 的 `BatchSpanProcessor`，通过 OTLP HTTP/proto 协议导出。关键实现细节：

- 使用 `opentelemetry_otlp` 的 `Protocol::HttpBinary`（protobuf over HTTP）
- 支持自定义 headers（用于 API key 注入）
- Langfuse adapter 将 endpoint 映射为 `{host}/api/public/otel/v1/traces`
- AgentOps adapter 将 endpoint 映射为 AgentOps 的 OTLP 接收地址

这个设计让核心导出逻辑与 vendor 完全解耦。接新平台只需新增一个 adapter 文件，不需要修改任何业务代码。

## 运维体系：Runbook、稳定性计划和回滚

MicroClaw 的运维文档定义了三个关键动作循环：

```
发现 ──────────────► 定位 ──────────────► 恢复
  │                    │                    │
  ├ /api/metrics       ├ OTLP trace        ├ replay DLQ
  ├ /api/metrics/      │ (wave 级粒度)     ├ 冻结 feature merge
  │   summary          ├ 日志分类          ├ 回滚二进制

  ├ subagent_runs      ├ session tree
  └ 控制面自检         └ scheduler DLQ
```

## 示例代码：为什么 wave 级事件是并行执行可观测性的最小单位？

并行工具执行如果只报"5 个工具用了 3 秒"，运维无法判断是一个慢工具拖了后腿还是所有工具都慢。只有把执行拆到 wave 级，才能分辨并行效率和单点瓶颈。

Rust 版本用 enum 变体直接展示 ToolWave 事件的结构。这和 MicroClaw 实际的 `AgentEvent` 定义完全一致。

```rust
enum AgentEvent {
    ToolWaveStart { wave: usize, tool_count: usize },
    ToolWaveComplete { wave: usize },
    // ...
}

fn emit_wave_events(wave: usize, tools: &[String], start: std::time::Instant) {
    println!("Wave {wave}: {count} tools starting", count = tools.len());
    // ... execute tools ...
    let elapsed_ms = start.elapsed().as_millis();
    println!("Wave {wave}: completed in {elapsed_ms}ms");
}
```

Python 版本保持同样的事件结构。关键是让读者看到：wave 是并行执行的最小可观测单位，没有它，trace 视图中的并行工具就是一团模糊的重叠。

```python
from dataclasses import dataclass
from time import monotonic


@dataclass
class WaveStartEvent:
    wave: int
    tool_count: int

@dataclass
class WaveCompleteEvent:
    wave: int

def execute_wave(wave: int, tools: list[str]) -> WaveCompleteEvent:
    start = monotonic()
    # ... execute tools in parallel ...
    elapsed_ms = int((monotonic() - start) * 1000)
    print(f"Wave {wave}: {len(tools)} tools completed in {elapsed_ms}ms")
    return WaveCompleteEvent(wave=wave)
```

## 关键权衡

### 决策一：把观测信号抽到独立 crate

优点是业务代码与导出逻辑分离，adapter 可独立演进。代价是 observability crate 也要维护自己的接口稳定性。

### 决策二：ToolWave 事件作为一等公民

优点是并行执行不再是黑盒。代价是事件量增加，SSE 和 RunHub 需要处理更多数据。

### 决策三：adapter 只注入 endpoint 和 headers

优点是接新平台不改核心逻辑。代价是 adapter 能力有限，无法做平台特定的信号转换。但这个限制恰恰是正确的边界。

### 决策四：subagent_runs 表独立追踪

优点是子代理的成本和可靠性有了专属视图。代价是 schema 维护成本增加。

### 决策五：RunHub 内存历史 + SQLite 持久化

优点是实时性和回看能力兼具。代价是需要管理两层存储的一致性和容量。

## 容易走错的地方

### 失败模式 1：只看基础设施指标，不看 runtime 和 wave 指标

这样常常会错过并行执行中的单点瓶颈。

### 失败模式 2：子代理运行不追踪

没有 `subagent_runs` 表，就无法回答"这个月子代理花了多少钱"这类基本问题。

### 失败模式 3：观测平台和业务代码耦合

把 Langfuse API 调用散落在 agent_engine 中，换平台时就要做大规模重构。

### 失败模式 4：有 dashboard，没有 runbook

事故一来，团队只能临场猜测操作步骤。

## 小结

MicroClaw 的可观测性设计做到了从"能看到指标"到"能看到并行执行的每个 wave、每个子代理运行、每一美元成本"的完整视图。`microclaw-observability` crate 的独立化让信号采集与导出分离，adapter 模式让平台切换成本趋近于零，ToolWave 事件让并行执行从黑盒变成了可调优的透明结构。

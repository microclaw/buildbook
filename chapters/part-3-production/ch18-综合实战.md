# Chapter 18 综合实战

## 这一章要回答什么问题

前面 17 章已经把定位、架构、实现、治理和生产约束讲清楚了。最后一章要回答的问题不再是“MicroClaw 是什么”，而是“如果今天开始落地，你究竟该怎么做，按什么顺序做，做到什么程度才算交付可用版本”。

这一章不给泛泛建议，而是给出一条更接近真实项目节奏的 30 天落地路线：

- Day 1：做出最小可运行版本
- Week 1：补齐持续使用所需的核心能力
- Month 1：把系统推到可托管状态

这一章读完后，你应该能把这条路线直接转成团队的启动计划，而不是只停留在技术理解层面。

## Day 1：先交付一个最小可运行版本

第一天的目标不是“把所有特性都开起来”，而是要得到一个真正能响应消息、能保存状态、能被观察的最小 runtime。

### 第一步：安装与环境自检

当前推荐路径是：

1. `microclaw setup`
2. `microclaw doctor`
3. `microclaw start`

如果是源码部署，则需要至少完成：

- Rust 工具链可用
- 配置文件已生成
- 至少一个渠道或 Web 已启用
- 至少一个 LLM provider 凭据已配置

这里绝对不要跳过 `doctor`。对新部署来说，最常见的失败并不是代码，而是环境缺失、路径错误或 sandbox / MCP 依赖不满足。

### 第二步：先用 Web 跑通主链路

如果你不是必须第一天就上 Telegram 或 Discord，最稳妥的顺序通常是：

- 先启用本地 Web 控制面
- 用 Web 创建会话并验证基础对话
- 确认数据库、session、metrics 和控制面可用

这样做的好处是调试路径最短。你可以直接在本地看到事件流、配置自检和历史记录，而不必一开始就同时处理外部平台的 webhook、token 和网络问题。

### 第三步：验证四个最小验收项

Day 1 至少要验收：

1. 一次基础对话成功。
2. 同一会话的多轮上下文可保持。
3. 一次简单工具调用能成功返回。
4. Web 控制面能看到会话和指标。

只要这四项未完成，就还不算“可运行版本”。

### 小例子：为什么 Day 1 就应该有一份最小健康检查？

很多团队会把 smoke test 推迟到“系统稳定以后”，结果第一周都在凭感觉判断到底哪里坏了。真正高效的做法，是 Day 1 就有一份能验证最小主链路的健康检查。

Rust 版本把 Day 1 的验收压成一个 `SmokeCheck` 结构。这样“能不能对话、能不能看到会话、能不能看到指标”变成显式检查项，而不是口头标准。

```rust
struct SmokeCheck {
    can_chat: bool,
    can_store_session: bool,
    can_read_metrics: bool,
}

impl SmokeCheck {
    fn passed(&self) -> bool {
        self.can_chat && self.can_store_session && self.can_read_metrics
    }
}
```

Python 版本用 `@dataclass` 表达同样的验收对象。对于综合实战章节来说，这种例子比一串抽象建议更能说明落地顺序。

```python
from dataclasses import dataclass


@dataclass
class SmokeCheck:
    can_chat: bool
    can_store_session: bool
    can_read_metrics: bool

    def passed(self) -> bool:
        return self.can_chat and self.can_store_session and self.can_read_metrics
```

## Week 1：把系统从 Demo 拉到持续可用

第一周的重点，是把“能对话”升级为“能长期使用”。

### 任务一：接入至少一个真实聊天渠道

到这一阶段，建议再接入一个外部渠道，例如 Telegram 或 Discord。目标不是多，而是验证统一内核是否真的能从 Web 平滑延展到聊天平台。

验收重点包括：

- 私聊与群聊语义
- mention 行为
- 长消息拆分
- bot 回复入库

### 任务二：启用记忆与显式 remember

第一周应该至少让团队验证：

- `AGENTS.md` 文件记忆已进入上下文
- 显式 `remember` 能写入结构化记忆
- 新会话能召回关键事实

如果这一步不做，系统很容易一直停留在“每次都像第一次见面”的假智能状态。

### 任务三：建立至少一个定时任务场景

建议选一个风险较低、收益明确的 once 或 daily 任务，例如：

- 每天汇总一次 TODO
- 到点提醒团队检查某项发布动作

这一步的目标，是尽早验证 Scheduler、消息回传和任务日志是否正常。

### 任务四：接入一个简单 Skill 或 Plugin

不要等到很后期才验证扩展机制。第一周就应选择一个最小扩展场景，例如：

- 一个内部 runbook 作为 context provider
- 一个简单的 slash command
- 一个常用工作流的技能包

这样可以提前暴露扩展治理问题，而不是等主链路已经很复杂时再发现边界不清。

## Month 1：把系统推到可托管状态

第一个月的目标，不是“功能越来越多”，而是“这个系统开始适合被托管和持续依赖”。

### 任务一：加固安全姿态

到这个阶段，至少应完成：

- 明确 control chat 列表
- 高风险工具确认保持开启
- 评估并启用 sandbox
- 收紧 `web_fetch` URL / 内容校验
- 完成 Web 密码与 API key 管理

如果仍然保持第一天的宽松姿态，系统越有用，风险越高。

### 任务二：接上可观测与告警

至少要做到：

- `/api/metrics`、`/api/metrics/summary` 正常
- 指标历史可查询
- 关键 burn alert 阈值被团队理解
- 如有条件，接出 OTLP 到外部平台

这一步完成后，系统才真正从“能跑”变成“能运营”。

### 任务三：建立回归清单和发布门禁

建议把 `docs/test/blackbox-core-20-cases.md` 中与你们使用场景最相关的用例固化成发布清单，并至少覆盖：

- 基础对话
- 工具成功 / 失败
- 跨 chat 权限
- once 任务
- DLQ 重放
- Metrics 合约

如果没有这一步，每次升级都会重新回到手工试错状态。

### 任务四：建立升级与回滚流程

到 Month 1，团队必须能够清楚完成：

- 升级前备份 SQLite
- 记录当前配置
- 执行最小升级验证
- 回滚到上一版本并恢复数据

这不是大公司的仪式感，而是所有长期运行系统的必要自救能力。

## 一个可用 MicroClaw 的交付物清单

如果把前面的 30 天路线压缩成最终交付物，一个真正可用的 MicroClaw 至少应包含：

1. 至少一个真实渠道和一个本地 Web 控制面。
2. 已验证的统一循环、会话恢复和记忆召回。
3. 至少一个稳定定时任务场景。
4. 已定义的高风险操作边界和 control chat。
5. 可查询的 metrics summary、runbook 和升级回滚流程。
6. 一套最小黑盒回归清单。

没有这些，系统最多是“有很多 feature 的 Demo”；具备这些，才更接近“可交付 runtime”。

## 示例代码：怎样搭出一个最小但可继续扩展的 Runtime 骨架？

最后一章最适合把全书收束成一个最小骨架。下面的代码都故意保持很短，但它们保留了本书一直强调的三件事：统一装配、显式依赖、异步主链路。

Rust 版本先定义模型边界，再让 `AppState` 持有状态并暴露一个异步运行入口。这样最小骨架虽然短，但后面接渠道、记忆、调度时都还有明确落点，而不是只能继续往 `main` 函数里堆代码。

```rust
#[async_trait::async_trait]
trait LlmProvider {
    async fn send(&self, prompt: &str) -> anyhow::Result<String>;
}

struct AppState {
    llm: Box<dyn LlmProvider>,
    tools: ToolRegistry,
}
```

最后一章这里先把最小 runtime 的边界压缩成一个对象。先看清运行时真正持有什么，后面的 `run_once` 和 `main` 才会显得是在驱动骨架，而不是堆几行启动代码。

```rust

impl AppState {
    fn new(config: Config) -> Self {
        Self {
            llm: create_provider(&config),
            tools: ToolRegistry::new(&config),
        }
    }

    async fn run_once(&self, prompt: &str) -> anyhow::Result<String> {
        let _available_tools = self.tools.list();
        self.llm.send(prompt).await
    }
}

#[tokio::main]
async fn main() -> anyhow::Result<()> {
    let config = Config::load()?;
    let state = AppState::new(config);
    let reply = state.run_once("hello from runtime").await?;
    println!("{reply}");
    Ok(())
}
```

Python 版本保留同样的结构，但用 `Protocol` 和 `@dataclass` 让依赖注入关系更加显式。这样即使只看十几行代码，读者也能看到一个可继续生长的 runtime 应该怎样起步。

```python
import asyncio
from dataclasses import dataclass
from typing import Protocol


class LlmProvider(Protocol):
    async def send(self, prompt: str) -> str: ...


@dataclass
class AppState:
    llm: LlmProvider
    tools: ToolRegistry
```

Python 也先让最小运行时对象站住，再补运行入口。这样即使代码很短，读者也能先看到系统骨架，再看到骨架怎样被真正跑起来。

```python

    async def run_once(self, prompt: str) -> str:
        _available_tools = self.tools.list()
        return await self.llm.send(prompt)


async def main() -> None:
    config = Config.load()
    state = AppState(llm=create_provider(config), tools=ToolRegistry(config))
    reply = await state.run_once("hello from runtime")
    print(reply)


asyncio.run(main())
```

## 容易走错的地方

### 失败模式 1：第一周就同时上所有渠道和所有扩展

这样通常会让你无法分辨问题来自内核、渠道、配置还是扩展生态。

### 失败模式 2：只顾功能，不建运维基础

到了 Month 1 才发现没有 metrics、没有回滚、没有自检，代价会非常高。

### 失败模式 3：把长期运行问题留到“上线后再说”

Scheduler、记忆污染、权限边界、控制面安全都不是上线后才自然出现的问题，而是从 Day 1 就在累积。

## 小结

真正把 MicroClaw 交付出来的关键，并不在于照着功能列表一路堆完，而是按正确顺序把主链路、长期能力和生产治理逐步固化下来。只要顺序正确，系统会越来越稳；顺序错了，再多 feature 也很难变成可托管产品。

这也是本书真正想传达的结论：构建一个多渠道 Agent Runtime，从来不只是把模型接进聊天窗口，而是把执行、状态、记忆、安全、观测和演进组织成一个长期可维护的系统。

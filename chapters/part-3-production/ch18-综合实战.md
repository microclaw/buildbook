# Chapter 18 综合实战

## 今天开始落地 MicroClaw，按什么顺序做？

前面 17 章已经把定位、架构、实现、治理和生产约束讲清楚了。最后一章回答的问题不再是"MicroClaw 是什么"，而是"如果今天开始落地，你究竟该怎么做，按什么顺序做，做到什么程度才算交付可用版本"。

MicroClaw 提供了完整的运维工具链：Setup wizard（11,000+ 行的全屏 TUI）、Doctor 跨平台诊断、Gateway 服务管理、Web 密码管理命令、Upgrade 自升级、Reembed 向量重建。这一章把这些工具编织进一条完整的落地路线。

- Day 1：用 setup + doctor + start 做出最小可运行版本
- Week 1：接入渠道、建立记忆、启用并行工具执行
- Month 1：推到可托管状态——安全、观测、门禁、升级全部就绪

## Day 1：先交付一个最小可运行版本

第一天的目标不是"把所有特性都开起来"，而是要得到一个真正能响应消息、能保存状态、能被观察的最小 runtime。

### 第一步：安装

选择最适合你环境的安装方式：

```bash
# macOS/Linux (Homebrew)
brew install microclaw/tap/microclaw

# Linux/macOS (one-line)
curl -sSf https://raw.githubusercontent.com/microclaw/microclaw/main/install.sh | bash

# Windows (PowerShell)
irm https://raw.githubusercontent.com/microclaw/microclaw/main/install.ps1 | iex

# Docker
docker compose up -d

# Nix
nix run github:microclaw/microclaw

# 源码（需要自定义 feature 组合时）
cargo install --path . --features full
```

### 第二步：Setup Wizard

```bash
microclaw setup
```

Setup wizard 是一个全屏 TUI 应用（11,000+ 行 Rust，使用 ratatui），引导你完成：

1. **LLM Provider 配置**：选择 provider（OpenAI、Anthropic、Ollama、openai-codex、qwen-portal 等），输入 API key，验证连通性
2. **渠道配置**：选择要启用的渠道（Telegram、Discord、Slack、Web 等），输入对应的 token 和配置
3. **Sandbox 配置**：选择 sandbox 模式（Off / Shared / Chat），检测 Docker/Podman 可用性
4. **高级选项**：SOUL.md 性格文件、MCP server 配置、记忆设置

Setup wizard 的关键设计特点：

- 动态渠道列表：每个渠道模块（Telegram、Slack、Feishu 等）通过 `SETUP_DEF` 常量声明自己的配置字段，wizard 动态渲染
- 实时验证：API key 输入后立即调用模型验证连通性
- 配置文件生成：完成后生成 `microclaw.yaml`
- `--enable-sandbox` 快捷模式：直接启用 sandbox 而不进入全屏向导

### 第三步：Doctor 诊断

```bash
microclaw doctor
```

Doctor 是最值得信赖的部署前自检工具。它输出结构化的诊断报告，覆盖：

- **平台信息**：OS、架构、是否在 WSL 环境中
- **配置文件**：路径是否存在、YAML 是否合法
- **LLM Provider**：API key 是否配置、能否连通
- **Sandbox**：Docker/Podman 是否可用、安全 profile 是否正确
- **渠道**：每个启用渠道的 token 是否有效
- **MCP**：配置的 MCP server 是否可达
- **数据库**：SQLite 文件是否可读写
- **Web UI**：端口是否被占用

每个检查项有四种状态：

- `PASS`：正常
- `MISS`：可选项未配置
- `WARN`：存在潜在风险
- `FAIL`：必须修复

```bash
# JSON 输出，适合自动化
microclaw doctor --json

# 指定配置文件
microclaw doctor --config /path/to/microclaw.yaml
```

绝对不要跳过 doctor。对新部署来说，最常见的失败并不是代码，而是环境缺失、路径错误或 sandbox/MCP 依赖不满足。

### 第四步：启动

```bash
microclaw start
```

### 第五步：验证四个最小验收项

Day 1 至少要验收：

1. 一次基础对话成功（通过 Web UI）
2. 同一会话的多轮上下文可保持
3. 一次简单工具调用能成功返回
4. Web 控制面能看到会话和指标

### 第六步：设置 Web 密码

```bash
# 方法一：命令行直接设置
microclaw web password "your-strong-password"

# 方法二：生成随机密码
microclaw web password-generate

# 方法三：通过 bootstrap token 在 Web UI 中设置
# （首次启动时控制台会输出 bootstrap token）
```

Web 密码管理是专用的 CLI 能力。`password-clear` 用于测试或重置场景：

```bash
microclaw web password-clear  # 清除密码哈希并撤销所有 session
```

### Day 1 为什么就应该有最小健康检查？

很多团队把 smoke test 推迟到"系统稳定以后"，结果第一周都在凭感觉判断到底哪里坏了。高效的做法是 Day 1 就有一份能验证最小主链路的健康检查。

Rust 版本把 Day 1 的验收压成一个 `SmokeCheck` 结构。

```rust
struct SmokeCheck {
    can_chat: bool,
    can_store_session: bool,
    can_read_metrics: bool,
    web_password_set: bool,
}

impl SmokeCheck {
    fn passed(&self) -> bool {
        self.can_chat && self.can_store_session
            && self.can_read_metrics && self.web_password_set
    }
}
```

Python 版本用 `@dataclass` 表达同样的验收对象。

```python
from dataclasses import dataclass


@dataclass
class SmokeCheck:
    can_chat: bool
    can_store_session: bool
    can_read_metrics: bool
    web_password_set: bool

    def passed(self) -> bool:
        return all([
            self.can_chat, self.can_store_session,
            self.can_read_metrics, self.web_password_set,
        ])
```

## Week 1：把系统从 Demo 拉到持续可用

第一周的重点，是把"能对话"升级为"能长期使用"。

### 任务一：接入至少一个真实聊天渠道

到这一阶段，建议接入一个外部渠道（Telegram、Discord、Slack、飞书、钉钉等）。MicroClaw 支持 16 个渠道。验收重点：

- 私聊与群聊语义
- mention 行为
- 长消息拆分
- bot 回复入库

### 任务二：启用并行工具执行

确认 `parallel_tool_max_concurrency` 已设置（默认值取决于配置）。通过 Web UI 的 SSE 事件流验证：

- ToolWaveStart / ToolWaveComplete 事件正确出现
- ReadOnly 工具确实在同一 wave 内并行
- 延迟相比串行有明显下降

### 任务三：启用记忆与 SOUL.md

第一周应该至少验证：

- `SOUL.md` 性格文件已被加载到系统 prompt
- 显式 `remember` 能写入结构化记忆
- 新会话能召回关键事实

如果启用了 `sqlite-vec` feature，还应验证向量记忆搜索：

```bash
# 如果需要重建向量索引
microclaw reembed
```

`microclaw reembed` 会遍历所有活跃记忆条目，使用当前配置的 embedding model 重新生成向量。这在切换 embedding provider 或升级 sqlite-vec 后特别有用。

### 任务四：建立至少一个定时任务场景

建议选一个风险较低、收益明确的 once 或 daily 任务。

### 任务五：接入一个 MCP Server 或 Skill

不要等到很后期才验证扩展机制。第一周就应选择一个最小扩展场景，提前暴露治理问题。

### 任务六：Gateway 服务化（可选）

如果系统需要长期运行：

```bash
microclaw gateway install
microclaw gateway start
microclaw gateway status
microclaw gateway logs
```

这比 `nohup microclaw start &` 更可靠，因为 Gateway 利用操作系统原生的服务管理：

- macOS：launchd 自动重启
- Linux：systemd 自动重启 + journal 日志
- Windows：Windows Service + 事件日志

## Month 1：把系统推到可托管状态

第一个月的目标，不是"功能越来越多"，而是"这个系统开始适合被托管和持续依赖"。

### 任务一：加固安全姿态

到这个阶段，至少应完成：

- Web 密码从默认值修改为强密码（或通过 bootstrap token 首次设置）
- 高风险工具确认保持开启
- 评估并启用 sandbox（至少 Shared 模式，多用户场景推荐 Chat 模式）
- API key 已为外部集成生成 scoped key
- `web_fetch` URL/内容校验已开启
- control chat 列表已明确

### 任务二：接上可观测与告警

至少要做到：

- `/api/metrics`、`/api/metrics/summary` 正常
- 指标历史可查询
- 如有条件，配置 OTLP 导出到外部平台

```yaml
# 接入 Langfuse 示例
observability:
  langfuse_host: "https://cloud.langfuse.com"
  langfuse_public_key: "pk-..."
  langfuse_secret_key: "sk-..."

# 接入 AgentOps 示例
observability:
  agentops_api_key: "your-agentops-key"

# 通用 OTLP 示例
observability:
  otlp_tracing_endpoint: "http://jaeger:4318/v1/traces"
  otlp_tracing_enabled: true
```

### 任务三：建立回归清单和发布门禁

固化发布清单，至少覆盖：

- 基础对话
- 工具成功/失败
- 并行工具执行（wave 事件）
- 跨 chat 权限
- 子代理启动和超时
- once 任务
- Metrics 合约

### 任务四：建立升级与回滚流程

```bash
# 升级
microclaw upgrade

# 升级前手动备份（推荐）
cp microclaw.db microclaw.db.backup-$(date +%Y%m%d)

# 如果需要回滚
microclaw gateway stop
cp microclaw.db.backup-YYYYMMDD microclaw.db
# 安装旧版本二进制
microclaw gateway start
```

`microclaw upgrade` 是自升级命令。它从 GitHub Releases 下载最新版本并替换当前二进制。跨平台实现：

- Unix：下载并执行 `install.sh`
- Windows：下载并执行 `install.ps1`（需要处理当前进程锁定问题）

到 Month 1，团队必须能清楚完成升级前备份、最小升级验证、回滚到上一版本并恢复数据。

### 任务五：子代理预算审查

如果使用了子代理，审查：

- `subagent_runs` 表中的成本分布
- 是否有超时或失败的子代理运行
- `max_concurrent` 和 `max_tokens_per_run` 是否需要调整

```bash
# 查询子代理成本（通过 SQLite）
sqlite3 microclaw.db "SELECT status, COUNT(*), SUM(cost_estimate) FROM subagent_runs GROUP BY status"
```

## 一个可用 MicroClaw 的交付物清单

如果把前面的 30 天路线压缩成最终交付物，一个真正可用的 MicroClaw 至少应包含：

1. 至少一个真实渠道和一个本地 Web 控制面，密码已从默认值修改。
2. 已验证的统一循环、会话恢复、记忆召回和并行工具执行。
3. 至少一个稳定定时任务场景。
4. 已定义的高风险操作边界、control chat 和 API key 管理。
5. 可查询的 metrics summary、OTLP 导出（如有外部平台）、RunHub 历史。
6. Gateway 服务化或等效的进程管理。
7. 升级和回滚流程已验证。
8. 一套最小黑盒回归清单。

没有这些，系统最多是"有很多 feature 的 Demo"；具备这些，才更接近"可交付 runtime"。

## 示例代码：怎样搭出一个最小但可继续扩展的 Runtime 骨架？

最后一章最适合把全书收束成一个最小骨架。下面的代码都故意保持很短，但它们保留了本书一直强调的三件事：统一装配、显式依赖、异步主链路。

Rust 版本包含了 MicroClaw 的关键要素：LLM provider、tool registry、turn queue 和 observability。

```rust
#[async_trait::async_trait]
trait LlmProvider {
    async fn send(&self, prompt: &str) -> anyhow::Result<String>;
}

struct AppState {
    llm: Box<dyn LlmProvider>,
    tools: ToolRegistry,
    turn_queue: Arc<ChatTurnQueue>,
    trace_exporter: Option<Arc<OtlpTraceExporter>>,
}

impl AppState {
    async fn run_once(
        &self, channel: &str, chat_id: i64, prompt: &str,
    ) -> anyhow::Result<String> {
        // per-chat turn serialization
        let _guard = self.turn_queue.acquire(channel, chat_id).await;
        let _tools = self.tools.list();
        self.llm.send(prompt).await
    }
}
```

Python 版本保留同样的结构。

```python
import asyncio
from dataclasses import dataclass
from typing import Protocol


class LlmProvider(Protocol):
    async def send(self, prompt: str) -> str: ...

@dataclass
class AppState:
    llm: LlmProvider
    tools: "ToolRegistry"
    turn_queue: "ChatTurnQueue"

    async def run_once(
        self, channel: str, chat_id: int, prompt: str,
    ) -> str:
        async with self.turn_queue.acquire(channel, chat_id):
            _tools = self.tools.list()
            return await self.llm.send(prompt)
```

## 关键权衡

### 决策一：分阶段交付而非一次全开

Day 1 只追求最小可运行（setup + doctor + start + 一个渠道），Week 1 才加入并行执行、记忆和多渠道，Month 1 才推到安全加固和可观测。优点是每个阶段都有明确的验收标准，出了问题可以定位到具体阶段。代价是看起来"慢"，但实际上避免了"功能全开但哪里都不稳"的困境。

### 决策二：依赖 setup wizard 而非手写配置

11,000 行的 setup wizard 处理了 provider 验证、渠道兼容性检查、sandbox 检测等大量边界情况。优点是新用户的首次体验不会因为配置格式错误而崩溃。代价是 setup wizard 本身需要持续维护，且不适合完全自动化的 CI 部署场景（此时应使用环境变量或预置配置文件）。

### 决策三：Gateway 服务化而非前台运行

`microclaw gateway install/start/stop/status/logs` 把 MicroClaw 变成系统服务。优点是进程崩溃后可以自动重启，日志有标准化的落盘和轮转。代价是调试时不如前台运行直观，需要通过 `gateway logs` 或 OTLP 导出查看状态。

### 决策四：升级前先备份再验证

`microclaw upgrade` 只是下载新版本；真正安全的升级还需要数据库备份、doctor 验证和回滚预案。优点是不会在升级失败后丢失数据。代价是升级流程比"下载覆盖"更复杂，但这正是生产系统应有的纪律。

## 容易走错的地方

### 失败模式 1：跳过 setup 和 doctor，直接手写配置文件

Setup wizard 的 11,000 行代码处理了大量边界情况（provider 验证、渠道兼容性、sandbox 检测）。手写配置很容易遗漏关键字段或用错格式。

### 失败模式 2：Week 1 不启用并行工具执行

这是最容易获得的性能收益，延迟可以降低 60%+，而且不需要改任何配置以外的东西。

### 失败模式 3：Month 1 仍然使用默认密码

Bootstrap token 机制就是为了消除这个风险。如果你仍然在用默认密码，系统越有用，风险越高。

### 失败模式 4：不建升级和回滚流程就上线

`microclaw upgrade` 只是自动化了下载新版本。没有备份、没有验证、没有回滚能力，升级就是赌博。

## 小结

真正把 MicroClaw 交付出来的关键，并不在于照着功能列表一路堆完，而是按正确顺序把主链路、长期能力和生产治理逐步固化下来。MicroClaw 提供了完整的运维工具链——Setup wizard 降低了配置门槛，Doctor 消除了环境不确定性，Gateway 标准化了服务管理，Upgrade 让版本迭代不再是手工操作，Reembed 让向量索引可以按需重建。只要顺序正确，系统会越来越稳；顺序错了，再多 feature 也很难变成可托管产品。

这也是本书真正想传达的结论：构建一个多渠道 Agent Runtime，从来不只是把模型接进聊天窗口，而是把执行、状态、记忆、安全、观测和演进组织成一个长期可维护的系统。

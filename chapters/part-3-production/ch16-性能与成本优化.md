# Chapter 16 性能与成本优化

## 性能瓶颈在哪里

Agent Runtime 的总耗时由四部分叠加：LLM 推理时间、上下文 token 数量、工具执行（受 wave 结构影响）、外部依赖与子代理嵌套。MicroClaw v0.1.57 的核心优化是 **wave-based 并行工具执行**、**per-chat turn serialization**、**token 预算分级**，三者结合可以把端到端延迟降到合理量级，而不必改提示词。

## 上下文预算

最容易吃掉成本的不是模型本身，而是越吹越大的对话历史。MicroClaw 的三道闸门：

```yaml
max_session_messages: 40       # 单会话保留消息上限
compact_keep_recent: 20        # compaction 时保留最近 N 条
memory_token_budget: 1500      # 召回记忆注入上限（token）
max_history_messages: 50       # 渲染到 LLM 上下文的硬上限
```

**Compaction**：超出 `max_session_messages` 时，把早期消息折叠成一段摘要，保留 `compact_keep_recent` 条原文。compaction 同时是性能、成本与质量优化——上下文短了，幻觉与无关召回都减少。

**图片在持久化前剥离**：`strip_images_for_session` 在写库前把消息里的 image 内容摘除，避免数据库体积失控；图片的「曾经存在」事实保留为占位文本。

## 并行工具执行

`tool_executor` 的 `partition_into_waves` 按 `ToolConcurrencyClass` 分组：

1. **Wave 1**：所有 `ReadOnly`（`read_file`、`grep`、`glob`、`web_fetch`、`web_search`、`get_current_time`、`calculate`、`structured_memory_search`、`subagents_list` 等）同 wave 并行（`tokio::JoinSet`）。
2. **后续 wave**：每个 `SideEffect` 独占一个 wave 顺序执行。
3. **独占 wave**：`Exclusive`（`bash`、`activate_skill`、`sessions_spawn`）独占一个 wave。

典型搜索请求 `3 grep + 1 read_file + 1 bash`：

| 模式 | wave 切分 | 估算耗时 |
|------|----------|---------|
| 串行 | 一个个跑 | 5 × 1.5s = 7.5s |
| wave 并行 | Wave 1: 4 ReadOnly 并行 → 1.5s；Wave 2: bash → 1.5s | ≈ 3s |

并行度上限：

```rust
fn default_parallel_tool_max_concurrency() -> usize {
    8
}
```

`parallel_tool_max_concurrency = 8` 同时控制三件事：操作系统文件描述符不被打爆、上游 API rate limit 不被反弹、SQLite 写锁竞争可控。

```rust
use tokio::sync::Semaphore;
use std::sync::Arc;

async fn execute_wave(tools: Vec<ToolCall>, max: usize) -> Vec<ToolResult> {
    let sem = Arc::new(Semaphore::new(max));
    let mut set = tokio::task::JoinSet::new();
    for tool in tools {
        let permit = sem.clone().acquire_owned().await.unwrap();
        set.spawn(async move {
            let result = run_tool(tool).await;
            drop(permit);
            result
        });
    }
    let mut out = Vec::new();
    while let Some(Ok(r)) = set.join_next().await {
        out.push(r);
    }
    out
}
```

允许业务通过配置覆盖默认分类：

```yaml
tool_concurrency_overrides:
  query_db: read_only        # 把自定义 MCP 工具标记为可并行读
  send_notification: exclusive
```

## per-chat Turn Serialization

| 维度 | 全局锁 | per-chat 锁 |
|------|--------|-------------|
| 并发能力 | = 1 | = 活跃 chat 数 |
| 跨 chat 阻塞 | 有 | 无 |
| 状态一致性 | 进程级 | 每 chat 独立 |

`ChatTurnQueue` 用 `(channel, chat_id)` 做 key，每个 slot 一个 `tokio::Mutex<()>`，超 `max_pending` 条排队就丢最早的。idle 超过 600 秒回收 slot，slot 超过 100 触发清理。

后到的消息不会立刻起一个 run，而是 `coalesce` 到 `pending_messages` 队列，下次 run 开始时一起 `drain_pending` 取出，合并为新输入。这避免了「用户连发 3 条」就启动 3 个独立 agent run 的灾难。

## 工具超时分级

```
工具输入 timeout_secs   ──► 最高优先（同次调用即时覆盖）
         │
         ▼ (未指定)
tool_timeout_overrides  ──► 中优先（按工具名配置）
         │
         ▼ (未指定)
default_tool_timeout_secs ──► 最低优先（全局默认）
```

```yaml
default_tool_timeout_secs: 30
tool_timeout_overrides:
  bash: 90
  browser: 60
  web_search: 15
```

每工具单独 timeout 比单一全局值更准——`web_search` 等几秒就该判超时，`bash` 给 90 秒不算激进。

## MCP 限流与熔断

每个 MCP server 在 `McpServerConfig` 上声明四件套：

```yaml
mcp_servers:
  - name: my-tool
    command: ./tool
    request_timeout_secs: 30
    max_concurrent_requests: 4
    rate_limit_per_minute: 60
    circuit_breaker_failure_threshold: 5
    circuit_breaker_cooldown_secs: 60
```

调用路径上有三道闸门：

1. `FixedWindowRateLimiter`：超 `rate_limit_per_minute` 直接快速失败。
2. `Semaphore`（容量 = `max_concurrent_requests`）：阻塞但不丢请求。
3. `CircuitBreakerState`：连续失败累计达 `failure_threshold` 进 open，冷却结束前所有请求快速失败。

三个独立计数器都进 `metrics_history` 的 `mcp_*_rejections` 列，可以分辨「上游限速」「下游堵」与「上游真挂了」。

## 子代理预算

```yaml
subagents:
  max_concurrent: 4              # 系统级 semaphore
  max_active_per_chat: 2         # 单 chat 上限
  run_timeout_secs: 900          # 15 分钟（默认 900）
  max_tokens_per_run: 400000     # 单次 token 预算（默认 400k）
  max_spawn_depth: 1             # 默认 1 层嵌套
  max_children_per_run: 5
```

agent 主循环里另有 `max_tool_iterations`（默认 100）防工具调用死循环；运行到 80% / 95% 时会注入 `<system_notice type="iteration_budget">` 提示模型收尾。

```rust
struct RunBudget {
    max_session_messages: usize,
    memory_token_budget: usize,
    default_tool_timeout_secs: u64,
    parallel_tool_max_concurrency: usize,
    subagent_max_concurrent: usize,
    subagent_max_tokens_per_run: i64,
    max_tool_iterations: usize,
}

impl RunBudget {
    fn should_compact(&self, msg_count: usize) -> bool {
        msg_count > self.max_session_messages
    }
    fn iter_pct(&self, current: usize) -> usize {
        if self.max_tool_iterations == 0 { 0 }
        else { (current * 100) / self.max_tool_iterations }
    }
}
```

## 控制面并发

Web 路径上几个隐式优化：

- `audit_logs` 写入异步 spawn，不阻塞 HTTP 响应。
- session token 与 API key 验证结果带短 TTL 缓存（避免每个请求都读 DB）。
- SSE/WebSocket 用独立任务处理，不与请求线程共享状态。

## 模型成本闭环

`AppState` 暴露 `Box<dyn LlmProvider>` 抽象，并通过两张运行时表精细化路由：

```rust
pub struct AppState {
    pub llm: Box<dyn LlmProvider>,
    pub llm_provider_overrides: Arc<RwLock<HashMap<String, String>>>,
    pub llm_model_overrides: Arc<RwLock<HashMap<String, String>>>,
    // ...
}
```

不同 chat 可以走不同 provider 或模型——例如「成本敏感的群组用 Haiku 类小模型，重要工作 chat 用 Sonnet 类」。

```yaml
model_prices:
  - model: "claude-sonnet-4-20250514"
    input_per_million_usd: 3.0
    output_per_million_usd: 15.0
  - model: "*"
    input_per_million_usd: 5.0
    output_per_million_usd: 15.0
```

每次 LLM 调用后记录 input/output token 数，匹配 `model_prices` 计入 `subagent_runs` 与 `metrics_history`。

## 重复调用抑制

`tool_cache::cache_key()` 在工具调用前计算归一化 key：把输入 JSON 按字段排序，去掉 `auth`、`request_id` 等无关字段，再算 hash。同一 wave 内、相同 key 的两次调用直接复用结果。

```rust
pub fn cache_key(tool_name: &str, input: &serde_json::Value) -> String {
    let normalized = canonicalize(input); // 排序、剔除无关字段
    let mut hasher = sha2::Sha256::new();
    hasher.update(tool_name.as_bytes());
    hasher.update(normalized.to_string().as_bytes());
    format!("{:x}", hasher.finalize())
}
```

## 优化顺序

1. 打开 trace + metrics，看清瓶颈到 wave 粒度
2. 启用并行工具执行（绝大多数场景延迟降 50% 以上）
3. 调上下文长度与 `memory_token_budget`
4. 调每工具 timeout 与 MCP `circuit_breaker / rate_limit`
5. 配置 `model_prices`，建立成本视图
6. 调子代理并发与 `max_tokens_per_run`
7. 数据库或单进程被证实为瓶颈，再考虑架构改动

## 容易走错的地方

**不启用并行工具执行**：白白损失 50% 以上延迟优化空间。

**用全局锁代替 per-chat 锁**：系统并发能力退化为 1。

**子代理无预算限制**：一次工具调用就能消耗整月 API 预算。

**只看平均耗时**：拖垮体验的总是某个 wave 中最慢的工具。结合 wave 级 OTLP span 才能定位。

**无差别全局超时**：`web_search` 给 90 秒等于不给超时。

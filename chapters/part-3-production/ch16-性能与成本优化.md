# Chapter 16 性能与成本优化

## 性能瓶颈在哪里

Agent Runtime 的总耗时由 LLM 推理、token 上下文长度、工具执行（含 wave 结构）、外部依赖和子代理运行叠加决定。核心优化：**wave-based 并行工具执行** + **per-chat turn serialization**。

## 并行工具执行

### wave-based 并行

`partition_into_waves` 按 concurrency class 分组：

1. **Wave 1**：所有 `ReadOnly` 工具并行（`tokio::JoinSet`）
2. **后续 wave**：`SideEffect` 逐个顺序
3. **独占 wave**：`Exclusive` 独占

### 收益量化

典型代码搜索：3 个 `grep` + 1 个 `read_file` + 1 个 `bash`

- **串行**：5 × 1.5s = 7.5s
- **wave 并行**：Wave 1（4 ReadOnly）≈ 1.5s + Wave 2（bash）≈ 1.5s = 3s
- **收益**：延迟降低 60%，无需改 prompt

### 配置覆盖

```yaml
tool_concurrency_overrides:
  query_db: "read_only"
```

### 并行度上限

`parallel_tool_max_concurrency` 防止文件描述符限制、API rate limit 反弹和 SQLite 写锁竞争。

```rust
use tokio::sync::Semaphore;
use std::sync::Arc;

async fn execute_wave(
    tools: Vec<String>,
    max_concurrency: usize,
) -> Vec<String> {
    let sem = Arc::new(Semaphore::new(max_concurrency));
    let mut set = tokio::task::JoinSet::new();
    for tool in tools {
        let permit = sem.clone().acquire_owned().await.unwrap();
        set.spawn(async move {
            let result = execute_tool(&tool).await;
            drop(permit);
            result
        });
    }
    let mut results = Vec::new();
    while let Some(Ok(result)) = set.join_next().await {
        results.push(result);
    }
    results
}
```

```python
import asyncio


async def execute_wave(
    tools: list[str], max_concurrency: int
) -> list[str]:
    sem = asyncio.Semaphore(max_concurrency)
    async def run(tool: str) -> str:
        async with sem:
            return await execute_tool(tool)
    return await asyncio.gather(*(run(t) for t in tools))
```

## per-chat Turn Serialization

| 维度 | 全局锁 | per-chat 锁 |
|------|--------|-------------|
| 并发能力 | = 1 | = 活跃 chat 数 |
| 跨 chat 阻塞 | 有 | 无 |
| 状态一致性 | 保证 | 保证（per-chat） |

资源管理：idle TTL 600 秒回收、超 100 slot 触发清理、`max_pending` 限制队列深度。

消息合并：agent 运行时后到的消息 coalesce 进 pending 队列，`drain_pending` 一次性取出合并为下次 run 输入。

## 子代理预算

```yaml
subagents:
  max_concurrent: 4          # 系统级 semaphore
  max_active_per_chat: 2     # 单 chat 上限
  run_timeout_secs: 300      # 5 分钟
  max_tokens_per_run: 50000  # token 预算
```

即使子代理陷入无限循环，损失也被限制在单次预算内。

## MCP 性能治理

| 机制 | 配置项 | 防护目标 |
|------|--------|---------|
| 并发限制 | `max_concurrent_requests` | 防并发堆积 |
| 队列等待 | `queue_wait_ms` | 防排队过长 |
| 速率限制 | `rate_limit_per_minute` | 避免上游限流 |
| 超时 | `request_timeout_secs` | 防单请求卡死 |
| 熔断 | circuit breaker | 连续失败自动切断 |

## 优化顺序

1. 打开 trace/metrics，看清瓶颈（含 ToolWave 粒度）
2. 启用并行工具执行
3. 控制上下文长度和记忆预算
4. 调整工具和 MCP 的 timeout/concurrency
5. 配置 `model_prices` 建立成本视图
6. 调整子代理并发和预算
7. 数据库或单进程被证实为瓶颈时，再考虑架构改动

## 示例代码：显式预算对象

```rust
struct RunBudget {
    max_session_messages: usize,
    memory_token_budget: usize,
    default_tool_timeout_secs: u64,
    parallel_tool_max_concurrency: usize,
    subagent_max_concurrent: usize,
    subagent_max_tokens_per_run: usize,
}

impl RunBudget {
    fn can_compact(&self, message_count: usize) -> bool {
        message_count > self.max_session_messages
    }
    fn wave_concurrency(&self) -> usize {
        self.parallel_tool_max_concurrency
    }
}
```

```python
from dataclasses import dataclass


@dataclass
class RunBudget:
    max_session_messages: int
    memory_token_budget: int
    default_tool_timeout_secs: int
    parallel_tool_max_concurrency: int
    subagent_max_concurrent: int
    subagent_max_tokens_per_run: int

    def can_compact(self, message_count: int) -> bool:
        return message_count > self.max_session_messages
    def wave_concurrency(self) -> int:
        return self.parallel_tool_max_concurrency
```

## 容易走错的地方

### 失败模式 1：不启用并行工具执行

白白浪费 60%+ 延迟优化空间。

### 失败模式 2：用全局锁代替 per-chat 锁

系统并发能力退化为 1。

### 失败模式 3：子代理无预算限制

一次调用消耗整月 API 预算。

### 失败模式 4：只看平均耗时

真正拖垮体验的是某个 wave 中最慢的工具。结合 ToolWave 事件和 OTLP span 定位。

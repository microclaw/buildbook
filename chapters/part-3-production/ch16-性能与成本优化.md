# Chapter 16 性能与成本优化

## 这一章要回答什么问题

Agent Runtime 的性能优化，和传统接口服务不完全一样。它的总耗时不是只由 CPU 或数据库决定，而是由几类成本叠加出来：

- LLM 推理时间
- token 上下文长度
- 工具执行时长
- 外部依赖等待时间
- 数据库存取与控制面并发

与此同时，成本也不只是服务器成本，还包括模型费用、网络请求费用、人工排障成本和误触发造成的浪费。MicroClaw 当前已经提供了不少用于控速控费的开关和观测口径。

这一章读完后，你应该已经知道：

1. 优化要先盯哪几个主瓶颈。
2. 上下文成本控制为什么是第一优先级。
3. 工具和 MCP 的超时、并发、预算如何共同工作。
4. 为什么成本优化必须以观测和默认值治理为前提。

## 先看真正的大头：大多数时间花在哪里

对 MicroClaw 这类系统来说，典型请求的总时间主要来自下面几段：

1. 模型调用
2. 工具执行
3. 多轮 tool loop 迭代
4. 会话压缩或记忆检索
5. 外部渠道发回消息

其中最常见的大头通常不是 SQLite，而是：

- 模型本身延迟
- 太长的上下文
- 某个慢工具
- MCP 远程调用抖动

这也是为什么优化不应从“先把数据库换掉”开始，而应先通过 trace 和 metrics 找到实际主耗时。

## 上下文成本控制：最先值得优化的地方

### `max_session_messages` 与 compaction

`config.rs` 里的默认值已经很明确：

- `max_session_messages = 40`
- `compact_keep_recent = 20`
- `memory_token_budget = 1500`

这说明 MicroClaw 默认就假设上下文需要被控制，而不是无限增长。Agent Runtime 里最隐蔽、但最昂贵的成本之一，就是不断膨胀的 prompt。它会同时带来：

- 更慢的模型响应
- 更高的 token 支出
- 更高的幻觉和上下文污染概率

compaction 因此不仅是性能优化，也是成本优化和质量优化。

### 记忆预算而不是记忆贪婪

结构化记忆越多，不代表每次都该注入越多。`memory_token_budget` 的存在，说明系统已经把记忆视为可预算资源。对生产系统来说，这非常关键，因为记忆的边际收益是递减的，而 token 成本不是。

优化时最值得做的，通常不是“把所有历史偏好都灌给模型”，而是提高 relevant recall，然后保持注入预算稳定。

### 小例子：为什么预算对象应该显式存在？

性能和成本优化最容易失控的地方，是预算规则散在各个配置项和调用点里，没人知道一轮请求到底背了多少成本约束。把预算收成对象，可以让团队先讨论策略，再讨论数字。

Rust 版本用 `RunBudget` 同时收纳上下文、记忆和工具预算。这样后续做超时选择、token 裁剪或子代理限制时，都有统一入口。

```rust
struct RunBudget {
    max_session_messages: usize,
    memory_token_budget: usize,
    default_tool_timeout_secs: u64,
}

impl RunBudget {
    fn can_compact(&self, message_count: usize) -> bool {
        message_count > self.max_session_messages
    }
}
```

Python 版本用 `@dataclass` 表达同样的预算对象。例子虽然短，但它强调的是一个运行时方法论：预算不该只是若干零散常量。

```python
from dataclasses import dataclass


@dataclass
class RunBudget:
    max_session_messages: int
    memory_token_budget: int
    default_tool_timeout_secs: int

    def can_compact(self, message_count: int) -> bool:
        return message_count > self.max_session_messages
```

## 工具执行：超时、默认值和局部覆盖

### 全局默认超时

MicroClaw 当前已经把 `default_tool_timeout_secs` 作为正式配置，默认值为 `30`。这一步非常重要，因为没有默认超时，慢工具会直接把整个请求拖死。

### per-tool override

同样重要的是 `tool_timeout_overrides.<tool_name>`。这让系统可以承认不同工具的耗时分布不同，例如：

- `bash` 可能需要更长预算
- `browser` 可能需要更长页面加载时间
- `web_fetch` 和 `web_search` 适合更短时限

这种分层预算比“一刀切 30 秒”更合理。

### 输入优先级

运行手册已经定义了 timeout 的优先级：

- 工具输入里的 `timeout_secs`
- `tool_timeout_overrides`
- `default_tool_timeout_secs`

这让局部任务仍然能在全局策略之上申请更精细预算，但前提是团队显式做出决定。

## MCP 与外部依赖：性能优化首先是避免被拖慢

MCP 的问题不只是慢，还可能出现：

- 并发堆积
- 队列等待过长
- 连续失败导致熔断
- 上游限流

所以 `src/mcp.rs` 的性能治理手段也很典型：

- `max_concurrent_requests`
- `queue_wait_ms`
- `rate_limit_per_minute`
- `request_timeout_secs`
- circuit breaker

这说明在 MicroClaw 的设计里，MCP 性能优化的第一目标不是“永远最快”，而是“最坏情况下也不要拖垮主链路”。对运行时来说，这是正确优先级。

## Web 与控制面的并发限制

性能优化不只是模型和工具。Web 控制面也有自己的资源保护边界，例如：

- `web_max_inflight_per_session`
- `web_max_requests_per_window`
- `web_rate_window_seconds`
- `web_run_history_limit`

这些配置的意义在于防止：

- 同一 session 触发过多并发运行
- 前端或脚本瞬间压爆控制面
- 流式历史无限增长导致内存压力

也就是说，性能优化在控制面上通常不是“让它接受更多请求”，而是“让它在可承受范围内稳定退化”。

## 模型成本：从 token 到美元的闭环

`config.rs` 中存在 `model_prices` 配置和对应的匹配函数。这说明 MicroClaw 已经开始把“模型调用量”进一步映射到“成本估算”。

这类能力非常重要，因为性能优化如果不和成本视图绑定，很容易走向单一方向：

- 快了，但更贵
- 便宜了，但质量大幅下降

真正可用的优化策略，应该同时观察：

- 输入 token
- 输出 token
- tool loop 次数
- 每种模型的单价

这样你才能回答一个真正的生产问题：某个优化到底是在省钱，还是只是在转移成本。

## Subagents 与并发预算

MicroClaw 的子代理体系也有明显的并发边界，例如：

- `subagents.max_concurrent`
- `subagents.max_active_per_chat`
- `subagents.run_timeout_secs`
- `subagents.max_tokens_per_run`

这些限制的价值非常直接：避免“后台并行很酷”最后演变成“主系统被子任务拖垮”。这再次说明，性能优化在 Agent Runtime 里经常表现为预算管理，而不是极限压榨吞吐。

## Observability 队列与历史保留

性能和成本优化还会落到观测系统自己身上。比如：

- `otlp_queue_capacity`
- OTLP retry / backoff
- `metrics_history_retention_days`

这些配置体现了一个常被忽略的现实：观测不是免费。导出过多、重试过猛、保留太久，都可能反过来给主系统造成额外压力。MicroClaw 通过显式配置这些参数，允许团队在“观测完整度”和“系统负担”之间做理性平衡。

## 优化顺序：先高收益低风险，再深层重构

如果要按照 MicroClaw 的经验给出一条优化顺序，最合理的次序通常是：

1. 先打开 trace 和 metrics，看清真正瓶颈。
2. 控制上下文长度和记忆预算。
3. 调整工具和 MCP 的 timeout / concurrency。
4. 优化最慢工具或最不稳定上游依赖。
5. 只有在数据库或单进程装配被证明确实成为瓶颈时，再考虑更重的架构改动。

这条顺序和本书前面的技术方法论完全一致：先优化最大约束，而不是先做最复杂的重构。

## 示例代码：为什么超时预算要按能力分层，而不是全局一个大数？

性能优化里最常见、也最有效的一类动作，就是把不同能力纳入不同预算，而不是给系统一个统一的大超时。统一大超时看似简单，实际上会把慢路径、坏路径和正常路径全部混在一起。

Rust 版本把超时选择抽成 trait，再由 policy struct 负责给不同工具分配预算。这样预算规则可以被单独调优和测试，而不用散落在每个调用点的字面量里。

```rust
#[async_trait::async_trait]
trait TimeoutPolicy {
    async fn timeout_for(&self, tool_name: &str) -> std::time::Duration;
}

struct DefaultTimeoutPolicy;

#[async_trait::async_trait]
impl TimeoutPolicy for DefaultTimeoutPolicy {
    async fn timeout_for(&self, tool_name: &str) -> std::time::Duration {
        let seconds = match tool_name {
            "bash" => 90,
            "browser" => 45,
            _ => 30,
        };
        std::time::Duration::from_secs(seconds)
    }
}
```

Python 版本保留相同的结构，但用 `Protocol` 和 `@dataclass` 把策略对象显式化。这样示例表达的重点不是某几个数字，而是“预算本身也是运行时策略”，应该有独立边界。

```python
from dataclasses import dataclass
from typing import Protocol


class TimeoutPolicy(Protocol):
    async def timeout_for(self, tool_name: str) -> int: ...


@dataclass
class DefaultTimeoutPolicy:
    async def timeout_for(self, tool_name: str) -> int:
        if tool_name == "bash":
            return 90
        if tool_name == "browser":
            return 45
        return 30
```

## 关键权衡

### 决策一：把上下文窗口控制作为默认行为

优点是同时降低延迟和 token 成本。代价是需要持续关注 compaction 质量。

### 决策二：通过 timeout 和 override 管理工具预算

优点是能快速止损慢工具。代价是错误设置过短时可能影响任务完成率。

### 决策三：把外部依赖性能问题视为韧性问题

优点是系统在依赖抖动时更稳。代价是限流、熔断和排队会让部分请求更早失败。

### 决策四：把成本估算纳入配置和观测

优点是优化不再只看“快不快”。代价是需要维护模型价格信息和成本口径。

## 容易走错的地方

### 失败模式 1：把优化等同于数据库或语言重构

在多数 Agent Runtime 中，最先该优化的不是这两者。

### 失败模式 2：为了追求质量，无上限地扩大上下文

这通常会让延迟、成本和噪声一起上升。

### 失败模式 3：只看平均耗时，不看最慢工具和外部依赖

真正拖垮体验的往往是尾部，而不是均值。

- 你是否为工具、MCP、子代理和控制面分别设置了并发与超时预算？
- 你是否在优化时同时看性能收益和成本变化？

## 小结

MicroClaw 的性能与成本优化路线很务实：把上下文、工具、外部依赖和并发预算当成主要约束，再通过观测来决定优化顺序。这种方法不会让系统一下子变成极限性能机器，但能让它在真实生产约束下逐步变得更快、更稳、更省。

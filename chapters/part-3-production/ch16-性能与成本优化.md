# Chapter 16 性能与成本优化

## Agent Runtime 的性能瓶颈在哪里，成本如何控制？

Agent Runtime 的性能优化和传统接口服务不完全一样。总耗时不是只由 CPU 或数据库决定，而是由几类成本叠加：

- LLM 推理时间
- token 上下文长度
- 工具执行时长（现在还包括并行 wave 的结构）
- 外部依赖等待时间
- 子代理运行时间和资源消耗

MicroClaw 在性能与成本方面最重要的两个设计是：**并行工具执行**（wave-based）作为主要优化手段，以及 **per-chat turn serialization** 替代全局锁，从根本上改善多 chat 并发性能。

这一章读完后，你应该已经知道：

1. 并行工具执行如何在单次 turn 内压缩延迟。
2. per-chat 粒度为什么比全局锁更优。
3. 子代理的预算和并发控制如何防止成本失控。
4. wave-based 批处理带来了哪些具体收益。

## 并行工具执行：核心性能优化

### 问题：为什么串行执行是瓶颈

如果没有并行调度，即使 LLM 一次返回 5 个工具调用，系统也只能逐个顺序执行。如果每个工具平均耗时 2 秒，5 个工具就是 10 秒。但实际上，其中 3 个可能是纯读操作（`read_file`、`grep`、`glob`），它们之间没有依赖，完全可以并行。

### 解法：wave-based 并行

`tool_executor.rs` 的 `partition_into_waves` 按 concurrency class 把工具调用分组：

1. **Wave 1**：所有 `ReadOnly` 工具并行执行（`tokio::JoinSet`）
2. **后续 wave**：`SideEffect` 工具逐个顺序执行
3. **独占 wave**：`Exclusive` 工具独占一个 wave

`parallel_tool_max_concurrency` 控制单个 wave 内的最大并行度。

### 收益量化

考虑一个典型的代码搜索场景：LLM 返回 3 个 `grep` + 1 个 `read_file` + 1 个 `bash`。

- **串行**：5 个工具 × 平均 1.5s = 7.5s
- **wave 并行**：Wave 1（4 个 ReadOnly 并行）≈ 1.5s + Wave 2（bash 串行）≈ 1.5s = 3s
- **收益**：延迟降低 60%

这种优化不需要模型或 prompt 做任何改变，完全是运行时层面的透明加速。

### 配置覆盖

`tool_concurrency_overrides` 允许把特定工具的默认 class 覆盖为其他级别。例如，如果某个自定义工具名为 `query_db` 但实际只做读操作，可以手动将其标记为 ReadOnly 以参与并行。

```yaml
tool_concurrency_overrides:
  query_db: "read_only"
```

### 并行度为什么应该有上限？

无限并行看起来最快，但实际上会遇到文件描述符限制、模型 API rate limit 反弹和共享资源竞争（如 SQLite 写锁）。`parallel_tool_max_concurrency` 的存在不是限制性能，而是让性能可预测。

Rust 版本用 Semaphore 控制并行度，这也是 MicroClaw 实际使用的模式。

```rust
use tokio::sync::Semaphore;
use std::sync::Arc;

async fn execute_wave(
    tools: Vec<String>,
    max_concurrency: usize,
) -> Vec<String> {
    let sem = Arc::new(Semaphore::new(max_concurrency));
    let mut set = tokio::task::JoinSet::new();
    for tool in tools {
        let permit = sem.clone().acquire_owned().await.unwrap();
        set.spawn(async move {
            let result = execute_tool(&tool).await;
            drop(permit);
            result
        });
    }
    let mut results = Vec::new();
    while let Some(Ok(result)) = set.join_next().await {
        results.push(result);
    }
    results
}
```

Python 版本用 `asyncio.Semaphore` 表达同样的思路。

```python
import asyncio


async def execute_wave(
    tools: list[str], max_concurrency: int
) -> list[str]:
    sem = asyncio.Semaphore(max_concurrency)
    async def run(tool: str) -> str:
        async with sem:
            return await execute_tool(tool)
    return await asyncio.gather(*(run(t) for t in tools))
```

## per-chat Turn Serialization：不是性能瓶颈，而是性能基础

### 全局锁的问题

如果使用全局互斥锁来防止并发竞态，后果是：

- Telegram chat 1 在运行 → Discord chat 2 被阻塞
- 高延迟的 agent run（比如需要浏览器工具）会阻塞所有其他 chat
- 系统的实际并发能力等于 1

### per-chat 粒度的解法

`ChatTurnQueue` 以 `(channel, chat_id)` 为 key，每个 chat 有独立的 `turn_lock`。效果：

- 不同 chat 完全独立，互不阻塞
- 同一 chat 内严格序列化，保证状态一致性
- 系统的实际并发能力 = 活跃 chat 数

### 资源管理

per-chat 锁的代价是每个活跃 chat 需要一个 `ChatSlot`（包含一个 `Arc<Mutex<()>>` 和一个 `Vec<PendingMessage>`）。`ChatTurnQueue` 通过以下机制管理资源：

- **idle TTL**：600 秒无活动的 slot 被回收
- **触发时机**：slot 数量超过 100 时触发清理
- **max_pending**：限制每个 chat 的待处理消息队列深度

这意味着即使有上千个 chat 曾经活跃过，内存占用也只和当前活跃的 chat 数成正比。

### 消息合并

`ChatTurnQueue` 的另一个性能特性是消息合并。当 agent 正在处理 chat 1 的请求时，用户在同一 chat 中又发了两条消息。这两条消息不会被丢弃，也不会触发新的 agent run，而是被 coalesce 进 pending 队列。当前 run 完成后，`drain_pending` 一次性取出所有待处理消息，合并为下一次 run 的输入。

这比"丢弃后到的消息"更友好，比"每条消息都排队执行"更高效。

## 上下文成本控制

### `max_session_messages` 与 compaction

```yaml
max_session_messages: 40
compact_keep_recent: 20
memory_token_budget: 1500
```

Agent Runtime 里最隐蔽但最昂贵的成本，是不断膨胀的 prompt。它同时带来更慢的模型响应、更高的 token 支出、更高的幻觉和上下文污染概率。

compaction 因此不仅是性能优化，也是成本优化和质量优化。

### 记忆预算而不是记忆贪婪

`memory_token_budget` 的存在，说明系统已经把记忆视为可预算资源。优化时最值得做的，通常不是"把所有历史偏好都灌给模型"，而是提高 relevant recall，然后保持注入预算稳定。

## 模型成本：从 token 到美元的闭环

`config.rs` 中的 `model_prices` 配置把 token 消耗映射为美元成本：

```yaml
model_prices:
  - model: "claude-sonnet-4-20250514"
    input_per_million_usd: 3.0
    output_per_million_usd: 15.0
  - model: "*"
    input_per_million_usd: 5.0
    output_per_million_usd: 15.0
```

通配符 `*` 提供了兜底定价。系统在每次 LLM 调用后记录 input/output token 数量，结合 `model_prices` 计算本次调用的成本估算。这个估算会被写入 `subagent_runs` 表（对子代理运行）和 metrics 系统。

有了这个闭环，团队可以回答：

- 这个 chat 今天花了多少钱？
- 子代理运行的平均成本是多少？
- compaction 每天省了多少 token？

## 子代理预算与并发控制

### 并发限制

```yaml
subagents:
  max_concurrent: 4          # 系统级 semaphore
  max_active_per_chat: 2     # 单 chat 上限
  run_timeout_secs: 300      # 5 分钟
  max_tokens_per_run: 50000  # token 预算
```

`max_concurrent` 使用 tokio Semaphore 实现，防止所有 chat 的子代理加起来超过系统承受能力。`max_active_per_chat` 防止单个 chat 独占所有子代理名额。

### 成本隔离

子代理的 token 预算（`max_tokens_per_run`）和运行超时（`run_timeout_secs`）共同构成了成本隔离边界。即使子代理陷入无限循环或过度使用工具，损失也被限制在单次运行的预算内。

### 权衡：并发度 vs 资源消耗

每个子代理在独立进程中运行，消耗独立的内存和 CPU。`max_concurrent = 4` 意味着最坏情况下有 4 个额外进程同时运行。在内存受限的环境（如低配家庭服务器），这个值可能需要下调。

## 工具执行：超时与分层预算

### 全局默认超时

`default_tool_timeout_secs = 30` 确保任何工具都不会无限等待。

### per-tool override

```yaml
tool_timeout_overrides:
  bash: 90
  browser: 60
  web_search: 15
```

不同工具的耗时分布差异巨大。`bash` 可能需要编译代码，`browser` 需要加载页面，而 `web_search` 应该很快返回。分层预算比一刀切更合理。

### 超时选取优先级

```
工具输入 timeout_secs  ──► 最高优先
         │
         ▼ (未设置)
tool_timeout_overrides ──► 中等优先
         │
         ▼ (未设置)
default_tool_timeout_secs ──► 最低优先（默认 30s）
```

## MCP 与外部依赖：避免被拖慢

MCP 的问题不只是慢，还可能出现并发堆积、队列等待过长、连续失败导致熔断、上游限流。`src/mcp.rs` 的性能治理手段：

| 机制 | 配置项 | 防护目标 |
|------|--------|---------|
| 并发限制 | `max_concurrent_requests` | 防止并发堆积 |
| 队列等待 | `queue_wait_ms` | 防止排队过长 |
| 速率限制 | `rate_limit_per_minute` | 避免上游限流 |
| 超时 | `request_timeout_secs` | 防止单请求卡死 |
| 熔断 | circuit breaker | 连续失败时自动切断 |

设计的第一目标不是"永远最快"，而是"最坏情况下不拖垮主链路"。

## Observability 队列与历史保留

观测不是免费的。相关的性能配置包括：

- `otlp_queue_capacity`：导出队列深度
- OTLP retry / backoff 策略
- `metrics_history_retention_days`：本地持久化保留天数

导出过多、重试过猛、保留太久，都可能反过来给主系统造成压力。

## 优化顺序：先高收益低风险，再深层重构

推荐的优化顺序是：

1. 先打开 trace 和 metrics，看清真正瓶颈（包括 ToolWave 级别的粒度）。
2. 启用并行工具执行，调整 `parallel_tool_max_concurrency`。
3. 控制上下文长度和记忆预算。
4. 调整工具和 MCP 的 timeout / concurrency。
5. 配置 `model_prices`，建立成本视图。
6. 调整子代理并发和预算。
7. 只有在数据库或单进程被证实成为瓶颈时，再考虑更重的架构改动。

## 示例代码：为什么预算对象应该显式存在？

性能和成本优化最容易失控的地方，是预算规则散在各个配置项和调用点里，没人知道一轮请求到底背了多少成本约束。把预算收成对象，可以让团队先讨论策略，再讨论数字。

Rust 版本用 `RunBudget` 同时收纳上下文、记忆、工具和子代理预算。注意并行度和子代理预算字段。

```rust
struct RunBudget {
    max_session_messages: usize,
    memory_token_budget: usize,
    default_tool_timeout_secs: u64,
    parallel_tool_max_concurrency: usize,
    subagent_max_concurrent: usize,
    subagent_max_tokens_per_run: usize,
}

impl RunBudget {
    fn can_compact(&self, message_count: usize) -> bool {
        message_count > self.max_session_messages
    }

    fn wave_concurrency(&self) -> usize {
        self.parallel_tool_max_concurrency
    }
}
```

Python 版本用 `@dataclass` 表达同样的预算对象。

```python
from dataclasses import dataclass


@dataclass
class RunBudget:
    max_session_messages: int
    memory_token_budget: int
    default_tool_timeout_secs: int
    parallel_tool_max_concurrency: int
    subagent_max_concurrent: int
    subagent_max_tokens_per_run: int

    def can_compact(self, message_count: int) -> bool:
        return message_count > self.max_session_messages

    def wave_concurrency(self) -> int:
        return self.parallel_tool_max_concurrency
```

## 关键权衡

### 决策一：wave-based 并行而非无限并行

优点是延迟大幅降低，同时保持了 SideEffect 工具的安全顺序执行。代价是 ReadOnly 分类需要准确，错误分类可能导致数据竞态。通过 `tool_concurrency_overrides` 可以手动纠正。

### 决策二：per-chat 锁而非全局锁

优点是不同 chat 之间零阻塞。代价是内存占用随活跃 chat 数增长，但通过 idle TTL 清理和 max_pending 限制，实际开销可控。

### 决策三：子代理预算作为硬性上限

优点是成本可预测、可控。代价是某些复杂任务可能在 token 预算内无法完成，需要调大或分拆。

### 决策四：model_prices 支持通配符

优点是即使没有精确定价的模型也能有成本估算。代价是通配符价格可能不准确。

## 容易走错的地方

### 失败模式 1：不启用并行工具执行

并行执行是最容易获得的性能收益。不启用它等于白白浪费了 60%+ 的延迟优化空间。

### 失败模式 2：用全局锁代替 per-chat 锁

全局锁让系统的实际并发能力退化为 1，即使有 100 个活跃 chat 也只能排队。

### 失败模式 3：子代理无预算限制

没有 token 上限和并发控制的子代理，可能在一次调用中消耗掉整个月的 API 预算。

### 失败模式 4：只看平均耗时，不看 wave 级别的工具延迟

真正拖垮体验的往往是某个 wave 中最慢的那个工具。结合 ToolWaveStart/ToolWaveComplete 事件和每个工具的独立 OTLP span duration 是定位这类问题的关键。

## 小结

MicroClaw 的性能与成本优化围绕两个核心设计展开：wave-based 并行工具执行在单次 turn 内压缩了 60%+ 的工具延迟，per-chat turn serialization 在多 chat 场景下将并发能力从 1 提升到了活跃 chat 数。配合 model_prices 成本映射和子代理预算控制，团队可以在延迟、成本和安全之间找到更精确的平衡点。

下一章我们讨论架构演进。因为当系统已经具备可运行、可观测和可控成本之后，团队马上会面临一个新的问题：下一步该往哪里拆、怎么拆、什么时候不该拆。

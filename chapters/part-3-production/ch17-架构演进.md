# Chapter 17 架构演进

## 这一章要回答什么问题

到了这一章，我们已经讨论了一个可用 MicroClaw 所需的大部分核心能力。真正的难题不再是“怎么从零搭起来”，而是“系统已经跑起来以后，接下来该怎么演进而不把自己拆散”。

MicroClaw 当前仓库本身就展示了一条很有价值的演进轨迹：

- 从较集中的 `src/*` 主链路出发
- 逐步把稳定能力抽到多个 crate
- 仍保持单二进制 runtime
- 再通过 A2A、MCP、ClawHub、Web 控制面为进一步扩展留接口

这一章读完后，你应该理解：

1. 什么信号说明系统该拆，什么信号说明还不该拆。
2. 配置、数据、协议三种兼容性该如何分别治理。
3. 本地、家庭服务器和云部署在架构上有什么不同侧重点。
4. 怎样设计一条“可持续演进而非持续返工”的 roadmap。

## 演进的第一原则：先拆边界，再拆进程

MicroClaw 当前最值得学习的地方，是它优先做了 crate 级边界治理，而不是直接把系统拆成多个服务。

这种顺序非常合理。因为多数系统真正的首要问题不是“必须多进程”，而是“职责已经混乱”。如果模块边界尚未清晰，过早拆服务通常只会把：

- 隐式耦合
- 状态不一致
- 调试复杂度

一起放大。

MicroClaw 先把 `storage`、`tools`、`channels`、`observability`、`core` 等稳定能力抽出来，让顶层 `src/` 继续负责高层编排。只有在这种情况下，未来讨论服务拆分才有基础。

## 什么时候应该保持单体

对当前阶段的 MicroClaw 来说，单体仍然有明显优势：

- 部署极简
- 本地数据一致性强
- 统一循环不被 RPC 切碎
- Web 控制面和后台任务能直接共享 `AppState`
- 调试路径短

当下面这些条件仍然成立时，保持单体通常是正确选择：

1. 核心瓶颈还不在进程边界。
2. 团队规模还不足以支撑大量服务治理。
3. 主要部署形态仍是单机、本地或轻量家庭服务器。
4. 运行时一致性比水平扩展更重要。

这不是保守，而是工程现实。

## 什么时候应该考虑拆分

相反，如果你开始持续遇到下面这些信号，才说明服务化可能值得认真考虑：

- 某个子系统的资源消耗长期压制主链路，例如浏览器自动化或大规模索引构建。
- 某个能力的发布节奏、依赖关系和主 runtime 已明显不同步。
- 单进程权限边界已经不足以隔离高风险能力。
- 某个外部接口需要单独扩缩容，例如公开 Web API 或专门的 A2A 网关。

也就是说，拆分的理由应该来自真实运行约束，而不是架构时髦值。

### 小例子：为什么兼容性检查也应该先抽成接口？

架构演进阶段最怕的是每次升级都临时拼一张检查清单，结果配置、数据和协议各看各的。把兼容性检查抽成明确接口，才能把升级风险变成持续治理对象。

Rust 版本把检查动作收进 `CompatibilityCheck` trait，再让 `UpgradePlan` 汇总结果。这样演进时的判断标准可以被复用、测试和自动化。

```rust
trait CompatibilityCheck {
    fn run(&self) -> anyhow::Result<()>;
}

struct UpgradePlan<C> {
    config_check: C,
    requires_backup: bool,
}
```

Python 版本用 `Protocol` 和 `@dataclass` 保持同样的结构。这个例子要强调的不是类型技巧，而是“兼容性本身也要工程化”。

```python
from dataclasses import dataclass
from typing import Protocol


class CompatibilityCheck(Protocol):
    def run(self) -> None: ...


@dataclass
class UpgradePlan:
    config_check: CompatibilityCheck
    requires_backup: bool
```

## 三种兼容性要分开治理

架构演进最容易出问题的地方，是把“兼容性”当成一个抽象口号。对 MicroClaw 来说，至少有三类兼容性要分别处理。

### 配置兼容

`config.rs` 是运行时入口之一，因此任何字段名、默认值、结构层次的变化都会影响升级体验。一个成熟做法应当包括：

- 保持旧字段兼容窗口
- 新旧配置同时给出迁移提示
- 在控制面自检中把危险旧配置显式标红

### 数据兼容

`docs/releases/upgrade-guide.md` 已经明确：数据库 migration 在启动时自动执行，回滚依赖 DB 备份而不是手工逆向 SQL。这个姿态很正确，因为数据层兼容一旦处理不好，后果通常比 API 变更更严重。

### 协议兼容

这里包括：

- Web API 合约
- MCP 接口行为
- session fork 等控制面语义
- A2A 消息格式

协议兼容的关键不是“永不变化”，而是要明确版本和迁移路径。A2A 文档中已经使用了 `microclaw-a2a/v1` 这样的协议标识，就是很好的开始。

## A2A 与外部边界：迈向分布式的最自然切口

`docs/a2a.md` 展示了一个很值得注意的演进方向：MicroClaw 并没有急着把所有能力拆成内部服务，而是先通过 A2A 提供了 instance-to-instance delegation 能力。

这条路线的好处是：

- 主 runtime 仍然内聚
- 但可以开始把专门工作委派给其他实例
- 分布式边界先出现在协议层，而不是先出现在仓库内部

这是一种非常务实的演进方式。因为它允许团队先验证“是否真的需要多实例协作”，再决定是否做更重的架构重构。

## 部署形态：本地、家庭服务器、云环境各看重什么

### 本地开发

重点通常是：

- 安装和启动简单
- `setup` 与 `doctor` 体验顺滑
- SQLite 与工作目录可直接检查

### 家庭服务器或轻量自托管

重点通常是：

- 单二进制可靠运行
- Web 控制面可访问
- sandbox 和备份策略明确
- 调度和升级路径简单

### 云环境

重点通常是：

- 配置和密钥管理规范
- 观测导出到外部平台
- A2A 或公开 API 的边界收敛
- 升级与回滚流程标准化

同一套 MicroClaw 内核能适配不同部署形态，前提恰恰是它先保持了较强的主链路一致性。

## Roadmap 应该围绕什么排优先级

MicroClaw 当前的文档和 RFC 已经能看出一种合理的 roadmap 节奏：

1. 先补稳定性与安全短板。
2. 再扩展控制面和可观测能力。
3. 在此基础上增加新的后台 lane 和协作协议。
4. 最后再考虑更大范围的平台化。

对大多数团队来说，这个顺序非常重要。因为 Agent Runtime 最大的诱惑就是不断扩能力面，但真正的系统价值常常取决于：

- 行为是否一致
- 升级是否安全
- 故障是否可恢复

所以 roadmap 不应只按“新功能吸引力”排序，而应按“主链路稳定收益”排序。

## 技术债治理：哪些债务值得优先还

在 MicroClaw 这类系统里，最值得优先处理的技术债通常不是“代码风格不统一”，而是：

- 模块边界模糊
- 兼容性路径不清
- 控制面和文档漂移
- 测试和运维脚本落后于实现
- 安全开关存在危险默认值

只有这些债先被控制住，后续加渠道、加协议、加生态才不会不断返工。

## 示例代码：为什么演进时应该先抽接口，再决定要不要拆进程？

很多演进问题并不需要立刻服务化。先把边界抽成稳定接口，通常是更低风险的第一步，因为你先验证的是职责是否清楚，而不是网络调用是否华丽。

Rust 版本把存储边界抽成 trait，再让上层 service 依赖这个抽象异步取数。这样哪怕后面真的从 SQLite 换成远程服务，改动也首先落在实现层，而不是把业务代码整体推倒重来。

```rust
#[async_trait::async_trait]
trait MemoryStore {
    async fn load(&self, chat_id: i64) -> anyhow::Result<Vec<String>>;
}

struct SqliteMemoryStore;

#[async_trait::async_trait]
impl MemoryStore for SqliteMemoryStore {
    async fn load(&self, _chat_id: i64) -> anyhow::Result<Vec<String>> {
        Ok(vec!["memory from sqlite".into()])
    }
}

struct MemoryService<S> {
    store: S,
}
```

先把抽象边界和服务对象单独放出来，能更清楚地说明“先抽接口”到底落在哪里。这样后面的 `load_context` 看起来就是在消费抽象，而不是顺手包一层函数。

```rust

impl<S: MemoryStore> MemoryService<S> {
    async fn load_context(&self, chat_id: i64) -> anyhow::Result<Vec<String>> {
        self.store.load(chat_id).await
    }
}
```

Python 版本也保持“接口先行、实现后置”的结构，但用 `Protocol` 和 `@dataclass` 让这种边界更一眼可见。这样示例要传达的重点就不是“怎么拆服务”，而是“为什么先把边界抽干净”。

```python
from dataclasses import dataclass
from typing import Protocol


class MemoryStore(Protocol):
    async def load(self, chat_id: int) -> list[str]: ...


@dataclass
class SqliteMemoryStore:
    async def load(self, chat_id: int) -> list[str]:
        return ["memory from sqlite"]


@dataclass
class MemoryService:
    store: MemoryStore
```

Python 也先定下接口和 service 的形状，再给出真正调用存储的动作。这样示例会把“职责边界”而不是“拆服务技巧”放在视觉中心。

```python

    async def load_context(self, chat_id: int) -> list[str]:
        return await self.store.load(chat_id)
```

## 关键权衡

### 决策一：优先做 crate 级边界治理

优点是为后续演进建立清晰边界。代价是短期内不会立刻得到“分布式平台”的外观。

### 决策二：单体保持为默认部署形态

优点是维护和调试简单。代价是水平扩展和权限隔离能力不是默认获得。

### 决策三：通过 A2A 等协议先试探外部分布式边界

优点是风险更低。代价是协议治理会先于内部服务化到来。

### 决策四：把兼容性拆成配置、数据、协议三条线

优点是升级管理更清晰。代价是文档和测试都要更细致。

## 容易走错的地方

### 失败模式 1：还没理清边界就急着拆服务

最终通常只是把单体里的混乱复制成了网络调用。

### 失败模式 2：把所有兼容性问题混为一谈

这样会导致配置迁移、数据升级和 API 变更互相拖累。

### 失败模式 3：roadmap 只按功能热度排序

系统会越来越“会做事”，但越来越不可靠。

## 读到这里，你应该能回答

- 你是否能说清当前阶段为什么还要保持单体，或者为什么已经到了该拆分的时点？
- 你是否分别治理配置、数据和协议兼容性？
- 你是否为本地、自托管和云部署定义了不同重点？
- 你是否让 roadmap 围绕主链路稳定性排序，而不是只围绕新功能排序？

## 小结

MicroClaw 的架构演进路线提醒我们，真正可持续的系统不是拆得最快的系统，而是边界最先被理清的系统。只有先把内核、兼容性和运维基础打稳，后面的多实例协作、平台化和生态扩展才不会变成反复返工。

# Chapter 17 架构演进

## 演进而不拆散

系统跑起来后，真正的难题是"如何演进而不拆散"。MicroClaw 的策略：8-crate workspace 按稳定性分层、A2A 实例间协作、ACP headless 控制、Gateway 服务管理标准化、feature flags 按需编译。

## 8-crate Workspace

```
[workspace]
members = [
    ".",                           # microclaw (顶层 binary)
    "crates/microclaw-core",       # 错误类型、共享类型（最底层）
    "crates/microclaw-clawhub",    # ClawHub 客户端
    "crates/microclaw-storage",    # SQLite 存储层
    "crates/microclaw-tools",      # 工具定义、sandbox、权限
    "crates/microclaw-channels",   # 渠道抽象层
    "crates/microclaw-app",        # 应用层工具
    "crates/microclaw-observability", # OTLP 导出、指标、trace
]
```

按稳定性分层：`core` 最底层最稳定 → `storage`/`tools`/`observability`/`channels` 各依赖 `core` → 顶层 binary 组装。改 observability 不触发 tools 重新编译。

```rust
// microclaw-core: 最稳定
pub mod error {
    pub enum MicroClawError { Config(String), Storage(String) }
}

// microclaw-storage: 只依赖 core
use microclaw_core::error::MicroClawError;
pub struct Database { /* ... */ }

// microclaw-tools: 只依赖 core
use microclaw_core::error::MicroClawError;
pub fn tool_concurrency_class(name: &str) -> ToolConcurrencyClass { /* ... */ }
```

```python
# microclaw_core/error.py — 最底层
class MicroClawError(Exception):
    pass

# microclaw_storage/db.py — 只依赖 core
from microclaw_core.error import MicroClawError

# microclaw_tools/runtime.py — 只依赖 core
from microclaw_core.error import MicroClawError
```

## A2A：实例间分布式边界

不急着内部拆微服务。A2A 让不同 MicroClaw 实例之间委派任务，主 runtime 内部仍然紧密内聚。

使用场景：本地实例委派浏览器任务给专用实例、云端委派文件操作给边缘实例、跨团队协作。

## ACP：Headless 控制

基于 stdio 的双向 JSON-RPC 协议，让 MicroClaw 可被另一个 agent 当作子代理驱动。

```bash
microclaw acp  # ACP server 模式启动
```

支持 `RequestPermission`、终端输出限制（256KB）、独立进程通信。

## Gateway：服务管理标准化

```bash
microclaw gateway install/start/stop/status/logs
```

| 平台 | 实现 |
|------|------|
| macOS | launchd plist |
| Linux | systemd unit |
| Windows | Windows Service |

`status --deep` 通过 WebSocket RPC 健康检查，`status --json` 适合监控集成。

## Feature Flags

```toml
[features]
default = []
sqlite-vec = ["microclaw-storage/sqlite-vec"]
journald = ["microclaw-app/journald"]
channel-matrix = ["dep:matrix-sdk"]
full = ["channel-matrix"]
```

默认最小编译。`matrix-sdk` 引入大量传递依赖，设为 optional 让默认编译时间可预测。

```rust
#[cfg(feature = "channel-matrix")]
use crate::channels::matrix;

const DYNAMIC_CHANNELS: &[DynamicChannelDef] = &[
    slack::SETUP_DEF,
    #[cfg(feature = "channel-matrix")]
    matrix::SETUP_DEF,
];
```

```python
import importlib

OPTIONAL_CHANNELS: dict[str, str] = {
    "matrix": "microclaw.channels.matrix",
}

def load_channel(name: str):
    module_path = OPTIONAL_CHANNELS.get(name)
    if module_path is None:
        raise ImportError(f"channel {name} not available")
    return importlib.import_module(module_path)
```

## 部署形态

| 维度 | 本地开发 | 家庭服务器 | 云部署 |
|------|---------|-----------|--------|
| 安装 | Homebrew/source | Gateway install | Docker/Nix |
| sandbox | Off | Shared/Chat | Chat |
| 观测 | 本地 metrics | 本地 + 可选 OTLP | OTLP 必选 |
| 鉴权 | 默认密码可接受 | 必须修改密码 | API key + TLS |
| 升级 | `microclaw upgrade` | Gateway 重启 | Rolling update |
| 备份 | 可选 | 必须 | 必须 + 异地 |

## 演进原则：先拆边界，再拆进程

```{=typst}
#pagebreak(weak: true)
```

**保持单体**：瓶颈不在进程边界、团队规模不支撑服务治理、主部署仍是单机。

**考虑拆分**：子系统资源消耗压制主链路、发布节奏不同步、单进程权限隔离不足、需要独立扩缩容。

## 三种兼容性

| 维度 | 影响范围 | 治理策略 |
|------|---------|---------|
| 配置兼容 | 字段名、默认值 | 旧字段兼容窗口、doctor 标红 |
| 数据兼容 | SQLite schema | 启动时自动 migration、备份回滚 |
| 协议兼容 | Web API、MCP、A2A、ACP | 版本标识 + 协商 |

## 容易走错的地方

### 失败模式 1：还没理清 crate 边界就急着拆服务

把单体混乱复制成网络调用。

### 失败模式 2：所有 feature 默认开启

编译时间膨胀，依赖树复杂化。默认最小化才正确。

### 失败模式 3：只支持一种安装方式

Homebrew 用户不想装 Rust 工具链，NixOS 用户不想用 Homebrew。

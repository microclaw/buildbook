# Chapter 17 架构演进

## 演进而不拆散

系统跑起来后，最难的是「演进而不拆散」。MicroClaw v0.1.57 的策略：8-crate workspace 按稳定性分层、单进程统一治理 + per-chat 串行 + 单 SQLite 是默认形态、A2A/ACP 各自承担两类天然分布边界、配置与数据兼容性靠代码内迁移而不是手工 SQL。

## 8-crate Workspace

```toml
[workspace]
members = [
    ".",                                # microclaw（顶层 binary）
    "crates/microclaw-core",            # 错误类型、共享类型
    "crates/microclaw-clawhub",         # ClawHub 客户端
    "crates/microclaw-storage",         # SQLite 存储层
    "crates/microclaw-tools",           # 工具定义、sandbox、权限
    "crates/microclaw-channels",        # 渠道抽象层
    "crates/microclaw-app",             # 应用层工具
    "crates/microclaw-observability",   # OTLP 导出、metrics、trace、log
]
```

按稳定性分层：`core` 最底层最稳定 → `storage` / `tools` / `channels` / `observability` 各自只依赖 `core` → 顶层 binary 组装它们。改 `microclaw-observability` 不会触发 `microclaw-tools` 重新编译。

```rust
// microclaw-core: 最稳定，错误类型与共享 newtypes
pub mod error {
    pub enum MicroClawError {
        Config(String),
        Storage(String),
        Tool(String),
    }
}

// microclaw-storage: 只依赖 core
use microclaw_core::error::MicroClawError;
pub struct Database { /* ... */ }

// microclaw-tools: 只依赖 core
use microclaw_core::error::MicroClawError;
pub fn tool_concurrency_class(name: &str) -> ToolConcurrencyClass { /* ... */ }

// microclaw（顶层）: 把 storage / tools / observability 等组装为 AppState
pub struct AppState {
    pub db: Arc<Database>,
    pub tools: ToolRegistry,
    pub trace_exporter: Option<Arc<OtlpTraceExporter>>,
    pub log_exporter: Option<Arc<OtlpLogExporter>>,
    // ...
}
```

## 何时保持单体

绝大多数场景下，「单进程 MicroClaw + 一个 SQLite」是成本最低的部署形态：

- **统一治理**：审批门、Hook、SandboxRouter、控制面鉴权都在同一个进程内，不存在跨服务的信任传递。
- **per-chat 串行**：`ChatTurnQueue` 用进程内 mutex，零网络抖动。
- **单 SQLite**：备份就是 `cp microclaw.db`，回滚就是把文件换回去。
- **零分布式协调**：调度任务、`subagent_runs`、`metrics_history` 全在同一份事务里，状态机不用考虑跨节点的最终一致。

## 何时拆边界

MicroClaw 提供两个**天然分布边界**，不需要内部强行拆微服务：

**A2A（Agent-to-Agent）**走 HTTP，是实例之间的协作协议：

```rust
pub const A2A_PROTOCOL_VERSION: &str = "microclaw-a2a/v1";
```

每个实例暴露 `/api/a2a/agent-card` 自描述：能力、本地 agent 名、签名信息。一个本地实例可以把「打开浏览器抓页面」这种重活委派给一台部署了浏览器扩展的远端实例；跨团队的 MicroClaw 也能通过 A2A 互发任务，不需要共享数据库。

**ACP（Agent-Client Protocol）**只走 stdio，由 IDE / 脚本驱动：

```bash
microclaw acp   # 启动为 ACP server，从 stdin 读，stdout 写
```

ACP 是嵌入式场景的入口——把 MicroClaw 当作一个被 IDE 拉起来的子进程。对应代码在 `src/acp.rs` 与 `src/acp_subagent.rs`，使用 `agent-client-protocol` crate，用 `protocol_version` 与对端协商版本。

| 协议 | 传输 | 角色 | 适用 |
|------|------|------|------|
| A2A | HTTP | 实例之间 peer | 跨主机协作、跨团队联通 |
| ACP | stdio | IDE/脚本驱动 | 嵌入到外部 agent 中作为子代理 |
| MCP | stdio / HTTP | MicroClaw 调外部工具 | 接入第三方工具生态 |

## 三种兼容性

| 维度 | 影响范围 | 治理策略 |
|------|---------|---------|
| 配置 | 字段名、默认值 | 全部字段加 `#[serde(default)]` |
| 数据 | SQLite schema | `db.rs` 内置迁移函数，启动时按版本号升 |
| 协议 | A2A、ACP、MCP | 协议名带版本，握手时协商 |

### 配置兼容

`Config` 每一个新增字段都强制带 `#[serde(default)]`，旧配置不会因为缺字段而启动失败：

```rust
#[derive(Deserialize)]
pub struct Config {
    #[serde(default = "default_max_tool_iterations")]
    pub max_tool_iterations: usize,
    #[serde(default = "default_parallel_tool_max_concurrency")]
    pub parallel_tool_max_concurrency: usize,
    #[serde(default)]
    pub working_dir_isolation: WorkingDirIsolation,
    // ... 全字段 default
}
```

旧字段废弃前进入「兼容窗口」：保留解析但 `microclaw doctor` 标 `WARN`，下个大版本再删。

### 数据兼容

schema 在代码内迁移，不用单独的 `.sql` 文件：

```rust
const SCHEMA_VERSION_CURRENT: i64 = 25;

fn apply_schema_migrations(conn: &Connection) -> Result<()> {
    let mut version = get_schema_version(conn)?;
    if version < 7 {
        conn.execute_batch(/* CREATE TABLE metrics_history ... */)?;
        set_schema_version(conn, 7)?;
        version = 7;
    }
    if version < 8 {
        conn.execute_batch(/* CREATE TABLE audit_logs ... */)?;
        set_schema_version(conn, 8)?;
        version = 8;
    }
    // ... 逐版本升到 25
    Ok(())
}
```

每个版本号对应一段「if version < N」的迁移块，存量库启动时只跑差量；新库直接跑到 `SCHEMA_VERSION_CURRENT`。`db_meta` 表持久化版本号，回滚需要先停服、用备份替换 db 文件。

### 协议兼容

A2A 协议名 `microclaw-a2a/v1`，握手时双方要求版本前缀一致；客户端可以发送 `accepts: ["microclaw-a2a/v1"]` 让未来的 v2 节点向后兼容。ACP 用 `agent-client-protocol` crate 自带版本协商；MCP 通过 `protocol_version` / `protocolVersion` 字段在 server 列表里覆盖默认值。

## 部署形态演进

| 维度 | 本地开发 | 家庭服务器 | 云部署 |
|------|---------|-----------|--------|
| 安装 | `cargo run` 或 `brew install` | `microclaw gateway install` | 容器 / GHCR 镜像 / Docker Compose |
| 进程管理 | 终端前台 | systemd / launchd / Windows Service | 容器 orchestrator |
| Sandbox | `Off` | `Shared` | `All`（每 chat 独立） |
| 观测 | 本地 metrics 即可 | 本地 + 可选 OTLP | OTLP 必选 |
| 鉴权 | 默认密码可接受 | 必须设强密码 | API key + TLS |
| 升级 | `microclaw upgrade` | `gateway restart` | rolling update |
| 备份 | 可选 | 必须本地备份 | 必须本地 + 异地 |

`microclaw gateway install` 会按平台生成对应的服务定义（macOS launchd plist、Linux systemd unit、Windows service），统一了「装服务」这件事，不必让用户手写 `.service` 文件。

## Feature Flags

```toml
[features]
default = []
sqlite-vec       = ["microclaw-storage/sqlite-vec"]
journald         = ["microclaw-app/journald"]
channel-matrix   = ["dep:matrix-sdk"]
full             = ["channel-matrix"]
```

默认最小编译。`matrix-sdk` 引入大量传递依赖，设为 optional 让常规编译时间可预测。代码侧用 `#[cfg(feature = "...")]` 守门：

```rust
#[cfg(feature = "channel-matrix")]
use crate::channels::matrix;

const DYNAMIC_CHANNELS: &[DynamicChannelDef] = &[
    slack::SETUP_DEF,
    discord::SETUP_DEF,
    feishu::SETUP_DEF,
    #[cfg(feature = "channel-matrix")]
    matrix::SETUP_DEF,
];
```

## 演进原则：先拆边界，再拆进程

**保持单体**：瓶颈不在进程边界、团队规模不要求服务治理、主部署仍是单机或单容器。

**考虑拆分**：子系统资源消耗压制主链路（例如浏览器扩展拖死整个 runtime）、发布节奏与主进程不同步、需要独立扩缩容、单进程权限隔离不足以承载多租户。

**拆分时优先用 A2A**：A2A 是已有的协议边界，把「特殊能力实例」做成独立 MicroClaw 节点，主实例通过 `a2a_send` 委派任务，比内部拆微服务的成本低一个数量级。

## 容易走错的地方

**还没理清 crate 边界就急着拆服务**：把单体的混乱复制成网络调用。先看 8-crate 的依赖图是否清晰，再说微服务。

**所有 feature 默认开启**：编译时间膨胀，依赖树复杂化。`default = []` 是正确起点。

**手写 `.sql` 迁移文件**：版本控制和回滚都难。`db.rs` 内置的「if version < N」模式可读、可单测、可回放。

**A2A 当成同步 RPC 用**：A2A 设计为异步消息委派，强行拿来当同步 RPC 会撞上超时和重试问题。

# Chapter 17 架构演进

## 系统跑起来之后，如何演进而不把自己拆散？

到了这一章，我们已经讨论了可用 MicroClaw 所需的大部分核心能力。真正的难题不再是"怎么从零搭起来"，而是"系统已经跑起来以后，接下来该怎么演进而不把自己拆散"。

MicroClaw v0.1.38 的架构相比 v0.1.16 经历了显著的结构化演进：

- 从较集中的 `src/*` 演进到 **8-crate workspace**
- **A2A** 成为实例间协作的天然分布式边界
- **ACP**（Agent Client Protocol）提供了 headless 控制能力
- **Gateway** 把服务管理标准化为 install/start/stop/status/logs
- **Feature flags** 让可选能力（Matrix 渠道、sqlite-vec 向量搜索）按需编译
- 从单一安装方式演进到 **Homebrew / Docker / Nix / one-line scripts** 全覆盖

这一章读完后，你应该理解：

1. 8-crate workspace 的职责边界为什么这样划分。
2. A2A 为什么是"最自然的分布式切口"。
3. 从本地开发到家庭服务器再到云部署，架构侧重点如何变化。
4. Feature flags 如何在编译期管理可选能力。

## 8-crate Workspace：职责边界的工程化

v0.1.38 的 workspace 结构是：

```
[workspace]
members = [
    ".",                           # microclaw (顶层 binary)
    "crates/microclaw-core",       # 错误类型、文本工具、共享类型
    "crates/microclaw-clawhub",    # ClawHub 技能市场客户端
    "crates/microclaw-storage",    # SQLite 存储层
    "crates/microclaw-tools",      # 工具定义、sandbox、权限
    "crates/microclaw-channels",   # 渠道抽象层
    "crates/microclaw-app",        # 应用层工具（日志、信号处理）
    "crates/microclaw-observability", # OTLP 导出、指标、trace
]
```

这 8 个 crate 的划分遵循了一个清晰的原则：**按稳定性和依赖方向分层**。

- `microclaw-core` 是最底层，被所有其他 crate 依赖，因此必须最稳定
- `microclaw-storage` 只依赖 `core`，提供纯数据访问
- `microclaw-tools` 依赖 `core`，定义工具能力和权限
- `microclaw-observability` 依赖 `core`，封装全部 OTLP 相关复杂度
- `microclaw-channels` 依赖 `core`，提供渠道抽象
- `microclaw-clawhub` 依赖 `core`，封装技能市场交互
- `microclaw-app` 依赖 `core`，提供应用层工具
- 顶层 `microclaw` 把所有 crate 组装成单一 binary

这种分层的工程价值在于：改动 observability 的 OTLP 导出逻辑不会触发 tools 或 storage 的重新编译；改动某个渠道的解析逻辑不会影响 observability crate。在一个已经有 80+ 个依赖的系统中，这种编译隔离节省的时间是实实在在的。

### Crate 边界为什么应该按稳定性而不是功能域划分？

很多团队按"功能域"划分 crate：chat crate、tool crate、admin crate。但如果 chat 和 admin 都依赖同一个 storage 抽象，改 storage 就会波及两个 crate。按稳定性分层则把最少变化的放最底层，变化频繁的放最上层，编译和依赖波及范围更可预测。

Rust 版本用依赖方向展示这个思路。

```rust
// microclaw-core: 最稳定，被所有 crate 依赖
pub mod error {
    pub enum MicroClawError {
        Config(String),
        Storage(String),
        // ...
    }
}

// microclaw-storage: 只依赖 core
use microclaw_core::error::MicroClawError;
pub struct Database { /* ... */ }

// microclaw-tools: 只依赖 core
use microclaw_core::error::MicroClawError;
pub fn tool_concurrency_class(name: &str) -> ToolConcurrencyClass { /* ... */ }
```

Python 版本用 package 层级表达同样的依赖方向。

```python
# microclaw_core/error.py — 最底层
class MicroClawError(Exception):
    pass

# microclaw_storage/db.py — 只依赖 core
from microclaw_core.error import MicroClawError

# microclaw_tools/runtime.py — 只依赖 core
from microclaw_core.error import MicroClawError
```

## A2A：最自然的分布式切口

MicroClaw 没有急着把内部能力拆成微服务。它选择了一条更务实的路线：通过 A2A（Agent-to-Agent）协议，让不同 MicroClaw 实例之间可以互相委派任务。

### 为什么是 A2A 而不是内部服务拆分

内部服务拆分的前提是模块边界已经清晰、通信协议已经稳定、运维能力已经就绪。对于一个仍在快速演进的系统，过早拆分通常意味着把单体内的隐式耦合变成跨服务的分布式耦合——调试更难，部署更复杂，但本质问题没有解决。

A2A 的路线完全不同：

- 主 runtime 内部仍然紧密内聚
- 分布式边界出现在**实例之间**，而不是实例内部
- 协议层（`microclaw-a2a/v1`）独立于实现细节
- 可以先验证"是否真的需要多实例协作"，再决定是否做更重的架构重构

### A2A 的实际使用场景

- 本地开发实例委派浏览器自动化任务给专用实例
- 云端实例把文件操作委派给有文件系统访问权的边缘实例
- 不同团队各自运行 MicroClaw 实例，通过 A2A 实现跨团队协作

## ACP：Headless 控制

`src/acp.rs` 和 `src/acp_subagent.rs` 实现了 Agent Client Protocol 支持。ACP 是一个基于 stdio 的双向 JSON-RPC 协议，它让 MicroClaw 可以被另一个 agent（如 Claude Code、Codex CLI）当作子代理来驱动。

```bash
microclaw acp  # 以 ACP server 模式启动
```

ACP 的架构意义在于：MicroClaw 不只是一个独立运行的 agent，还可以是其他系统的"能力提供者"。这种可嵌入性是 v0.1.38 架构的重要特征。

关键实现细节：

- 通过 `agent-client-protocol` crate（v0.10.3）实现协议
- 子代理在独立进程中运行，通过 stdin/stdout 通信
- 支持 `RequestPermission` 机制让父代理控制权限
- 终端输出有大小限制（`ACP_TERMINAL_OUTPUT_LIMIT_BYTES = 256KB`）

## Gateway：服务管理的标准化

`src/gateway.rs` 把 MicroClaw 的服务管理标准化为一组统一命令：

```bash
microclaw gateway install    # 安装为系统服务
microclaw gateway start      # 启动
microclaw gateway stop       # 停止
microclaw gateway status     # 状态查询
microclaw gateway logs       # 查看日志
```

Gateway 的跨平台实现是一个值得关注的工程细节。它通过编译期条件分支支持三个平台：

- **macOS**：launchd plist（`ai.microclaw.gateway`）
- **Linux**：systemd unit（`microclaw-gateway.service`）
- **Windows**：Windows Service（`MicroClawGateway`）

每个平台有独立的 `RuntimeStatus` 结构体，但对外暴露的 CLI 接口完全一致。这让用户不需要关心底层服务管理机制的差异。

`status --deep` 模式通过 WebSocket 连接到运行中的实例，执行 RPC 健康检查。`status --json` 输出结构化状态，适合监控系统集成。

## Feature Flags：编译期的可选能力

```toml
[features]
default = []
sqlite-vec = ["microclaw-storage/sqlite-vec"]
journald = ["microclaw-app/journald"]
channel-matrix = ["dep:matrix-sdk"]
full = ["channel-matrix"]
```

这组 feature flags 体现了一个务实原则：**MCP 是默认能力，Matrix 是可选能力**。

- `default = []`：最小编译，适合 CI 和轻量部署
- `sqlite-vec`：启用向量搜索，需要 sqlite-vec 扩展
- `channel-matrix`：启用 Matrix 渠道，引入 matrix-sdk 依赖
- `journald`：启用 systemd journal 日志
- `full`：全部可选能力

这种设计的好处是：默认构建足够轻量，特定能力的依赖不会拖累不需要它的用户。`matrix-sdk` 单独就引入了大量传递依赖，把它设为 optional 让默认编译时间更可预测。

## 安装方式的多样化

v0.1.38 支持多种安装路径：

### Homebrew（macOS/Linux）

```bash
brew install microclaw/tap/microclaw
```

最简单的安装方式，适合个人开发。

### Docker

```bash
docker compose up -d
```

`Dockerfile` 和 `docker-compose.yaml` 已经在仓库根目录。适合无 Rust 工具链的环境和一致的部署体验。

### Nix

`flake.nix` 和 `flake.lock` 提供了可复现的构建。适合 NixOS 用户和需要严格构建可复现性的场景。

### One-line Scripts

```bash
# Unix
curl -sSf https://raw.githubusercontent.com/microclaw/microclaw/main/install.sh | bash
# Windows
irm https://raw.githubusercontent.com/microclaw/microclaw/main/install.ps1 | iex
```

适合快速试用和 CI 环境。

### 源码构建

```bash
cargo install --path .
# 或带全部 feature
cargo install --path . --features full
```

适合需要自定义 feature 组合的场景。

## 部署形态：从本地到云

### 本地开发

重点：

- `microclaw setup` 引导式配置
- `microclaw doctor` 自检
- `microclaw start` 前台运行
- SQLite 文件和工作目录可直接检查
- Web UI 在 `localhost` 运行

### 家庭服务器

重点：

- `microclaw gateway install && microclaw gateway start`
- Web 控制面可通过局域网访问
- sandbox 模式开启（Chat 模式提供多用户隔离）
- SQLite 定期备份
- Web 密码必须从默认值修改

### 云部署

重点：

- OTLP 导出到外部平台（Langfuse、AgentOps、自建 Jaeger）
- API key 管理：生成 scoped key 供外部系统调用
- 标准化升级/回滚流程
- A2A 可能需要 TLS 和认证
- model_prices 配置用于成本监控
- `microclaw upgrade` 或容器镜像更新

### 三种形态的架构差异

| 维度 | 本地开发 | 家庭服务器 | 云部署 |
|------|---------|-----------|--------|
| 安装方式 | Homebrew/source | Gateway install | Docker/Nix |
| sandbox | Off | Shared/Chat | Chat |
| 观测导出 | 本地 metrics | 本地 + 可选 OTLP | OTLP 必选 |
| 鉴权 | 默认密码可接受 | 必须修改密码 | API key + TLS |
| 升级 | `microclaw upgrade` | Gateway 重启 | Rolling update |
| 备份 | 可选 | 必须 | 必须 + 异地 |

## 演进原则：先拆边界，再拆进程

MicroClaw v0.1.38 最值得学习的地方，是它优先做了 crate 级边界治理和协议级外部接口（A2A、ACP），而不是直接把系统拆成多个服务。

这种顺序非常合理。因为多数系统真正的首要问题不是"必须多进程"，而是"职责已经混乱"。如果模块边界尚未清晰，过早拆服务通常只会把隐式耦合、状态不一致和调试复杂度一起放大。

### 什么时候应该保持单体

- 核心瓶颈还不在进程边界
- 团队规模还不足以支撑大量服务治理
- 主要部署形态仍是单机
- 运行时一致性比水平扩展更重要

### 什么时候应该考虑拆分

- 某个子系统的资源消耗长期压制主链路（如浏览器自动化）
- 某个能力的发布节奏和主 runtime 明显不同步
- 单进程权限边界不足以隔离高风险能力
- 需要独立扩缩容（如公开 Web API 或 A2A 网关）

## 三种兼容性分别治理

### 配置兼容

`config.rs` 是运行时入口之一，任何字段名、默认值、结构层次的变化都影响升级。成熟做法：保持旧字段兼容窗口、新旧配置同时迁移提示、doctor 自检标红危险旧配置。

### 数据兼容

数据库 migration 在启动时自动执行，回滚依赖 SQLite 备份。v0.1.38 新增了 `subagent_runs` 等表，migration 脚本需要处理这些 schema 变更。

### 协议兼容

包括 Web API 合约、MCP 接口行为、A2A 消息格式、ACP 协议版本。A2A 使用了 `microclaw-a2a/v1` 协议标识，ACP 使用了 `agent-client-protocol` 的版本协商机制。

## 示例代码：为什么 feature flag 应该在编译期而不是运行期切换？

运行期 feature toggle 的代码总是存在，只是被 `if` 跳过。编译期 feature flag 通过 `#[cfg(feature = "...")]` 让代码在不需要的二进制中完全不存在——减小二进制体积，消除不可达代码的维护负担。

Rust 版本展示了 MicroClaw 实际的 feature gate 模式。

```rust
// 只在启用 channel-matrix feature 时编译
#[cfg(feature = "channel-matrix")]
use crate::channels::matrix;

// setup wizard 中的渠道列表也按 feature 条件编译
const DYNAMIC_CHANNELS: &[DynamicChannelDef] = &[
    slack::SETUP_DEF,
    #[cfg(feature = "channel-matrix")]
    matrix::SETUP_DEF,
    // ...
];
```

Python 版本用 import guard 近似表达同样的意图。

```python
import importlib

OPTIONAL_CHANNELS: dict[str, str] = {
    "matrix": "microclaw.channels.matrix",
}

def load_channel(name: str):
    module_path = OPTIONAL_CHANNELS.get(name)
    if module_path is None:
        raise ImportError(f"channel {name} not available")
    return importlib.import_module(module_path)
```

## 关键权衡

### 决策一：8-crate workspace 按稳定性分层

优点是编译隔离和依赖方向清晰。代价是 crate 间接口需要更谨慎的版本管理。

### 决策二：A2A 作为分布式边界

优点是主 runtime 内聚不变，分布式复杂度被推到实例间。代价是 A2A 协议治理需要先于内部服务化到来。

### 决策三：Gateway 标准化服务管理

优点是三平台一致体验。代价是每个平台的服务集成代码需要独立维护。

### 决策四：Feature flags 在编译期控制可选能力

优点是默认构建轻量。代价是用户需要了解 feature 组合才能获得全部能力。

### 决策五：多种安装方式并存

优点是降低不同用户群体的入门门槛。代价是每种安装路径都需要独立测试和维护。

## 容易走错的地方

### 失败模式 1：还没理清 crate 边界就急着拆服务

最终通常只是把单体里的混乱复制成了网络调用。

### 失败模式 2：所有 feature 默认开启

这会让编译时间膨胀，依赖树复杂化，二进制体积增大。默认最小化才是正确姿态。

### 失败模式 3：只支持一种安装方式

这会把大量潜在用户挡在门外。Homebrew 用户不想装 Rust 工具链，NixOS 用户不想用 Homebrew。

## 小结

MicroClaw v0.1.38 的架构演进展示了一条务实路线：先通过 8-crate workspace 在内部建立清晰的职责边界，再通过 A2A 和 ACP 在外部建立协议级的分布式接口，通过 Gateway 标准化服务管理，通过 feature flags 控制编译范围，通过多种安装方式降低入门门槛。这条路线的核心思想是：先拆边界，再拆进程；先保持内聚，再按需分布。

# Chapter 15 测试策略

## 确定性与非确定性的分治

Agent Runtime 同时包含确定性部分（数据库、权限、配置、并发控制）和非确定性部分（LLM 输出、子代理嵌套、真实渠道发送）。用测 CRUD 的方式测 Agent，要么测不到关键路径，要么全是脆弱断言。MicroClaw v0.1.57 的策略是：**确定性面用单元测试覆盖到行**，**非确定性面用合约式断言只测结构**，**门禁阶段用 `cargo deny` + `cargo clippy` 守住供应链与代码质量**。

## 确定性测试面

| 模块 | 测什么 | 为什么是确定性 |
|------|--------|---------------|
| 领域类型转换 | `Message`/`Tool`/`Provider` 之间序列化 | 纯函数 |
| 调度时间计算 | `next_run_at` 在不同时区、cron、once 模式下的取值 | 给定时间和表达式输入 → 输出确定 |
| 工具元数据校验 | `ToolRisk`、`ToolConcurrencyClass`、`ToolExecutionPolicy` | 静态匹配函数 |
| `tool_executor` 波次划分 | `partition_into_waves` 按 ReadOnly / SideEffect / Exclusive 分组 | 纯函数，覆盖率应接近 100% |
| 记忆 reflector 解析 | LLM 返回的反思 JSON 解析与 schema 校验 | 给定输入 JSON → 解析结果固定 |
| ChatTurnQueue | `acquire` / `try_start_or_enqueue` / `drain_pending` 的状态机 | 单元可控 |
| run_control | `register_run` / `abort_runs` / `is_cancelled` | 原子标志的状态翻转 |

### 波次划分单元测试

```rust
use microclaw_tools::runtime::ToolConcurrencyClass::{self as Cls, *};

fn partition_into_waves(classes: &[Cls]) -> Vec<Vec<usize>> {
    let mut readonly = Vec::new();
    let mut waves = Vec::new();
    for (i, c) in classes.iter().enumerate() {
        match c {
            ReadOnly => readonly.push(i),
            SideEffect => waves.push(vec![i]),
            Exclusive => waves.push(vec![i]),
        }
    }
    if !readonly.is_empty() {
        waves.insert(0, readonly);
    }
    waves
}

#[test]
fn waves_split_by_class() {
    let classes = vec![ReadOnly, ReadOnly, SideEffect, Exclusive];
    let waves = partition_into_waves(&classes);
    assert_eq!(waves, vec![vec![0, 1], vec![2], vec![3]]);
}

#[test]
fn empty_input_yields_empty_waves() {
    assert!(partition_into_waves(&[]).is_empty());
}

#[test]
fn all_readonly_collapses_to_one_wave() {
    assert_eq!(partition_into_waves(&[ReadOnly; 5]).len(), 1);
}
```

### run_control 取消传播

```rust
#[tokio::test]
async fn abort_marks_in_flight_run() {
    let (run_id, flag, _notify) =
        register_run("test", 101, Some("u1".into())).await;
    assert!(!is_cancelled(&flag));

    let aborted = abort_runs("test", 101).await;
    assert_eq!(aborted, 1);
    assert!(is_cancelled(&flag));
    assert!(is_aborted_source_message("test", 101, "u1").await);

    unregister_run("test", 101, run_id).await;
}
```

### ChatTurnQueue 竞态窗口

```rust
#[tokio::test]
async fn try_start_or_enqueue_is_atomic() {
    let q = ChatTurnQueue::new(20);
    let g1 = q.try_start_or_enqueue("tg", 1, msg("a")).await;
    assert!(g1.is_some());
    let g2 = q.try_start_or_enqueue("tg", 1, msg("b")).await;
    assert!(g2.is_none()); // 第二条被 enqueue
    drop(g1);
    let drained = q.drain_pending("tg", 1).await;
    assert_eq!(drained.len(), 1);
}
```

`acquire` 与 `try_start_or_enqueue` 必须分开测：前者是「等到能拿就拿」，后者是「拿不到就排队」，两者的原子性是 turn 串行的核心。

## 非确定性面：测结构不测文风

LLM 输出无法字面比对。MicroClaw 的合约式断言：

**该测**：模型选择了正确分支（哪个工具）、`stop_reason` 安全处理、审批状态机、工具失败后会话能恢复、compaction 后消息结构合法、wave 分区结果。

**不该测**：最终回答是不是某段中文。

```rust
#[test]
fn llm_chooses_read_first_for_search_intent() {
    let resp = mock_llm_with_tool_calls(&["grep", "read_file"]);
    let plan = parse_tool_plan(&resp).unwrap();
    assert!(plan.iter().any(|t| t.name == "grep"));
    assert!(matches!(plan[0].risk(), ToolRisk::Low));
}
```

## 状态机回归

三个状态机必须每次发布都跑一遍：

| 状态机 | 关键流转 | 必须断言 |
|--------|---------|---------|
| `scheduled_tasks` | `pending → running → completed`／`failed → retry`／`paused`／`cancelled` | DLQ 写入条件、retry 上限、`pause` 不丢任务 |
| `subagent_runs` | `running → completed`／`timeout`／`cancelled`／`failed` | timeout 后进程被回收、token 计数被结算 |
| `run_control` cancel | `register → abort_runs → is_cancelled` | 取消标志能被工具循环看到 |

## 发布门禁：CI 必跑四件套

```
cargo fmt --check
cargo clippy --all-targets --all-features -- -D warnings
cargo test  --all-features
cargo deny  check
```

`deny.toml` 已经在仓库里：

```toml
[advisories]
version = 2
yanked = "deny"

[licenses]
allow = [
  "MIT", "Apache-2.0", "BSD-2-Clause", "BSD-3-Clause",
  "ISC", "MPL-2.0", "Zlib", "Unicode-DFS-2016",
]
confidence-threshold = 0.8

[bans]
multiple-versions = "allow"
wildcards = "deny"
```

`wildcards = "deny"` 是关键一条：禁止 `*` 版本通配，避免传递依赖把 runtime 拖到不可预测的版本上。

## 文档一致性回归

文档与代码经常各自漂移：

- `docs/tools/*.md` 中的工具列表与 `crates/microclaw-tools/src/runtime.rs` 中的内置工具集（约 50 个）必须一致；CI 用一个小脚本 grep 内置工具名做交叉对比。
- `docs/configuration/*.md` 中的字段和 `Config` 结构体的 `#[serde(default)]` 字段必须一致。
- CLI 子命令（`setup` / `doctor` / `start` / `acp` / `gateway` / `web` / `skill` / `hooks` / `reembed` / `upgrade`）在 `--help` 输出与 README 命令速查必须一致。

## 容易走错的地方

**不测 wave 分区**：`tool_concurrency_overrides` 配置错误时，SideEffect 工具被错误塞进并行 wave，生产中才暴露。

**子代理只测 happy path**：不测超时和取消，生产中出现僵死子代理消耗预算。

**依赖审计只做一次**：新依赖随时可能拉来安全公告或许可证冲突。`cargo deny` 要进 CI。

**忽略 ChatTurnQueue 竞态**：单测 `acquire` 和 `try_start_or_enqueue` 分别 OK，但不测两者交错的竞态会漏掉真实 bug。

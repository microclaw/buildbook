# Chapter 15 测试策略

## 这一章要回答什么问题

Agent Runtime 的测试比普通 CRUD 系统难，原因很简单：它同时包含确定性部分和非确定性部分。

- 数据库、权限、配置解析、调度状态机大多是确定性的。
- 模型输出、长对话、工具编排则带有明显非确定性。

如果团队只会写传统单元测试，就会在 LLM 行为上失去把控；如果只做人工体验测试，又会在配置、权限和数据回归上不断踩坑。MicroClaw 当前的测试和文档体系，展示了一种比较平衡的路线。

这一章读完后，你应该理解：

1. 哪些能力必须做强确定性测试。
2. 对 LLM 非确定性应该怎样设计断言。
3. 为什么黑盒回归和发布门禁同样重要。
4. 文档与生成物一致性检查为什么也是测试的一部分。

## 先把确定性部分测到够硬

MicroClaw 当前的测试样本虽然不是全景，但已经明确体现了优先级。

### 配置验证

`tests/config_validation.rs` 覆盖了：

- 最小配置解析
- 默认值行为
- 完整 YAML 解析
- 路径派生
- 边界值与兼容行为

这类测试非常关键。因为很多运行时问题不是代码 bug，而是配置解释与默认值发生漂移。把配置测试做扎实，等于先堵住一大类线上事故。

### 权限模型

`tests/tool_permissions.rs` 专门验证 `ToolAuthContext` 和跨 chat 访问控制。这类测试特别重要，因为权限问题一旦回归，后果通常比普通功能 bug 更严重。

### 数据库生命周期

`tests/db_integration.rs` 覆盖了消息、session、scheduled tasks、task run logs 等跨表生命周期操作。这种测试直接对应运行时最核心的持久化语义：

- 会话能不能正确保存和恢复
- 任务状态是否按预期迁移
- 重启后数据是否还在

这就是典型“必须高度确定性”的测试面。

### 小例子：为什么测试替身要围绕状态边界搭，而不是围绕文案搭？

当系统里既有确定性逻辑又有模型非确定性时，最稳的测试替身应该模拟“状态怎么变”，而不是模拟“最后说了哪句话”。这样你保护的是 runtime 规则，而不是一段脆弱文案。

Rust 版本用一个固定返回 `stop_reason` 的 fake model。测试关心的是循环进入哪条分支，而不是模型输出看起来像不像真人。

```rust
enum StopReason {
    EndTurn,
    ToolUse,
}

struct FakeModel {
    stop_reason: StopReason,
}

impl FakeModel {
    async fn next_stop_reason(&self) -> StopReason {
        match self.stop_reason {
            StopReason::EndTurn => StopReason::EndTurn,
            StopReason::ToolUse => StopReason::ToolUse,
        }
    }
}
```

Python 版本保留同样的最小替身结构，用 `@dataclass` 明确把测试控制点放在状态字段上。这样示例会更贴近本章的核心主张：测结构，不测修辞。

```python
from dataclasses import dataclass
from enum import Enum


class StopReason(str, Enum):
    END_TURN = "end_turn"
    TOOL_USE = "tool_use"


@dataclass
class FakeModel:
    stop_reason: StopReason

    async def next_stop_reason(self) -> StopReason:
        return self.stop_reason
```

## 对 LLM 非确定性，要测结构，不测文风

一到 Agent 项目，很多团队就会走两个极端：

- 要么放弃测试，说“LLM 本来就不稳定”。
- 要么写死整段输出文案，结果几乎每次都脆断。

MicroClaw 更合理的方向，是测试结构化行为，而不是测试措辞风格。

### 该测什么

- 是否进入了正确的工具调用分支
- `stop_reason` 是否被安全处理
- 高风险审批是否走了预期状态机
- 工具失败后会话是否还能继续
- context compaction 后消息结构是否仍合法

这些都是结构性断言，不依赖模型用哪一句修辞。

### 不该测什么

- 最终回答是否一字不差
- 某段自然语言解释是否完全一致

对这类系统来说，真正重要的是行为一致，不是措辞一致。

## Agent Engine 的回归重点

`agent_engine.rs` 自身已经包含大量测试，围绕的都是高风险状态机分支，例如：

- 高风险工具审批自动重试
- 等待用户确认
- `stop_reason=tool_use` 但无工具调用
- 工具调用循环和中止路径

这些测试的共同特点是：它们盯住的是最容易让 runtime 失控的转折点。对 Agent Runtime 来说，这是正确的测试投入方向。与其写大量重复的 happy path 文案断言，不如把核心状态转换测清楚。

## 黑盒回归：从用户视角验证链路

`docs/test/blackbox-core-20-cases.md` 给出了一个非常有价值的实践：把关键能力整理成手工黑盒清单。当前已经覆盖：

- 基础对话链路
- 多轮上下文保持
- 工具成功与失败
- 跨会话权限拦截
- once 任务
- DLQ 与 replay
- MCP 基础调用
- metrics summary / history 合约
- 会话分叉
- 显式记忆写入与召回

这类文档的意义，不在于替代自动化，而在于把“发布前到底要验证什么”具象化。对于仍在快速演进的 Agent 系统，这样的清单尤其重要，因为它能让团队对关键能力形成共同认知。

## 发布门禁：测试不应停在本地

MicroClaw 的发布文档已经把更严格的校验纳入正式流程，例如：

- `cargo fmt`
- `cargo clippy --all-targets`
- `cargo test`
- Web 前端构建
- 文档生成物检查
- 升级路径验证
- 配置自检审查

`docs/releases/pr-release-checklist.md` 与 `docs/operations/stability-plan-2026-q1.md` 一起表达了同一个原则：测试不只是“代码能跑通”，还包括：

- 升级是否安全
- 权限和 CSRF 是否正确
- Metrics 和 Sessions API 合约是否稳定
- 文档是否与代码同步

这类门禁对 Agent Runtime 很关键，因为系统的很多重要能力都体现在配置、文档、工具定义和控制面接口上，而不只在 Rust 函数里。

## 文档漂移也是一种回归

很多团队低估了文档漂移的危害。对 MicroClaw 这种同时依赖 README、generated docs、runbook、book 稿件的项目来说，文档一旦过时，用户和维护者就会按错误心智操作系统。

当前仓库已经体现出“把文档一致性纳入测试”的方向，例如：

- `docs/generated/tools.md`
- `docs/generated/config-defaults.md`
- `docs/generated/provider-matrix.md`
- release checklist 中的生成物校验

本书自己的构建仓库也有类似思路：`make audit` 检查无用途文件，构建脚本固定输入顺序。这类“看起来不像测试”的机制，本质上也是防回归。

## 测试金字塔在 Agent Runtime 里怎么落地

如果把 MicroClaw 的经验抽象成一张测试金字塔，大致会是这样：

### 第一层：单元与组件测试

适合配置解析、权限判断、schema 兼容、状态转换这类高度确定逻辑。

### 第二层：集成测试

适合数据库生命周期、调度状态变更、控制面 API 合约、auth scope 等跨模块协作能力。

### 第三层：黑盒和发布回归

适合验证完整链路、用户可见行为、升级兼容和运维路径。

### 非确定性专项测试

单独处理 LLM 行为，重点验证结构和安全边界，而不是写死文案。

这比套用传统 Web 项目的测试习惯更适合 Agent Runtime。

## 示例代码：为什么测试应该锁定状态转换，而不是锁定文案？

对 Agent Runtime 来说，测试更适合锁定“允许还是阻止”“进入哪个状态”“是否写回结果”，而不是逐字锁定自然语言输出。后者很脆弱，前者才是系统真正必须守住的行为。

Rust 版本仍然沿用 `ApprovalGate` 这种可替换结构，但把断言聚焦在“未批准时必须报错”这个状态结果上。这样测试保护的是规则，而不是某次模型碰巧生成的句子。

```rust
#[async_trait::async_trait]
trait ApprovalPolicy {
    async fn confirm(&self, action: &str) -> anyhow::Result<bool>;
}

struct ApprovalGate<P> {
    policy: P,
}

impl<P: ApprovalPolicy> ApprovalGate<P> {
    async fn authorize(&self, action: &str) -> anyhow::Result<()> {
        if self.policy.confirm(action).await? {
            return Ok(());
        }
        anyhow::bail!("approval required for {action}")
    }
}
```

这一段先把审批策略和 gate 的状态转换固定下来。测试章节里最重要的是先看清“规则如何表达”，然后再看具体测试如何验证它。

```rust

struct RejectAllPolicy;

#[async_trait::async_trait]
impl ApprovalPolicy for RejectAllPolicy {
    async fn confirm(&self, _action: &str) -> anyhow::Result<bool> {
        Ok(false)
    }
}

#[tokio::test]
async fn approval_gate_blocks_without_confirmation() {
    let gate = ApprovalGate { policy: RejectAllPolicy };
    let result = gate.authorize("bash rm -rf /tmp").await;
    assert!(result.is_err());
}
```

Python 版本保留同样的意图，但用 `@dataclass` 承载 gate，并用异步测试直接验证状态转换结果。这样即使示例语言不同，测试关注点仍然是一致的：规则是否被正确执行。

```python
from dataclasses import dataclass
from typing import Protocol


class ApprovalPolicy(Protocol):
    async def confirm(self, action: str) -> bool: ...


@dataclass
class ApprovalGate:
    policy: ApprovalPolicy

    async def authorize(self, action: str) -> None:
        if await self.policy.confirm(action):
            return
        raise RuntimeError(f"approval required for {action}")
```

Python 这里同样先把规则对象和授权动作单独展示，再放测试本身。这样读者不会把“被测对象”和“测试脚手架”混在一起看。

```python


@dataclass
class RejectAllPolicy:
    async def confirm(self, action: str) -> bool:
        return False


async def test_approval_gate_blocks_without_confirmation() -> None:
    gate = ApprovalGate(policy=RejectAllPolicy())
    try:
        await gate.authorize("bash rm -rf /tmp")
        assert False, "expected error"
    except RuntimeError as exc:
        assert "approval required" in str(exc)
```

## 关键权衡

### 决策一：把配置、权限、存储视为强确定性测试重点

优点是能稳定挡住高风险回归。代价是需要先投入不少基础设施测试，而不是只追求用户可见功能。

### 决策二：对 LLM 行为做结构化断言

优点是测试更稳健。代价是不能期待“一次把自然语言回答完全锁死”。

### 决策三：保留黑盒回归清单

优点是发布前关键链路一目了然。代价是人工执行成本存在，需要持续维护。

### 决策四：把文档与生成物一致性纳入门禁

优点是减少知识漂移。代价是发布流程更长，对文档纪律要求更高。

## 容易走错的地方

### 失败模式 1：因为 LLM 有非确定性，就放弃自动化测试

这样做会让权限、配置、调度、状态机这些本可稳定验证的部分也失去保护。

### 失败模式 2：试图把自然语言回答逐字固定

这类测试往往维护成本极高，信噪比很差。

### 失败模式 3：只测代码，不测升级、文档和控制面合约

对 MicroClaw 这类系统来说，这些“外围”恰恰是生产事故高发区。

## 读到这里，你应该能回答

- 你是否区分了确定性逻辑和非确定性逻辑的测试方法？
- 你是否为配置、权限、数据库和调度状态机建立了强确定性测试？
- 你是否用结构化断言测试 LLM 行为，而不是锁死自然语言输出？
- 你是否把黑盒回归、发布门禁和文档一致性视为测试体系的一部分？

## 小结

MicroClaw 的测试策略告诉我们，Agent Runtime 并不意味着“无法测试”。真正的关键在于承认系统既有确定性部分，也有非确定性部分，并针对不同部分采用不同的验证方法。只有这样，系统才能在快速迭代中保持稳定，而不是每次发布都靠祈祷。

下一章我们转向性能与成本优化。因为当系统足够稳定后，团队很快会碰到另一个现实问题：它能不能在不牺牲质量的前提下跑得更快、花得更少。

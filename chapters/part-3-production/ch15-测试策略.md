# Chapter 15 测试策略

## 如何测试一个同时包含确定性和非确定性部分的系统？

Agent Runtime 的测试比普通 CRUD 系统难，原因很简单：它同时包含确定性部分和非确定性部分。数据库操作、权限校验、配置解析、调度状态机、并发控制——这些大多是确定性的，可以用传统的断言来覆盖。但模型输出、长对话、工具编排带有明显的非确定性，逐字匹配毫无意义。如果你用测 CRUD 的方式去测 Agent，要么测不到关键路径，要么测到的全是脆弱断言。

MicroClaw v0.1.38 的测试体系在 v0.1.16 的基础上有了显著扩展：`deny.toml` 引入了依赖审计，GitHub CI/CD 实现了自动化门禁，并行工具执行带来了全新的测试挑战，子代理生命周期需要端到端验证。

这一章读完后，你应该理解：

1. `deny.toml` 和 CI/CD 如何构成发布安全网。
2. 并行工具执行（wave 模型）带来了哪些新的测试需求。
3. 子代理生命周期测试为什么必须覆盖超时和取消路径。
4. per-chat turn serialization 的正确性如何验证。

## 依赖审计：deny.toml

v0.1.38 引入了 `deny.toml`，通过 `cargo deny` 对依赖进行三维审计：

### 安全公告（advisories）

```toml
[advisories]
version = 2
yanked = "deny"
ignore = ["RUSTSEC-2026-0049"]  # 临时豁免等待上游修复
```

`yanked = "deny"` 意味着使用已被上游撤回的版本会直接失败。`ignore` 列表明确记录了每个临时豁免的原因，让团队在每次审计时都能追溯决策。

### 许可证合规（licenses）

```toml
[licenses]
allow = [
  "MIT", "Apache-2.0", "BSD-2-Clause", "BSD-3-Clause",
  "ISC", "MPL-2.0", "Zlib", ...
]
confidence-threshold = 0.8
```

对于商业部署场景，许可证合规不是可选项。`deny.toml` 把这个检查从人工 review 变成了自动化门禁。

### 依赖重复检查（bans）

```toml
[bans]
multiple-versions = "allow"
wildcards = "deny"
```

当前 `multiple-versions = "allow"` 是务实选择——Matrix、Telegram、OTLP 等依赖栈不可避免地引入了重复传递依赖。但 `wildcards = "deny"` 禁止了 `*` 版本通配，防止依赖在构建时出现不可预测的版本漂移。

## CI/CD：自动化门禁

GitHub CI 把以下检查串成发布管线：

1. `cargo fmt --check`：格式一致性
2. `cargo clippy --all-targets`：静态分析
3. `cargo test`：全量测试
4. `cargo deny check`：依赖审计
5. Web 前端构建验证
6. 文档生成物检查

这条管线确保了每个 PR 在合入前都经过完整的质量门禁。对于 8-crate workspace 来说，CI 还需要验证跨 crate 的接口兼容性。

## 并行工具执行的测试挑战

v0.1.38 引入的 wave-based 并行执行给测试带来了三类新问题。

### Wave 分区正确性

`partition_into_waves` 函数需要验证：

- 全 ReadOnly 工具 → 单个 wave 并行
- 混合 ReadOnly + SideEffect → ReadOnly 先并行，SideEffect 后顺序
- Exclusive 工具 → 独占 wave
- 单工具调用 → 不触发并行路径
- 空调用列表 → 返回空 wave 列表
- `tool_concurrency_overrides` 配置是否生效

这类测试是纯确定性的，可以用传统单元测试完全覆盖。

### 并行执行的状态安全

当多个 ReadOnly 工具并行执行时，`ToolBatchContext` 中的共享状态（`failed_tools`、`failed_tool_details`）需要安全更新。测试需要验证：

- 并行工具的结果不会互相覆盖
- 一个工具失败不会影响其他并行工具的执行
- `waiting_for_user_approval` 在并行场景下的语义是否正确

### ToolWave 事件的完整性

每个 wave 开始和结束时必须发出对应事件，测试需要验证：

- 事件顺序：WaveStart(1) → ... → WaveComplete(1) → WaveStart(2) → ...
- tool_count 准确
- duration_ms 合理（不为 0，不超过总超时）

### 并行测试为什么要验证"分区逻辑"而不是"并行本身"？

测试并行系统最常见的陷阱是试图验证"这些操作确实同时在跑"——这类断言既脆弱又依赖时序。更有效的做法是把并行系统拆成两部分：分区逻辑（确定性）和并行执行引擎（用已有的 tokio JoinSet 保证）。测试重点放在前者。

Rust 版本展示了 wave 分区的确定性测试，不涉及任何异步或时序。

```rust
#[derive(Clone, Debug, PartialEq)]
enum ConcurrencyClass { ReadOnly, SideEffect, Exclusive }

fn partition_into_waves(classes: &[ConcurrencyClass]) -> Vec<Vec<usize>> {
    let mut readonly = Vec::new();
    let mut sideeffect = Vec::new();
    let mut exclusive = Vec::new();
    for (i, c) in classes.iter().enumerate() {
        match c {
            ConcurrencyClass::ReadOnly => readonly.push(i),
            ConcurrencyClass::SideEffect => sideeffect.push(i),
            ConcurrencyClass::Exclusive => exclusive.push(i),
        }
    }
    let mut waves = Vec::new();
    if !readonly.is_empty() { waves.push(readonly); }
    for idx in sideeffect { waves.push(vec![idx]); }
    for idx in exclusive { waves.push(vec![idx]); }
    waves
}

#[test]
fn test_mixed_partition() {
    use ConcurrencyClass::*;
    let classes = vec![ReadOnly, ReadOnly, SideEffect, Exclusive];
    let waves = partition_into_waves(&classes);
    assert_eq!(waves, vec![vec![0, 1], vec![2], vec![3]]);
}
```

Python 版本保持同样的纯函数测试结构。

```python
from enum import Enum


class ConcurrencyClass(str, Enum):
    READ_ONLY = "read_only"
    SIDE_EFFECT = "side_effect"
    EXCLUSIVE = "exclusive"

def partition_into_waves(classes: list[ConcurrencyClass]) -> list[list[int]]:
    readonly = [i for i, c in enumerate(classes) if c == ConcurrencyClass.READ_ONLY]
    sideeffect = [i for i, c in enumerate(classes) if c == ConcurrencyClass.SIDE_EFFECT]
    exclusive = [i for i, c in enumerate(classes) if c == ConcurrencyClass.EXCLUSIVE]
    waves: list[list[int]] = []
    if readonly:
        waves.append(readonly)
    waves.extend([idx] for idx in sideeffect)
    waves.extend([idx] for idx in exclusive)
    return waves

def test_mixed_partition() -> None:
    classes = [
        ConcurrencyClass.READ_ONLY, ConcurrencyClass.READ_ONLY,
        ConcurrencyClass.SIDE_EFFECT, ConcurrencyClass.EXCLUSIVE,
    ]
    assert partition_into_waves(classes) == [[0, 1], [2], [3]]
```

## 子代理生命周期测试

v0.1.38 的 session-native subagent 引入了新的测试面。

### 正常完成路径

验证子代理从启动到完成的全生命周期：

- ACP 协议握手成功
- 任务执行并返回结果
- `subagent_runs` 表正确记录 status=completed
- token 使用量和成本估算被记录

### 超时路径

`subagents.run_timeout_secs` 到期时：

- 子代理进程被终止
- `subagent_runs` 记录 status=timeout
- 父 agent 收到超时错误并继续运行（不崩溃）

### 取消路径

父 agent 的 run 被 `run_control.abort_runs` 取消时：

- 子代理的 `local_cancel` AtomicBool 被设置
- ACP 轮询检测到取消并停止执行
- 子代理进程被清理

### 并发限制

当已达到 `max_concurrent` 上限时：

- 新的子代理请求应等待或拒绝
- 不应出现死锁（子代理之间不互相等待）

## per-chat Turn Serialization 测试

`chat_turn_queue.rs` 自身包含了丰富的测试用例，覆盖了关键的正确性属性：

### 基本获取释放

```rust
#[tokio::test]
async fn test_acquire_release_basic() {
    let q = make_queue();
    let guard = q.acquire("telegram", 1).await;
    assert!(guard.is_some());
    drop(guard);
    let guard2 = q.acquire("telegram", 1).await;
    assert!(guard2.is_some());
}
```

### 并发阻塞

验证同一 chat 的第二次 acquire 确实被阻塞，直到第一个 guard 被 drop。

### 不同 chat 独立

验证不同 `chat_id` 之间不会互相阻塞——这是 per-chat 模型相比全局锁的核心优势。

### 队列容量管理

验证 `max_pending` 限制生效，超出时丢弃最旧消息。

### 原子性操作

`try_start_or_enqueue` 消除了"检查是否忙"和"入队或获取锁"之间的竞态窗口。这个接口的测试需要验证在高并发下不会丢消息或产生重复执行。

## 先把确定性部分测到够硬

MicroClaw 的 `tests/` 目录包含三个集成测试文件：

### 配置验证

`tests/config_validation.rs` 覆盖了最小配置解析、默认值行为、完整 YAML 解析、路径派生、边界值与兼容行为。配置测试非常关键，因为很多运行时问题不是代码 bug，而是配置解释与默认值发生漂移。

### 权限模型

`tests/tool_permissions.rs` 专门验证 `ToolAuthContext` 和跨 chat 访问控制。这类测试特别重要，因为权限问题一旦回归，后果通常比普通功能 bug 更严重。

### 数据库生命周期

`tests/db_integration.rs` 覆盖了消息、session、scheduled tasks、task run logs、`subagent_runs` 等跨表生命周期操作。

## 对 LLM 非确定性，要测结构，不测文风

### 该测什么

- 是否进入了正确的工具调用分支
- `stop_reason` 是否被安全处理
- 高风险审批是否走了预期状态机
- 工具失败后会话是否还能继续
- context compaction 后消息结构是否仍合法
- 并行工具调用的 wave 分区是否正确

### 不该测什么

- 最终回答是否一字不差
- 某段自然语言解释是否完全一致

## 发布门禁：测试不应停在本地

MicroClaw 的发布文档把更严格的校验纳入正式流程：

- `cargo fmt` + `cargo clippy --all-targets`
- `cargo test`（含 workspace 全 crate）
- `cargo deny check`（依赖审计）
- Web 前端构建
- 文档生成物检查
- 升级路径验证
- 配置自检审查

这类门禁对 Agent Runtime 很关键，因为系统的很多重要能力体现在配置、文档、工具定义和控制面接口上，而不只在 Rust 函数里。

## 示例代码：为什么 run_control 的测试要验证取消传播？

run_control 最关键的正确性属性不是"能注册"，而是"取消能传播到所有活跃 run"。如果取消标志没有正确设置，用户按 /stop 后 agent 仍然继续执行。

Rust 版本直接展示了 MicroClaw 实际的测试模式：注册 run，取消，验证 flag。

```rust
#[tokio::test]
async fn test_abort_runs_marks_cancelled() {
    let (run_id, cancelled, _notify) =
        register_run("test", 101, Some("u1".to_string())).await;
    assert!(!is_cancelled(&cancelled));

    let aborted = abort_runs("test", 101).await;
    assert_eq!(aborted, 1);
    assert!(is_cancelled(&cancelled));
    assert!(is_aborted_source_message("test", 101, "u1").await);

    unregister_run("test", 101, run_id).await;
}
```

Python 版本保持同样的断言结构。

```python
import asyncio
from dataclasses import dataclass, field
from threading import Event


@dataclass
class RunControl:
    cancelled: Event = field(default_factory=Event)

async def test_abort_propagates() -> None:
    rc = RunControl()
    assert not rc.cancelled.is_set()
    rc.cancelled.set()  # 模拟 abort
    assert rc.cancelled.is_set()
```

## 关键权衡

### 决策一：deny.toml 作为 CI 必过门禁

优点是依赖安全问题在合入前就被拦截。代价是偶尔需要处理上游尚未修复的 advisory（通过显式 ignore + 注释管理）。

### 决策二：并行执行测试聚焦分区逻辑

优点是测试稳定、不依赖时序。代价是不直接验证"确实并行了"——但这交给 tokio JoinSet 的正确性保证。

### 决策三：子代理生命周期做端到端测试

优点是覆盖了真实的 ACP 协议交互。代价是测试需要启动子进程，速度较慢。

### 决策四：per-chat 并发测试用 spawned task 验证阻塞

优点是能真正验证锁的语义。代价是需要 sleep 等待来判断阻塞状态，存在时序敏感性（通过足够的 margin 缓解）。

## 容易走错的地方

### 失败模式 1：不测并行工具执行的 wave 分区

这样当 `tool_concurrency_overrides` 配置错误时，可能把 SideEffect 工具错误地分到并行 wave，导致数据竞态。

### 失败模式 2：子代理测试只覆盖 happy path

不测超时和取消路径，生产中就会出现僵死的子代理进程。

### 失败模式 3：依赖审计只做一次，不放进 CI

新引入的依赖可能带来安全公告或许可证冲突，只有 CI 持续检查才能及时发现。

### 失败模式 4：忽略 ChatTurnQueue 的竞态窗口

`try_start_or_enqueue` 存在的意义就是消除 `enqueue_if_busy` + `acquire` 之间的竞态。如果测试只覆盖了分开的接口而不覆盖原子接口，可能遗漏竞态 bug。

## 小结

MicroClaw v0.1.38 的测试策略围绕三个新维度展开：`deny.toml` 把依赖安全纳入了自动化门禁，wave-based 并行执行带来了分区正确性和事件完整性的新测试面，子代理生命周期和 per-chat turn serialization 要求端到端地验证并发语义。系统不是不可测试的——关键在于识别哪些部分是确定性的（用传统断言），哪些是并发语义的（用阻塞验证和状态检查），哪些依赖 LLM（用结构化断言）。

下一章我们转向性能与成本优化。因为当系统足够稳定后，团队很快会碰到另一个现实问题：它能不能在不牺牲质量的前提下跑得更快、花得更少。

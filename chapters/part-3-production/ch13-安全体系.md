# Chapter 13 安全体系

## 这一章要回答什么问题

一旦 Agent 能执行命令、读写文件、抓网页、发消息、接插件，它就不再是单纯的聊天产品，而是一个带有真实操作权的运行时系统。到了这一阶段，“能跑”已经不构成判断标准，真正的问题变成了：它能否在默认情况下保持足够安全，并在风险升高时提供清晰的控制点。

MicroClaw 的安全设计很有代表性，因为它没有寄希望于某一个神奇开关，而是采用多层防线：

- 审批门
- 执行策略
- sandbox
- path guard
- URL 与内容校验
- 控制面鉴权
- 审计与自检

这一章读完后，你应该已经能回答两个关键问题：MicroClaw 当前的威胁模型是什么，以及这些风险是如何被拆解到不同控制层里的。

## 先画威胁模型：系统里到底有什么值得保护

对 MicroClaw 来说，真正需要保护的资产至少包括以下几类。

### 宿主机执行环境

包括工作目录、宿主机文件系统、命令执行能力以及通过 shell 可以进一步触达的系统资源。这部分资产一旦越权，后果通常最严重。

### 运行时状态数据

`microclaw.db` 中保存的不只是聊天记录，还包括：

- 会话状态
- 结构化记忆
- 调度任务
- API key 和 session
- 指标历史
- 审计日志
- 子代理运行记录

这意味着数据库既是功能核心，也是高价值攻击目标。

### 凭据与配置

模型 API key、渠道 token、MCP server 认证头、ClawHub token 等都属于敏感信息。配置误泄露往往比代码 bug 更快造成事故。

### 渠道和 chat 边界

一旦跨 chat 权限控制失效，系统就可能把本应属于一个会话的能力投射到另一个会话，形成数据泄露或主动消息滥用。

### 扩展生态入口

MCP server、Skills、Plugins、Hooks、ClawHub 安装包都属于供应链和执行面结合的敏感入口。它们的风险不是单点，而是“外部能力直接进入主链路”。

## 核心攻击面：风险主要从哪里进来

### 工具执行

`bash`、文件写入、插件命令执行是最显眼的高风险入口。它们拥有直接副作用，一旦被误用或越权，损害面非常大。

### 文件路径与挂载路径

即使没有 shell，错误的路径解析也可能让文件工具接触到：

- `~/.ssh`
- `~/.aws`
- `.env`
- 密钥文件
- 符号链接跳转后的敏感路径

### Web 抓取与提示注入

`web_fetch` 的风险不只是 SSRF，还包括把恶意网页内容直接返回给模型，诱导后续执行危险操作。

### 控制面接口

Web API 一旦缺乏鉴权、作用域控制、CSRF 防护或速率限制，就会从“本地控制面”退化成“本地高危入口”。

### 扩展供应链

安装 Skills、加载 Plugins、连接外部 MCP server，本质上都是把外部定义的能力或内容纳入 runtime。这里的风险来自来源可信度、版本漂移和运行权限。

### 小例子：为什么风险分级必须在执行前完成？

安全控制最怕的一种失败，是系统先开始执行，事后才发现这原来是高风险动作。把风险分级前置成独立步骤，才能让审批、sandbox 和审计围绕同一判断展开。

Rust 版本先定义 `RiskLevel`，再让分类器根据动作名称给出等级。这样后续 gate 判断依赖的是统一分类结果，而不是到处散落的字符串比较。

```rust
enum RiskLevel {
    Low,
    High,
}

trait RiskClassifier {
    fn classify(&self, action: &str) -> RiskLevel;
}

fn needs_confirmation(level: RiskLevel) -> bool {
    matches!(level, RiskLevel::High)
}
```

Python 版本用 `Enum` 和 `Protocol` 保持同样的结构。这样读者会更容易看出，审批链的起点其实不是“弹个确认框”，而是先把动作风险说清楚。

```python
from enum import Enum
from typing import Protocol


class RiskLevel(str, Enum):
    LOW = "low"
    HIGH = "high"


class RiskClassifier(Protocol):
    def classify(self, action: str) -> RiskLevel: ...


def needs_confirmation(level: RiskLevel) -> bool:
    return level is RiskLevel.HIGH
```

## 第一层防线：风险分级与审批门

MicroClaw 当前最重要的默认安全立场之一，是高风险工具必须经过审批。`agent_engine.rs` 在工具执行遇到 `approval_required` 时，会根据配置和用户最近的明确确认语义决定是否继续。

核心配置是：

- `high_risk_tool_user_confirmation_required = true`

这意味着高风险动作默认不是“模型想做就做”，而是“用户必须明确放行”。这类控制必须在 runtime 层实现，而不是依赖 prompt 中的道德劝告。因为真正需要的是可执行约束，而不是语言约束。

审批门的价值有三点：

1. 把高风险动作从默认路径移出。
2. 让用户确认成为可观测事件。
3. 为自动化环境保留清晰开关，而不是隐式豁免。

## 第二层防线：执行策略与 sandbox

### 执行策略标签

`docs/security/execution-model.md` 把工具执行策略分为：

- `host_only`
- `sandbox_only`
- `dual`

当前 `bash` 是 `dual`，而文件写入与编辑等工具仍以 `host_only` 为主。这不是缺陷，而是现实折中：不同工具对宿主机路径和执行环境的依赖程度不同，不能用一把尺子强行套住。

### sandbox 的真实角色

MicroClaw 当前支持：

- `auto`
- `docker`
- `podman`

并提供：

- `hardened`
- `standard`
- `privileged`

三种安全 profile。它的推荐姿态也很清楚：

- 入门阶段可保持 `sandbox=off`
- 生产或高风险环境启用 sandbox，并配合 allowlist

这说明 sandbox 在 MicroClaw 里既重要，又没有被神话。它是关键防线，但不是唯一防线；没有审批、路径限制和权限模型，单靠容器并不能解决所有问题。

## 第三层防线：路径保护与挂载校验

文件工具真正危险的地方，不是“能写文件”，而是“能写到哪里”。`microclaw-tools` 把这部分风险控制抽成了底层能力，包括：

- 敏感路径 deny list
- symlink 校验
- 可选 path allowlist
- 可选 mount allowlist

`docs/security/execution-model.md` 明确列出了会被拦截的敏感组件，例如 `.ssh`、`.aws`、`.gnupg`、`.kube`、`.docker`、`.env` 等。这种策略非常务实，因为绝大多数误操作和很多攻击链，最终都会落到“路径边界是否被严格处理”上。

## 第四层防线：网络抓取双重校验

`web_fetch` 的安全设计是当前系统里最值得借鉴的细节之一。它采用的是两阶段控制：

1. URL policy check
2. 抓取后 content validation

这比单纯做 host denylist 或单纯做内容清洗都更完整。

### URL 校验解决什么问题

它主要防止：

- 本地地址和元数据地址访问
- 非允许 scheme 的请求
- 未进入 allowlist 的目标域名

### 内容校验解决什么问题

它主要防止把恶意 prompt 注入文本直接回流给模型。对于 Agent 来说，这一步非常关键，因为抓到的网页内容往往会进入下一轮推理上下文。

生产环境里，最稳妥的姿态通常是：

- 打开 URL validation
- 配置显式 denylist
- 有条件时启用 allowlist
- 对 feed sync 采用更保守的故障策略

## 第五层防线：控制面鉴权与作用域

`docs/rfcs/0001-authn-authz-model.md` 已经把控制面安全从单一 token 模式推进到更成熟的方向，包括：

- session cookie
- API key
- scopes
- 登录和敏感接口节流

当前 scope 语义包括：

- `operator.read`
- `operator.write`
- `operator.admin`
- `operator.approvals`

这意味着 Web 控制面已经不再是“登录了就什么都能做”，而是开始具备职责分层。对生产部署来说，这很重要，因为自动化脚本、只读观察者和高危审批操作者不应共享同一权限面。

此外，控制面还需要关注：

- HTTP-only session cookie
- CSRF token
- API key 哈希存储
- 显式 revoke 与 expiry 检查

这些都是一个真正可托管系统的最低要求。

## 第六层防线：扩展与供应链治理

MicroClaw 的安全问题不只发生在内建工具上，也发生在扩展生态上。

### MCP

风险在于把外部 server 直接接入主链路。对应控制包括：

- 超时
- 并发限制
- rate limit
- circuit breaker

### Skills

风险在于技能来源、平台兼容和依赖满足情况。对应控制包括：

- frontmatter 元数据
- availability diagnostics
- enable / disable 状态

### Plugins

风险在于本地命令执行和上下文注入。对应控制包括：

- allowed channels
- require_control_chat
- execution_policy

### ClawHub

风险在于安装来源与版本漂移。对应控制包括：

- security warnings
- lockfile
- 生产环境固定版本

这说明 MicroClaw 对扩展的态度是：可以接，但必须带着边界接。

## 审计、自检与可恢复性

真正成熟的安全体系，不只是阻止危险动作，还要让系统在事后可回溯、在部署前可自查。

### 审计

存储层中已经存在 `AuditLogRecord` 结构，说明系统正在把关键行为纳入审计视角。对高风险 runtime 来说，这一点非常关键，因为很多安全问题不是完全阻止，而是需要在事故后回答：

- 谁触发了什么动作
- 作用到了哪里
- 结果是成功还是失败

### 配置自检

控制面自检可以让运维在上线前发现：

- 高风险配置
- 不安全默认值
- 缺失依赖
- 权限策略不完整

### 升级与回滚

`docs/releases/upgrade-guide.md` 明确建议升级前备份 SQLite 数据库，并把“恢复 DB 备份”作为主要回滚路径。这种设计虽然传统，却是最可靠的恢复策略。

## 示例代码：为什么高风险操作必须经过审批，而不是让模型直接执行？

最关键的安全控制之一，就是把“模型提出动作”与“系统允许动作”明确拆开。只有审批链独立存在，运行时才有机会把危险操作卡在真正执行之前。

Rust 版本把审批策略抽成 trait，再用 `ApprovalGate` 统一执行业务动作前的判定。这样审批不是一条零散的 `if`，而是可以被替换、测试和审计的正式边界。

```rust
#[async_trait::async_trait]
trait ApprovalPolicy {
    async fn confirm(&self, action: &str) -> anyhow::Result<bool>;
}

struct ApprovalGate<P> {
    policy: P,
}

impl<P: ApprovalPolicy> ApprovalGate<P> {
    async fn authorize(&self, action: &str) -> anyhow::Result<()> {
        if self.policy.confirm(action).await? {
            return Ok(());
        }
        anyhow::bail!("approval required for {action}")
    }
}
```

Python 版本保留相同的结构，但用 `Protocol` 和 `@dataclass` 显式描述审批边界。这样读者会更容易意识到：安全控制真正要管理的是执行前的决策点，而不是异常发生后的补救。

```python
from dataclasses import dataclass
from typing import Protocol


class ApprovalPolicy(Protocol):
    async def confirm(self, action: str) -> bool: ...


@dataclass
class ApprovalGate:
    policy: ApprovalPolicy

    async def authorize(self, action: str) -> None:
        if await self.policy.confirm(action):
            return
        raise RuntimeError(f"approval required for {action}")
```

## 关键权衡

### 决策一：默认把高风险工具纳入审批链

优点是用户始终有最终控制权。代价是自动化体验会稍慢，某些场景需要显式放权。

### 决策二：sandbox 作为关键但非唯一防线

优点是兼顾入门易用和生产加固。代价是安全姿态需要根据环境主动升级，不能永远停留在默认配置。

### 决策三：文件、网络、控制面分别设置不同控制层

优点是风险更细粒度。代价是系统安全模型比“一个开关全解决”更复杂。

### 决策四：把扩展生态纳入主安全模型

优点是不会出现“内建能力很安全，扩展能力裸奔”的断层。代价是扩展接入门槛更高。

## 容易走错的地方

### 失败模式 1：把安全完全等同于 sandbox

没有审批、路径保护和控制面鉴权，容器化本身不足以建立可信边界。

### 失败模式 2：长期使用引导态安全配置

默认密码、宽松 allowlist、关闭审批等姿态适合试验，不适合托管。

### 失败模式 3：扩展能力接入时只验证功能，不验证来源和权限

这会把外部供应链风险直接带入主链路。

## 小结

MicroClaw 当前的安全体系并不追求“一次性解决所有安全问题”，而是通过多层控制把高风险动作逐步压缩在更小、更可审计的边界内。对一个拥有执行能力的 Agent Runtime 来说，这种分层治理比任何单点技术都更重要。

下一章我们继续从生产视角前进，讨论可观测性与运维。因为安全只能告诉你“尽量别出事”，而真正的生产系统还必须做到“出事时能发现、能定位、能恢复”。

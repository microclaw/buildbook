# Chapter 13 安全体系

## 威胁模型

Agent 一旦能执行命令、读写文件、发消息、接插件，就是带有真实操作权的运行时系统。MicroClaw 的安全体系覆盖七层防线。

| 资产类别 | 包含内容 | 风险后果 |
|---------|---------|---------|
| 宿主机执行环境 | 工作目录、文件系统、shell | 越权后果最严重 |
| 运行时状态数据 | `microclaw.db`：会话、记忆、调度、API key 哈希、审计日志 | 高价值攻击目标 |
| 凭据与配置 | 模型 API key、渠道 token、MCP 认证头 | 配置误泄露比代码 bug 更快造成事故 |
| 渠道与 chat 边界 | 跨 chat 权限隔离 | `ChatTurnQueue` 在运行时保障 |
| 扩展生态入口 | MCP/Skills/Plugins/Hooks/ClawHub | 供应链风险 |

## 核心攻击面

| 攻击面 | 风险 | 控制 |
|-------|------|------|
| 工具执行 | 直接副作用 | Concurrency class 分级（ReadOnly / SideEffect / Exclusive） |
| 并发竞态 | 同一 chat 多条消息并发 | `ChatTurnQueue` per-chat turn serialization |
| Web 控制面 | 缺鉴权时退化为高危入口 | Argon2 密码、API key scope、CSRF、登录限流 |
| 子代理越权 | 不受控递归执行 | `max_concurrent`、`max_tokens_per_run`、`max_spawn_depth` |
| 扩展供应链 | 外部能力纳入 runtime | 超时、并发限制、rate limit、circuit breaker、lockfile |

## 第一层：运行取消与 turn 隔离

### run_control：随时可以停下来

`run_control` 用 `(channel, chat_id)` 为每次 agent run 注册 `AtomicBool` 取消标志。调用 `abort_runs` 后，工具执行循环在下一个 tool boundary 处干净终止。

```rust
// run_control.rs 核心签名
pub async fn register_run(
    channel: &str, chat_id: i64,
    source_message_id: Option<String>,
) -> (u64, Arc<AtomicBool>, Arc<Notify>);

pub async fn abort_runs(channel: &str, chat_id: i64) -> usize;

pub fn is_cancelled(flag: &AtomicBool) -> bool;
```

### ChatTurnQueue：同一 chat 内最多一个活跃 run

保证同一 `(channel, chat_id)` 任何时刻只有一个 agent run 在执行，后到消息 coalesce 进 `pending_messages` 队列。

- **per-chat 粒度**：不同 chat 完全独立，无全局锁
- **RAII guard**：`TurnGuard` drop 时自动释放
- **60 秒超时**：防递归死锁
- **原子操作**：`try_start_or_enqueue` 消除检查-获取竞态窗口

```rust
pub async fn acquire(&self, channel: &str, chat_id: i64) -> Option<TurnGuard>;
pub async fn try_start_or_enqueue(
    &self, channel: &str, chat_id: i64, msg: PendingMessage,
) -> Option<TurnGuard>;
pub async fn drain_pending(&self, channel: &str, chat_id: i64) -> Vec<PendingMessage>;
```

Rust 用 RAII guard 保证锁生命周期与 run 精确对齐：

```rust
struct TurnGuard {
    _guard: tokio::sync::OwnedMutexGuard<()>,
    key: (String, i64),
}

impl Drop for TurnGuard {
    fn drop(&mut self) {
        // 自动释放 per-chat turn lock

```

```python
from dataclasses import dataclass
from contextlib import asynccontextmanager
from asyncio import Lock
from typing import AsyncIterator


@dataclass
class ChatSlot:
    turn_lock: Lock
    pending: list[str]

@asynccontextmanager
async def acquire_turn(slot: ChatSlot) -> AsyncIterator[None]:
    await slot.turn_lock.acquire()
    try:
        yield
    finally:
        slot.turn_lock.release()
```

## 第二层：风险分级与审批门

高风险工具必须经过审批（`high_risk_tool_user_confirmation_required = true`）。`Exclusive` 类工具默认需要审批，`ReadOnly` 类通常不需要。

## 第三层：执行策略与 sandbox

**Sandbox 模式**：`Off`（入门）、`Shared`（共享容器）、`Chat`（每 chat 独立容器，多用户场景关键）。

**执行策略**：`host_only`、`sandbox_only`、`dual`（根据配置决定）。

**路径保护**：敏感路径 deny list（`.ssh`、`.aws`、`.gnupg` 等）、symlink 校验、可选 allowlist。

## 第四层：控制面鉴权

### 密码管理（Argon2）

```rust
pub(super) fn make_password_hash(password: &str) -> String {
    let salt = SaltString::encode_b64(uuid::Uuid::new_v4().as_bytes())
        .unwrap_or_else(|_| SaltString::from_b64("AAAA...").unwrap());
    Argon2::default()
        .hash_password(password.as_bytes(), &salt)
        .map(|h| h.to_string())
        .unwrap_or_default()
}
```

旧格式 `v1$salt$hash` 登录时自动升级为 Argon2。

### Bootstrap Token

首次启动生成一次性 token → 管理员通过 `X-Bootstrap-Token` 设置初始密码 → token 立即失效。消除"默认密码"反模式。

### API Key 生命周期

- **生成**：`mk_<uuid>` 格式，服务端只存 SHA-256 哈希
- **撤销**：标记 `revoked_at`
- **轮换**：原子创建新 key 并撤销旧 key

Scope：`operator.read` / `operator.write` / `operator.admin` / `operator.approvals`

### CSRF 双 cookie + 登录限流

登录后设置 `mc_session`（HttpOnly）+ `mc_csrf`（JS 可读），写操作需 CSRF token 匹配。60 秒内最多 5 次登录尝试。

## 第五层：子代理权限边界

- `subagents.max_concurrent`：系统级 semaphore
- `subagents.max_active_per_chat`：单 chat 上限
- `subagents.run_timeout_secs`：运行时间上限
- `subagents.max_tokens_per_run`：token 预算
- ACP `RequestPermission` 机制申请高风险操作权限
- `subagent_runs` 表记录运行 ID、状态、输出摘要和成本

## 第六层：网络抓取双重校验

1. URL policy check：防本地地址、非允许 scheme、域名限制
2. 抓取后 content validation：防 prompt 注入

## 第七层：扩展供应链治理

| 扩展类型 | 风险 | 控制 |
|---------|------|------|
| MCP | 外部 server 接入主链路 | 超时、并发、rate limit、circuit breaker |
| Skills | 来源可信度 | frontmatter 元数据、availability diagnostics |
| Plugins | 本地命令执行 | allowed channels、require_control_chat |
| Hooks | 拦截/修改运行流 | 超时、命令白名单、输出大小限制 |
| ClawHub | 安装来源与版本漂移 | security warnings、lockfile、固定版本 |

## 审计与自检

- **审计**：`audit_log` 记录 actor、action、target、status
- **配置自检**：`microclaw doctor` 发现高风险配置和缺失依赖
- **升级回滚**：`microclaw upgrade` + SQLite 备份

## 示例代码：API key 只存哈希

```rust
use sha2::{Digest, Sha256};

fn generate_api_key() -> (String, String, String) {
    let raw_key = format!("mk_{}", uuid::Uuid::new_v4().simple());
    let prefix = raw_key.chars().take(10).collect::<String>();
    let mut hasher = Sha256::new();
    hasher.update(raw_key.as_bytes());
    let hash = format!("{:x}", hasher.finalize());
    (raw_key, prefix, hash)  // raw_key 只返回给用户一次
}
```

```python
import hashlib
import uuid
from dataclasses import dataclass


@dataclass
class ApiKeyResult:
    raw_key: str    # 只返回给用户一次
    prefix: str     # 用于管理界面显示
    hash: str       # 存入数据库

def generate_api_key() -> ApiKeyResult:
    raw_key = f"mk_{uuid.uuid4().hex}"
    prefix = raw_key[:10]
    key_hash = hashlib.sha256(raw_key.encode()).hexdigest()
    return ApiKeyResult(raw_key=raw_key, prefix=prefix, hash=key_hash)
```

## 容易走错的地方

### 失败模式 1：把安全完全等同于 sandbox

没有 turn serialization、审批、路径保护和控制面鉴权，容器化本身不足以建立可信边界。

### 失败模式 2：忽视并发竞态

多 chat 并发场景下状态混乱。`ChatTurnQueue` 解决的正是这个问题。

### 失败模式 3：API key 明文存储

数据库泄露 = 直接交出控制权。只存哈希是最低要求。

### 失败模式 4：子代理无预算限制

一次看似无害的工具调用，可能在几分钟内消耗掉整个月的 API 预算。`max_tokens_per_run` 和 `max_spawn_depth` 把风险限制在可预测范围。

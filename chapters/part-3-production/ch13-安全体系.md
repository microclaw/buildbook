# Chapter 13 安全体系

## Agent 有了真实操作权之后，默认安全基线是什么？

一旦 Agent 能执行命令、读写文件、抓网页、发消息、接插件，它就不再是单纯的聊天产品，而是一个带有真实操作权的运行时系统。"能跑"不再构成判断标准，真正的问题是：它能否在默认情况下保持足够安全，并在风险升高时提供清晰的控制点。

MicroClaw v0.1.38 的安全体系相比早期版本有了质的变化。最显著的几项演进包括：

- `run_control` 模块实现了 per-chat 运行取消
- `ChatTurnQueue` 以 per-chat 粒度实现 turn serialization，从根本上消除了并发竞态
- API key 生命周期管理：生成、撤销、轮换
- Bootstrap token 实现了无密码首次部署的安全引导
- Argon2 密码哈希替代了早期的 SHA-256 salt 方案
- CSRF 双 cookie 防护
- 子代理（subagent）权限隔离

这一章读完后，你应该已经能回答两个关键问题：MicroClaw 当前的威胁模型是什么，以及这些风险是如何被拆解到不同控制层里的。

## 威胁模型：系统里有什么值得保护？

对 MicroClaw 来说，真正需要保护的资产至少包括以下几类。

### 宿主机执行环境

包括工作目录、宿主机文件系统、命令执行能力以及通过 shell 可以进一步触达的系统资源。这部分资产一旦越权，后果通常最严重。

### 运行时状态数据

`microclaw.db` 中保存的不只是聊天记录，还包括：

- 会话状态与 session cookie
- 结构化记忆
- 调度任务
- API key 哈希与 session 表
- 指标历史
- 审计日志
- 子代理运行记录（`subagent_runs` 表）

数据库既是功能核心，也是高价值攻击目标。

### 凭据与配置

模型 API key、渠道 token、MCP server 认证头、ClawHub token 等都属于敏感信息。配置误泄露往往比代码 bug 更快造成事故。

### 渠道和 chat 边界

一旦跨 chat 权限控制失效，系统就可能把本应属于一个会话的能力投射到另一个会话，形成数据泄露或主动消息滥用。v0.1.38 通过 `ChatTurnQueue` 把这层边界做到了运行时级别。

### 扩展生态入口

MCP server、Skills、Plugins、Hooks、ClawHub 安装包都属于供应链和执行面结合的敏感入口。它们的风险不是单点，而是"外部能力直接进入主链路"。

## 核心攻击面：风险从哪里进来？

### 工具执行

`bash`、文件写入、插件命令执行是最显眼的高风险入口。它们拥有直接副作用，一旦被误用或越权，损害面非常大。v0.1.38 的 44 个内建工具每一个都有 concurrency class 分级（ReadOnly / SideEffect / Exclusive），这不只是性能分类，也是安全分类。

### 并发竞态

v0.1.16 时代的全局锁模型在并发场景下存在两个问题：要么锁得太粗导致所有 chat 互相阻塞，要么锁得不够导致同一 chat 内多条消息同时驱动 agent 循环。v0.1.38 通过 `ChatTurnQueue` 以 `(channel, chat_id)` 为粒度实现了精确的 turn serialization。这个设计从源头消除了同一聊天内的并发竞态。

### Web 控制面接口

Web API 一旦缺乏鉴权、作用域控制、CSRF 防护或速率限制，就会从"本地控制面"退化成"本地高危入口"。

### 子代理越权

Session-native subagent 通过 ACP 协议驱动独立进程。如果不限制子代理的工作目录、token 预算和并发数量，父 agent 的一次工具调用就可能触发不受控的递归执行。

### 扩展供应链

安装 Skills、加载 Plugins、连接外部 MCP server，本质上都是把外部定义的能力或内容纳入 runtime。这里的风险来自来源可信度、版本漂移和运行权限。

## 第一层防线：运行取消与 turn 隔离

v0.1.38 最重要的安全基础设施之一，是 `run_control.rs` 和 `chat_turn_queue.rs` 的配合。

### run_control：随时可以停下来

`run_control` 用 `(channel, chat_id)` 作为 key，为每次 agent run 注册一个 `AtomicBool` 取消标志和 `Notify` 唤醒器。任何时候调用 `abort_runs(channel, chat_id)` 都会：

1. 将该 chat 下所有活跃 run 标记为 `cancelled = true`
2. 通知所有等待中的 waiter
3. 记录被取消的 source message id，防止后续重试

工具执行循环在每个 iteration 开头检查 `is_cancelled(flag)`，一旦为 true 立刻中止。这意味着用户在 Web UI 或聊天渠道中发送 `/stop` 时，系统能在下一个 tool boundary 处干净地终止运行。

```rust
// run_control.rs 核心签名
pub async fn register_run(
    channel: &str, chat_id: i64,
    source_message_id: Option<String>,
) -> (u64, Arc<AtomicBool>, Arc<Notify>);

pub async fn abort_runs(channel: &str, chat_id: i64) -> usize;

pub fn is_cancelled(flag: &AtomicBool) -> bool;
```

### ChatTurnQueue：同一 chat 内最多一个活跃 run

`ChatTurnQueue` 是 v0.1.38 并发模型的核心。它保证同一 `(channel, chat_id)` 在任何时刻只有一个 agent run 在执行。后到的消息不会被丢弃，而是被 coalesce 进 `pending_messages` 队列，等当前 run 结束后再处理。

关键设计选择：

- **per-chat 粒度**：不同 chat 之间完全独立，不存在全局锁。Telegram chat 1 的长运行不会阻塞 Discord chat 2。
- **RAII guard**：`TurnGuard` 在 drop 时自动释放锁，避免忘记解锁。
- **60 秒超时**：如果等待锁超过 60 秒（可能是递归调用导致的死锁），系统会放弃等待并记录警告，而不是永远阻塞。
- **容量管理**：`max_pending` 限制队列深度，超出时丢弃最旧的消息。
- **原子操作**：`try_start_or_enqueue` 在单个操作中完成"尝试获取锁或入队"，消除了检查和获取之间的竞态窗口。

```rust
// chat_turn_queue.rs 核心接口
pub async fn acquire(&self, channel: &str, chat_id: i64) -> Option<TurnGuard>;
pub async fn try_start_or_enqueue(
    &self, channel: &str, chat_id: i64, msg: PendingMessage,
) -> Option<TurnGuard>;
pub async fn drain_pending(&self, channel: &str, chat_id: i64) -> Vec<PendingMessage>;
```

这两个模块配合使用：`ChatTurnQueue` 保证序列化，`run_control` 保证可取消。这是 v0.1.38 安全模型中最基础、最不可绕过的一层。

### 并发控制为什么必须在 runtime 层实现？

如果并发控制依赖模型的"自我约束"（比如在 prompt 中说"不要同时处理两条消息"），那它根本不是控制，而是建议。真正的并发安全必须在可执行的运行时机制中实现。

Rust 版本用 RAII guard 表达这个关键思想：锁的获取和释放由类型系统保证，不可能被遗忘。

```rust
struct TurnGuard {
    _guard: tokio::sync::OwnedMutexGuard<()>,
    key: (String, i64),
}

impl Drop for TurnGuard {
    fn drop(&mut self) {
        // 自动释放 per-chat turn lock
    }
}
```

Python 版本用 async context manager 保持同样的结构。这样读者会更清楚看到：turn serialization 的核心不是"加个锁"，而是保证锁的生命周期与 run 的生命周期精确对齐。

```python
from dataclasses import dataclass
from contextlib import asynccontextmanager
from asyncio import Lock
from typing import AsyncIterator


@dataclass
class ChatSlot:
    turn_lock: Lock
    pending: list[str]

@asynccontextmanager
async def acquire_turn(slot: ChatSlot) -> AsyncIterator[None]:
    await slot.turn_lock.acquire()
    try:
        yield
    finally:
        slot.turn_lock.release()
```

## 第二层防线：风险分级与审批门

MicroClaw 当前最重要的默认安全立场之一，是高风险工具必须经过审批。`agent_engine.rs` 在工具执行遇到 `approval_required` 时，会根据配置和用户最近的明确确认语义决定是否继续。

核心配置是：

- `high_risk_tool_user_confirmation_required = true`

审批门的价值有三点：

1. 把高风险动作从默认路径移出。
2. 让用户确认成为可观测事件。
3. 为自动化环境保留清晰开关，而不是隐式豁免。

v0.1.38 的 tool concurrency class 也与风险分级深度关联。`Exclusive` 类工具（如 `bash`）不仅在执行时独占 wave，其高风险性质也意味着默认需要审批。`ReadOnly` 类工具可以并行，同时也通常不需要审批确认。

## 第三层防线：执行策略与 sandbox

### sandbox 模式演进

v0.1.38 的 sandbox 配置从 `SandboxMode` enum 出发，支持：

- `Off`：关闭，适合入门和信任环境
- `Shared`：所有 chat 共享同一个容器实例
- `Chat`：每个 chat 独立容器，提供更强的隔离

`Chat` 模式是 v0.1.38 新增的关键安全能力。它意味着不同对话之间即使都在执行 `bash`，也不会共享文件系统或进程空间。这对多用户场景（家庭服务器、团队共享实例）尤为重要。

### 执行策略标签

工具执行策略分为：

- `host_only`：直接在宿主机执行
- `sandbox_only`：只在容器内执行
- `dual`：根据配置决定

当前 `bash` 是 `dual`，而文件写入与编辑等工具仍以 `host_only` 为主。这不是缺陷，而是现实折中：不同工具对宿主机路径和执行环境的依赖程度不同。

### 路径保护与挂载校验

文件工具真正危险的地方，不是"能写文件"，而是"能写到哪里"。`microclaw-tools` 把这部分风险控制抽成了底层能力，包括：

- 敏感路径 deny list（`.ssh`、`.aws`、`.gnupg`、`.kube`、`.docker`、`.env`）
- symlink 校验
- 可选 path allowlist
- 可选 mount allowlist

## 第四层防线：控制面鉴权体系

v0.1.38 的 Web 控制面鉴权经历了从"单一 token"到完整身份系统的演进。

### 密码管理与 Argon2

`web/middleware.rs` 中的 `make_password_hash` 使用 Argon2 算法：

```rust
pub(super) fn make_password_hash(password: &str) -> String {
    let salt = SaltString::encode_b64(uuid::Uuid::new_v4().as_bytes())
        .unwrap_or_else(|_| SaltString::from_b64("AAAA...").unwrap());
    Argon2::default()
        .hash_password(password.as_bytes(), &salt)
        .map(|h| h.to_string())
        .unwrap_or_default()
}
```

`verify_password_hash` 同时兼容旧的 `v1$salt$hash` 格式。如果登录时检测到旧格式，系统会自动升级为 Argon2 哈希——这是一种非常务实的渐进迁移策略。

### Bootstrap Token

新部署时数据库中没有密码哈希，此时系统无法通过常规认证设置初始密码。v0.1.38 引入了 bootstrap token：

1. 首次启动时，系统在内存中生成一个一次性 token 并输出到控制台
2. 管理员通过 `X-Bootstrap-Token` 头调用 `POST /api/auth/set-password` 设置初始密码
3. 密码设置成功后，bootstrap token 立即失效

这个设计避免了"默认密码"这个经典安全反模式。

### API Key 生命周期

`web/auth.rs` 实现了完整的 API key 管理：

- **生成**：`POST /api/auth/api-keys`，返回 `mk_<uuid>` 格式的密钥，服务端只存储 SHA-256 哈希
- **撤销**：`DELETE /api/auth/api-keys/:id`，标记 `revoked_at`
- **轮换**：`POST /api/auth/api-keys/:id/rotate`，原子地创建新 key 并撤销旧 key，支持继承或更新 scope

每个 key 携带 scope 列表，取值范围：

- `operator.read`
- `operator.write`
- `operator.admin`
- `operator.approvals`

scope 验证逻辑在 `AuthIdentity::allows` 中统一执行，`operator.admin` 隐含所有权限。

### CSRF 防护

登录成功后，系统同时设置两个 cookie：

- `mc_session`：HttpOnly, SameSite=Strict
- `mc_csrf`：非 HttpOnly（允许 JavaScript 读取），SameSite=Strict

写操作需要 CSRF token 匹配。这是标准的双 cookie 模式，在单页应用场景下比 Synchronizer Token 模式更简洁。

### 登录限流

`AuthHub` 维护了 per-client 的登录尝试计数器。默认策略是 60 秒内最多 5 次尝试，超限返回 `429 Too Many Requests`。

## 第五层防线：子代理权限边界

v0.1.38 引入的 session-native subagent 通过 ACP 协议启动独立进程。安全边界体现在：

### 并发控制

- `subagents.max_concurrent`：系统级 semaphore 限制同时运行的子代理数量
- `subagents.max_active_per_chat`：单个 chat 内的子代理上限
- `subagents.run_timeout_secs`：单次子代理运行的时间上限

### Token 预算

- `subagents.max_tokens_per_run`：限制单次子代理运行的 token 消耗

### 权限隔离

子代理继承父 agent 的 `ToolAuthContext`，但在独立进程中执行。ACP 协议要求子代理通过 `RequestPermission` 机制申请高风险操作权限，父 agent 可以拒绝。

### 运行追踪

所有子代理运行记录写入 `subagent_runs` 表，包含运行 ID、状态、输出摘要和成本。这既是审计需要，也是成本控制的基础。

## 第六层防线：网络抓取双重校验

`web_fetch` 的安全设计采用两阶段控制：

1. URL policy check：防止本地地址访问、非允许 scheme、未进入 allowlist 的域名
2. 抓取后 content validation：防止恶意 prompt 注入文本回流给模型

生产环境里，最稳妥的姿态通常是打开 URL validation、配置显式 denylist、有条件时启用 allowlist。

## 第七层防线：扩展与供应链治理

### MCP

风险在于把外部 server 直接接入主链路。对应控制包括超时、并发限制、rate limit、circuit breaker。

### Skills

风险在于技能来源、平台兼容和依赖满足情况。对应控制包括 frontmatter 元数据、availability diagnostics、enable/disable 状态。

### Plugins

风险在于本地命令执行和上下文注入。对应控制包括 allowed channels、require_control_chat、execution_policy。

### Hooks

v0.1.38 的 hooks 系统定义了三个事件点（`BeforeLLMCall`、`BeforeToolCall`、`AfterToolCall`），每个 hook 可以拦截或修改运行流。Hook 本身也需要安全治理：超时、命令白名单、输出大小限制。

### ClawHub

风险在于安装来源与版本漂移。对应控制包括 security warnings、lockfile、生产环境固定版本。

## 审计、自检与可恢复性

### 审计

系统在关键操作后调用 `audit_log`，记录 actor、action、target、status。API key 创建、撤销、轮换，密码设置，登录登出都会产生审计事件。

### 配置自检

`microclaw doctor` 能在上线前发现高风险配置、不安全默认值、缺失依赖、权限策略不完整。Doctor 的检查覆盖了跨平台差异（macOS、Linux、Windows/WSL）。

### 升级与回滚

`microclaw upgrade` 提供了标准化的升级路径。数据库 migration 在启动时自动执行，回滚依赖 SQLite 备份。

## 示例代码：为什么 API key 必须只存哈希，永远不存明文？

API key 管理中最容易犯的错误，是在数据库中保存明文 key。一旦数据库泄露，所有 key 立刻失效。正确做法是只保存哈希，在验证时比较哈希。

Rust 版本展示了 MicroClaw 的实际策略：生成时返回明文给用户（仅此一次），数据库只存 SHA-256 哈希和前缀。

```rust
use sha2::{Digest, Sha256};

fn generate_api_key() -> (String, String, String) {
    let raw_key = format!("mk_{}", uuid::Uuid::new_v4().simple());
    let prefix = raw_key.chars().take(10).collect::<String>();
    let mut hasher = Sha256::new();
    hasher.update(raw_key.as_bytes());
    let hash = format!("{:x}", hasher.finalize());
    (raw_key, prefix, hash)  // raw_key 只返回给用户一次
}
```

Python 版本保持同样的结构。前缀用于管理界面显示（"以 mk_abc123 开头的 key"），而哈希用于验证——这样即使数据库被导出，攻击者也无法还原出可用的 key。

```python
import hashlib
import uuid
from dataclasses import dataclass


@dataclass
class ApiKeyResult:
    raw_key: str    # 只返回给用户一次
    prefix: str     # 用于管理界面显示
    hash: str       # 存入数据库

def generate_api_key() -> ApiKeyResult:
    raw_key = f"mk_{uuid.uuid4().hex}"
    prefix = raw_key[:10]
    key_hash = hashlib.sha256(raw_key.encode()).hexdigest()
    return ApiKeyResult(raw_key=raw_key, prefix=prefix, hash=key_hash)
```

## 关键权衡

### 决策一：per-chat turn serialization 而非全局锁

优点是不同 chat 之间完全独立，不存在跨 chat 阻塞。代价是需要管理每个 chat 的独立锁状态和 pending queue，内存占用随活跃 chat 数线性增长（通过 idle TTL 清理缓解）。

### 决策二：Bootstrap token 替代默认密码

优点是从根本上消除了"忘记改默认密码"的风险。代价是首次部署流程稍复杂，需要从控制台获取 token。

### 决策三：Argon2 哈希与自动升级

优点是密码存储达到了当前最佳实践。代价是 Argon2 计算成本高于 SHA-256，但这正是抵御暴力破解的核心特性。自动升级策略让迁移对用户完全透明。

### 决策四：子代理在独立进程中执行

优点是权限隔离更彻底。代价是进程启动开销和 IPC 成本。但对于安全性要求，这个代价非常值得。

### 决策五：把扩展生态纳入主安全模型

优点是不会出现"内建能力很安全，扩展能力裸奔"的断层。代价是扩展接入门槛更高。

## 容易走错的地方

### 失败模式 1：把安全完全等同于 sandbox

没有 turn serialization、审批、路径保护和控制面鉴权，容器化本身不足以建立可信边界。

### 失败模式 2：忽视并发竞态

很多 Agent 系统在单用户测试时表现正常，但在多 chat 并发场景下出现状态混乱。`ChatTurnQueue` 解决的正是这个问题。

### 失败模式 3：API key 明文存储

一旦数据库泄露，明文 key 等于直接交出控制权。只存哈希是最低要求。

### 失败模式 4：子代理无预算限制

这是一个非常具体的成本风险。假设主 Agent 收到"帮我调研这个问题"的请求，调用 `sessions_spawn` 创建了一个子代理。如果子代理没有 token 预算限制，它可以无限地调用模型、执行工具、甚至继续 spawn 更深层的子代理。一次看似无害的工具调用，可能在几分钟内消耗掉整个月的 API 预算。MicroClaw 通过 `max_tokens_per_run` 和 `max_spawn_depth` 把这个风险限制在可预测的范围内，但如果你在配置中把这些值设得过高或完全禁用，风险会迅速回来。

- 你是否把审批、sandbox、路径控制、网络校验和控制面鉴权视为多层防线？

## 证据来源（v0.1.38）

- 核心源码路径：`src/run_control.rs`、`src/chat_turn_queue.rs`、`src/web/auth.rs`、`src/web/middleware.rs`、`src/tool_executor.rs`、`src/acp_subagent.rs`、`src/agent_engine.rs`

- 关键配置项：`src/config.rs` 中与高风险审批、sandbox mode（Shared/Chat）、子代理预算、API key scope 和 CSRF 相关的默认值

## 小结

MicroClaw v0.1.38 的安全体系在 v0.1.16 的基础上实现了三个关键跃迁：从"防止错误操作"到"从并发根源消除竞态"（ChatTurnQueue），从"单一 token 鉴权"到"完整身份和 API key 生命周期"，从"单体安全边界"到"子代理独立隔离"。这些变化共同构成了一个多层、可审计、可自检的安全体系，让 Agent Runtime 真正具备了托管级别的安全姿态。

下一章我们继续从生产视角前进，讨论可观测性与运维。因为安全只能告诉你"尽量别出事"，而真正的生产系统还必须做到"出事时能发现、能定位、能恢复"。

## <a id="ch23"></a>第23章 设计决策备忘录：80 条 ADR 风格取舍记录

### <a id="ch23-1"></a>23.1 阅读说明

本章采用轻量 ADR（Architecture Decision Record）体例。每条记录包含：

1. 决策主题。
2. 背景问题。
3. 方案选择。
4. 主要收益。
5. 代价与后续动作。

这些记录可直接作为团队内部架构评审模板。

### <a id="ch23-2"></a>23.2 决策记录

#### ADR-1: 共享 Agent Loop

- 背景问题：
  在持续迭代过程中，团队发现当前系统在“功能可用”与“长期可维护”之间存在张力。短期内的快速补丁能解决单点问题，但会逐渐侵蚀核心一致性，最终带来跨模块行为漂移和运维复杂度上升。

- 方案选择：
  采用“共享 Agent Loop”作为本阶段主策略，并将相关规则放入运行时主路径，而不是仅写入文档约定。所有新功能必须先对齐该策略，再进入实现和发布流程。

- 主要收益：
  1. 减少平台分叉，让系统行为更可预测。
  2. 把隐性经验沉淀为显性约束，降低人员变动影响。
  3. 让测试和观测可以围绕稳定契约建立，而不是追逐临时行为。
  4. 为后续扩展（渠道、工具、策略）提供统一演进支点。

- 代价与风险：
  1. 需要更多约束治理，短期内学习成本会上升。
  2. 某些历史路径需要迁移，存在兼容性治理工作量。
  3. 发布节奏可能在初期放缓，因为需要补充验证和回归。
  4. 若缺乏执行纪律，策略可能停留在“名义存在”。

- 落地动作：
  1. 在代码层加入强约束与结构化错误返回。
  2. 在文档层同步更新配置说明与 runbook。
  3. 在测试层增加回归用例覆盖关键分支。
  4. 在运维层增加对应指标与告警阈值。

- 验证标准：
  1. 关键路径行为在多渠道保持一致。
  2. 异常发生时可在固定时间内定位到模块层级。
  3. 变更后未出现同类回归或同类事故显著下降。
  4. 团队成员可用同一模板解释该决策与取舍。

  在持续迭代过程中，团队发现当前系统在“功能可用”与“长期可维护”之间存在张力。短期内的快速补丁能解决单点问题，但会逐渐侵蚀核心一致性，最终带来跨模块行为漂移和运维复杂度上升。

- 方案选择：
  采用“sandbox fail-closed”作为本阶段主策略，并将相关规则放入运行时主路径，而不是仅写入文档约定。所有新功能必须先对齐该策略，再进入实现和发布流程。

- 主要收益：
  1. 避免隐式降级，让系统行为更可预测。
  2. 把隐性经验沉淀为显性约束，降低人员变动影响。
  3. 让测试和观测可以围绕稳定契约建立，而不是追逐临时行为。
  4. 为后续扩展（渠道、工具、策略）提供统一演进支点。

- 代价与风险：
  1. 要求更完整监控，短期内学习成本会上升。
  2. 某些历史路径需要迁移，存在兼容性治理工作量。
  3. 发布节奏可能在初期放缓，因为需要补充验证和回归。
  4. 若缺乏执行纪律，策略可能停留在“名义存在”。

- 落地动作：
  1. 在代码层加入强约束与结构化错误返回。
  2. 在文档层同步更新配置说明与 runbook。
  3. 在测试层增加回归用例覆盖关键分支。
  4. 在运维层增加对应指标与告警阈值。

- 验证标准：
  1. 关键路径行为在多渠道保持一致。
  2. 异常发生时可在固定时间内定位到模块层级。
  3. 变更后未出现同类回归或同类事故显著下降。
  4. 团队成员可用同一模板解释该决策与取舍。

### <a id="ch23-3"></a>23.3 决策管理建议

1. 每个季度挑选 5 条关键 ADR 做有效性复审。
2. 若决策已失效，要显式标注“废弃”并给替代方案。
3. ADR 变更必须关联测试与文档改动，避免纸面决策。
4. 对高风险 ADR 增加“演练证明”要求。

### <a id="ch23-4"></a>23.4 本章小结

把设计决策写下来，不会自动提高系统质量，但会显著提高团队对质量的可重复执行能力。这是 Agent 系统从“个人技巧”走向“组织能力”的关键一步。
